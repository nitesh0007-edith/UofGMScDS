DATABASE QUERY PROCESSING
⚙️ QUERY PROCESSING WORKFLOW

Typical Query Processing Pipeline

[Query] →
    Parser → [AST] →
        Planner/Rewriter/Algebriser → [Logical Plan] →
            Optimiser → [Physical Plan] →
                Code Generation/Execution → [Result Set]
🎯 Key Principle: Multiple equivalent logical plans exist for any query. The optimizer's job is to select the physical plan with the lowest estimated cost!
📊 1. EXTERNAL SORTING (Basic Tool)
Why Sorting is Fundamental
⚠️ Key Limitation:
Cannot store entire relation file in memory for sorting - must use external sorting!
  • CREATE INDEX: Requires sorting by indexed field
  • ORDER BY: Explicit sorting requirement
  • SELECT DISTINCT: Eliminate duplicates via sorting
  • JOIN operations: Often expedited by sorting both relations
  • GROUP BY: Aggregation facilitated by sorting
🔄 External Sorting Algorithm
Principle: Divide & Conquer

Sorting Phase:

  • Divide: Split file into smaller sub-files that fit in memory
  • Sort: Use internal sorting algorithm (quick-sort, bubble-sort)
  • Write: Write sorted sub-files back to disk

Merging Phase:

  • Load: Load sorted blocks from sub-files into memory
  • Merge: Use merge-list/parallel-merge algorithm
  • Write: Write merged sorted results back to disk
  • Iterate: Multiple passes until entire file is sorted
External Sort Cost Formula:
$$\text{Cost} = (2 \cdot b) + (2 \cdot b \cdot \log_M(L))$$

Where:
- $b$ = number of file blocks
- $M$ = degree of merging (number of sorted blocks merged per pass)
- $L$ = number of initial sorted sub-files
⚡ Performance Notes:
  • $M = 2$ gives worst-case performance (binary merge)
  • $M > 2$ enables M-way merging for better performance
  • Higher $M$ means fewer passes required
🔍 2. STRATEGIES FOR SELECT
SELECT * FROM relation WHERE selection-condition
📋 S1. Linear Search (Serial Scan)
  • Method: Retrieve every record and test selection condition
  • Precondition: None
  • Use Case: No access path available
Expected Cost:
$$\text{Cost} = \frac{b}{2}$$ where $b$ = number of blocks in file
📝 Example:
SELECT * FROM EMPLOYEE WHERE SSN = '12345678'
Without any index: Must scan entire file
🔢 S2. Binary Search
  • Method: Binary search on ordering key
  • Precondition: File sorted by search key
  • Use Case: Equality condition on ordering key
Expected Cost (sorted file):
$$\text{Cost} = \lceil \log_2(b) \rceil$$
Expected Cost (unsorted file - must sort first):
$$\text{Cost} = \lceil \log_2(b) \rceil + 2b + 2b \cdot \log_M(L)$$
🗝️ S3. Primary Index or Hash Function (Key)
  • Method: Use primary index or hash function
  • Precondition: Primary Index of level $t$ over key, OR hash function exists
  • Use Case: Equality on key attribute
Expected Cost:
$$\text{Cost} = t + 1$$ where $t$ = number of index levels
📝 Example:
SELECT * FROM EMPLOYEE WHERE SSN = '12345678'
With Primary Index: Navigate index levels + 1 data block access
📈 S4. Primary Index in Range
  • Method: Use index to find first record, then scan sequentially
  • Precondition: Primary Index over key
  • Use Case: Range query: $>, \geq, <, \leq$ on key
Expected Cost:
$$\text{Cost} = (t + 1) + O(b)$$
⚠️ Important:
Do NOT use hashing for range queries! Hash functions destroy ordering.
📍 S5. Clustering Index (Multiple Records)
  • Method: Use clustering index to retrieve contiguous cluster
  • Precondition: Clustering Index of level $t$ over non-key
  • Use Case: Equality on non-key attribute
Expected Cost:
$$\text{Cost} = (t + 1) + O\left(\frac{b}{c}\right)$$ where $c$ = number of distinct values of non-key attribute
📝 Example:
SELECT * FROM EMPLOYEE WHERE DNO = 5
Retrieves all employees in department 5 (clustered together)
🔗 S6. Secondary Index (B+ Tree) - Equality
On Key Field (Unique)
  • Method: Use B+ tree index to find unique record
  • Precondition: File NOT ordered by key (secondary access)
Expected Cost:
$$\text{Cost} = t + 1$$
On Non-Key Field
  • Method: B+ leaf points to group of pointers
  • Structure: Two-level indirection (index → pointer block → data blocks)
Expected Cost:
$$\text{Cost} = t + 1 + O(b)$$
📊 S7. Secondary Index (B+ Tree) - Range
  • Method: Serial scan through linked B+ leaf nodes
  • Advantage: B+ leaf nodes are sorted and linked
Methodology:
1. Find first leaf node (e.g., Salary = 10K)
2. Load pointer cluster for that value
3. Follow leaf-next-pointers to subsequent nodes
4. Stop at last leaf node in range (e.g., Salary > 40K)
Expected Cost:
$$\text{Cost} = t + 1 + O(b \times n)$$ where $n$ = number of distinct values in range
🔀 3. COMPLEX SELECT STRATEGIES
∨ Disjunctive SELECT (OR)
Result must contain tuples satisfying the UNION of conditions
📝 Example:
SELECT * FROM EMPLOYEE WHERE SALARY > 10000 OR NAME LIKE '%Chris%'

Strategy:

  • IF: Access path exists for ALL attributes:
    • Use each index to retrieve records for each condition
    • UNION all result sets
  • ELSE: If ANY attribute lacks access path → Linear search unavoidable!
∧ Conjunctive SELECT (AND)
📝 Example:
SELECT * FROM EMPLOYEE WHERE SALARY > 40000 AND NAME LIKE '%Chris%'

Strategy:

  • Step 1: Use index on one attribute to retrieve candidate set
  • Step 2: Check candidates against other conditions
  • Optimization Question: Which index to use first?
🎯 Optimization Strategy:
Use the index that returns the SMALLEST result set first (highest selectivity)!
This is determined by selectivity estimation.
🔗 4. STRATEGIES FOR JOIN
⚠️ Critical Observation:
JOIN is the most resource-consuming operator in query processing!
Focus: Two-way equijoin (joining two relations with equality '=')
SELECT * FROM EMPLOYEE E, DEPARTMENT D WHERE E.DNO = D.DNUMBER

Five Fundamental Join Strategies:

  1. Naïve Join (no access path)
  2. Nested-Loop Join (no access path)
  3. Index-based Nested-Loop Join (requires index/B+ Tree)
  4. Sort-Merge Join (requires sorted relations)
  5. Hash Join (requires hashing)
❌ Naïve Join
Approach
  1. Step 1: Compute Cartesian product $R \times S$
  2. Step 2: Scan result and check join condition
Algorithm Naïve Join: T = Cartesian(R × S) Scan T, tuple t ∈ T at a time: t = (r, s) If r.A = s.B then add (r, s) to result Else go to next tuple t ∈ T
⚠️ Outcome:
Very inefficient! Typically the result is a tiny subset of the Cartesian product.
Cost: Extremely high - generates all possible combinations first.
🔄 Nested-Loop Join
Block-Based Algorithm
Algorithm Nested-Loop Join: For each block chunk from outer relation R: For each block from inner relation S: For each tuple r in R-block: For each tuple s in S-block: If r.A = s.B then add (r,s) to result

Memory Buffer Allocation:

  • 1 block for reading inner relation
  • 1 block for writing join result
  • $n_B - 2$ blocks for outer relation chunk
Cost Formula:
$$\text{Cost} = n_{\text{outer}} + n_{\text{inner}} \times \left\lceil \frac{n_{\text{outer}}}{n_B - 2} \right\rceil$$
📊 Example:
Given: $n_E = 2000$ blocks, $n_D = 10$ blocks, $n_B = 7$ blocks

Strategy 1 (E outer, D inner):
$$\text{Cost} = 2000 + 10 \times \left\lceil \frac{2000}{5} \right\rceil = 6{,}000 \text{ blocks}$$ Strategy 2 (D outer, E inner):
$$\text{Cost} = 10 + 2000 \times \left\lceil \frac{10}{5} \right\rceil = 4{,}010 \text{ blocks}$$
🎯 Lesson Learned:
The relation with FEWER blocks should be in the outer loop!
This minimizes the number of times the inner relation is read.
🗂️ Index-Based Nested-Loop Join
Method
  • Requirement: Index on joining attribute of one relation
  • Strategy: For each tuple in outer relation, use index to find matching tuples
Algorithm Index-Based Nested-Loop Join: For each tuple r ∈ R: Use index f(B) to find all tuples s ∈ S where s.B = r.A For each matching tuple s: Add (r, s) to result
Cost Formula:
$$\text{Cost} = n_{\text{outer}} + r_{\text{outer}} \times (x_{\text{index}} + 1)$$ where $r_{\text{outer}}$ = number of tuples, $x_{\text{index}}$ = index levels
📊 Example Comparison:
Scenario: JOIN Employee and Department on Manager SSN
- Employee: $r_E = 6000$ tuples, $n_E = 2000$ blocks, B+ Tree on SSN ($x_E = 4$)
- Department: $r_D = 50$ tuples, $n_D = 10$ blocks, B+ Tree on Mgr_SSN ($x_D = 2$)

Strategy 1 (E outer, use D index):
$$\text{Cost} = 2000 + 6000 \times (2 + 1) = 20{,}000 \text{ blocks}$$ Problem: Not all employees are managers!

Strategy 2 (D outer, use E index):
$$\text{Cost} = 10 + 50 \times (4 + 1) = 260 \text{ blocks}$$ Better: Every department has exactly one manager!
🎯 Lesson Learned:
Use the PRIMARY KEY index of the referenced relation pointed to by the FOREIGN KEY!
This ensures every probe finds exactly one match (referential integrity).
⬆️⬇️ Sort-Merge Join
Precondition
📋 Requirement:
Both relations MUST be physically sorted on their joining attributes!
Method
  1. Step 1: Load pair of sorted blocks $\{R.\text{block}, S.\text{block}\}$ into memory
  2. Step 2: Linearly scan both blocks concurrently (list-merge mode)
  3. Step 3: Store matching tuples in output buffer
✅ Advantage: Each file scanned only ONCE!
Cost (if already sorted):
$$\text{Cost} = n_E + n_D$$
Cost (if NOT sorted - must sort first):
$$\text{Cost} = n_E + n_D + 2n_E + 2n_E \log_2\left(\left\lceil \frac{n_E}{n_B} \right\rceil\right) + 2n_D + 2n_D \log_2\left(\left\lceil \frac{n_D}{n_B} \right\rceil\right)$$
📊 Example:
$n_E = 2000$ blocks, $n_D = 10$ blocks, $n_B = 7$ blocks

If already sorted:
$$\text{Cost} = 2000 + 10 = 2{,}010 \text{ blocks}$$ If must sort first:
$$\text{Cost} = 2010 + 4000 + 4000\log(286) + 20 + 20\log(2) = 38{,}690 \text{ blocks}$$ Only 5.1% devoted to actual join!
⚠️ Lesson Learned:
Think before you sort! Sorting overhead can dominate join cost.
Only use sort-merge if relations are already sorted or will be reused sorted.
# Hash Join
Precondition
  • Use SAME hash function $h$ on joining attributes of both relations
  • Partition both relations into $M$ buckets using $h$
  • Assumption: Smaller relation fits in memory
Algorithm Hash Join: /* Partitioning Phase */ For each tuple r ∈ R: Compute y = h(r.A) Place r into bucket y in memory /* Probing Phase */ For each tuple s ∈ S: Compute y = h(s.B) // same hash function! Find bucket y in memory For each tuple r in bucket y: If s.B = r.A then add (r,s) to result
Best Case (smaller relation fits in memory):
$$\text{Cost} = n_E + n_D$$ Condition: $n_B > n_D + 2$
Normal Case (doesn't fit in memory):
$$\text{Cost} = 3(n_E + n_D)$$ Breakdown:
- Read both relations: $n_E + n_D$
- Write partitioned buckets: $n_E + n_D$
- Read buckets for probing: $n_E + n_D$
📊 Example:
$n_E = 2000$ blocks, $n_D = 10$ blocks

Normal Case:
$$\text{Cost} = 3(2000 + 10) = 6{,}030 \text{ blocks}$$
🎯 Optimization:
Hash the SMALLER relation first!
This maximizes the chance it fits in memory (best case).
📈 5. JOIN COST COMPARISON
📊 Complete Example:
Setup: $n_E = 2000$ blocks, $n_D = 10$ blocks, $n_B = 7$ blocks
$r_E = 6000$ tuples, $r_D = 50$ tuples, $x_E = 4$, $x_D = 2$
Join Strategy Cost (Block Accesses) Precondition
Naïve Join $n_E \times n_D = 20{,}000$ None
Nested-Loop (best) $n_D + n_E \times \lceil n_D/(n_B-2) \rceil = 4{,}010$ None
Index Nested-Loop (best) $n_D + r_D \times (x_E + 1) = 260$ Index on join key
Sort-Merge (sorted) $n_E + n_D = 2{,}010$ Both sorted
Sort-Merge (unsorted) $38{,}690$ Must sort first
Hash Join $3(n_E + n_D) = 6{,}030$ Same hash function
🎯 Key Insights:
  • Index-based is BEST when index exists and join selectivity is high
  • Sort-Merge is BEST when relations are already sorted
  • Hash Join is GOOD for equi-joins with no existing order
  • Nested-Loop is ACCEPTABLE when one relation is very small
  • Naïve is NEVER GOOD - always avoid!
💡 Optimization Challenge:
Predict join cost and choose the best strategy based on:
- Available access paths (indexes)
- Relation sizes
- Join selectivity
- Available memory
- Physical organization (sorted/hashed)
🎓 6. KEY TAKEAWAYS
🌟 Core Principles:
  1. Query Processing is Multi-Stage: Parse → Plan → Optimize → Execute
  2. Cost is Measured in Block Accesses: I/O operations dominate CPU cost
  3. Multiple Strategies Exist: Always multiple ways to execute a query
  4. Optimization is Critical: Cost difference can be orders of magnitude
  5. Indexes are Powerful: Can reduce cost from $O(n)$ to $O(\log n)$
  6. Context Matters: Best strategy depends on data characteristics
Decision Framework

For SELECT Queries:

  1. Check if index exists on search attribute
  2. Check if file is sorted on search attribute
  3. Consider selectivity (how many records match)
  4. Choose strategy with lowest estimated cost

For JOIN Queries:

  1. Check for indexes on join attributes
  2. Check if relations are sorted on join attributes
  3. Compare relation sizes
  4. Consider join selectivity
  5. Evaluate memory availability
  6. Choose strategy with lowest estimated cost
🏆 ULTIMATE LESSON:

Query optimization is about making intelligent choices among multiple execution strategies.

The optimizer's job is to:
1. Generate alternative execution plans
2. Estimate cost of each plan
3. Select the plan with minimum estimated cost

Cost estimation requires:
- Statistics about data (table sizes, index levels, distinct values)
- Understanding of algorithm costs
- Knowledge of available access paths

The difference between a good and bad plan can be
orders of magnitude in execution time! 🚀
