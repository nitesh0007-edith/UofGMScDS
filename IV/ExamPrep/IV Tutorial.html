01. Introduction to Information Visualisation

What it is, why it works, and how it differs from scientific visualisation.

Definition Information Visualisation (InfoVis) = the use of computer-supported, interactive, visual representations of abstract data to amplify cognition.

InfoVis vs SciVis — the key contrast

Both are visualisation, but they sit on opposite sides of one question: does the data already have a natural place in space?

Why does visualisation work?

  • Humans have remarkable visual abilities — we scan, recognise and recall images fast.
  • Visual displays exploit the very high bandwidth of the human visual system.
  • Patterns, clusters, gaps and outliers leap out visually but stay hidden in raw numbers.

Exploration vs Explanation

AspectExploratoryExplanatory
GoalDiscover unknown patternsCommunicate known findings
AudienceThe analyst (you)Other people
InteractionHighly interactiveOften static
DesignFlexible, multiple viewsPolished, focused narrative
Beginner takeaway A scatter plot of GDP vs life expectancy = InfoVis. A 3D MRI brain scan = SciVis. The test: did nature give you the spatial layout, or do you have to invent one?

02. Visual Variables — Bertin's Semiology of Graphics

The "alphabet" of every chart you'll ever draw.

In 1967, French cartographer Jacques Bertin wrote Semiologie Graphique. He asked: what are the basic ingredients we have for encoding data on a 2D plane? His answer is still the foundation of the field.

The two categories

1. Location (positional) variables — where the mark sits on the plane (x, y). This is the most powerful channel.
2. Retinal variables (6) — properties that "elevate" a mark with a different pattern of light: size, shape, hue, value, orientation, texture.

The four perceptual properties

Each variable has up to four properties. Knowing which variable does which property is a classic exam question.

PropertyMeans…Which variables?
AssociativeAll values feel "equal" — none dominatesPosition, shape, orientation, hue, texture
DissociativeOne value dominates the eyeSize, value
SelectiveYou can focus on one value despite changes in othersALL except shape
OrderedValues are perceived as rankedSize, value, texture
QuantitativeNumerical magnitude can be estimatedPosition and size only
Two facts to memorise
  1. Shape is the only non-selective variable. If you scatter different shapes among other shapes, you can't easily pick out "all triangles" at a glance.
  2. Position and size are the only quantitative variables. So when you need a real number to be readable, use bars (length on a common scale) or scatter plots.

Matching variables to data types

Data typeBest variablesWhy
Nominal (categories)Shape, hue, orientation, textureAssociative — no value dominates
Ordinal (ranked)Value, sizeDissociative and ordered
QuantitativePosition, sizeMagnitude can be estimated

Figure and Ground

The figure is what stands out; the ground recedes. Dissociative variables (size, value) are how you make a figure pop out.

04. The Visualisation Pipeline

Every visualisation system goes through the same four stages.

Marks and Channels (Munzner)

Marks are geometric primitives:

  • Points (0D) — individual items
  • Lines (1D) — connections
  • Areas (2D) — regions
  • Volumes (3D) — solids

Channels are visual properties applied to marks (position, length, angle, area, hue, luminance, motion, shape, …).

Channel effectiveness — Cleveland & McGill ranking

This ranking tells you which channels let humans estimate numbers most accurately. Memorise it.

Rank (Quantitative)ChannelExample
1Position on a common scaleBar chart, scatter plot
2Position on unaligned scalesSmall multiples
3LengthStacked bars
4Angle / slopePie chart
5AreaBubble chart
6Volume / curvature3D bubble
7Luminance / saturationHeatmap

For categorical data, the ranking is: spatial region → hue → motion → shape.

Why bar charts beat pie charts Bars use position on a common scale (rank 1). Pies use angle (rank 4). The same numbers are simply easier to read as bars.

05. Interaction — Shneiderman's Taxonomy

"The Eyes Have It" (Shneiderman, 1996) — the most quoted paper in the field.

"Overview first, zoom and filter, then details-on-demand."

This single sentence is the Visual Information-Seeking Mantra. Treat it as a default design recipe.

The seven tasks

Mnemonic: O Z F D R H E — "Oz Filed His Receipt".

TaskWhat it doesConcrete example
OverviewSee the whole collectionPlot all 10,000 points
ZoomMagnify items of interestPinch-zoom on a cluster
FilterHide uninteresting itemsShow only 2024 records
Details-on-demandPull up details when wantedHover tooltip
RelateShow relationships between itemsHighlight a node's neighbours
HistoryUndo / redo / replayStep backwards through filters
ExtractSave a sub-collection or queryExport filtered CSV

Common interaction patterns

  • Overview + Detail — minimap plus main view (Google Maps).
  • Focus + Context — fisheye lens distorts, but keeps surrounding context visible.
  • Brushing & Linking — select in one view, the same items light up in all linked views.
  • Dynamic Queries — sliders, toggles, immediate visual feedback (Shneiderman's Homefinder).
  • Direct Manipulation — visible objects + rapid, reversible, incremental actions.

06. Data Selection & Filtering — Heer's Generalised Selection

From "click on dots" to "select all things like this one".

Heer et al. argue that traditional pointer-based selection is too limited. Real users want declarative selections — like an SQL query, but built interactively from examples.

Four kinds of selection

TypeHow it worksExample
PointClick an itemOne data point
RegionLasso / rubber-band area"All items in East L.A."
AttributePick by data attribute"All gunshot victims"
GeneralisedSpecific → broaden interactively"All items blue like this one"

Set operations on selections

  • Union — combine two selections (OR)
  • Intersection — items in both (AND)
  • Difference — in first but not second
  • Complement — everything NOT selected

Dynamic Queries vs Query Relaxation

Dynamic QueriesQuery Relaxation
Top-down: start broad, narrow downBottom-up: start specific, broaden
Slider-based filteringClick-and-generalise
Predefined query dimensionsEmergent query dimensions
Why this matters Because selections are now queries, not lists of IDs, they survive when the data updates — a brushed selection in one view continues to point at meaningful items in another view, even if the underlying data changes.

07. Visual Perception & Pre-attentive Processing

Why some things "pop out" and others don't.

Definition Pre-attentive processing = visual features detected by the low-level visual system before conscious attention, typically in under 200–250 ms, regardless of the number of distractors. Often called "pop-out" (Treisman, 1985).

Pop-out demo

Treisman's Feature Integration Theory

  1. Pre-attentive stage — features (colour, orientation, size) are processed in parallel across the whole visual field.
  2. Attentive stage — features are bound together by focused attention. This is serial and slower.
  3. Conjunctions (red AND circle) require the attentive stage → no pop-out.

Gestalt Principles

PrincipleIdeaIn visualisation
ProximityClose items are groupedSpacing in small multiples
SimilaritySimilar-looking items are groupedColour-coded categories
ContinuityEye follows smooth pathsLine charts, flow lines
ClosureMind completes shapesImplied boundaries
ConnectednessConnected items groupNetwork edges
EnclosureItems in a boundary groupBounding boxes
Common fateItems moving together groupAnimated transitions
Figure / groundForeground vs backgroundContrast, layering

Weber's Law

ΔS / S = k. The smallest change you can detect is proportional to the size of the stimulus. Small differences are easier at small magnitudes. This is one reason position on a common scale wins — its just-noticeable-difference is small.

Stevens' Power Law

P = k · Sn — perceived magnitude is a power function of physical magnitude. The exponent n depends on the channel:

ChannelExponent nEffect
Length≈ 1.0Perceived proportionally — accurate
Area≈ 0.7Underestimated
Volume≈ 0.6Even more underestimated
Brightness≈ 0.5Heavily underestimated

Practical consequence: a bubble whose area is twice as big does not look twice as big — it looks ~1.6× as big. Bar lengths are honest; bubble areas are not.

08. Colour Theory

Three colour spaces, three colour-map types, and one notorious villain.

Colour spaces

  • RGB — additive, used by displays. Equal RGB steps ≠ equal perceptual steps.
  • HSL / HSV — Hue, Saturation, Lightness/Value. More intuitive for designers.
  • CIELAB (Lab*) — designed to be perceptually uniform. Preferred for scientific colour-map design.

Three types of colour map

Why the rainbow is bad

  • Not perceptually uniform — some bands look much wider than others.
  • Creates false boundaries where hues change abruptly (e.g. yellow → green).
  • Hue is not naturally ordered — viewers can't say "is this more or less than that?".
  • Use viridis, inferno, magma or cividis instead — perceptually uniform and colour-blind safe.

Colour blindness

  • ~8% of males, ~0.5% of females have colour vision deficiency.
  • Protanopia / Deuteranopia — red/green confusion (most common).
  • Tritanopia — blue/yellow confusion (rare).
  • Rule: never use red/green as the only differentiator. Add a redundant cue (shape, texture, label).

09. Depicting Quantitative Data

Why bars beat pies (and most other classic charts you'll be asked to critique).

Tufte's principles (memorise these phrases)

  • Data-ink ratio = ink for data ÷ total ink. Maximise it. Erase non-data ink.
  • Chartjunk — moiré patterns, heavy gridlines, 3D effects, decorative clipart. Remove.
  • Lie factor = (size of effect in graphic) ÷ (size of effect in data). Should be ≈ 1.0. Anything far from 1 is misleading.
  • Small multiples — many copies of the same chart, each showing a different subset. Powerful for comparison.

Cleveland & McGill perceptual hierarchy (again)

Most → least accurate for quantitative judgement: position on a common scale > position on unaligned scales > length > angle/slope > area > volume > colour shading.

Watch out — pie chart trap Pie charts are intuitive but use angle (rank 4). Limit to ~5 slices, never compare two pies side-by-side, and prefer a bar chart whenever you actually need to read the numbers.

10. Multidimensional Data

When you have 5, 20 or 100 attributes per record, what do you do?

Parallel Coordinates

Each dimension becomes a vertical axis; each item becomes a polyline that snakes across all the axes.

Strengths: handles 10–20 dimensions, reveals correlations between adjacent axes. Weaknesses: cluttered with many items; only adjacent-axis correlations are visible — axis order matters.

Scatterplot Matrix (SPLOM)

Grid of every pairwise scatter plot. For n dimensions you get n × (n−1) / 2 unique scatter plots, or n² cells in the full grid (with the diagonal showing the variable name or histogram).

Worked example 9 dimensions → 9 × 9 = 81 cells in a square SPLOM. (Unique pairs would be 9 × 8 / 2 = 36.) Be careful which the question is asking for.

Other techniques

  • Star plot / radar — radial axes; profile shape per item. Good for comparing 5–10 items.
  • Chernoff faces — face features (eyes, mouth, nose) encode dimensions. Exploits face recognition; arbitrary mapping.
  • Heatmaps — matrices coloured by value; need meaningful row/column ordering.
  • Glyphs — small composite marks where multiple variables encode multiple dimensions.

Dimensionality reduction

MethodIdeaPreserves
PCALinear projection; max varianceGlobal variance
MDSPreserves pairwise distancesGlobal distances
t-SNENon-linear; emphasises local neighbourhoodsLocal clusters (NOT inter-cluster distance)
UMAPNon-linear; balances local + globalLocal and some global structure
Spring modelForce-based layoutPairwise similarity (slow)
Spring model complexity A naïve spring model is roughly O(n²) per iteration in the number of objects. Doubling 5,000 → 10,000 makes each iteration roughly 4× slower (2² = 4). Triple it → 9× slower. This is a favourite exam calculation.

11. Cognitive Amplification — Larkin & Simon (1987)

"Why a Diagram is (Sometimes) Worth Ten Thousand Words" — the theoretical foundation of the field.

The big claim

Diagrams aren't inherently better than text. The advantage depends on computational efficiency — how easily an information-processing system (a person, or a computer) can extract and use the information.

Two kinds of equivalence

Definitions Informational equivalence — two representations contain the same information; you can derive each from the other.
Computational equivalence — two representations let you reach answers with the same effort.
Two representations can be informationally equivalent yet computationally very different.

Sentential vs Diagrammatic

SententialDiagrammatic
OrganisationSequential / linearIndexed by location on a plane
InformationExplicit in propositionsOften implicit in spatial layout
SearchSequentialBy location — usually one place
InferenceCompute spatial relations symbolicallyRead spatial relations off directly

Three advantages of diagrams

  1. Locality — related information is grouped spatially, so search is short.
  2. Minimising labelling — location itself carries information; you need fewer variable names.
  3. Perceptual inference — some inferences happen automatically in the visual system (e.g. "those two lines cross").

The "(Sometimes)" caveat

Diagrams are not always better — when the problem requires algebraic manipulation, when the spatial layout doesn't match the problem structure, when the diagram is poorly designed, or when content is truly sequential (e.g. a proof).

13. Networks and Graphs

Two big choices: node-link or matrix?

Layout algorithms

  • Force-directed (spring-embedder) — nodes repel, edges pull. Aesthetic; clusters emerge. Slow; non-deterministic.
  • Hierarchical (Sugiyama) — assign nodes to layers, minimise edge crossings. Good for DAGs.
  • Circular — all nodes on a circle.
  • Matrix — adjacency matrix. No edge crossings; needs row/column ordering.

Scalability tricks

Aggregation, edge bundling, filtering, fisheye distortion, semantic zoom.

15. Rules of Thumb

A short list of principles you can quote in any answer.

Tufte

  1. Above all else, show the data.
  2. Maximise the data-ink ratio.
  3. Erase non-data ink and redundant data ink.
  4. Avoid chartjunk (heavy grids, 3D, decoration).
  5. Use small multiples for comparisons.
  6. Keep the lie factor close to 1.0.

Munzner

  1. No unjustified 3D — only use 3D for inherently 3D data.
  2. Eyes beat memory — put information side by side, not in sequence.
  3. Expressiveness — encode all and only the data.
  4. Effectiveness — pick the most accurate channel for the data type.

Shneiderman

  1. Overview first, zoom & filter, then details-on-demand.
  2. Rapid, incremental, reversible actions.
  3. Dynamic queries with immediate feedback.
  4. Tight coupling between views.

Common pitfalls

17. Sample Questions with Worked Answers

Click "Show answer" to reveal a written-style response. The format below mirrors the 15-mark questions you'll see in the exam.

Sample Question 1 · 15 marks · (official extended sample)

Outline the basic concept of Stevens' Psychophysical Power Law, and why it is important in the design of information visualisation systems. In your answer, outline three different common styles of chart or graphical encoding, and critique each of them in terms of this power law.

Model answer

Stevens' Power Law states that the perceived magnitude P of a stimulus is a power function of the physical magnitude S, i.e. P = k · Sn. The exponent n depends on the channel: roughly 1.0 for length, 0.7 for area, 0.6 for volume and 0.5 for brightness. So when channels with low exponents encode quantitative data, viewers systematically underestimate the magnitudes.

This matters for visualisation because the whole point of a chart is to let the eye recover numerical relationships from visual ones. If the channel we use distorts that recovery, the chart actively misleads. Stevens' law is therefore one reason why Cleveland & McGill's accuracy hierarchy puts position and length at the top and area, volume and colour-saturation near the bottom.

1. Bar chart (length / position). Bars use position on a common scale and length, both with n ≈ 1. Viewers can read ratios accurately — a bar twice as tall looks twice as big. This is the canonical "honest" chart, and Stevens' law explains why.

2. Bubble chart (area). Area has n ≈ 0.7. A bubble whose area is double its neighbour's is perceived as only ~1.6× larger. Authors sometimes scale by diameter instead, which makes the lie even worse. Critique: bubble charts are great for ordering and clustering but poor for precise quantitative comparison; if the message is "X is twice Y", use a bar.

3. Choropleth / heatmap (colour luminance or saturation). Brightness has n ≈ 0.5 — heavily compressive — and many maps add a non-uniform palette such as the rainbow on top of that. Viewers cannot reliably read magnitudes, only rough ordering. Critique: use perceptually uniform sequential maps (viridis, cividis), and accept that you are giving up precision; pair with a bar chart or tooltip when actual numbers matter.

Conclusion. Stevens' law gives a principled basis for the effectiveness ranking: encode quantitative data with channels whose exponent is close to 1 (position, length); reserve compressive channels (area, volume, luminance) for ordinal cues or coarse overviews, and provide redundant encodings if accurate reading is needed.

Sample Question 2 · 15 marks

Imagine a data set of cars' current locations and speeds, in some urban area, as tracked by a sensor network. Demonstrate the generalised selection interaction technique using this data set. Explain what semantic model of the data you would use, and how you would apply generalised selection to it, to improve interaction and analysis (when compared to normal direct selection of data objects).

Model answer

Semantic model. Treat the data as a relation Car(id, x, y, speed, time, road_segment, vehicle_type). Each row is one sensor reading. A selection is not just a set of row IDs but a declarative query over these attributes (Heer et al.).

Direct selection limitation. A user lasso-ing five fast cars on the map produces a selection of five IDs. The moment the cars move or new readings arrive, the selection no longer means anything sensible — it points at stale data, and the user has to re-select.

Generalised selection. The user clicks one fast-moving car. The system shows generalisation handles: "all cars on this road", "all cars with speed > 50 km/h", "all cars of this vehicle type". The user picks "speed > 50 km/h on this road segment". The selection is now stored as SELECT * WHERE road_segment = 'X' AND speed > 50.

Benefits. (1) The selection persists as data updates — new fast cars on the segment join automatically. (2) It can be combined with set operations: union with "speed > 50 on segment Y", intersection with "between 17:00 and 18:00". (3) It transfers across linked views — brushing on the map highlights the same items in a speed-vs-time scatter plot, because both views share the underlying query. (4) Users build more accurate selections than via lassoing, because they declare intent rather than guessing pixel boundaries.

Comparison summary. Direct selection is point-and-click and breaks under change; generalised selection is declarative, robust to updates, composable, and supports the wider Shneiderman mantra (filter → details-on-demand) more naturally.

Sample Question 3 · 15 marks

Imagine a data set with ~1,000,000 objects and ~100 dimensions. You have one visualisation made with a spring-model dimensional reduction. You also have several other chart types available (bar charts, scatter plots, selectable lists). Describe, in general terms, how and why you would link multiple views of the data.

Model answer

Why link views? No single view can show 100 dimensions and 10⁶ items effectively. The spring-model reduction gives a useful 2D overview but loses individual dimension semantics. Linked views combine the strengths of multiple representations — Munzner's "eyes beat memory" — and operationalise Shneiderman's mantra: overview (spring layout) → zoom & filter (selectable list, sliders) → details-on-demand (bar chart of one item's attributes).

How to link. Treat selection as a declarative query (Heer) shared across all views. Brushing in one view highlights or filters the same items in every other view. Concretely:

  • Spring-model scatter — overview; brushing a cluster sends the underlying IDs as a query.
  • Bar charts of individual dimensions — show the distribution of one of the 100 attributes; bars for selected items can be stacked or colour-coded.
  • Scatter plots of pairs — pick two of the 100 dimensions; brushing here re-highlights the spring-model view and reveals what the cluster means.
  • Selectable list — full search/sort/filter on attributes; selection feeds back to every visual view.

Why this matters. Spring-model layouts answer "what's similar to what?" but never "why?". Linked bar / scatter plots answer "why?" by exposing original dimensions. The combination supports cluster discovery (in the spring view) and cluster interpretation (in the bar/scatter views).

Performance note. With 10⁶ items, linking must use efficient indices (declarative queries, not per-item highlighting), and views should aggregate at low zoom (binning, density) and reveal individuals only on demand.

Sample Question 4 · 15 marks

Your local health service wishes to offer patients access to a database of medical records (conditions, prescriptions, hospital visits, measurements over time). The current system is a web app exposing raw SQL. Outline an alternative design based on dynamic queries, and present arguments for it compared with the SQL app.

Model answer

Why SQL is wrong here. Patients are not analysts. Writing SELECT requires schema knowledge, syntactic correctness, and ignores the Shneiderman mantra entirely — there is no overview, no incremental refinement, and errors are silent.

Dynamic-query design.

  1. Overview view. A timeline (x = time, y = visit) showing all recorded events as glyphs coloured by type (condition, prescription, measurement, visit). Sparklines along the bottom show measurement trends (heart rate, weight).
  2. Filter widgets. Sliders for date range; checkboxes for event types; a search box for prescription names. Each widget updates the timeline immediately — Shneiderman's "rapid, incremental, reversible".
  3. Linked detail panel. Clicking any glyph opens a details-on-demand panel with the full record. Brushing a date range filters all panels at once.
  4. Generalised selection. Clicking one prescription offers "all prescriptions of this drug" or "all prescriptions for this condition" — Heer-style relaxation.

Arguments vs SQL.

  • Discoverable. Visible widgets show what is filterable; SQL requires knowing the schema.
  • Reversible. Sliders go back; a wrong SQL query produces a wrong (but plausible) result.
  • Cognitively amplifying. Patterns (drug interactions, seasonal flare-ups) are visible spatially — Larkin & Simon's locality and perceptual inference.
  • Safe for non-experts. No SQL injection, no syntax errors, no data leakage from accidental joins.
  • Faster feedback loop. Immediate visual response encourages exploration; SQL's compile-run cycle discourages it.

Trade-offs. Dynamic queries cover a curated query space — power users may want a free-form mode. A "show SQL" overlay can satisfy both audiences.

Sample Question 5 · short-answer practice

Explain why "shape" is sometimes called the only non-selective visual variable, and give a design implication.

Model answer

A variable is selective if you can mentally isolate one value despite variations in other variables — e.g. spotting "all the red marks" among many marks of different shapes. Shape fails this test: when squares, circles and triangles are scattered together, you cannot easily focus on "all the squares" without scanning serially. Bertin therefore classifies shape as the only non-selective retinal variable.

Design implication. Don't use shape alone to encode a category that the user will need to filter on visually — combine it with hue or position, or use a redundant encoding. Shape is fine for low-cardinality identification of individual marks (e.g. labelling outliers), but is poor for at-a-glance group selection.

Sample Question 6 · short-answer practice

(a) How many scatter plots are in a SPLOM for 9-dimensional data? (b) A scientist needs both fine-grained local detail and accurate inter-cluster distances. Which dimensionality reduction methods are appropriate, which are not, and why?

Model answer

(a) A full SPLOM is a 9 × 9 grid = 81 cells (the diagonal usually shows the variable name or histogram). The number of unique off-diagonal pairs is 9 × 8 / 2 = 36, which appear twice (once above and once below the diagonal).

(b) Spring model and UMAP attempt to preserve both local and (approximately) global structure, so they are reasonable choices. t-SNE is not appropriate: it preserves local neighbourhoods very well but inter-cluster distances in a t-SNE plot are not meaningful — clusters that look far apart may be neighbours in the original space and vice versa. So if the scientist relies on t-SNE distances between clusters, they will be misled.

Sample Question 7 · short-answer practice

Explain pre-attentive processing. Why does a single red dot pop out among blue dots, but a red circle does not pop out among red squares and blue circles?

Model answer

Pre-attentive processing is the rapid (under ~200 ms) detection of visual features by the early visual system, before focused attention. Per Treisman's Feature Integration Theory, individual features (hue, orientation, size, motion) are processed in parallel across the entire visual field.

A red dot among blue dots differs in one feature (hue), so the red dot pops out — search time is roughly constant regardless of the number of distractors.

A red circle among red squares and blue circles is a conjunction: the target shares hue with one distractor group and shape with the other. No single feature isolates it. Locating it requires the slower attentive stage, with serial search across items — search time scales roughly linearly with the number of distractors.

Design implication. Pop-out works only for single-feature differences. To make a target stand out among many, vary one strong feature (hue + position is best); avoid asking the user to find conjunctions visually.

Sample Question 8 · critique practice

A paint company plans to map its 10 most popular paint colours onto a world map of sales by country, with a legend. Critique this design.

Model answer

Three problems, in increasing severity:

  1. Hue is not ordered. Sales values are quantitative and ordered, but hue is associative — viewers cannot say whether "magenta" is bigger or smaller than "teal". A sequential luminance map (light → dark of one hue) would respect the ordering.
  2. Ten colours is too many. The 5–8-colour rule of thumb for distinguishable hues is exceeded; some colours will be confused.
  3. The colours are chosen for branding, not for perception. "Most popular paint colours" likely includes near-neutrals and similar pastels, which are nearly indistinguishable on a map.

Recommendation. If precise reading of sales is required, abandon the brand-colour scheme and use a perceptually uniform sequential map (e.g. viridis). If brand identity must dominate, accept that the map only supports rough comparison and add bar charts for precise figures.

Sample Question 9 · short-answer practice

You are asked to link a bar chart of yearly sales with a map of shops where each mark shows staff count. Both come from the same database. What is the first technical step?

Model answer

The two views currently share no selectable attribute — yearly sales totals are not per-shop, and staff counts are per-shop, not per-year. You cannot brush from one to the other meaningfully.

The right first step is to change each chart so it has its own selectable list of attributes from the shared database — let the user choose which attribute drives bar height in the bar chart and which drives mark size in the map. Once both views can be configured to expose a common attribute (e.g. "sales by shop"), brushing & linking becomes meaningful.

This is more flexible than forcing both charts to use the same attribute (option b), and avoids the false claim that linking is impossible (option a).

Sample Question 10 · theory practice

Explain the distinction between informational and computational equivalence (Larkin & Simon, 1987), and why it matters for visualisation design.

Model answer

Two representations are informationally equivalent if all the information in one can be derived from the other and vice versa. They are computationally equivalent only if extracting that information takes the same effort in both. Two representations can be informationally equivalent yet computationally very different.

Example. A list of (city, latitude, longitude) tuples and a map dot for each city contain the same information. But "which two cities are closest?" is trivial on the map (perceptual inference) and laborious on the list (compute distances pairwise). The map is computationally cheaper for that question.

Why it matters. Visualisation design is, in Larkin & Simon's terms, a search for representations that are computationally efficient for the user's tasks. Three sources of efficiency: locality (related info nearby in space), minimised labelling (location replaces names) and perceptual inference (the visual system does work the cognitive system would otherwise have to do). The whole effectiveness ranking, the case for diagrams over text, and the case for linked views over single tables all rest on this distinction.
