Chapter 01

Introduction to Information Visualisation

Information Visualisation (InfoVis) is the practice of representing abstract data using interactive, computer-supported visual displays so we can understand it better. Think of it as turning numbers into pictures — but done carefully and scientifically.

📖 Official Definition (Memorise This):
"The use of computer-supported, interactive, visual representations of abstract data to amplify cognition."

InfoVis vs Scientific Visualisation (SciVis)

This is a classic exam distinction. The key difference is whether the data has a natural spatial meaning.

AspectInfoVisSciVis
Data typeAbstract data (no inherent space)Physically-based data (has real space)
Spatial layoutDesigner must choose how to arrange itData already has a natural spatial form
ExampleScatter plot of GDP vs life expectancy3D MRI brain scan
ExampleBar chart of sales figuresWind flow visualisation over a map
🎯 Real-World Example

Imagine you have a spreadsheet with 10,000 customer records (age, income, location, purchases). This is InfoVis — there's no natural picture here. You must design how to show it. Contrast with a CT scan of your lungs — the data literally has a 3D physical position. That's SciVis.

Why Do We Visualise?

👁️
Human Perception
We process visual information incredibly fast — much faster than reading numbers.
🔍
Pattern Discovery
Patterns, clusters, gaps, and outliers hidden in raw data become obvious in a good chart.
🧠
Amplify Cognition
Offloads memory and computation to the visual system, freeing our brain for higher thinking.
📢
Communication
Share findings with others in an accessible, persuasive format.

Exploration vs Explanation

AspectExploratoryExplanatory
GoalDiscover unknown patternsCommunicate known findings
AudienceAnalyst (yourself)Others (stakeholders, public)
InteractionHighly interactiveOften static
DesignFlexible, try many viewsPolished, focused narrative
ExampleData scientist exploring a new datasetA newspaper infographic
🎓 Exam Tip: Be ready to classify a given scenario as exploratory OR explanatory, and explain why. The key question is: "Do you already know what you want to show, or are you finding out?"
Chapter 02

Visual Variables — Bertin's Semiology of Graphics

Jacques Bertin (1918–2010) was a French cartographer who published Sémiologie Graphique in 1967. He established the scientific foundation for how we map data to visual properties. His framework is still the gold standard today.

💡 Core Idea: A "visual variable" is any visual property you can change to convey information. When you make a dot bigger, or red instead of blue, or triangular instead of round — you're using visual variables.

The Two Main Categories

Bertin's Visual Variable Framework

The 4 Perceptual Properties

Each visual variable has one or more of these properties, which tell us what kind of data they can encode:

PropertyMeaningWhich Variables?Use For
Associative All variations look equally "important" — none dominates Location, shape, orientation, hue, texture Nominal / categorical data
Dissociative One variation dominates — your eye is drawn to it Size, colour value Creating emphasis or hierarchy
Selective You can isolate one group despite variation in others ALL except Shape Filtering — "show me just the red ones"
Ordered Variations are perceived as ranked/graduated Size, colour value, texture Ordinal data
Quantitative You can estimate the numerical ratio Position, size Numeric/quantitative data
🎓 #1 Exam Favourite: Shape is the ONLY non-selective variable. This means if you colour dots by category, you can find the red ones by sight. But if you shape-code them, you can't isolate circles from squares without looking at every one carefully. This is why shape is poor for selective tasks.

Which Variable for Which Data Type?

Data TypeBest VariablesWhy
Nominal (categories, no order)Shape, Hue, Orientation, TextureAssociative — no category should look more important
Ordinal (ordered, not numeric)Colour Value, SizeOrdered — darker = more, bigger = more
Quantitative (numeric)Position, SizeQuantitative — we can judge exact ratios

Bertin's Effectiveness Table

🎯 Concrete Example

Scenario: You want to show which country each data point belongs to (nominal), and also their GDP rank (ordinal).

  • For country: Use hue (red = USA, blue = UK, green = France) — associative, no country looks dominant
  • For GDP rank: Use size or colour value — bigger/darker = higher rank
  • For population (quantitative): Use position on y-axis or size
🎓 Exam Tip: Position and size are the ONLY two variables that are quantitative. All others cannot accurately convey numeric magnitude.
Chapter 03

Data Types and Data Attributes

Before you can design a visualisation, you need to understand your data. Two frameworks are key here: Munzner's attribute types and Shneiderman's data types.

Munzner's Attribute Types (The What)

These describe the mathematical nature of data values:

TypeDescriptionExamplesValid Operations
Categorical / Nominal No implicit ordering between values Country names, fruit types, job roles = and ≠ only
Ordinal Has order, but differences aren't meaningful Low/Medium/High, T-shirt sizes (S/M/L/XL) =, ≠, <, >
Quantitative Numeric with measurable distances Temperature, salary, age, distance =, ≠, <, >, +, −
🎯 Tricky Example

A satisfaction score from 1 to 5 seems numeric but is actually ordinal. The difference between 1 and 2 isn't necessarily the same as between 4 and 5. You can say 3 > 2, but you can't say 3 is "twice as satisfied" as 1.5.

A temperature in Kelvin is quantitative — the difference between 200K and 300K is exactly the same magnitude as 300K to 400K.

Shneiderman's 7 Data Types (The Shape of the Data)

This framework asks: "What structure does the data have?" It's different from asking about individual attribute values.

Shneiderman's 7 Data Types — Visual Overview
🎓 Exam Tip: Know all 7 data types by heart and give an example visualisation for each. A common question asks you to classify a new dataset into one of these types, then suggest an appropriate visualisation.

Each Stage Explained

🎯 Running Example: Hospital Patient Data

Let's trace patient records (10 million rows, 50 columns) through the pipeline:

  • Stage 1 (Raw Data): CSV files from 100 hospitals, inconsistent formats, some missing values
  • Stage 2 (Data Transform): Remove duplicates, fill missing ages with median, aggregate to daily counts, filter to last 5 years
  • Stage 3 (Visual Mapping): Map DATE to x-axis position, ADMISSION COUNT to bar height, HOSPITAL to colour hue
  • Stage 4 (View Transform): User zooms into Q3 2024, pans left to Q2
  • Stage 5 (Rendering): Browser draws the final bar chart as coloured rectangles on screen

Marks and Channels (Munzner)

The visual mapping stage uses two concepts: marks (what geometric shape represents an item) and channels (visual properties that encode data).

Marks vs Channels
🎓 Exam Tip: Position on a common scale is ALWAYS the most effective channel for quantitative data. This is why bar charts and scatter plots work so well — they use position. Pie charts use angle (rank 4), which is why bars beat pies.
Chapter 05

Interaction Techniques — Shneiderman's Taxonomy

Ben Shneiderman gave us the most famous principle in all of information visualisation:

"Overview first, zoom and filter, then details-on-demand" — Ben Shneiderman, "The Eyes Have It" (1996)

This mantra describes how users should be able to navigate any complex visualisation. Start with the big picture, narrow it down, then get the specifics when needed.

🎯 Applying the Mantra — Google Maps
  • Overview: You open Maps and see your whole city
  • Zoom: You pinch to zoom into your neighbourhood
  • Filter: You toggle "Restaurants" to filter points of interest
  • Details-on-demand: You tap a restaurant to see its hours, ratings, and menu

Shneiderman's 7 Tasks

TaskDescriptionExample
1. OverviewGain a view of the entire collectionSee all 10,000 data points at once in a scatter plot
2. ZoomZoom into items of interestMagnify a dense cluster in a network graph
3. FilterRemove uninteresting itemsShow only records from the year 2024
4. Details-on-demandSelect an item to get full detailsHover on a bar to see the exact sales value
5. RelateView relationships among itemsHighlight connections from a selected node in a network
6. HistoryKeep history for undo and replayStep back through your exploration states
7. ExtractExtract sub-collections for reuseExport a filtered subset as a CSV file
💡 Mnemonic: "OZF-DRHE" — Overview, Zoom, Filter, Details-on-demand, Relate, History, Extract

Key Interaction Techniques

The 4 Major Interaction Patterns
🎓 Exam Tip: The mantra "Overview first, zoom and filter, then details-on-demand" could appear in almost ANY question. Be able to apply it to a given scenario — hospitals, cars, social networks, whatever. Always apply this structure.
Chapter 06

Data Selection and Filtering — Heer et al.

Heer et al. introduced Generalised Selection via Interactive Query Relaxation. The core insight: clicking on individual items or drawing a lasso is too primitive. We need a smarter, more expressive way to select data.

💡 Core Idea: Instead of selecting individual items (point selection), model selection as a declarative query (like SQL) that can be generalised and relaxed interactively.

What is Query Relaxation?

You start with a very specific selection and gradually broaden it to include related items:

🎯 Example: Crime Data
  1. Start: You click on one specific data point — a robbery on Main Street on 1st March
  2. Relax once: The system generalises to "all robberies on Main Street"
  3. Relax again: Generalises to "all robberies in this neighbourhood"
  4. Relax more: "All violent crimes in this neighbourhood"
  5. Relax fully: "All crimes in this city"

At each step, you choose how far to generalise. This is the "relaxation" — loosening the query constraints.

Types of Selection

TypeDescriptionExample
Point selectionClick on individual itemsClick on one dot in a scatter plot
Region selectionDraw a lasso/rectangle around itemsSelect all items in East London on a map
Attribute selectionSelect by data attribute values"All gunshot victims aged 20–30"
Generalised selectionRelax from specific to general"All items with similar colour to this one"

Dynamic Queries vs Query Relaxation — Key Difference

Dynamic QueriesQuery Relaxation
DirectionStart broad → narrow downStart specific → broaden out
MechanismSliders, buttons, togglesClick item → generalise interactively
ApproachTop-down (you define criteria first)Bottom-up (data guides the query)
Query typePredefined dimensionsEmergent from the data
ExampleProperty search with price sliderClick a crime, generalise to find similar crimes
🎓 Exam Tip: Heer's key contribution is coupling declarative query structure with interactive generalisation. The system helps you refine your query by showing what "similar" items exist, making selections more accurate than simple point/region selection.

Selection Operations

  • Union: Combine two selections — "show all red items OR all large items" (OR)
  • Intersection: Items in both — "show items that are red AND large" (AND)
  • Difference: Items in first but not second — "red items that are NOT large"
  • Complement: Everything NOT selected — "all items except the red ones"

Visual Perception and Pre-attentive Processing

Understanding how the human visual system works is essential to designing effective visualisations. The brain processes visual information in two stages — a fast, automatic stage and a slower, conscious one.

Pre-attentive Processing

📖 Definition: Visual features that are detected by the low-level visual system before conscious attention — typically in under 200–250 milliseconds, regardless of how many other items are present.

This is often called the "pop-out" effect — the feature literally "pops out" at you without effort.

Pre-attentive Pop-Out Demonstration

📏

↗️

🔺

A circle among squares (simple shapes only)

Treisman's Feature Integration Theory

  1. Pre-attentive stage: Individual features (colour, orientation) are processed in parallel across your entire visual field — very fast
  2. Attentive stage: Features are "bound" together through focused attention — serial scanning, much slower
  3. Key implication: If you're searching for a single feature (red), it pops out. If you're searching for a combination (red AND circular), you must serially scan each item.

Gestalt Principles

These describe how the brain naturally groups visual elements:

PrincipleDescriptionUse in Visualisation
ProximityItems close together are perceived as groupedSpacing in small multiples, clustering
SimilarityItems that look alike are perceived as groupedColour coding categories
ContinuityEye follows smooth pathsLine charts, flow diagrams
ClosureMind completes incomplete shapesImplied boundaries
ConnectednessConnected items are perceived as a groupLinks in network diagrams
EnclosureItems inside a boundary are a groupBounding boxes, panel borders
Common fateItems moving together are a groupAnimated transitions
Figure-groundPerception separates foreground from backgroundContrast, layering

Weber's Law & Stevens' Power Law

Weber's Law: ΔS / S = k — the "Just Noticeable Difference" is proportional to the stimulus size. A 1g difference between 10g and 11g is easy to detect. The same 1g difference between 1000g and 1001g is imperceptible.
Stevens' Power Law: Perceived magnitude P = k × Sn
— Length: n ≈ 1.0 (perceived accurately)
— Area: n ≈ 0.7 (underestimated by ~30%)
— Volume: n ≈ 0.6 (even more underestimated)
— Brightness: n ≈ 0.5
This explains why area-based charts (bubble charts, treemaps) are less accurate than bar charts.

Colour Map Types

The 3 Types of Colour Maps

Why the Rainbow Colour Map is BAD

⚠️ The Rainbow Map Problem: Despite being everywhere, the rainbow (spectrum) colour map is one of the worst choices for data visualisation because:
  • Not perceptually uniform — some bands appear wider/narrower than they really are
  • Creates false boundaries where hue changes abruptly (yellow→green boundary looks like a real threshold)
  • Obscures the actual data structure by adding artificial visual "cliffs"
  • Fails completely for colour-blind users (red-green transitions)
Use instead: Viridis, Inferno, Magma, Plasma (perceptually uniform, colour-blind safe)

Colour Blindness Design Guidelines

📊 Key Statistics

~8% of males and ~0.5% of females have some form of colour vision deficiency. This means in a lecture hall of 100 people, roughly 5–8 cannot see your red-green colour coding!

  • Never use red and green as the only differentiator
  • Use redundant encoding — colour + shape, or colour + texture
  • Use colour-blind-safe palettes (e.g., ColorBrewer, Viridis palette family)
  • Test with colour blindness simulators (e.g., Coblis online)

Depicting Quantitative Data

Cleveland & McGill's Perceptual Accuracy Hierarchy

This is one of the most important rankings in visualisation. It tells you how accurately humans can read different visual encodings:

Perceptual Accuracy Ranking (Most → Least Accurate)
🎓 Why Bar Charts Beat Pie Charts: Bar charts use channel ① (position on common scale). Pie charts use channel ④ (angle). Humans are much better at judging position than angle. This is the scientific reason pie charts are generally bad.

Fundamental Chart Types

ChartData TypeStrengthWeakness
Bar chartCategorical × QuantitativePrecise comparison via aligned lengthsOnly 1–2 variables
Scatter plotQuantitative × QuantitativeShows correlation, clusters, outliersOverplotting with large N
Line chartTemporal/Ordered × QuantitativeShows trends over timeImplies continuity between points
Pie chartParts of a whole (nominal)Intuitive for proportionsHard to compare slices; limit to ~5
HistogramDistribution of one variableShows shape of distributionBin size choice affects interpretation
Box plotDistribution summaryShows median, quartiles, outliersHides distribution shape (bimodality)

Tufte's Design Principles

📐 Edward Tufte's Core Rules
  • Data-ink ratio = Data ink / Total ink. Maximise it. Remove all "non-data ink" (heavy grids, redundant labels, 3D effects).
  • Chartjunk: Any visual element that doesn't carry data — 3D effects, moiré patterns, decorative images, excessive gridlines
  • Lie factor = (size of effect in graphic) / (size of effect in data). Should be close to 1.0. A bar chart that doubles a value but the bar only grows by 50% has a lie factor of 0.5.
  • Small multiples: Repeated similar charts showing different subsets — enables direct visual comparison
Chapter 10

Depicting Multidimensional Data

When your data has many attributes (dimensions), you can't simply map each one to X and Y — you've run out of axes! These techniques handle data with 5, 10, even 100+ dimensions.

Parallel Coordinates

Each dimension gets its own vertical axis. Each data item is drawn as a line (polyline) connecting its value on every axis.

Parallel Coordinates — How They Work
💡 Reading Parallel Coords:
  • Parallel lines between two axes → positive correlation
  • Crossing lines (X-shape) → negative correlation
  • Brushing: Draw a range on one axis → those lines are highlighted everywhere
  • Major weakness: Only reveals correlations between adjacent axes. Axis ordering matters hugely!

Scatterplot Matrix (SPLOM)

A grid showing every possible pairwise scatter plot. For N dimensions, you get N×(N−1) plots (or N×(N−1)/2 if you count each pair once).

📐 SPLOM Formula

For 9-dimensional data: 9 × (9−1) = 72 scatter plots (if including both directions), or 9×8/2 = 36 unique pairs.

⚠️ This was an actual exam question! Question 9 in your past paper.

Comparison of Multidimensional Techniques

TechniqueMax Practical DimsBest ForKey Weakness
Parallel Coordinates10–20Finding correlations, clustersAdjacent axes only; clutter with many items
SPLOM8–12All pairwise relationshipsScales poorly — N² plots
Star Plot / Radar5–10Comparing individual profilesArea distorted by axis ordering
Chernoff Faces10–15Informal pattern detectionArbitrary mapping; some features more salient

Dimensionality Reduction Methods

📐
PCA
Linear projection. Preserves global variance. Good when data has linear structure.
🌐
t-SNE
Non-linear. Preserves local neighbourhood structure. Great for cluster visualisation.
🗺️
UMAP
Preserves both local AND global structure better than t-SNE. Faster.
📏

Preserves pairwise distances. "Spring model" — nodes repel/attract based on similarity.
🎯 Exam Trap — t-SNE vs Spring Model

A question may ask: "Scientists need both fine-grained local detail AND accurate high-level cluster relationships. Which algorithm is NOT a good choice?"

Answer: The spring model (basic MDS) — it doesn't preserve local neighbourhood structure well. t-SNE preserves local structure. UMAP preserves both local and global. Spring model/MDS focuses on preserving overall distances, but local detail can be lost.

Chapter 11

Cognitive Amplification — Larkin & Simon (1987)

Paper: "Why a Diagram is (Sometimes) Worth Ten Thousand Words" — Jill Larkin & Herbert Simon

📖 Central Argument: Diagrams are NOT always better than text. The advantage depends on computational efficiency — how easily an information-processing system can extract and use the information. The key distinction is between informational equivalence and computational equivalence.

The Two Key Concepts

Informational vs Computational Equivalence

The Three Advantages of Diagrams

📍
1. Locality
Related information is grouped spatially. You don't need to search — it's all in one place.
🏷️
2. Minimise Labelling
Location itself carries information. No need for explicit variable names like "the longer side is..."
⚡
3. Perceptual Inference
Some inferences are made directly by the visual system (e.g., "these two lines intersect") without conscious calculation.

The "(Sometimes)" Caveat

⚠️ Diagrams are NOT always better!
  • When the problem requires symbolic/algebraic manipulation (e.g., solving equations)
  • When the spatial layout does NOT map naturally to the problem structure
  • When the diagram is poorly designed (misleading layout)
  • When information is best expressed sequentially (e.g., a logical proof)
🎓 Exam Tip: The key takeaway is: two representations can be informationally equivalent (contain identical information) but differ hugely in computational efficiency (how easy they are to use). This is the theoretical foundation for why visualisation amplifies cognition.
Chapter 12

Design and Evaluation

Munzner's Nested Model of Visualisation Design

The 4-Level Nested Model

The nesting matters: errors at outer levels can't be fixed by getting inner levels right. Even if you implement a perfect algorithm, if you solved the wrong problem, it fails.

Evaluation Methods

MethodWhat You MeasureStrengthWeakness
Controlled Experiments (Quantitative) Task time, error rate, accuracy. Compare Technique A vs B statistically. Objective, reproducible, statistically rigorous Lab setting ≠ real world; limited task scope
Qualitative (Think-aloud, Interview) HOW and WHY users behave. Rich insights, unexpected issues. Reveals unexpected problems; rich context Subjective, hard to generalise, small samples
Heuristic Evaluation Experts review against established principles Fast, cheap, catches many issues early Expert opinions may not reflect real users
Case Study / Deployment Real users doing real tasks over extended time Highest ecological validity Very hard to control; difficult to generalise

Evaluation Metrics

  • Task completion time: Efficiency
  • Error rate: Accuracy
  • Insight count: How many discoveries did users make?
  • Subjective satisfaction: Likert scale questionnaires (e.g., "I found this easy to use: 1–7")
  • Learning curve: How quickly can new users become proficient?
🎓 Exam Tip: Be able to explain when to use quantitative vs qualitative evaluation. Quantitative: when you want to prove Technique A is faster than B with statistical confidence. Qualitative: when you want to understand WHY users struggle, or what unexpected behaviours emerge.
Chapter 13

Network and Graph Visualisation

Key Terminology

  • Node (vertex): An entity (person, webpage, computer)
  • Edge (link): A relationship between two nodes
  • Directed graph: Edges have direction (Twitter follows)
  • Undirected graph: Edges are symmetric (Facebook friends)
  • Weighted graph: Edges have numeric values (flight distances)
  • Tree: A connected graph with no cycles (hierarchical structure)

Node-Link vs Matrix Representation

Two Ways to Show the Same Graph
🎓 Node-link vs Matrix — When to Use Which:
  • Node-link: Sparse graphs, path-tracing tasks ("is there a route from A to D?"), topology exploration
  • Matrix: Dense graphs, connection-checking ("are these two nodes connected?"), large N, no edge crossings

Chapter 15

Rules of Thumb for Visualisation

Tufte's Principles (Summary)

📊
Show the Data
"Above all else, show the data." Everything else is secondary.
✏️
Data-ink Ratio
Maximise proportion of ink devoted to actual data. Remove all non-data ink.
🚫
No Chartjunk
3D effects, moiré patterns, heavy grids, decorations — delete them all.
📋
Small Multiples
Use repeated similar charts for comparison rather than animation.
⚖️
Lie Factor ≈ 1.0
Visual effect size should equal data effect size. Never exaggerate.

Munzner's Key Principles

⚠️ No Unjustified 3D: 3D introduces occlusion (objects blocking each other) and perspective distortion. Only use 3D when the data is inherently 3D (e.g., molecular structures, MRI scans). Never use 3D just to make something look impressive.
💡 Eyes Beat Memory: Put information side-by-side rather than requiring users to remember. Small multiples > animation. Linked views > switching between tabs.
Expressiveness + Effectiveness:
  • Expressiveness: Encode ALL and ONLY the data. Don't imply relationships that don't exist (e.g., don't use an ordered colour scale for nominal data).
  • Effectiveness: Of all the correct (expressive) encodings, choose the most effective for the data type.

Common Design Pitfalls

PitfallProblemFix
Truncated y-axisExaggerates differences — lie factor >> 1Start bar chart y-axis at zero
Pie charts for comparisonAngle perception is poor (rank 4); hard to compareUse bar charts instead
Rainbow colour mapNon-uniform; creates false boundariesUse Viridis, Inferno, Magma
Unnecessary 3DOcclusion, perspective distortionUse 2D unless data is inherently 3D
Dual y-axesCan create misleading apparent correlationsUse two separate charts
OverplottingCan't see density or distribution in dense scatter plotsUse transparency, jittering, or aggregation

Q1. Explain Larkin & Simon's distinction between informational equivalence and computational equivalence. Use an example to illustrate why a diagram is sometimes—but not always—more useful than text. [15 marks]
Click to reveal model answer

Informational Equivalence: Two representations are informationally equivalent if all information in one can be derived from the other and vice versa. A map and a set of written directions to the same location may contain identical information — both tell you how to get from A to B.

Computational Equivalence: Even if two representations are informationally equivalent, they may NOT be computationally equivalent — meaning they differ in how much cognitive effort is required to extract and use the information.

Why diagrams help: Larkin & Simon identify three advantages: (1) Locality — related information is grouped spatially, reducing search; (2) Minimising labelling — location itself carries information without needing explicit text labels; (3) Perceptual inference — some inferences are made automatically by the visual system (e.g., seeing two lines cross) without conscious computation.

Example: Consider a circuit diagram vs a text description of the same circuit. Both contain the same information. But to determine which components are connected, in the text you must mentally track "resistor R1 connects to capacitor C2 which connects to..." requiring extensive working memory. In the diagram, you simply follow the wires — a perceptual operation, not a cognitive one.

The "(Sometimes)" caveat: Diagrams are NOT always better. For symbolic manipulation (solving an algebraic equation), text/equations are more appropriate. If the spatial layout doesn't map naturally to the problem (a poor diagram), it can hinder rather than help. The key criterion is whether the visual representation exploits human perceptual abilities to reduce computational load.

Conclusion: The value of visualisation is not inherent in being visual — it lies in whether the representation makes relevant information accessible with minimal cognitive effort.

Q2. Describe the generalised selection interaction technique (Heer et al.). How does it differ from dynamic queries? Apply it to a dataset of cars' locations and speeds in an urban area. [15 marks]
Click to reveal model answer

Generalised Selection: Heer et al. propose modelling user selection as a declarative query over data attributes, not just as a set of individually selected items. The key innovation is "query relaxation" — starting from a specific selection and interactively broadening it to include related items.

Semantic Model for Car Data: Each car is an object with attributes: location (lat/lng), speed (km/h), direction, vehicle type, timestamp. Relationships exist between cars: proximity, shared road segments, similar speeds.

Applying Generalised Selection:

  • Initial selection: User clicks on one specific car — a taxi stopped at Oxford Street
  • Relax once: System generalises to "all stopped vehicles (speed < 5 km/h) within 200m"
  • Relax further: "All stopped vehicles on Oxford Street" — reveals a traffic jam pattern
  • Relax more: "All vehicles with speed < 20 km/h in the city centre" — shows congestion zones

At each step, the user chooses how far to generalise. Selections can be combined with Union (jams OR accidents), Intersection (slow AND large vehicles), etc.

Contrast with Dynamic Queries: Dynamic queries work top-down — the user defines a filter range with sliders ("show cars with speed 0–30 km/h") and observes the result. Generalised selection works bottom-up — the user starts by pointing at an interesting item and the system helps find more items like it. Dynamic queries have predefined dimensions; generalised selection discovers dimensions from the data itself.

Advantages: Generalised selection enables discovery-oriented exploration — users find patterns they didn't know to look for. It also produces more accurate selections by anchoring on real examples rather than abstract thresholds.

Q3. Outline Shneiderman's Visual Information-Seeking Mantra and his seven task types. Design a visualisation for a medical records database (patient conditions, prescriptions, measurements over time) using dynamic queries as an alternative to SQL. [15 marks]
Click to reveal model answer

The Mantra: "Overview first, zoom and filter, then details-on-demand." This describes the ideal interaction sequence: start with a complete view of the data, narrow it progressively, and retrieve specific information only when needed.

Seven Tasks: Overview, Zoom, Filter, Details-on-demand, Relate, History, Extract (OZF-DRHE).

Dynamic Query Design for Medical Records:

Overview: A timeline view showing all patients as horizontal bars, colour-coded by primary condition. This reveals the distribution of conditions and time ranges at a glance.

Dynamic Query Controls (vs SQL):

  • Condition slider: Select one or multiple conditions (e.g., "diabetes") — results update instantly
  • Age range slider: Filter patients aged 40–60 — results update in real-time
  • Date range selector: Focus on hospital visits in the last 12 months
  • Measurement range: "Patients with systolic BP > 140" — slider with heart-rate graph updating live
  • Toggle buttons: "Has prescription for X", "Had surgery", "Emergency admission"

Details-on-demand: Clicking any patient bar reveals their full timeline — conditions, prescriptions, measurements — using small multiples for different vitals.

Relate: Brushing and linking: select patients on the timeline → linked scatter plot updates to show correlation between their age and blood pressure.

Advantages over SQL: (1) Immediate visual feedback — results appear as you adjust sliders, enabling exploration of the data space. (2) No knowledge of SQL syntax required — accessible to non-technical clinicians. (3) Users can discover unexpected patterns (a physician might notice "all diabetic patients on Drug X have elevated BP" by seeing the visual correlation). (4) Query relaxation and refinement is natural — drag the slider slightly to see the effect. SQL requires rewriting and re-executing queries.

Q4. Outline Stevens' power law and its importance in visualisation design. Critique bar charts, pie charts, and bubble charts in terms of this law. [15 marks]
Click to reveal model answer

Stevens' Power Law: Perceived magnitude P = k × Sn, where S is the physical stimulus magnitude, n is the exponent characteristic of that sensory dimension, and k is a scaling constant. The law states that our perception of physical stimuli follows a power function, not a linear one.

Key exponents: Length/position: n ≈ 1.0 (accurate). Area: n ≈ 0.7 (underestimated by ~30%). Volume: n ≈ 0.6. Brightness: n ≈ 0.5.

Importance for Visualisation: The exponent determines how accurately users can extract quantitative information from a visual channel. Channels with n = 1 (position, length) allow accurate magnitude judgements. Channels with n < 1 cause systematic underestimation — a bubble twice as large as another will be perceived as only 1.6× larger (20.7 ≈ 1.6).

Critique of Bar Charts: Bar height uses length (n ≈ 1.0), which is among the most accurate channels. Users can readily judge that one bar is twice another. Bar charts also use position on a common baseline (rank 1 in Cleveland & McGill hierarchy). Verdict: Excellent for quantitative comparison.

Critique of Pie Charts: Pie slices use angle (n ≈ not directly in Stevens, but angle perception is known to be inaccurate). Cleveland & McGill rank angle at position 4. Users systematically misestimate slice proportions, especially when comparing non-adjacent slices. Verdict: Poor for quantitative comparison; bar charts are almost always preferable.

Critique of Bubble Charts: Bubble size encodes data via area (n ≈ 0.7). A bubble representing a value of 100 will be perceived as representing only about 1000.7/some_baseline of that. Designers sometimes try to compensate by encoding the data in the radius rather than the area, but this creates even worse distortions (since perceived area = actual area, not radius). Verdict: Systematically inaccurate; users underestimate larger values, leading to misleading impressions of the data.

Conclusion: Stevens' law provides a psychophysical basis for Cleveland & McGill's empirically derived hierarchy. Position and length are best because their exponents are close to 1.0. Area and volume are worse because their exponents cause systematic perceptual distortion.

Q5. Describe how you would visualise a dataset of 1,000,000 objects with 100 dimensions using multiple linked views, including a dimensionality reduction. [15 marks]
Click to reveal model answer

Overview of the Problem: With 1M objects and 100 dimensions, no single visualisation can show everything. We need a coordinated multi-view system following Shneiderman's mantra: overview first, zoom and filter, then details-on-demand.

View 1 — Dimensionality Reduction Overview (Spring Model / UMAP): Apply dimensionality reduction to project all 1M objects to a 2D scatter plot. UMAP is preferred for this scale — it preserves both local neighbourhood structure and global cluster relationships, unlike t-SNE which can misrepresent inter-cluster distances. This provides the overview — users immediately see high-level clusters and outliers without examining all 100 dimensions.

Why Linking Matters: The reduced 2D view loses individual dimension interpretability. To restore meaning, we link it to other views — selecting a cluster in the overview reveals what it means in original dimensions.

View 2 — Parallel Coordinates: Linked to the spring model view. Brushing a cluster in the overview highlights those items as polylines in the parallel coordinates plot, showing their values across (a subset of) the 100 dimensions. Users can then brush on a specific axis in parallel coordinates to further refine the selection. This allows pattern discovery — "this cluster has high values on dimensions 15, 32, and 67."

View 3 — Bar Charts / Histograms per Dimension: Selectable list of dimensions with distribution histograms for each. When a cluster is selected in the spring model, the histograms update to show the selected subset's distribution vs. the whole dataset. Users can then add dynamic query filters ("only objects where dim_15 > 0.6") which propagate to all views.

View 4 — Selectable Detail List: A sortable table showing individual object records for the currently selected subset. Supports details-on-demand — drill down to examine any individual item.

Interaction Design: Brushing and linking connects all views bidirectionally. History supports undo. Extract allows exporting selected subsets. The mantra is satisfied: spring model = overview; zoom into regions; filter via parallel coordinates and dynamic queries; detail list = details-on-demand.

Exam Strategy Summary

📋 For Any IV Exam Question:
  1. Describe: State data types → choose marks and channels → justify with effectiveness ranking → describe interaction (apply Shneiderman) → mention evaluation
  2. Critique: What works → what's wrong (wrong variable for data type, chartjunk, misleading scales) → specific improvements → reference principles (Tufte, Munzner, Bertin)
  3. Compare: Structured comparison — expressiveness, effectiveness, scalability, tasks supported → when each is preferred
  4. Theory: Define precisely → give concrete example → explain WHY it matters → reference relevant paper/person
🎓 Final Tip: When in doubt, connect your answer back to the Visual Information-Seeking Mantra and the channel effectiveness ranking. These two frameworks answer almost every design question. Always justify visual encoding choices by referencing data types and perceptual properties.
