DATABASE INDEXING
📚 1. PHYSICAL STORAGE FUNDAMENTALS
Storage Hierarchy
  • Primary Memory: Fast, expensive, volatile (RAM)
  • Secondary Memory: Slower, moderate cost, persistent (SSD/HDD)
  • Tertiary Memory: Slowest, cheapest, archival (Tape)
⚠️ Key Bottleneck:
Data must be loaded from secondary to primary memory - secondary storage access time becomes the bottleneck!
Physical Representation
  • Block: Basic unit of storage (e.g., 4KB, 512 bytes)
  • Record: Individual data entry stored in blocks
  • File: Collection of blocks containing records
  • Blocking Factor: Number of records per block
Blocking Factor (bfr):
$$\text{bfr} = \left\lfloor \frac{B}{R} \right\rfloor$$
Number of Blocks (b):
$$b = \left\lceil \frac{r}{\text{bfr}} \right\rceil$$
Cost Function:
$$\text{Cost} = \# \text{ of block accesses (read/write)}$$
File Organizations
Type Use Case Search Cost Characteristics
Heap Files Full table scans $O(b)$ - Linear Random order, no optimization
Sequential Files Ordered retrieval, range queries $O(\log b)$ - Binary search Sorted by key field
Hash Files Exact match queries $O(1)$ - Constant Fast lookup, poor for ranges
Indexes Multiple access paths $O(\log m)$ - Tree-based Metadata structure, fast updates
🎯 2. INDEX DESIGN OBJECTIVES
Primary Access Path: Specific file type provides primary access on ONE field
Secondary Access Path (Index): Any file type with access on MULTIPLE fields

📊 COST

Additional metadata file on disk

Storage overhead

Maintenance complexity

⚡ BENEFIT

Significantly accelerate search

Avoid linear scans

Enable efficient queries

Key Principles of Indexing
  • Principle #1: Index created over predefined field(s)
  • Principle #2: Index is a separate file from the data file
  • Principle #3: All index entries are unique and sorted w.r.t. the index field
Key Hypotheses
  • Hypothesis #1: Index file occupies less blocks than data file
    Reason: Index entries $\{k_i, p_i\}$ are smaller than full records
  • Hypothesis #2: Searching through index is faster
    Reason: Index is ordered, enabling binary/tree-based search
🔑 3. MAJOR INDEX TYPES
📌 PRIMARY INDEX
  • Definition: Index on ordering-key field of sequential file
  • Structure: Fixed-length pairs $\{k_i, p_i\}$
  • Sparseness: SPARSE - One entry per data block
  • Anchor: First record in each block (with key value $k_i$)
Index Blocking Factor (ibfr):
$$\text{ibfr} = \left\lfloor \frac{B}{V + P} \right\rfloor$$
Number of Index Blocks (ib):
$$\text{ib} = \left\lceil \frac{b}{\text{ibfr}} \right\rceil$$
Search Cost:
$$\text{Cost} = \left\lceil \log_2(\text{ib}) \right\rceil + 1 \text{ block accesses}$$
📊 Example:
File: $r = 300{,}000$ records, $R=100$ bytes, $B=4{,}096$ bytes, $V=9$ bytes, $P=6$ bytes

$$\text{bfr} = \left\lfloor \frac{4096}{100} \right\rfloor = 40 \text{ records/block}$$ $$b = \left\lceil \frac{300{,}000}{40} \right\rceil = 7{,}500 \text{ blocks}$$ $$\text{ibfr} = \left\lfloor \frac{4096}{9 + 6} \right\rfloor = 273 \text{ entries/block}$$ $$\text{ib} = \left\lceil \frac{7{,}500}{273} \right\rceil = 28 \text{ blocks}$$ $$\text{Search Cost} = \left\lceil \log_2(28) \right\rceil + 1 = 5 + 1 = 6 \text{ I/O}$$ Storage overhead: $\frac{28}{7{,}500} \times 100\% = 0.37\%$
Speedup: vs Sequential (13 I/O) = 53%
⚡ Theorem:
Create primary index IFF a block can accommodate at least TWO index entries: $$\text{ibfr} \geq 2$$
📍 CLUSTERING INDEX
  • Definition: Index on ordering NON-KEY field
  • Structure: $\{\text{cluster-value}, \text{block-pointer}\}$
  • Sparseness: SPARSE - One entry per distinct value
  • Clusters: File organized into value-based clusters
Index Entries:
$$n = \text{number of distinct values}$$
Index Blocks:
$$\text{ib} = \left\lceil \frac{n}{\text{ibfr}} \right\rceil$$
Search Cost:
$$\text{Cost} = \left\lceil \log_2(\text{ib}) \right\rceil + \frac{b}{n} \text{ block accesses}$$
📊 Example:
File: $r = 300{,}000$ records on DNO (Department Number)
Distinct values: $n = 10$ departments

$$\text{Index entries} = 10$$ $$\text{Index blocks: } \text{ib} = \left\lceil \frac{10}{273} \right\rceil = 1 \text{ block}$$ $$\text{Blocks per cluster} = \frac{7{,}500}{10} = 750$$ $$\text{Search cost} = 1 + 750 = 751 \text{ I/O (81.79\% speedup)}$$ Linear Search: $4{,}125$ block accesses
📐 Create Condition:
Create clustering index of $m$ blocks IFF: $$m < 2^{\frac{b(n+1)}{2n}}$$
Range Queries
Cost Formula:
$$\text{Cost} = \log_2(m) + (L + 1) \times \frac{b}{n}$$ where $L$ = number of additional clusters in range

Benefit Condition:
$$\log_2(m) + (L + 1) \times \frac{b}{n} < b \iff m < 2^{\frac{b(n-(L+1))}{n}}$$
🔗 SECONDARY INDEX
  • Definition: Index on NON-ORDERING field (ordered or unordered file)
  • Types:
    • [S1] On non-ordering KEY field (unique)
    • [S2] On non-ordering NON-KEY field
[S1] Secondary Index on Key Field
  • Sparseness: DENSE - One entry per record
  • Reason: File not ordered, cannot use anchor records
  • Structure: $\{\text{key-value}, \text{block-pointer}\}$
Index Entries:
$$n = r \text{ (one per record - DENSE!)}$$
Index Blocks:
$$\text{ib} = \left\lceil \frac{r}{\text{ibfr}} \right\rceil$$
Search Cost:
$$\text{Cost} = \left\lceil \log_2(\text{ib}) \right\rceil + 1 \text{ block access}$$
📊 Example:
File: $r = 300{,}000$ records

$$\text{Index entries} = 300{,}000 \text{ (dense!)}$$ $$\text{ib} = \left\lceil \frac{300{,}000}{273} \right\rceil = 1{,}099 \text{ blocks}$$ $$\text{Search cost} = \left\lceil \log_2(1{,}099) \right\rceil + 1 = 11 + 1 = 12 \text{ I/O}$$ Storage overhead: $\frac{1{,}099}{7{,}500} \times 100\% = 14\%$
Speedup: 99.68% vs heap file (3,750 I/O)
[S2] Secondary Index on Non-Key Field
  • Challenge: Multiple records per index value
  • Solution: Two-level indirection
  • Level 1: Cluster of block-pointers per value
  • Level 2: Block-pointers to actual data blocks
💡 Key Trick: Index entry → Block of pointers → Data blocks
This pattern is reusable in many indexing scenarios!
Search Cost:
$$\text{Cost} = 1 \text{ (index lookup)} + 1 \text{ (pointer block)} + k \text{ (data blocks)}$$ where $k$ = number of blocks containing the value
🏗️ MULTILEVEL INDEX
💡 Key Insight: An index is an ordered file → Can build an index on an index!
  • Level-1 (Base): Original index with $b$ entries
  • Level-2: Primary index on Level-1 with $b_1$ entries
  • Level-t: Continue until top level has 1 block
Level Calculations:
$$b_1 = \left\lceil \frac{b}{m} \right\rceil \text{ (Level-1 blocks)}$$ $$b_2 = \left\lceil \frac{b_1}{m} \right\rceil = \left\lceil \frac{b}{m^2} \right\rceil \text{ (Level-2 blocks)}$$ $$b_t = \left\lceil \frac{b}{m^t} \right\rceil \text{ (Level-t blocks)}$$
Optimal Number of Levels:
$$1 \leq \frac{b}{m^t} \implies t = \log_m(b)$$
Search Cost:
$$\text{Cost} = t + 1 \text{ block accesses}$$ where $m$ = index blocking factor (fan-out)
📊 Example:
Setup: $p=23$ (order), 69% full $\implies$ fan-out $= 0.69 \times 23 = 16$

Level-4 (Root): 1 node, 15 keys, 16 pointers
Level-3: 16 nodes, $16 \times 15 = 240$ keys
Level-2: 256 nodes, $256 \times 15 = 3{,}840$ keys
Level-1: 4,096 nodes, $4{,}096 \times 15 = 61{,}440$ keys

$$\text{Total capacity} = 15 + 240 + 3{,}840 + 61{,}440 = 65{,}535 \text{ records}$$ $$\text{Search cost} = 3 + 1 = 4 \text{ I/O}$$
🎓 Historical Note:
Logarithms invented by John Napier (1550-1617) in Scotland (Uni St Andrews)!

Base-$m$ logarithm splits search space into $m$ subspaces:
$$t = \log_m(b) < \log_2(b) \text{ when } m > 2$$
🌳 4. TREE-BASED INDEXES
🔷 B-TREE
Structure & Properties
  • Node Structure: $\{P_1, (K_1, Pr_1), P_2, (K_2, Pr_2), \ldots, (K_{q-1}, Pr_{q-1}), P_q\}$
  • Order p: Maximum number of tree pointers per node ($q \leq p$)
  • Keys: $K_1 < K_2 < \ldots < K_{q-1}$ (sorted)
  • Data Pointers: $Pr_i$ points to block with key $K_i$
  • Tree Pointers: $P_i$ points to subtree where $K_{i-1} < X < K_i$
Order Calculation:
Node Size: $$p \times P + (p-1) \times V + (p-1) \times Pr \leq B$$ Solving for $p$: $$p \leq \frac{B + V + Pr}{P + V + Pr}$$
Search Algorithm
  1. Start at root node
  2. Find appropriate range: $K_{i-1} < \text{search\_key} \leq K_i$
  3. Follow tree pointer $P_i$ or access data pointer $Pr_i$
  4. Repeat until key found or leaf reached
⚠️ Limitations:
  • Can become unbalanced → varying leaf depths
  • Doesn't adjust to key distribution
  • Complex maintenance (split/merge)
  • Maintenance complexity: $O(p^t)$
🔶 B+ TREE (Recommended!)
🎯 Core Innovation: Separate internal nodes from leaf nodes
Remove data pointers from internal nodes → Higher fan-out!
Node Types

Internal Nodes

  • Structure: $\{P_1, K_1, P_2, K_2, \ldots, P_{q-1}, K_{q-1}, P_q\}$ where $q \leq p$
  • Purpose: Guide search process
  • Content: Only keys and tree pointers (no data pointers!)
  • Advantage: More tree pointers → higher fan-out
  • Pointer Rule: For subtree at $P_i$: $K_{i-1} < X \leq K_i$ for $1 < i < q$
Internal Node Order:
Node Size: $$p \times P + (p-1) \times V \leq B$$ Solving for $p$: $$p \leq \frac{B + V}{P + V}$$

Leaf Nodes

  • Structure: $\{(K_1, Pr_1), (K_2, Pr_2), \ldots, (K_{q-1}, Pr_{q-1}), P_{\text{next}}\}$ where $q \leq p$
  • Purpose: Store all keys and data pointers
  • Content: Key-value pairs + pointer to next leaf
  • Advantage: Sequential access via linked list
Leaf Node Order:
Node Size: $$p_L \times (Pr + V) + P \leq B$$ Solving for $p_L$: $$p_L \leq \frac{B - P}{Pr + V}$$
Key Properties
  • ✅ All leaf nodes at same level → BALANCED
  • ✅ Leaf nodes linked → efficient range queries
  • ✅ All keys appear in leaf nodes
  • ✅ Some keys replicated in internal nodes (for routing)
  • ✅ Higher fan-out than B-Tree → shorter tree
📊 Comparison Example:
Setup: $B=512$, $V=9$, $P=6$, $Pr=7$

B-Tree Order: $$p = \left\lfloor \frac{512 + 9 + 7}{6 + 9 + 7} \right\rfloor = \left\lfloor \frac{528}{22} \right\rfloor = 24$$ B+ Tree Internal Order: $$p = \left\lfloor \frac{512 + 9}{6 + 9} \right\rfloor = \left\lfloor \frac{521}{15} \right\rfloor = 34$$ B+ Tree Leaf Order: $$p_L = \left\lfloor \frac{512 - 6}{7 + 9} \right\rfloor = \left\lfloor \frac{506}{16} \right\rfloor = 31$$
Metric B-Tree B+ Tree
Order (p) 24 34 (internal)
Leaf Order ($p_L$) - 31
3-Level Capacity 65,535 keys 255,507 keys
Improvement Baseline 290% more!
🏆 Why B+ Tree is Superior:
  1. Higher Fan-out: More children per node → shorter tree
  2. Better Caching: Internal nodes smaller → more fit in memory
  3. Range Queries: Sequential scan via leaf linked list
  4. Consistent Performance: All searches traverse to same depth
  5. Storage Efficient: Data pointers only in leaves
B-Tree vs B+ Tree Summary
Aspect B-Tree B+ Tree
Data Pointers In all nodes Only in leaf nodes
Keys Distributed throughout All in leaf nodes
Leaf Nodes Not linked Linked list
Fan-out Lower Higher
Tree Height Taller Shorter
Range Queries Complex traversal Simple sequential scan
Search Uniformity Variable path length All paths same length

Trade-off Analysis Framework
📐 Design Decision Framework:
Balance between search speed and storage/maintenance overhead

📊 OVERHEAD FACTORS

  • Storage space (% of data file)
  • Index maintenance (insert/delete/update)
  • Number of index levels
  • Fill factor (unused space)

⚡ SPEED FACTORS

  • Block accesses (I/O operations)
  • Fan-out (search space reduction)
  • Tree height/levels
  • Query type (point vs range)
Key Performance Formulas
Index Blocking Factor (Fan-out):
$$m = \left\lfloor \frac{B}{V + P} \right\rfloor$$
Number of Index Levels:
$$t = \left\lceil \log_m(r) \right\rceil$$
B+ Tree Search Cost:
$$\text{Cost} = t + 1 \text{ (tree traversal + data access)}$$
Storage Overhead:
$$\text{Overhead} = \frac{\text{Index Blocks}}{\text{Data Blocks}} \times 100\%$$
Speedup Percentage:
$$\text{Speedup} = \frac{\text{Baseline} - \text{Index}}{\text{Baseline}} \times 100\%$$
🎯 Design Guidelines:
  1. For Small Files ($<10$K records): Primary/Secondary index sufficient
  2. For Medium Files (10K-1M records): Multilevel or B+ Tree recommended
  3. For Large Files ($>1$M records): B+ Tree strongly recommended
  4. For Write-Heavy Workloads: Consider simpler indexes (lower maintenance)
  5. For Read-Heavy Workloads: Invest in complex indexes (better search)
  6. For Range Queries: B+ Tree or Clustering Index optimal
  7. For Point Queries: Hash or Secondary Index efficient
⚙️ 6. PRACTICAL CONSIDERATIONS
Fill Factor Impact
  • Definition: Percentage of node capacity actually used
  • Typical Value: 69% (balance between space and flexibility)
  • Impact on Capacity: Lower fill factor → more nodes needed
  • Impact on Maintenance: Higher fill factor → more splits/merges
Effective Fan-out:
$$\text{Effective Fan-out} = \lfloor 0.69 \times p \rfloor$$
Effective Capacity:
$$\text{Effective Capacity} = \lfloor 0.69 \times p_L \rfloor \text{ keys per leaf}$$
Index Maintenance Operations
➕ INSERT Operation
  1. Search for appropriate leaf node
  2. If space available → insert and done
  3. If leaf full → SPLIT:
    • Create new leaf node
    • Redistribute keys
    • Update parent with new separator key
    • Propagate split upward if parent full
➖ DELETE Operation
  1. Search for key in leaf node
  2. Remove key and pointer
  3. If underflow ($<$ min keys) → MERGE:
    • Borrow from sibling if possible
    • Otherwise merge with sibling
    • Update parent separator key
    • Propagate merge upward if needed
🔄 UPDATE Operation
  • Non-Key Update: Direct update in data block
  • Key Update: Delete old + Insert new (expensive!)
  • For Primary Index: May require anchor record update

🏭 Industry Standard:
B+ Tree is the de facto standard for database indexing due to:
  • Excellent read performance
  • Good write performance
  • Optimal for both point and range queries
  • Balanced tree guarantees consistent performance
  • Efficient use of disk blocks
When NOT to Use Indexes
  • 🚫 Small tables: Linear scan may be faster than index overhead
  • 🚫 Write-intensive tables: Index maintenance cost too high
  • 🚫 Low cardinality columns: Few distinct values (e.g., boolean)
  • 🚫 Frequently updated columns: Constant index reorganization
  • 🚫 Full table scans: When query needs most rows anyway
💡 Rule of Thumb: Index when selectivity $> 5\%$
(i.e., query returns less than 5% of table rows)
🎓 7. KEY TAKEAWAYS & SUMMARY
🌟 Core Concepts:
  1. Indexes are metadata structures that accelerate search at the cost of storage
  2. Trade-off is fundamental: Speed ↔ Overhead (storage & maintenance)
  3. Index types differ by: ordering field, key/non-key, sparse/dense
  4. Multilevel indexes reduce search from $O(n)$ to $O(\log n)$
  5. B+ Trees are optimal for most database scenarios
Index Selection Decision Tree
🎯 How to Choose:
  1. Step 1: Is the file ordered by this field?
    • Yes, and field is key → Primary Index
    • Yes, and field is non-key → Clustering Index
    • No → Go to Step 2
  2. Step 2: Is the field a key (unique)?
    • Yes → Secondary Index (Type S1)
    • No → Secondary Index (Type S2)
  3. Step 3: Is file large ($>100$K records)?
    • Yes → Use Multilevel or B+ Tree
    • No → Single-level index sufficient
  4. Step 4: Are range queries common?
    • Yes → B+ Tree (linked leaves)
    • No → B-Tree or Multilevel acceptable
Critical Formulas Cheat Sheet
Blocking Factor:
$$\text{bfr} = \left\lfloor \frac{B}{R} \right\rfloor \quad \text{[data file]}$$ $$\text{ibfr} = \left\lfloor \frac{B}{V + P} \right\rfloor \quad \text{[index file]}$$
Number of Blocks:
$$b = \left\lceil \frac{r}{\text{bfr}} \right\rceil \quad \text{[data blocks]}$$ $$\text{ib} = \left\lceil \frac{n}{\text{ibfr}} \right\rceil \quad \text{[index blocks]}$$
Search Costs:
$$\text{Linear: } \frac{b}{2}$$ $$\text{Binary: } \left\lceil \log_2(b) \right\rceil$$ $$\text{Primary Index: } \left\lceil \log_2(\text{ib}) \right\rceil + 1$$ $$\text{Multilevel: } t + 1 \text{ where } t = \log_m(b)$$
B+ Tree Orders:
$$\text{Internal: } p \leq \frac{B + V}{P + V}$$ $$\text{Leaf: } p_L \leq \frac{B - P}{Pr + V}$$
🏆 ULTIMATE LESSON:

Database indexing is about making intelligent trade-offs.
Understand your workload patterns (read vs write, point vs range),
your data characteristics (size, cardinality, update frequency),
and choose the index type that optimizes for your specific use case.

When in doubt: B+ Tree is your best friend! 🌳
