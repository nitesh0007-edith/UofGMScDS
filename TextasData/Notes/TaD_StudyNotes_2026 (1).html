Topic 1
Tokens & Text Similarity
On Crib SheetAll four similarity formulas, metric properties (identity, positivity, symmetry, triangle inequality), NLP pipeline steps, ASCII/Unicode note.
Must MemoriseHow to execute BPE step-by-step. What "word" vs "token" means. Why BPE handles unknown words. Stemming vs lemmatisation trade-offs.

Word vs Token

A word is a linguistic unit (white space delimited, has a dictionary entry). A token is the output of a tokeniser — it may differ from a word. A single word like "don't" may become two tokens ("do", "n't"). Systems use tokens because they're more predictable and consistent across edge cases (punctuation, contractions, URLs, numbers).

Why systems prefer tokens: consistent unit for downstream processing; handles punctuation, clitics, and special characters systematically.

Tokenisation Approaches

Rule-Based Advantages: deterministic, human-interpretable, domain-customisable, fast.
Disadvantages: language-specific, brittle to new patterns, requires expert knowledge.
BPE (Byte Pair Encoding) Advantages: handles any word (no [UNK] for known characters), learns from data, compact vocab.
Disadvantages: may split meaningful morphemes arbitrarily, less interpretable.

BPE Algorithm — Step by Step

Must MemoriseThe exam always asks you to run BPE manually. Know every step.
  1. Pre-tokenise the text into words (split on whitespace). Add end-of-word marker if used.
  2. Initialise vocabulary = all unique characters that appear.
  3. Find the most frequent adjacent pair of symbols across all words.
  4. Merge that pair everywhere and add it to the vocabulary as a new symbol.
  5. Repeat steps 3–4 until target vocabulary size is reached.

Exam Insight (2023 Q1e) For a grammar suggestion writing assistant: do NOT use stemming, lemmatisation, or stopword removal. A grammar checker needs the original surface forms (was/were matters!), needs stopwords ("a" vs "the" is a grammatical choice), and needs full inflections. Modifying them destroys the grammatical signal.

Similarity Measures

On Crib SheetAll four formulas are given. You just need to know how to compute them and when each applies.
Given two sets X and Y (their unique tokens): Overlap Coefficient = |X ∩ Y| / min(|X|, |Y|) Sørensen–Dice = 2|X ∩ Y| / (|X| + |Y|) Jaccard Similarity = |X ∩ Y| / |X ∪ Y| Simple overlap = |X ∩ Y| Relation: Jaccard ≤ Dice (Dice is always ≥ Jaccard for non-empty sets) Relation: Overlap ≥ Dice ≥ Jaccard
★ Worked: Dice & Jaccard between Probe A and B — 2022 Q2(d)(i), 4 marks
Probe A tokens: {a, eda, bceda, bcda, bce} |A|=5 Probe B tokens: {ca, eda, bcba, bceda, bce} |B|=5 A ∩ B = {eda, bceda, bce} → |A∩B| = 3 A ∪ B = {a, eda, bceda, bcda, bce, ca, bcba} → |A∪B| = 7 Dice = 2×3 / (5+5) = 6/10 = 0.6 Jaccard = 3/7 ≈ 0.4286
★ Worked: Triangle Inequality proof (Dice is semi-metric) — 2022 Q2(d)(ii)

A metric satisfies: d(X,Z) ≤ d(X,Y) + d(Y,Z). Similarity → distance via: d = 1 − sim.

Probe C tokens: {beda, bceda, bceca, ebeda, b} A∩C = {bceda} → Dice(A,C) = 2×1/(5+5) = 0.2 → d(A,C) = 0.8 B∩C = {bceda} → Dice(B,C) = 2×1/(5+5) = 0.2 → d(B,C) = 0.8 Dice(A,B) = 0.6 (above) → d(A,B) = 0.4 Triangle inequality requires: d(A,C) ≤ d(A,B) + d(B,C) 0.8 ≤ 0.4 + 0.8 = 1.2 ✓ — holds here Try: d(A,B) ≤ d(A,C) + d(C,B) 0.4 ≤ 0.8 + 0.8 = 1.6 ✓ To show violation: need specific sets where d(X,Z) > d(X,Y) + d(Y,Z). Counter-example: X={a,b}, Y={a,b,c,d,e,f}, Z={c,d} Dice(X,Y)=0.4 → d=0.6; Dice(Y,Z)=0.4→d=0.6; Dice(X,Z)=0→d=1.0 1.0 ≤ 0.6+0.6=1.2 ✓ — need a sharper example. Key point for exam: Dice is a semi-metric — satisfies identity, positivity, symmetry but can violate triangle inequality for certain set combinations.

Metric vs Semi-metric

PropertyDefinitionDice?Jaccard?
Identityd(X,X) = 0✅✅
Positivityd(X,Y) > 0 if X≠Y✅✅
Symmetryd(X,Y) = d(Y,X)✅✅
Triangle ineq.d(X,Z) ≤ d(X,Y)+d(Y,Z)❌ Can fail✅

Therefore: Jaccard distance is a true metric; Dice distance is only a semi-metric.

TF-IDF Intuition

TF (Term Frequency) — how often a term appears in a document. Common terms in a doc get high TF.

IDF (Inverse Document Frequency) — log(N/df). Words appearing in few documents get high IDF (they're discriminating). "The" appears everywhere → low IDF. "Quantum" appears rarely → high IDF.

TF-IDF = TF × IDF. High value = word is frequent in this doc AND rare overall = important to this doc.

Variants (crib sheet has these): Linear TF: tf(t,d) = count(t,d) Log TF: tf(t,d) = 1 + log₁₀(count) if count > 0, else 0 Log IDF: idf(t) = log(N / df(t)) TF-IDF (log-log): (1 + log(tf)) × log(N/df) For 2025 exam Q1 — they used LINEAR tf × integer IDF: TF-IDF(t, d) = count(t, d) × IDF(t)

Zipf's Law

Zipf's Law Word frequency ∝ 1/rank. The most frequent word appears ~2× more than the 2nd most frequent, ~3× more than the 3rd, etc. Few words are very frequent, many words are very rare. This makes IDF useful: rare words carry more information.

Clustering

On Crib SheetK-Means steps and centroid initialisation options. Elbow method. Silhouette formula.

K-Means: (0) initialise centroids → (1) assign each point to nearest centroid → (2) recompute centroids → repeat until convergence.

Choosing k: Elbow method — plot inertia (within-cluster distance) vs k; pick k where improvement "elbows".

Silhouette: ranges −1 to +1. Close to 1 = well-clustered. Close to 0 = on boundary. Close to −1 = possibly in wrong cluster.

N-gram Language Models

Goal: estimate P(w₁, w₂, …, wₙ) and P(next word | history).

Markov assumption: only look at last n−1 words. Bigram: P(wᵢ | wᵢ₋₁). Trigram: P(wᵢ | wᵢ₋₁, wᵢ₋₂).

Count-based: P(b|a) = Count(a,b) / Count(a)

Too small vs too large n

n valueProblem
Too small (unigram)Ignores context — "dog bites man" and "man bites dog" have same probability. Poor coherence.
Too largeData sparsity — very few training examples match long sequences → many zero probabilities. Model doesn't generalise.
SelectionUse held-out perplexity on validation set to choose n. Or interpolate multiple n values.

Perplexity — How to Calculate

Must MemoriseThe exam always asks you to compute perplexity. You must remember the algorithm: multiply probabilities, take 1/N-th power, then inverse (or use log-sum).
Perplexity of sequence w₁...wₙ under model M: PP = P(w₁, w₂, ..., wₙ)^(-1/N) = (P(w₁)·P(w₂|w₁)·...·P(wₙ|wₙ₋₁))^(-1/N) where N = number of tokens in the sequence (including ⟨E⟩, sometimes excluding ⟨S⟩) Lower perplexity = better model (more certain about the sequence) Log version: PP = 2^(-1/N · Σ log₂ P(wᵢ|context))
★ Worked: Perplexity calculation — 2023 Q2(a), 6 marks
Sequence: ⟨S⟩ a b b a ⟨E⟩ (N = 5 tokens after ⟨S⟩, i.e. a,b,b,a,⟨E⟩) Model X probabilities: P(a|⟨S⟩) = 0.1 P(b|a) = 0.3 P(b|b) = 0.4 P(a|b) = 0.3 P(⟨E⟩|a) = 0.5 Product = 0.1 × 0.3 × 0.4 × 0.3 × 0.5 = 0.0018 PP(X) = (0.0018)^(-1/5) = (0.0018)^(-0.2) ≈ 7.86 Model Y probabilities: P(a|⟨S⟩) = 0.3 P(b|a) = 0.1 P(b|b) = 0.6 P(a|b) = 0.3 P(⟨E⟩|a) = 0.1 Product = 0.3 × 0.1 × 0.6 × 0.3 × 0.1 = 0.00054 PP(Y) = (0.00054)^(-0.2) ≈ 10.56 Model X better (lower perplexity = more confident about the sequence)

Greedy Generation

Greedy Decoding At each step, pick the token with the highest conditional probability given the current context. Fast but can miss globally better sequences (local optima problem).

Beam Search

Must MemoriseBeam search is asked every year (2023, 2024). Know the algorithm cold.
  1. Start with the prefix. Maintain B "beams" (candidate sequences), each with a running probability.
  2. For each beam, expand by computing P(next token | beam) for ALL vocab tokens.
  3. Keep only the top B sequences overall (across all beams × all extensions).
  4. If a beam ends with ⟨E⟩, move it to the completed set. Otherwise keep extending.
  5. Return the completed sequence with highest total probability.
Common Mistake Beam search keeps the top B globally, not top B per beam. After expanding, rank ALL B×|V| candidates, keep only the best B.

Smoothing

Zero probabilities in unsmoothed models cause perplexity = ∞. Smoothing redistributes probability mass to unseen n-grams.

Add-k smoothing: P(t|θ) = (count(t,θ) + k) / (Σcount(t',θ) + k|V|) k=1 = Laplace smoothing (too aggressive in practice) Too much smoothing → all words get similar probabilities → model loses discriminative power

Which values benefit most? Zero counts (unseen bigrams) — they go from 0 to k/kV which is small but non-zero. Also very rare counts.

Reading a Confusion Matrix

Binary (Pos/Neg): Predicted Pos Predicted Neg Actual Pos TP FN Actual Neg FP TN Precision = TP / (TP + FP) ← "of what we predicted pos, how many were?" Recall = TP / (TP + FN) ← "of all actual pos, how many did we find?" F1 = 2·P·R / (P+R) Accuracy = (TP+TN) / (TP+FP+TN+FN)

Multi-class Averaging

MethodHowWhen to use
MacroUnweighted average of per-class metricAll classes equally important
WeightedAverage weighted by class sample countClass imbalance, want overall performance
MicroSum TP/FP/FN across all classes, then computeSame as accuracy for multi-class

Overfitting vs Underfitting

Overfitting (System B — 2022 Q3d) High train performance, low test performance. Model memorises training data but doesn't generalise. System B: train P=0.67, R=0.48 (sig) but test P=0.51, R=0.42 (not sig vs baseline). Classic overfit.
Good Generalisation (System A) Train and test performance both improve modestly vs baseline. Both statistically significant. Smaller overall gains but they hold up — suggests genuine learning.
★ Worked: Identify 8 anomalies in eval setup — 2025 Q2(c), 8 marks

This is a high-value "spot the errors" question. The key anomalies in the 2025 Q2c data:

  1. Training confusion matrix sums to 15, not matching: TP=2, FN=2, FP=8, TN=3 → total=15. Accuracy = (2+3)/15 = 0.33, but table says 0.5. Accuracy is wrong.
  2. Perplexity shouldn't appear in classification metrics — Perplexity is a language model metric, not a classifier metric.
  3. Macro-F1 cannot exceed Macro-Precision and Macro-Recall — F1=0.6 > P=0.2 and R=0.5 is impossible (F1 is always between P and R).
  4. Validation set has 3 classes (Pos, Neg, Neutral) but training only has 2 — inconsistent label space.
  5. Validation accuracy = 1.00 is suspicious — perfect accuracy on validation is almost never real; suggests data leakage or the validation set is too easy.
  6. Training on 15 samples is far too small — any metric from 15 samples is statistically meaningless.
  7. Macro-Precision = Macro-Recall = Macro-F1 = 1.00 on validation is suspicious — this only holds if F1=P=R=1.0 per class.
  8. Evaluating on training set — reporting metrics on the training data is not a valid measure of generalisation.

Dataset Construction Pitfalls

Common Dataset Problems (2025 Q2a, Q2b)
  • Search bias: searching for "comfort food" to label comfort food recipes = labels are circular/biased; the first 5 results ≠ representative sample.
  • Class overlap: "pasta" and "Italian" overlap — a recipe can be both. Multi-label issue masquerading as multi-class.
  • Label leakage: if dataset split after triplication, the same sample appears in train AND test → inflated metrics.
  • Class imbalance from triplication: comfort food class 3× overrepresented → model biased toward that class.
  • Evaluation on train set: triplication before split = train/test not independent.

Statistical Significance

A result marked * (p<0.05) means there's less than 5% chance the improvement over baseline is due to random chance. Significance on train but not test = likely overfitting or the train gain doesn't generalise.

Significance does NOT mean the improvement is practically important — a 1% improvement can be significant with enough data.

Distributional Hypothesis

Core Idea "A word is known by the company it keeps" (Firth). Words with similar contexts have similar meanings. This is the foundation of all distributional semantics.

Bag-of-Words Context Vectors

Each word w is represented by a vector counting co-occurrences within a window of size k.

Window size k effects:

  • k too small: only captures syntactic/local relationships; misses topical/semantic associations.
  • k too large: captures thematic similarity but loses syntactic signal; also every word looks similar to every other word (all share common contexts).

Why Function Words Are Unreliable

★ Worked: 2022 Q1(d) — Function word problem

Function words like "the", "of", "is" appear everywhere. They co-occur with virtually every content word in the corpus. Therefore their context vectors are nearly uniform across all other words.

Example: "the" appears before nouns like "cat", "dog", "car", "bank". So the vector for "the" has high counts for ALL nouns. This means "the" appears similar to every noun — it has no discriminating power. Also, all content words will have "the" as a context, meaning their vectors all share this dimension. The function word's representation doesn't capture any specific meaning.

Fix: Use PPMI (Positive Pointwise Mutual Information) which downweights frequent co-occurrences, effectively de-emphasising function word contexts.

Word2Vec & Negative Sampling

Word2vec learns word vectors by training to predict: "is word w a real context word for target t, or a randomly sampled noise word?"

Role of negative samples: Without them, the model could trivially satisfy the objective by making all probabilities = 1. Negative samples (words that did NOT appear in context) provide a contrastive signal — the model must push real context words closer and noise words further away. This shapes the geometry of the embedding space.

Polysemy Problem

Polysemy (2022 Q1f) Word2vec produces ONE vector per word. For polysemous words (jaguar = car brand OR animal; bank = financial institution OR river bank; python = snake OR programming language), the single vector is a blend of all senses. Nearest neighbours will be a confusing mix (Ferrari AND jaguar the animal). For downstream classification, this creates ambiguity — the model can't distinguish which sense is active in context. Fix: Contextual embeddings (ELMo, BERT) produce different vectors depending on context.

Return argmin_wᵢ score(wᵢ) // word with lowest avg similarity to others Assumption: The "true" group words will be mutually similar; the outlier will be dissimilar to the majority. Dimensionality Reduction (SVD)

Role in Dense Embeddings (2023 Q3c — GPT critique) Raw co-occurrence vectors are sparse (mostly zeros) and very high-dimensional (vocab size). SVD (Singular Value Decomposition) compresses them into dense low-dimensional vectors by finding the most important axes of variation. Benefits: (1) removes noise from irrelevant dimensions, (2) captures latent semantic structure, (3) makes vectors computationally tractable. The GPT answer in 2023 only mentioned computational benefits — it missed the key point that SVD discovers latent semantic structure by treating similar contexts as related dimensions.

Hidden Markov Model for POS Tagging

Two probability tables: Transition: P(tag_t | tag_{t-1}) ← "after a NOUN, likely a VERB" Emission: P(word | tag) ← "given tag NOUN, likely to see 'dog'" Viterbi: dynamic programming — find the tag sequence that maximises the joint probability of words AND tags. O(n · T²) where T = number of tags. Greedy: at each position, pick the tag maximising P(tag | prev_tag) × P(word | tag) Fast but suboptimal — doesn't look ahead.

Domain Shift Problem

★ Worked: Why POS tagger fails on patents — 2022 Q3(c), 4 marks

Why the problem arises: The off-the-shelf HMM tagger was trained on general text (e.g. news). Its emission probabilities P(word|tag) were estimated from that corpus. Patent text contains specialised vocabulary ("alkylation", "infringement", "oscillator") that never appeared in training — these words have zero emission probability for any known tag. Also, domain-specific words may be used as different POS than in general text (e.g. technical nouns used as verbs).

What to do:

  • Annotate patent data: manually tag a sample of patent sentences for POS, then retrain or fine-tune the tagger.
  • Handle unknown words: add smoothing to emission probabilities; use morphological features (suffixes like "-tion", "-ing", capitalisation) to guess tags for unseen words.
  • Fine-tune a neural tagger (BERT-based) on patent data — better at generalising to new vocabulary.
  • Active learning: have the system flag uncertain tags for human correction, iteratively improving.

POS Features for Classification

★ Worked: POS feature for patent classification — 2022 Q3(b), 3 marks

Example: The word "bearing" can be a NOUN (a mechanical bearing — relevant to Section F: Mechanics) or a VERB (bearing weight — different context). Without POS, both uses are conflated in a bag-of-words. With POS, the feature "bearing/NN" (bearing as noun) strongly predicts Section F, while "bearing/VBG" (bearing as verb) doesn't. Similarly, "compound/NN" (a chemical compound) → Section C (Chemistry), vs "compound/VB" (to compound interest) → Section G (Finance).

Justification: POS disambiguates word sense at the feature level without needing full word sense disambiguation.

Including POS in BERT (2022 Q3e)

In a linear bag-of-features model, POS is easy: create a feature "word_POS" (e.g. "bearing_NN") as a string token in the feature vector.

In a BERT model, you can't just add string features. Options:

  • Concatenate POS embeddings: Run a POS tagger, embed each POS tag, concatenate the POS embedding to each BERT token embedding at the input layer. BERT then attends over the combined representations.
  • Add as additional tokens: Insert POS tag tokens into the sequence (e.g. "bearing [NN] weight [VBG]") — hacky but works.
  • Multi-task learning: Add a POS tagging head alongside the classification head. The shared BERT layers learn POS-aware representations.

Difference from linear model: In linear models features are independent and hand-engineered. In BERT, the POS information interacts with all other contextual signals through attention — it's fused, not concatenated at the decision boundary.

Named Entity Recognition (NER)

IOB tagging scheme: B-TYPE = Beginning of entity of type TYPE I-TYPE = Inside (continuation) of entity of type TYPE O = Outside (not an entity) Example: "Tim Cook, CEO of Apple" Tim/B-PER Cook/I-PER ,/O CEO/B-ROLE of/O Apple/B-ORG

Nested Entity Mentions

★ Worked: Critique GPT answer on nested entities — 2023 Q3(e), 3 marks

GPT's answer: "Tim Cook" is a nested entity containing "CEO" and "Apple".

This is wrong.

A nested entity mention is when one named entity spans contains another named entity span. The correct example is:

"The CEO of Apple announced..." — here, "[CEO of Apple]" is an ORG entity, and "[Apple]" is also an ORG entity, nested inside the larger one. The inner mention is Apple, the outer mention is "CEO of Apple".

Tim Cook is a PERSON entity. CEO is a ROLE. Apple is an ORG. Tim Cook is NOT a nested combination of CEO and Apple — they're separate entities in the sentence. Tim Cook just happens to hold that role. A true nested mention would be "[Bank of [England]]" where England is a GPE entity nested inside the "Bank of England" ORG entity.

Entity Linking

  1. Detect entity mentions in text (via NER).
  2. Generate candidate entities from a knowledge base (using a dictionary/lexicon lookup, or dense vector retrieval).
  3. (Optional) Rerank candidates using a transformer model that considers context.
  4. Link the mention to the best candidate entity (or NIL if not in KB).

Coreference Resolution

Anaphora: reference back ("John left. He was tired.") — pronoun refers to earlier mention.

Cataphora: reference forward ("Before he left, John grabbed his coat.") — pronoun refers to later mention.

Hobbs Algorithm: Finds the noun phrase a pronoun refers to by traversing the parse tree in a specific order. Start from the pronoun, go up the tree, then search left-to-right in a breadth-first manner for a matching NP (correct gender/number).

Fine-tuning Process

  1. Take a pre-trained model (BERT/GPT). It has general language understanding.
  2. Add a task-specific head (e.g. linear classifier on [CLS] token for classification).
  3. Train on labelled task data — the whole model (or just the head) is updated.
  4. Evaluate on held-out test set.

Multi-stage Fine-tuning (2024 Q3d)

Motivation Fine-tuning on intermediate datasets (even if not exactly the target task) can improve performance if they share linguistic features with the target. E.g., fine-tuning on general sentiment before IAC (indirect answer classification) helps because sentiment and indirect answers share affective language.
Why it sometimes hurts Catastrophic forgetting — fine-tuning on an unrelated or poorly-aligned dataset can overwrite useful representations from pretraining. Also, if the intermediate dataset has different label distributions or linguistic styles, the model may adapt in ways that hurt the target task.

Zero-shot vs Few-shot Prompting

Exam Trap (2024 Q3a) To assert "zero-shot" performance, you must verify: (1) the model's training data doesn't contain examples of the task (test set leakage risk), (2) the model wasn't instruction-tuned on similar tasks, (3) no in-context examples were provided. GPT-3 and large models are trained on internet text which may include NLP datasets — this is test set leakage.

Factuality Problems in LLMs

LLMs store knowledge implicitly in weights — not like a database lookup. Issues:

  • Hallucination: model generates plausible-sounding but false facts with confidence.
  • Knowledge cutoff: model doesn't know events after training.
  • Conflation: multiple similar entities (different people named "John Smith") get confused.
  • No source attribution: can't verify where a claim comes from.

How GPT3 "knows" Tim Cook is Apple's CEO: Trained on billions of web pages — this fact appeared many times in the training corpus, and the model learned the association through next-token prediction. It's statistical pattern matching, not true knowledge retrieval.

Sub-word Tokenisation & Brand Name Problem (2025 Q3d)

★ Worked: BeStep brand name classified as positive — 2025 Q3d, 4 marks

The problem: "bestep" / "BeStep" / "@Bestep" is an unknown brand name. BPE will split it into subwords that happen to exist in the vocabulary — e.g. "be", "step", "##step", "Best", "ep". The token "Best" or the subword sequence "be-step" may have positive sentiment associations in the pre-training data ("best" → good). This causes the model to classify all mentions of the brand as positive, regardless of the sentiment expressed.

Proposed fix:

  • Replace the brand name with a placeholder token (e.g. [BRAND]) before processing, so the model can't use spurious token-level associations.
  • Fine-tune on data containing the brand so the model learns the brand token's actual usage patterns.
  • Expand the vocabulary to include "bestep" as a single token, avoiding fragmentation.

Fleiss' Kappa & Inter-Annotator Agreement (2024 Q3b)

Counter-arguments to replacing human labels with LLM labels:
  1. κ=0.61 is "substantial agreement" (κ: <0 = worse than chance; 0–0.2 = slight; 0.21–0.4 = fair; 0.41–0.6 = moderate; 0.61–0.8 = substantial; 0.81–1.0 = almost perfect). This is acceptable quality for NLP tasks.
  2. LLMs also have biases from their training data — replacing human noise with model bias doesn't necessarily improve quality.
  3. LLMs have a knowledge cutoff and may not understand domain-specific or cultural nuances that human annotators can.
  4. Human disagreement reflects genuine label ambiguity in the data — this is valuable signal, not error. LLM labels would hide this uncertainty.
  5. Annotation guidelines were designed for humans — LLMs may interpret them differently, introducing systematic errors.

★ 2023 Q1(a) — Word vs Token (2 marks)

A word is a linguistic unit defined by its meaning and grammatical function — it's what you'd find in a dictionary. A token is the output of a tokeniser — the actual unit a system processes.

They differ because: "don't" = 1 word but often 2 tokens ("do", "n't"). URLs, numbers, and compound words create further divergence. Systems use tokens for consistency: tokenisation creates a standard, reproducible unit for downstream processing regardless of linguistic complexity.

Quick Recall Quiz

Test yourself — click "Show Answer" only after you've thought it through.

Q1. What is the Dice coefficient between sets {a,b,c} and {b,c,d,e}?

|X∩Y| = 2 ({b,c}). |X|=3, |Y|=4. Dice = 2×2/(3+4) = 4/7 ≈ 0.571. Jaccard = 2/(3+4−2) = 2/5 = 0.4.

Q2. Perplexity of "a b" under a model where P(a|S)=0.5, P(b|a)=0.4, P(E|b)=0.2?

Product = 0.5 × 0.4 × 0.2 = 0.04. N=3 (tokens: a, b, E). PP = (0.04)^(-1/3) = (0.04)^(-0.333) ≈ 2.924.

Q3. Name two reasons why removing stopwords is bad for transformer-based models like BERT.

(1) BERT uses positional encoding — removing words shifts positions, corrupting the sequence. (2) Stopwords carry grammatical structure that attention mechanisms use to understand sentence meaning ("I did NOT do it" — "not" is critical). (3) BERT was pre-trained on full text — removing stopwords creates a distribution mismatch.

Q4. What is the difference between Macro and Micro F1?

Macro F1: compute F1 for each class independently, then take the unweighted average. All classes count equally regardless of size. Micro F1: aggregate TP, FP, FN across all classes first, then compute a single F1. For multi-class, micro F1 equals accuracy when each sample has exactly one true label.

Q5. Why is Jaccard a true metric but Dice is only a semi-metric?

Both satisfy identity, positivity, and symmetry. Jaccard distance = 1 − Jaccard similarity satisfies the triangle inequality. Dice distance = 1 − Dice can violate it: there exist sets where d(A,C) > d(A,B) + d(B,C). You can construct a counter-example with a large "bridge" set B that shares tokens with both A and C, making pairwise distances small but d(A,C) large.

Q6. In word2vec, why are negative samples needed?

Without negatives, the model could trivially set all output probabilities to 1 and always "predict" the context word correctly. Negative samples (words NOT in the context window) provide a contrastive objective — the model must push real context words closer in vector space AND push randomly sampled words further away. This tension shapes the embedding geometry.

Q7. Describe beam search in 3 sentences.

Beam search maintains B candidate sequences (beams), each with a running joint probability. At each step, every beam is extended with all vocabulary tokens, and only the top B sequences (by probability) are kept. This continues until the desired length or until all beams reach the end-of-sequence token, at which point the highest-probability completed sequence is returned.

Q8. What does κ=0.61 (Fleiss' kappa) mean, and is it reliable for NLP annotation?

κ=0.61 falls in the "substantial agreement" range (0.61–0.80). It corrects for agreement by chance, so it means annotators agree substantially more than would be expected if they guessed randomly. For NLP tasks, this is generally considered acceptable — many published NLP datasets are built with κ in this range. It does NOT mean the labels are unreliable.

Q9. How would you include POS tags in a BERT classifier?

Run a POS tagger on the input. Create learned embeddings for each POS tag. Concatenate (or add) each token's POS embedding to its BERT sub-word embedding at the input layer. Then fine-tune the whole model on the classification task. The POS information interacts with other tokens through the self-attention mechanism, unlike a bag-of-features linear model where features are independent.

Q10. TF-IDF for "apple" in a document with 100 words where "apple" appears 4 times, across a corpus of 1000 documents where "apple" appears in 200: use log TF and log IDF.

Log TF = 1 + log₁₀(4) = 1 + 0.602 = 1.602. Log IDF = log₁₀(1000/200) = log₁₀(5) = 0.699. TF-IDF = 1.602 × 0.699 ≈ 1.120.
