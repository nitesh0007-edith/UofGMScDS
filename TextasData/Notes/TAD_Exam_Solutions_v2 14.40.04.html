Part (c) 4 marks
Compute the Overlap Coefficient and Jaccard Similarity between DocX and DocY.
Solution
Important
Set similarity uses sets of unique tokens (ignoring counts).
Set(DocX) = {A, C, G, H} → |DocX| = 4 Set(DocY) = {A, B, D, E, G, H, I, J} → |DocY| = 8 Intersection = {A, G, H} → |Intersection| = 3 Union = {A, B, C, D, E, G, H, I, J} → |Union| = 9
Overlap Coefficient $\text{Overlap} = \frac{|A \cap B|}{\min(|A|, |B|)} = \frac{3}{\min(4, 8)} = \frac{3}{4} = \mathbf{0.75}$
Jaccard Similarity $\text{Jaccard} = \frac{|A \cap B|}{|A \cup B|} = \frac{3}{9} = \frac{1}{3} \approx \mathbf{0.333}$
Why the difference?
Overlap Coefficient divides by the smaller set, so it's high when one set is a subset of the other. Jaccard divides by the union, penalising non-shared elements more.
Part (d) 2 marks
Compute sim(DocX, DocY) where sim(A,B) = |A−B| / |A∩B|
Solution
A - B = elements in DocX NOT in DocY = {A, C, G, H} - {A, B, D, E, G, H, I, J} = {C} → |A - B| = 1 A ∩ B = {A, G, H} → |A ∩ B| = 3
Result $\text{sim}(\text{DocX}, \text{DocY}) = \frac{|A - B|}{|A \cap B|} = \frac{1}{3} \approx \mathbf{0.333}$
Part (e) 6 marks
Describe three potential problems with the proposed similarity measure from (d).
Solution
  1. Not Symmetric: sim(A,B) ≠ sim(B,A) because |A−B| ≠ |B−A| in general. For example, |DocX−DocY|=1 but |DocY−DocX|=5. A good similarity measure should give the same value regardless of argument order.
  2. Division by Zero: If A ∩ B = ∅ (the sets share no elements), the denominator is 0 and the measure is undefined. This is problematic because completely dissimilar documents are exactly the case we need to handle.
  3. Counter-intuitive Values:
    • Identical sets give sim = 0 (since |A−A| = 0). Normally, identical items should have maximum similarity, not zero.
    • Values can exceed 1 (when |A−B| > |A∩B|), which is unusual for a "similarity" measure that should be bounded [0,1].
    • Higher values actually mean less similar (more differences relative to overlap), the opposite of what "similarity" intuitively means.
Question 2: Text Classification 20 marks
Part (a) 8 marks
Building a multi-class recipe classifier (comfort food, easy-to-make, Italian, pasta) by searching the website and taking the first 5 results per class. Discuss four problems.
Solution
  1. Overlapping / Non-Mutually-Exclusive Classes: A recipe can belong to multiple categories (e.g., "easy Italian pasta" is comfort food, easy-to-make, Italian, AND pasta). Using search results as labels assumes each recipe belongs to only one class, creating noisy labels.
  2. Very Small Dataset: Only 5 recipes per class = 20 total samples. This is far too small to train a reliable classifier. The model will not learn generalisable patterns and will likely overfit.
  3. Search Engine Bias: The search engine may prioritise popular, recent, or SEO-optimised recipes rather than representative ones. The top 5 results are unlikely to be a representative sample of each category.
  4. Noisy / Incorrect Labels: A recipe returned by searching "comfort food" might only mention the term without actually being comfort food (e.g., "this is NOT your typical comfort food"). The search treats keyword presence as ground truth, leading to mislabelled data.
Part (b) 4 marks
"Comfort food" samples were triplicated before data splitting. What effect on training and evaluation?
Solution
  1. Class Imbalance: Comfort food now has 15 samples vs 5 for each other class. The classifier will be biased towards predicting "comfort food" more often.
  2. Data Leakage: Since triplication occurred before splitting, the same comfort food samples will appear in both training and test sets. The model will be evaluated on data it already saw during training.
  3. Inflated Evaluation Metrics: Because of the data leakage, accuracy on the test set will appear artificially high — the model "memorised" these samples and can predict them correctly without learning genuine patterns.
  4. Poor Generalisation: The model over-represents comfort food patterns. When deployed on real data with a different class distribution, it will perform much worse than the evaluation suggested.
Part (c) 8 marks
Identify eight anomalies in the dataset, metrics, and training/evaluation procedure (Tables 1-4).
Solution
Approach
Manually verify each metric from the confusion matrix and check for logical inconsistencies.
  1. Different number of classes: Training CM has 2 classes (Pos, Neg), but validation CM has 3 classes (Pos, Neg, Neutral). A model trained on 2 classes cannot predict a 3rd class it never saw.
  2. Accuracy is wrong (training):
    From CM: correct = 2 + 3 = 5, total = 15 Accuracy = 5/15 = 0.333 Reported: 0.5 ← WRONG
  3. Perplexity is not a classification metric: Perplexity measures language model quality, not classifier performance. It should not appear in a classification evaluation table.
  4. Macro-Precision is wrong (training):
    Pos Precision = TP/(TP+FP) = 2/(2+8) = 0.2 Neg Precision = TN/(TN+FN) = 3/(3+2) = 0.6 Macro-Precision = (0.2 + 0.6)/2 = 0.4 Reported: 0.2 ← WRONG
  5. Macro-Recall is wrong (training):
    Pos Recall = 2/(2+2) = 0.5 Neg Recall = 3/(8+3) = 3/11 = 0.273 Macro-Recall = (0.5 + 0.273)/2 = 0.386 Reported: 0.5 ← WRONG
  6. Macro-F1 is impossible: F1 is the harmonic mean of Precision and Recall. It CANNOT be larger than both Precision (0.2) and Recall (0.5). Reported F1 = 0.6 > 0.5 ← mathematically impossible.
  7. Perfect validation metrics (data leakage / overfitting): The validation set achieves 100% accuracy with perfect precision, recall, and F1 — while training accuracy is only 33%. This strongly suggests data leakage or some other critical error.
  8. No test set: A proper evaluation requires train/validation/test splits. Only train and validation results are shown; there is no held-out test set for final unbiased evaluation.
Question 3: Sentiment Analysis 20 marks
Part (a) 2 marks
Rank three approaches by computational cost of deployment (least to most expensive): BERT-base, Logistic Regression + TF-IDF, LLAMA-70B.
Solution
Ranking (Least → Most Expensive) Logistic Regression (TF-IDF) < BERT-base < LLAMA-70B
  • Logistic Regression + TF-IDF: Cheapest. Simple linear model, TF-IDF is a lightweight transformation. Can run on CPU with minimal resources.
  • BERT-base: Medium cost. ~110M parameters. Requires a GPU for reasonable speed but manageable.
  • LLAMA-70B: Most expensive. 70 billion parameters — needs multiple high-end GPUs just for inference.
Part (b) 4 marks
Describe a procedure for determining which solution is most effective.
Solution
1
Collect labelled data: Gather social media posts mentioning BeStep and manually annotate them with sentiment labels (positive, negative, neutral, mixed).
2
Split data: Divide into train, validation, and test sets (e.g., 70/15/15).
3
Train/configure each model: Fine-tune logistic regression and BERT-base on training data. Design prompts for LLAMA-70B (zero-shot or few-shot).
4
Evaluate all three on the same held-out test set using metrics like Macro-F1, Accuracy, and per-class Precision/Recall. Use statistical significance tests to compare.
5
Select the best model based on effectiveness on the test set, taking into account cost-effectiveness trade-offs.
Part (c) 4 marks
Give one problem for logistic regression in another language. Give a different problem for BERT-base. Suggest a solution for both.
Solution
  • Logistic Regression problem: TF-IDF features are language-specific. The vocabulary, word boundaries, and distribution patterns differ entirely across languages. A model trained on English TF-IDF features cannot be applied to French or Mandarin.
  • BERT-base problem: Standard BERT-base was pre-trained only on English text. Its tokenizer and learned representations do not understand other languages.
  • Solution for both: Use a multilingual pre-trained model such as mBERT or XLM-RoBERTa. These models are pre-trained on 100+ languages and can perform cross-lingual transfer.
Part (d) 4 marks
A BERT model classifies every mention of "BeStep" as positive. The problem is sub-word tokenisation. Describe and suggest a fix.
Solution
The Problem
Sub-word tokenisers (like WordPiece used by BERT) break unknown words into smaller pieces. "BeStep" might be tokenized as ["Be", "##step"] or ["Best", "##ep"]. The sub-token "Best" overlaps with the positive sentiment word "best", causing the model to associate ANY mention of the brand with positive sentiment, regardless of context.

Similarly, "bestep" (lowercase) could become ["best", "##ep"], again injecting a positive signal.

Solution
Add "BeStep" (and its variations like "bestep", "@Bestep") as special tokens in the tokenizer vocabulary. This ensures the brand name is treated as a single, neutral token rather than being broken into meaningful sub-words. Then fine-tune the model so it learns that this token is semantically neutral.
Part (e) 6 marks
BeStep wants fine-grained sentiment for each of their 18 products. Describe an approach.
Solution
Approach: Aspect-Based Sentiment Analysis (ABSA)
1
Product/Aspect Extraction: Identify WHICH product is mentioned in each post. Use NER or keyword/pattern matching with a dictionary of BeStep's 18 product names and their known variations.
2
Aspect-Level Sentiment: For each identified product mention, classify the sentiment specifically about that product. A single post might mention multiple products with different sentiments.
3
Integration with existing system: Use the existing sentiment classifier as a first pass for overall post sentiment. Then apply the ABSA module to extract per-product sentiment. Pipeline: (1) filter posts, (2) identify products, (3) classify sentiment per product mention.
4
Aggregation & Dashboard: Aggregate per-product sentiment scores over time to give BeStep a dashboard showing sentiment trends for each of their 18 products individually.

Question 1: Tokenisation 20 marks
Part (a) 4 marks
Why are stopwords normally removed in bag-of-words? Why NOT removed for transformers? Give an example.
Solution

Why remove stopwords in BoW: Stopwords (e.g., "the", "is", "a") are extremely frequent and appear in almost every document. In a BoW representation, they dominate the feature space but carry little discriminative meaning.

Why keep stopwords in transformers: Transformers use contextual representations where word order and grammatical structure matter. Stopwords provide crucial grammatical information.

Example
• "The dog bit the man" vs "The man bit the dog"
• BoW (without stopwords): both → {dog, bit, man} — identical! BoW can't tell them apart.
• Transformer: keeps "the" and word positions, understanding that subject and object differ.
Part (b) 4 marks
Is [UNK] token necessary for subword tokenisers like BPE? Consider Unicode encoding.
Solution

The colleague is largely correct but with caveats:

  • Why they seem correct: BPE can break any word down to individual characters (or bytes). If the base vocabulary includes all byte values (0-255), then ANY input can be encoded.
  • Why [UNK] might still be needed: If BPE operates at the character level (not byte level) and the base vocabulary doesn't include all possible Unicode characters (over 140,000), then rare characters from unseen scripts cannot be represented.
Key Insight
Byte-level BPE (like GPT-2 uses) truly doesn't need [UNK] because Unicode text can always be represented as raw bytes. Character-level BPE may still encounter unknown characters if the character vocabulary is incomplete.
Part (c) 6 marks
Perform BPE sub-word tokenisation on "mississippi pie" using the given rules. Show all steps.
Solution
How BPE Works
Apply merge rules in order. For each rule, scan the token sequence and merge ALL occurrences of that pair.

Initial character tokens:

mississippi: [m, i, s, s, i, s, s, i, p, p, i] pie: [p, i, e]

Rule 1: i, s → is

mississippi: [m, is, s, is, s, i, p, p, i] pie: [p, i, e] (no i,s pair)

Rule 2: p, i → pi

mississippi: [m, is, s, is, s, i, p, pi] pie: [pi, e]

Rule 3: i, e → ie

mississippi: [m, is, s, is, s, i, p, pi] (no i,e pair) pie: [pi, e] (token is "pi" not "i", so no match!) NO CHANGES

Rule 4: p, ie → pie

No "ie" tokens exist anywhere. NO CHANGES

Rule 5: p, pi → ppi

mississippi: [m, is, s, is, s, i, ppi] pie: [pi, e] (no p,pi pair)

Rule 6: si, ppi → sippi

No "si" token exists (we have "s" and "i" as separate tokens). NO CHANGES
Final Tokenisation mississippi → [m, is, s, is, s, i, ppi]
pie → [pi, e]
Key Takeaway
The ORDER of BPE merges matters! In "pie", the "i" got consumed by rule 2 (p,i → pi) before rule 3 (i,e → ie) could apply. So "pie" is never formed as a single token.

Part (d) 4 marks
Propose a two-character text that a bigram model would be unable to generate. Explain why. Propose a fix.
Solution
Proposed text "AA"
Character 'A' appears only once (position 8), followed by 'B' (position 9). So: P(A|A) = 0 (the bigram "AA" never appears) To generate "AA" after the text: - First token: P(A|B) = 1/9 (possible) - Second token: P(A|A) = 0 ← IMPOSSIBLE

Fix: Apply Laplace (add-1) smoothing to all bigram counts. This adds 1 to every count, ensuring no bigram has zero probability. Then use random sampling (instead of greedy) to allow low-probability sequences to be generated occasionally.

Question 3: Indirect Answer Classification (IAC) 20 marks
Part (a) 2 marks
You get higher LLM effectiveness on Circa's test set than Louis et al. What must you consider before claiming "zero-shot"?
Solution

You must consider data contamination: the LLM may have seen the Circa dataset during pre-training. Large language models are trained on massive internet corpora that may include benchmark datasets. If Circa was in the training data, the model has effectively "memorised" the answers, making its performance not truly zero-shot.

Part (b) 6 marks
Colleague says Circa's Fleiss kappa of 0.61 means labels are unreliable. They suggest using an LLM instead. Give three counter-arguments.
Solution
  1. 0.61 kappa is "substantial agreement": According to Landis & Koch's scale, kappa of 0.61-0.80 is "substantial agreement." Indirect answers are inherently subjective — some disagreement is expected.
  2. LLMs hallucinate and have biases: LLMs can generate confident but incorrect labels. They have systematic biases and no guarantee of consistency — the same input may get different labels on different runs.
  3. No ground truth for LLM labels: How would you evaluate LLM-generated labels? You'd need human judgement anyway. Human annotations provide a principled framework (multiple annotators, agreement metrics) that LLM labels lack.
Part (c) 6 marks
Why might the "unmatched" evaluation setting be insufficient? Propose a better technique.
Solution

Problem: The "unmatched" setting splits by scenario but different scenarios may still be semantically similar. For example, "Want to go for dinner?" and "Want to grab lunch?" are different scenarios but very similar in meaning. A model might generalise not because it learned IAC but because test scenarios overlap textually with training scenarios.

Proposed Improvement
Use text similarity-based adversarial splitting. Compute semantic similarity between all scenario pairs, then split train/test to maximise the semantic distance between them. This ensures genuinely different test scenarios for rigorous evaluation.
Part (d) 6 marks
The paper tests multiple stages of fine-tuning. What is the motivation? Why does effectiveness degrade with some datasets?
Solution

Motivation (Transfer Learning): First fine-tune on a related task with more available data, then further fine-tune on the target IAC task. This "intermediate fine-tuning" teaches useful linguistic skills that transfer to IAC.

Why effectiveness degrades with some datasets:

  • Domain/Task Mismatch: If the intermediate dataset's task is too different from IAC, the model learns irrelevant or harmful features.
  • Catastrophic Forgetting: Fine-tuning on an intermediate task can overwrite useful representations learned during pre-training.

Question 1: Tokenisation 20 marks
Part (a) 2 marks
What is the difference between a word and a token? Why do systems use tokens?
Solution
  • Word: A meaningful unit of language as understood by humans (e.g., "running", "don't").
  • Token: The unit a computational system operates on. It can be a word, sub-word, character, or any defined text segment (e.g., "don't" → ["do", "n't"]).

Why use tokens: "Word" is ambiguous — is "don't" one word or two? Tokens provide a well-defined, controlled, finite vocabulary that a system can consistently process.

Part (b) 2 marks
Why do tokenisers differ across languages? Give an example.
Solution

Languages have fundamentally different structures, writing systems, and word boundary conventions.

Example
English uses spaces between words, so whitespace tokenisation works well. But Chinese has no spaces between words: "我喜欢猫" (I like cats) — a whitespace tokeniser would treat the entire sentence as one token! Chinese requires dictionary-based or statistical segmentation.
Part (c) 8 marks
Build a BPE vocabulary of size 8 for "mississippi is pie". Show all steps.
Solution
How BPE Training Works
1. Start with individual characters as vocabulary
2. Count all adjacent character pairs
3. Merge the most frequent pair → add to vocabulary
4. Repeat until target vocabulary size is reached

Initial vocabulary (6 unique characters): {m, i, s, p, e, _}

Initial tokenisation: m i s s i s s i p p i _ i s _ p i e

Merge 1: Most frequent pair = (i,s) count 3

Merge into "is" Vocab = {m, i, s, p, e, _, is} → size 7 After merge: m is s is s i p p i _ is _ p i e

Merge 2: Most frequent pair = (is,s) count 2

Merge into "iss" Vocab = {m, i, s, p, e, _, is, iss} → size 8 ✔ Final: m iss iss i p p i _ is _ p i e
Final BPE Vocabulary (size 8) {m, i, s, p, e, _ (space), is, iss}

Two advantages of BPE over rule-based, and two advantages of rule-based over BPE.

Part (e) 4 marks
For an automatic writing/grammar assistant, would you use stemming, lemmatisation, and/or stopword removal?
Solution

No to ALL three:

  • No stemming: Stemming removes grammatical information. A grammar assistant NEEDS exact word forms to detect errors like "he run" (should be "runs").
  • No lemmatisation: Lemmatisation converts words to base form. A grammar checker needs the actual tense, number, and form.
  • No stopword removal: Stopwords ARE grammar. "The", "a", "is", "was" are crucial for grammatical correctness. Missing articles and wrong auxiliary verbs are exactly what a grammar checker must detect.
Key Principle
These preprocessing techniques are designed for tasks like information retrieval. A grammar assistant needs the exact surface form of the text.
Question 2: Language Models 20 marks
Given

Part (a) 6 marks
Calculate perplexity of "<S> a b b a <E>" for both models. Which is better?
Solution
Perplexity Formula
$PP = P(\text{sequence})^{-1/N}$ where N = number of predicted tokens (everything except <S>) = 5

Model X:

P(a|<S>) = 0.1, P(b|a) = 0.3, P(b|b) = 0.4, P(a|b) = 0.3, P(<E>|a) = 0.5 P_X = 0.1 × 0.3 × 0.4 × 0.3 × 0.5 = 0.0018 PP_X = (0.0018)^(-1/5) = (555.56)^(0.2) ≈ 3.54

Model Y:

P(a|<S>) = 0.3, P(b|a) = 0.1, P(b|b) = 0.6, P(a|b) = 0.3, P(<E>|a) = 0.1 P_Y = 0.3 × 0.1 × 0.6 × 0.3 × 0.1 = 0.00054 PP_Y = (0.00054)^(-1/5) = (1851.85)^(0.2) ≈ 4.50
Conclusion Model X (PP ≈ 3.54) better represents the sequence.
Lower perplexity = better fit = the model is less "surprised" by the sequence.

Part (c) 3 marks
Why is smoothing applied? Which values benefit most? What are negatives of too much smoothing?
Solution
  • Why smoothing: To avoid zero probabilities. If a bigram never appears in training, P=0, making perplexity infinite for any sequence containing it.
  • Which values benefit most: The zero-probability entries in Model Y: P(a|a)=0.0, P(c|b)=0.0, P(a|c)=0.0. These make it impossible to generate certain sequences.
  • Too much smoothing: The distribution becomes nearly uniform — all tokens become equally likely regardless of context, erasing the patterns learned from data.
Part (d) 3 marks
Problems of very small or very large n in n-gram models? How to select best n?
Solution
Too Small n (e.g., 1)Too Large n (e.g., 10)
No context captured; unigram treats each token independently. Underfitting.Data sparsity: Most n-grams never appear. Model memorises training sequences (overfitting). Exponential memory cost.

How to select n: Evaluate perplexity on a held-out validation set for different values of n. Choose the n that gives the lowest validation perplexity. Typically n=3 to 5 works well.

Question 3: Factuality of Language Model Responses 20 marks
Task
Critique GPT3's responses and correct factual errors.
Part (a) 3 marks
GPT3 says language models measure "Perplexity" and "Coverage". Critique this.
Solution

Errors:

  • Perplexity is NOT a probability — it's an evaluation metric derived from probabilities.
  • "Coverage" is not a standard language model concept. GPT3 fabricated this.

Correct Answer: The two probabilities are:

  1. Joint Probability $P(w_1, w_2, \ldots, w_n)$: The probability of an entire sequence.
  2. Conditional Probability $P(w_n | w_1, \ldots, w_{n-1})$: The probability of the next word given all preceding words.
Part (b) 2 marks
GPT3 compares TF and IDF. Critique.
Solution

Mostly correct but with imprecisions:

  • TF issue: GPT3 defines TF as normalised (count/total words). The simplest TF is just the raw count. Multiple TF variants exist.
  • IDF: Essentially correct: IDF = log(N / df).
  • Missing: GPT3 doesn't mention TF and IDF are typically used together as TF-IDF = TF × IDF.
Part (c) 4 marks
GPT3 explains dimensionality reduction for dense word embeddings. Critique.
Solution

Major errors in framing:

  • Wrong framing: GPT3 says dimensionality reduction "accelerates training." The primary purpose is NOT speed — it IS the core mechanism for creating dense embeddings. SVD/PCA applied to a word co-occurrence matrix is the process of building dense embeddings from sparse vectors.
  • "Contextual information" claim is misleading: Dimensionality reduction captures latent semantic relationships — words appearing in similar contexts get similar dense vectors, even if they never co-occurred directly.
  • What's correct: Reducing noise from irrelevant dimensions is a valid point.
Part (d) 3 marks
GPT3 explains why HMMs are used for POS tagging. Critique.
Solution

Partially correct but incomplete:

  • Correct: HMMs capture transition probabilities P(tag_i | tag_{i-1}). The Viterbi algorithm point is also correct.
  • Missing key component: Emission probabilities P(word | tag). HMMs model TWO things: (1) Transition: how likely is a noun after a determiner? (2) Emission: how likely is "cat" to be a noun? Without emission probabilities, the model cannot connect words to tags at all.

Question 1: Distributional Semantics and Word Embedding 18 marks
Part (a) 5 marks
Given word vectors, describe pseudo-code for finding the outlier word from a list.
Solution
Algorithm: Find Outlier Word INPUT: List of words W = {w1, w2, ..., wn} with vectors {v1, v2, ..., vn} STEP 1: Compute the centroid (average vector) centroid = (1/n) × ∑ vi for all i from 1 to n STEP 2: Compute similarity of each word to the centroid FOR each word wi: simi = cosine_similarity(vi, centroid) STEP 3: Return the word with the LOWEST similarity outlier = argmin_i (simi) RETURN outlier
Example
For {linux, windows, solaris, android, java}: vectors for the four OS names will be close together. "java" (a programming language) will be farthest from the centroid → detected as outlier.
Part (b) 2 marks
What happens if context window size k is too large or too small?
Solution
k too small (e.g., 1)k too large (e.g., 100)
Only immediate neighbours captured. Tends to capture syntactic relationships. Misses broader semantic connections.Distant, unrelated words get counted as context. Too much noise. All word vectors start looking similar. Captures broad topical similarity but loses precise semantic meaning.
Part (c) 5 marks
Describe pseudo-code for distributional semantics vectors in a single pass through a collection.
Solution
Algorithm: Build Distributional Semantics Vectors (Single Pass) DATA STRUCTURES: word_vectors: HashMap<String, HashMap<String, Integer>> (each word maps to a sparse vector counting co-occurring context words) INPUT: Collection of documents, window size k FOR each document D in collection: tokens = tokenize(D) FOR i = 0 to len(tokens) - 1: target_word = tokens[i] FOR j = max(0, i-k) to min(len(tokens)-1, i+k): IF j ≠ i: context_word = tokens[j] word_vectors[target_word][context_word] += 1 RETURN word_vectors
Why HashMap?
O(1) lookup/update and naturally handles sparsity — only non-zero entries are stored.
Part (d) 2 marks
Why are vectors of function words (e.g., "of", "the") unreliable?
Solution

Function words appear in almost every context with almost every word. Their distributional vectors end up very similar to each other, because they co-occur with everything indiscriminately.

Example
"the" appears before nouns, adjectives, verbs, etc. Its context vector has high counts for virtually all words, making it nearly identical to "a" or "of." These vectors don't capture any specific meaning — they just reflect high frequency.
Part (e) 2 marks
What is the role of negative samples in word2vec's objective function?
Solution

Without negative samples, the objective only maximises the probability of observed word pairs (pushing co-occurring words together). This leads to a trivial solution where all vectors collapse to the same point.

Negative samples provide contrastive examples: randomly sampled words that are NOT in the context. The objective pushes these non-co-occurring word pairs apart, creating meaningful distinctions in the embedding space.

Part (f) 2 marks
What about word2vec's output for polysemous words like "jaguar", "bank", "python"?
Solution

Word2vec produces a single vector per word regardless of how many senses it has. For polysemous words, this vector is an average across all senses.

Example for "python"
Nearest neighbours might include BOTH "snake", "anaconda" (animal sense) AND "java", "programming" (language sense). The vector sits "between" both senses.

Problem for text classification: If a document uses "python" only as a programming language, the vector also encodes the snake sense, introducing noise and reducing classification accuracy.

Part (d)(i) 4 marks
Calculate Sørensen-Dice and Jaccard between Probe A and Probe B.
Solution
Set A = {a, eda, bceda, bcda, bce} |A| = 5 Set B = {ca, eda, bcba, bceda, bce} |B| = 5 A ∩ B = {eda, bceda, bce} |A ∩ B| = 3 A ∪ B = {a, eda, bceda, bcda, bce, ca, bcba} |A ∪ B| = 7
Sørensen-Dice $\text{Dice} = \frac{2|A \cap B|}{|A| + |B|} = \frac{2 \times 3}{5+5} = \frac{6}{10} = \mathbf{0.6}$
Jaccard Similarity $\text{Jaccard} = \frac{|A \cap B|}{|A \cup B|} = \frac{3}{7} \approx \mathbf{0.429}$
Part (d)(ii) 3 marks
Calculate Dice between Probe C and Probes A/B. Show Dice is a semi-metric (breaks triangle inequality).
Solution
Set C = {beda, bceda, bceca, ebeda, b} |C| = 5 A ∩ C = {bceda} |A ∩ C| = 1 B ∩ C = {bceda} |B ∩ C| = 1
Dice Results Dice(A, C) = 2×1 / (5+5) = 0.2
Dice(B, C) = 2×1 / (5+5) = 0.2
Converting to distance: d(X,Y) = 1 - Dice(X,Y) d(A,B) = 1 - 0.6 = 0.4 d(A,C) = 1 - 0.2 = 0.8 d(B,C) = 1 - 0.2 = 0.8 Triangle inequality: d(X,Z) ≤ d(X,Y) + d(Y,Z) d(A,B) = 0.4 ≤ 0.8 + 0.8 = 1.6 ✔ d(A,C) = 0.8 ≤ 0.4 + 0.8 = 1.2 ✔ d(B,C) = 0.8 ≤ 0.4 + 0.8 = 1.2 ✔
Discussion
In this specific example, the triangle inequality holds. However, Sørensen-Dice is known theoretically to be a semi-metric — it satisfies non-negativity, identity of indiscernibles, and symmetry, but does NOT always satisfy the triangle inequality. Counterexamples exist with different set configurations.
Question 3: Natural Language Processing (Patent Classification) 18 marks
Part (a) 4 marks
Justify case normalisation and stemming for patent classification. Provide examples.
Solution

Case Normalisation:

  • Why: Without it, "Heating", "HEATING", and "heating" are three different features. Normalisation unifies them.
  • Example: Patents about "Solar HEATING systems" and "Heating elements" both belong to Section F. Without normalisation, the classifier sees different features.

Stemming:

  • Why: Reduces vocabulary and groups related forms. "heating", "heated", "heater" all stem to "heat".
  • Example: Patents using "transport", "transportation", "transporting" all relate to Section B. Stemming unifies these features.
Part (b) 3 marks
Give an example where word + POS tag helps distinguish between two patent sections.
Solution
Example: The word "light"
• "light" (NOUN) — refers to illumination → Section F (Lighting; Heating) or Section G (Instrumentation)

• "light" (ADJECTIVE) — means lightweight → Section B (Transporting) or Section A (Human necessities)

Without POS, the classifier sees the same feature "light" for both meanings. With POS, "light/NOUN" and "light/ADJ" become separate features.
Part (c) 4 marks
A 97%-accuracy HMM POS tagger makes errors on patent text. Why? How to fix?
Solution

Why it fails:

  • Tagger trained on general-domain text. Patent language uses specialised vocabulary and unusual sentence structures the tagger never encountered.
  • Unknown words: Terms like "photolithographic", "thermoelectric" have zero or poor emission probabilities.
  • Different syntax: Patent claims use unique structures that differ from training data.

How to fix:

  • Domain adaptation: Annotate patent texts with POS tags and retrain/fine-tune the HMM.
  • Add patent vocabulary: Supplement the emission probability table with domain-specific words.
