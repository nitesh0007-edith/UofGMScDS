Part (c)[4 marks]
Compute the Overlap Coefficient and Jaccard Similarity between DocX and DocY.

Solution:

Important: Set similarity uses sets of unique tokens (ignoring counts).
Set(DocX) = {A, C, G, H} → |DocX| = 4 Set(DocY) = {A, B, D, E, G, H, I, J} → |DocY| = 8 Intersection = {A, G, H} → |Intersection| = 3 Union = {A, B, C, D, E, G, H, I, J} → |Union| = 9
Overlap Coefficient = |A ∩ B| / min(|A|, |B|) = 3 / min(4, 8) = 3/4 = 0.75
Jaccard Similarity = |A ∩ B| / |A ∪ B| = 3 / 9 = 1/3 ≈ 0.333
Why the difference? Overlap Coefficient divides by the smaller set, so it's high when one set is a subset of the other. Jaccard divides by the union, penalising non-shared elements more.
Part (d)[2 marks]
Compute sim(DocX, DocY) where sim(A,B) = |A−B| / |A∩B|

Solution:

A - B = elements in DocX NOT in DocY = {A, C, G, H} - {A, B, D, E, G, H, I, J} = {C} → |A - B| = 1 A ∩ B = {A, G, H} → |A ∩ B| = 3
sim(DocX, DocY) = |A - B| / |A ∩ B| = 1 / 3 ≈ 0.333
Part (e)[6 marks]
Describe three potential problems with the proposed similarity measure from (d).

Solution:

  1. Not Symmetric: sim(A,B) ≠ sim(B,A) because |A−B| ≠ |B−A| in general. For example, |DocX−DocY|=1 but |DocY−DocX|=5. A good similarity measure should give the same value regardless of argument order.
  2. Division by Zero: If A ∩ B = ∅ (the sets share no elements), the denominator is 0 and the measure is undefined. This is problematic because completely dissimilar documents are exactly the case we need to handle.
  3. Counter-intuitive Values:
    • Identical sets give sim = 0 (since |A−A| = 0). Normally, identical items should have maximum similarity, not zero.
    • Values can exceed 1 (when |A−B| > |A∩B|), which is unusual for a "similarity" measure that should typically be bounded [0,1].
    • Higher values actually mean less similar (more differences relative to overlap), which is the opposite of what "similarity" intuitively means.
Question 2: Text Classification 20 marks
Part (a)[8 marks]
Building a multi-class recipe classifier (comfort food, easy-to-make, Italian, pasta) by searching the website and taking the first 5 results per class. Discuss four problems.

Solution:

  1. Overlapping / Non-Mutually-Exclusive Classes: A recipe can belong to multiple categories (e.g., "easy Italian pasta" is comfort food, easy-to-make, Italian, AND pasta). Using search results as labels assumes each recipe belongs to only one class, creating noisy labels.
  2. Very Small Dataset: Only 5 recipes per class = 20 total samples. This is far too small to train a reliable classifier. The model will not learn generalisable patterns and will likely overfit.
  3. Search Engine Bias: The search engine may prioritise popular, recent, or SEO-optimised recipes rather than representative ones. The top 5 results are unlikely to be a representative sample of each category.
  4. Noisy / Incorrect Labels: A recipe returned by searching "comfort food" might only mention the term without actually being comfort food (e.g., "this is NOT your typical comfort food"). The search treats keyword presence as ground truth, leading to mislabelled data.
Part (b)[4 marks]
"Comfort food" samples were triplicated before data splitting. What effect on training and evaluation?

Solution:

  1. Class Imbalance: Comfort food now has 15 samples vs 5 for each other class. The classifier will be biased towards predicting "comfort food" more often.
  2. Data Leakage: Since triplication occurred before splitting, the same comfort food samples will appear in both training and test sets. The model will be evaluated on data it already saw during training.
  3. Inflated Evaluation Metrics: Because of the data leakage, accuracy on the test set will appear artificially high — the model "memorised" these samples and can predict them correctly without learning genuine patterns.
  4. Poor Generalisation: The model over-represents comfort food patterns. When deployed on real data with a different class distribution, it will perform much worse than the evaluation suggested.
Part (c)[8 marks]
Identify eight anomalies in the dataset, metrics, and training/evaluation procedure (Tables 1-4).

Solution:

Approach: Manually verify each metric from the confusion matrix and check for logical inconsistencies.
  1. Different number of classes: Training CM has 2 classes (Pos, Neg), but validation CM has 3 classes (Pos, Neg, Neutral). A model trained on 2 classes cannot predict a 3rd class it never saw.
  2. Accuracy is wrong (training):
    From CM: correct = 2 + 3 = 5, total = 15 Accuracy = 5/15 = 0.333 Reported: 0.5 ← WRONG
  3. Perplexity is not a classification metric: Perplexity measures language model quality, not classifier performance. It should not appear in a classification evaluation table.
  4. Macro-Precision is wrong (training):
    Pos Precision = TP/(TP+FP) = 2/(2+8) = 0.2 Neg Precision = TN/(TN+FN) = 3/(3+2) = 0.6 Macro-Precision = (0.2 + 0.6)/2 = 0.4 Reported: 0.2 ← WRONG

    Pos Recall = 2/(2+2) = 0.5 Neg Recall = 3/(8+3) = 3/11 = 0.273 Macro-Recall = (0.5 + 0.273)/2 = 0.386 Reported: 0.5 ← WRONG

  7. Perfect validation metrics (data leakage / overfitting): The validation set achieves 100% accuracy with perfect precision, recall, and F1 — while training accuracy is only 33%. This strongly suggests data leakage, the validation set being the training set, or some other critical error.

Question 3: Sentiment Analysis 20 marks
Part (a)[2 marks]
Rank three approaches by computational cost of deployment (least to most expensive): BERT-base, Logistic Regression + TF-IDF, LLAMA-70B.

Solution:

Least Expensive → Most Expensive:
Logistic Regression (TF-IDF) < BERT-base < LLAMA-70B
  • Logistic Regression + TF-IDF: Cheapest. Simple linear model, TF-IDF is a lightweight transformation. Can run on CPU with minimal resources.
  • BERT-base: Medium cost. ~110M parameters. Requires a GPU for reasonable speed but manageable for most setups.
  • LLAMA-70B: Most expensive. 70 billion parameters — needs multiple high-end GPUs just for inference. Massive memory and compute requirements.
Part (b)[4 marks]
Describe a procedure for determining which solution is most effective.

Solution:

1
Collect labelled data: Gather social media posts mentioning BeStep and manually annotate them with sentiment labels (positive, negative, neutral, mixed). Ensure sufficient samples per class.
2
Split data: Divide into train, validation, and test sets (e.g., 70/15/15).
3
Train/configure each model: Fine-tune logistic regression and BERT-base on training data. Design prompts for LLAMA-70B (zero-shot or few-shot).
4
Evaluate all three on the same held-out test set using metrics like Macro-F1, Accuracy, and per-class Precision/Recall. Use statistical significance tests to compare.
5
Select the best model based on effectiveness on the test set, taking into account cost-effectiveness trade-offs.
Part (c)[4 marks]
Give one problem for logistic regression in another language. Give a different problem for BERT-base. Suggest a solution for both.

Solution:

  • Logistic Regression problem: TF-IDF features are language-specific. The vocabulary, word boundaries, and distribution patterns differ entirely across languages. A model trained on English TF-IDF features cannot be applied to French or Mandarin — you'd need to build everything from scratch for each language.
  • BERT-base problem: Standard BERT-base was pre-trained only on English text. Its tokenizer and learned representations do not understand other languages. Even if you could tokenize the text, BERT's knowledge of linguistic patterns is English-only.
  • Solution for both: Use a multilingual pre-trained model such as mBERT (multilingual BERT) or XLM-RoBERTa. These models are pre-trained on text from 100+ languages and can perform cross-lingual transfer — fine-tune on English sentiment data and apply to other languages.
Part (d)[4 marks]
A BERT model classifies every mention of "BeStep" as positive. The problem is sub-word tokenisation. Describe and suggest a fix.

Solution:

The Problem: Sub-word tokenisers (like WordPiece used by BERT) break unknown words into smaller pieces. "BeStep" might be tokenized as ["Be", "##step"] or ["Best", "##ep"]. The sub-token "Best" overlaps with the positive sentiment word "best", causing the model to associate ANY mention of the brand with positive sentiment, regardless of context.

Similarly, "bestep" (lowercase) could become ["best", "##ep"], again injecting a positive signal.

Solution: Add "BeStep" (and its variations like "bestep", "@Bestep") as special tokens in the tokenizer vocabulary. This ensures the brand name is treated as a single, neutral token rather than being broken into meaningful sub-words. Then fine-tune the model so it learns that this token is semantically neutral (just a brand name).
Part (e)[6 marks]
BeStep wants fine-grained sentiment for each of their 18 products. Describe an approach.

Solution:

Approach: Aspect-Based Sentiment Analysis (ABSA)
1
Product/Aspect Extraction: First identify WHICH product is mentioned in each post. Use Named Entity Recognition (NER) or keyword/pattern matching with a dictionary of BeStep's 18 product names and their known variations/abbreviations.
2
Aspect-Level Sentiment: For each identified product mention, classify the sentiment of the text specifically about that product. A single post might mention multiple products with different sentiments (e.g., "I love the BeStep Pro but the BeStep Lite is awful").
3
Integration with existing system: Use the existing sentiment classifier as a first pass for overall post sentiment. Then apply the ABSA module to extract per-product sentiment. This can be a pipeline: (1) filter posts mentioning BeStep, (2) identify products mentioned, (3) classify sentiment per product mention.
4
Aggregation & Dashboard: Aggregate per-product sentiment scores over time to give BeStep a dashboard showing sentiment trends for each of their 18 products individually.

2024 - April/May Exam Session (60 Marks)

Question 1: Tokenisation 20 marks
Part (a)[4 marks]
Why are stopwords normally removed in bag-of-words? Why NOT removed for transformers? Give an example.

Solution:

Why remove stopwords in BoW: Stopwords (e.g., "the", "is", "a") are extremely frequent and appear in almost every document. In a BoW representation, they dominate the feature space but carry little discriminative meaning. Removing them reduces noise and focuses on content words that distinguish documents.

Why keep stopwords in transformers: Transformers use contextual representations where word order and grammatical structure matter. Stopwords provide crucial grammatical information.

Example:
• "The dog bit the man" vs "The man bit the dog"
• BoW (without stopwords): both → {dog, bit, man} — identical! BoW can't tell them apart anyway.
• Transformer: keeps "the" and word positions, understanding that subject and object differ — these are very different sentences!
Part (b)[4 marks]
Is [UNK] token necessary for subword tokenisers like BPE? Consider Unicode encoding.

Solution:

The colleague is largely correct but with caveats:

  • Why they seem correct: BPE can break any word down to individual characters (or bytes). If the base vocabulary includes all byte values (0-255), then ANY input can be encoded as a sequence of bytes — nothing is truly "unknown."
  • Why [UNK] might still be needed: If BPE operates at the character level (not byte level) and the base vocabulary doesn't include all possible Unicode characters (there are over 140,000), then rare characters from scripts not seen during training (e.g., rare CJK characters, emojis, ancient scripts) cannot be represented and would need [UNK].
Key Insight: Byte-level BPE (like GPT-2 uses) truly doesn't need [UNK] because Unicode text can always be represented as a sequence of raw bytes. Character-level BPE may still encounter unknown characters if the character vocabulary is incomplete.
Part (c)[6 marks]
Perform BPE sub-word tokenisation on "mississippi pie" using the given rules. Show all steps.

Solution:

How BPE works: Apply merge rules in order. For each rule, scan the token sequence and merge ALL occurrences of that pair.

Initial character tokens:

mississippi: [m, i, s, s, i, s, s, i, p, p, i] pie: [p, i, e]

Rule 1: i, s → is

mississippi: [m, is, s, is, s, i, p, p, i] pie: [p, i, e] (no i,s pair)

Rule 2: p, i → pi

mississippi: [m, is, s, is, s, i, p, pi] (only the last p,i pair merges; "i,p" at position 6-7 is NOT "p,i") pie: [pi, e]

Rule 3: i, e → ie

mississippi: [m, is, s, is, s, i, p, pi] (no i,e pair) pie: [pi, e] (token is "pi" not "i", so no match!) NO CHANGES

Rule 4: p, ie → pie

No "ie" tokens exist anywhere. NO CHANGES

Rule 5: p, pi → ppi

mississippi: [m, is, s, is, s, i, ppi] (the p and pi at positions 7-8 merge) pie: [pi, e] (no p,pi pair)

Rule 6: si, ppi → sippi

No "si" token exists (we have "s" and "i" as separate tokens). NO CHANGES
Final Tokenisation:
mississippi → [m, is, s, is, s, i, ppi]
pie → [pi, e]
Key Takeaway: The ORDER of BPE merges matters! In "pie", the "i" got consumed by rule 2 (p,i → pi) before rule 3 (i,e → ie) could apply. So "pie" is never formed as a single token. This is an intentional aspect of the question.

Part (d)[4 marks]
Propose a two-character text that a bigram model would be unable to generate. Explain why. Propose a fix.

Solution:

Proposed text: "AA"

Why it cannot be generated:

Character 'A' appears only once in the text (position 8), followed by 'B' (position 9). So: P(A|A) = 0 (the bigram "AA" never appears) To generate "AA" after the text: - First token: P(A|B) = 1/9 (possible) - Second token: P(A|A) = 0 ← IMPOSSIBLE Since the probability is exactly zero, no generation method (greedy, beam search, or sampling) can produce this.

Fix: Apply Laplace (add-1) smoothing to all bigram counts. This adds 1 to every count, ensuring no bigram has zero probability. Then use random sampling (instead of greedy) to sample from the probability distribution, which allows low-probability sequences to be generated occasionally.

Question 3: Indirect Answer Classification (IAC) 20 marks
Part (a)[2 marks]
You get higher LLM effectiveness on Circa's test set than Louis et al. What must you consider before claiming "zero-shot"?

Solution:

You must consider data contamination: the LLM may have seen the Circa dataset (or similar IAC data) during pre-training. Large language models are trained on massive internet corpora that may include benchmark datasets, academic papers discussing them, or similar data. If Circa was in the training data, the model has effectively "memorised" the answers, making its performance not truly zero-shot. You should check whether the dataset was available online before the model's training data cutoff.

Part (b)[6 marks]
Colleague says Circa's Fleiss kappa of 0.61 means labels are unreliable. They suggest using an LLM instead. Give three counter-arguments.

Solution:

  1. 0.61 kappa is "substantial agreement": According to Landis & Koch's scale, kappa of 0.61-0.80 is "substantial agreement." Indirect answers are inherently subjective — some disagreement among annotators is expected and doesn't mean labels are unreliable.
  2. LLMs hallucinate and have biases: LLMs can generate confident but incorrect labels. They have systematic biases from training data (e.g., cultural, linguistic biases) and no guarantee of consistency — the same input may get different labels on different runs. At least human annotations have measured inter-annotator agreement.
  3. No ground truth for LLM labels: How would you evaluate the quality of LLM-generated labels? You'd need human judgement to validate them anyway. Human annotations provide a principled framework (multiple annotators, agreement metrics) that LLM labels lack. Additionally, the LLM's own training may not include expertise in pragmatic language understanding needed for IAC.
Part (c)[6 marks]
Why might the "unmatched" evaluation setting be insufficient? Propose a better technique.

Solution:

Problem: The "unmatched" setting splits by scenario (question-context pairs) so the test set uses different scenarios than training. However, different scenarios may still be semantically similar. For example, "Want to go for dinner?" and "Want to grab lunch?" are different scenarios but very similar in meaning and language. A model might generalise not because it learned IAC but because the test scenarios overlap textually with training scenarios.

Proposed improvement: Use text similarity-based adversarial splitting. Compute semantic similarity (e.g., using sentence embeddings) between all scenario pairs, then split train/test to maximise the semantic distance between them. This ensures the test set contains scenarios that are genuinely different from training, providing a more rigorous evaluation of generalisation.
Part (d)[6 marks]
The paper tests multiple stages of fine-tuning. What is the motivation? Why does effectiveness degrade with some datasets?

Solution:

Motivation (Transfer Learning): The idea is to first fine-tune on a related task with more available data (e.g., Natural Language Inference, dialogue understanding), then further fine-tune on the target IAC task. This "intermediate fine-tuning" teaches the model useful linguistic skills (e.g., understanding entailment, pragmatics) that transfer to IAC, especially helpful when IAC training data is limited.

Why effectiveness degrades with some datasets:

  • Domain/Task Mismatch: If the intermediate dataset's task is too different from IAC, the model learns features that are irrelevant or even harmful. For example, a sentiment analysis dataset teaches the model to focus on sentiment words, which is not what IAC needs.
  • Catastrophic Forgetting: Fine-tuning on an intermediate task can overwrite useful representations learned during pre-training, degrading performance on the target task when the intermediate task is dissimilar.

2023 - April Exam Session (60 Marks)

Question 1: Tokenisation 20 marks
Part (a)[2 marks]
What is the difference between a word and a token? Why do systems use tokens?

Solution:

  • Word: A meaningful unit of language as understood by humans (e.g., "running", "don't").
  • Token: The unit a computational system operates on. It can be a word, sub-word, character, or any defined text segment (e.g., "don't" → tokens: ["do", "n't"]).

Why use tokens: "Word" is ambiguous — is "don't" one word or two? Is "New York" one or two? Tokens provide a well-defined, controlled, finite vocabulary that a system can consistently process. They handle these ambiguities through explicit rules or learned operations.

Part (b)[2 marks]
Why do tokenisers differ across languages? Give an example.

Solution:

Languages have fundamentally different structures, writing systems, and word boundary conventions.

Example: English uses spaces between words, so whitespace tokenisation works reasonably well: "I like cats" → ["I", "like", "cats"]. But Chinese has no spaces between words: "我喜欢猫" (I like cats) — a whitespace tokeniser would treat the entire sentence as one token! Chinese requires dictionary-based or statistical segmentation.

2. Handles unknown words: Can represent any word by breaking it into known sub-word units, eliminating out-of-vocabulary problems. 2. No training data needed: Works immediately based on defined rules, useful when training data is scarce or unavailable.
Part (e)[4 marks]
For an automatic writing/grammar assistant, would you use stemming, lemmatisation, and/or stopword removal?

Solution:

No to ALL three:

  • No stemming: Stemming removes grammatical information. "running" → "run", "better" → "bet". A grammar assistant NEEDS exact word forms to detect errors like "he run" (should be "runs"). Stemming would destroy the evidence.
  • No lemmatisation: Similarly, lemmatisation converts words to base form ("went" → "go"). A grammar checker needs to know the actual tense, number, and form used to identify mistakes.
  • No stopword removal: Stopwords ARE grammar. "The", "a", "is", "was" are crucial for grammatical correctness. Missing articles ("I went to store") and wrong auxiliary verbs ("He are going") are exactly what a grammar checker must detect.
Key Principle: These preprocessing techniques are designed for tasks like information retrieval where you want to match concepts. A grammar assistant needs the exact surface form of the text to identify and suggest corrections.

Part (c)[3 marks]
Why is smoothing applied? Which values benefit most? What are negatives of too much smoothing?

Solution:

  • Why smoothing: To avoid zero probabilities. If a bigram never appears in training, it gets P=0, making perplexity infinite for any sequence containing it. Smoothing assigns a small probability to unseen events.
  • Which values benefit most: The zero-probability entries in Model Y: P(a|a)=0.0, P(c|b)=0.0, P(a|c)=0.0. These make it impossible to generate certain sequences. Model X has no zeros (except P(<E>|<S>) which is logically appropriate).
  • Too much smoothing: The distribution becomes nearly uniform — all tokens become equally likely regardless of context. This erases the patterns learned from data, making the model unable to distinguish between likely and unlikely sequences.

Problems of very small or very large n in n-gram models? How to select best n?

Solution:

Too Small n (e.g., 1)Too Large n (e.g., 10)

Underfitting: too simple to capture language patterns Requires exponentially more memory to store all n-gram counts

Question 3: Factuality of Language Model Responses 20 marks
Task: Critique GPT3's responses and correct factual errors.
Part (a)[3 marks]
GPT3 says language models measure "Perplexity" and "Coverage". Critique this.

Solution:

Errors in GPT3's response:

  • Perplexity is NOT a probability — it's an evaluation metric derived from probabilities. The question asks about probabilities that language models measure.
  • "Coverage" is not a standard language model concept. GPT3 fabricated this.

Correct Answer: The two probabilities are:

  1. Joint Probability P(w₁, w₂, ..., wₙ): The probability of an entire sequence of words occurring. Language models compute this as the product of conditional probabilities.
  2. Conditional Probability P(wₙ | w₁, ..., wₙ₋₁): The probability of the next word given all preceding words. This is the fundamental computation a language model performs at each step.
Part (b)[2 marks]
GPT3 compares TF and IDF. Critique.

Solution:

GPT3's response is mostly correct but with an imprecision:

  • TF description issue: GPT3 defines TF as "dividing count by total words in document" — this is normalised TF, one variant. The simplest TF is just the raw count (how many times the term appears). There are multiple TF variants (raw, normalised, log-scaled, boolean) and GPT3 presents only one as if it's the only definition.
  • IDF description: Essentially correct: IDF = log(N / df) where N is total number of documents and df is how many documents contain the term.
  • Missing: GPT3 doesn't mention that TF and IDF are typically used together as TF-IDF = TF × IDF to weight terms.
Part (c)[4 marks]
GPT3 explains dimensionality reduction for dense word embeddings. Critique.

Solution:

Major errors in GPT3's framing:

  • Wrong framing of the role: GPT3 says dimensionality reduction "accelerates training." The primary purpose is NOT speed — it IS the core mechanism for creating dense embeddings. Dimensionality reduction (e.g., SVD/PCA applied to a word co-occurrence matrix) is the process of building dense embeddings from sparse vectors, not a helper for some other training process.
  • "Contextual information" claim is misleading: Sparse word vectors (like co-occurrence vectors) don't inherently encode "contextual information" that dimensionality reduction reveals. Rather, dimensionality reduction captures latent semantic relationships — words that appear in similar contexts get similar dense vectors, even if they never co-occurred directly.
  • What's correct: Reducing noise from irrelevant dimensions is a valid point. High-dimensional sparse vectors have many zero/noisy entries; dimensionality reduction finds the most meaningful patterns in fewer dimensions.
Part (d)[3 marks]
GPT3 explains why HMMs are used for POS tagging. Critique.

Solution:

GPT3's response is partially correct but incomplete:

  • Correct: HMMs capture transition probabilities P(tagᵢ | tagᵢ₋₁) — the probability of one POS tag following another. The Viterbi algorithm point is also correct.
  • Missing key component: GPT3 completely omits emission probabilities P(word | tag) — the probability of a word being generated by a given POS tag. HMMs model TWO things:
    1. Transition: How likely is a noun after a determiner?
    2. Emission: How likely is "cat" to be a noun?
    Without emission probabilities, the model cannot connect words to tags at all.
Part (e)[3 marks]
GPT3 gives an example of nested entity mentions. Critique.

Solution:

GPT3's example and explanation are WRONG:

  • GPT3 claims "Tim Cook" is a nested entity made of "CEO" and "Apple" — this is completely incorrect. "Tim Cook" is a PERSON entity, and "CEO" is a role/title, not an entity contained within it.

Correct Example:

• "University of Glasgow" is an ORGANISATION entity

Part (f)[5 marks]
GPT3 explains how it encodes knowledge and challenges. Critique.

Solution:

Issues with GPT3's response:

  • "It can look at data about Apple": This is misleading. GPT3 does NOT "look at" data at inference time. Knowledge is encoded implicitly in the model's parameters (weights) during pre-training. The model memorises facts from training data and reproduces them from memory, not by consulting a database.

Question 1: Distributional Semantics and Word Embedding 18 marks
Part (a)[5 marks]
Given word vectors, describe pseudo-code for finding the outlier word from a list.

Solution:

Intuition: The outlier is the word whose vector is most different from all the others. We find the "average" word and see which word is farthest from it.
Algorithm: Find Outlier Word INPUT: List of words W = {w₁, w₂, ..., wₙ} with vectors {v₁, v₂, ..., vₙ} STEP 1: Compute the centroid (average vector) centroid = (1/n) × ∑ vᵢ for all i from 1 to n STEP 2: Compute similarity of each word to the centroid FOR each word wᵢ: simᵢ = cosine_similarity(vᵢ, centroid) STEP 3: Return the word with the LOWEST similarity outlier = argminᵢ (simᵢ) RETURN outlier
Example: For {linux, windows, solaris, android, java}: vectors for the four OS names will be close together, pulling the centroid near them. "java" (a programming language) will be farthest from this centroid → detected as outlier.
Part (b)[2 marks]
What happens if context window size k is too large or too small?

Solution:

k too small (e.g., 1)k too large (e.g., 100)
Only immediate neighbours are captured. Tends to capture syntactic relationships (words that go right next to each other). Misses broader semantic/topical connections. Distant, unrelated words get counted as context. Too much noise dilutes meaningful co-occurrence patterns. All word vectors start looking similar because every word co-occurs with every other word. Captures very broad topical similarity but loses precise semantic meaning.
Part (c)[5 marks]
Describe pseudo-code for distributional semantics vectors in a single pass through a collection.

Solution:

Algorithm: Build Distributional Semantics Vectors (Single Pass) DATA STRUCTURES: - word_vectors: HashMap<String, HashMap<String, Integer>> (each word maps to a sparse vector counting co-occurring context words) INPUT: Collection of documents, window size k FOR each document D in collection: tokens = tokenize(D) FOR i = 0 to len(tokens) - 1: target_word = tokens[i] // Look at words within window of size k FOR j = max(0, i-k) to min(len(tokens)-1, i+k): IF j ≠ i: // skip the word itself context_word = tokens[j] word_vectors[target_word][context_word] += 1 RETURN word_vectors
Why HashMap? Using a HashMap (dictionary) for the inner vectors gives O(1) lookup/update and naturally handles sparsity — only non-zero entries are stored. This is memory-efficient since most word pairs never co-occur.
Part (d)[2 marks]
Why are vectors of function words (e.g., "of", "the") unreliable?

Solution:

Function words like "the" and "of" appear in almost every context with almost every word. Their distributional vectors end up being very similar to each other and to most content words, because they co-occur with everything indiscriminately.

Example: "the" appears before nouns ("the cat"), adjectives ("the big"), verbs ("the running"), etc. Its context vector has high counts for virtually all words, making it nearly identical to the context vector of "a" or "of." These vectors don't capture any specific meaning — they just reflect high frequency, making them unreliable for semantic tasks.
Part (e)[2 marks]
What is the role of negative samples in word2vec's objective function?

Solution:

Without negative samples, the objective only maximises the probability of observed word pairs (push co-occurring words together). This leads to a trivial solution where all vectors collapse to the same point — if every word has the same vector, all dot products are maximal.

Negative samples provide contrastive examples: randomly sampled words that are NOT in the context. The objective pushes these non-co-occurring word pairs apart. This creates meaningful distinctions in the embedding space — similar words cluster together while dissimilar words are separated.

Part (f)[2 marks]
What about word2vec's output for polysemous words like "jaguar", "bank", "python"?

Solution:

Word2vec produces a single vector per word regardless of how many senses it has. For polysemous words, this vector is an average across all senses.

Example for "python":
• Nearest neighbours might include BOTH "snake", "anaconda", "reptile" (animal sense) AND "java", "programming", "code" (language sense)
• The vector sits somewhere "between" both senses in the embedding space

Problem for text classification: If a document uses "python" only as a programming language, but the vector also encodes the snake sense, it introduces noise. The classifier receives a mixed signal rather than the precise meaning, potentially reducing classification accuracy.

Part (d)(i)[4 marks]
Calculate Sørensen-Dice and Jaccard between Probe A and Probe B.

Solution:

Probe A tokens: a, eda, bceda, eda, bcda, bce Unique Set A = {a, eda, bceda, bcda, bce} → |A| = 5 Probe B tokens: ca, eda, bcba, eda, bceda, eda, bce Unique Set B = {ca, eda, bcba, bceda, bce} → |B| = 5 Intersection A ∩ B = {eda, bceda, bce} → |A ∩ B| = 3 Union A ∪ B = {a, eda, bceda, bcda, bce, ca, bcba} → |A ∪ B| = 7
Sørensen-Dice = 2|A ∩ B| / (|A| + |B|) = 2×3 / (5+5) = 6/10 = 0.6
Jaccard = |A ∩ B| / |A ∪ B| = 3/7 ≈ 0.429
Part (d)(ii)[3 marks]
Calculate Dice between Probe C and Probes A/B. Show Dice is a semi-metric (breaks triangle inequality).

Solution:

Probe C tokens: beda, bceda, bceca, ebeda, bceda, b Unique Set C = {beda, bceda, bceca, ebeda, b} → |C| = 5 A ∩ C = {bceda} → |A ∩ C| = 1 B ∩ C = {bceda} → |B ∩ C| = 1
Dice(A, C) = 2×1 / (5+5) = 2/10 = 0.2
Dice(B, C) = 2×1 / (5+5) = 2/10 = 0.2

Showing it's a semi-metric (triangle inequality):

Converting to distance: d(X,Y) = 1 - Dice(X,Y) d(A,B) = 1 - 0.6 = 0.4 d(A,C) = 1 - 0.2 = 0.8 d(B,C) = 1 - 0.2 = 0.8 Triangle inequality requires: d(X,Z) ≤ d(X,Y) + d(Y,Z) for ALL triples. d(A,B) = 0.4 ≤ 0.8 + 0.8 = 1.6 ✔ d(A,C) = 0.8 ≤ 0.4 + 0.8 = 1.2 ✔ d(B,C) = 0.8 ≤ 0.4 + 0.8 = 1.2 ✔
Discussion: In this specific example, the triangle inequality holds. However, Sørensen-Dice is known theoretically to be a semi-metric — it satisfies non-negativity, identity of indiscernibles, and symmetry, but does NOT always satisfy the triangle inequality. Counterexamples exist with different set configurations. The key insight is that Dice distance (1 - Dice coefficient) is not a proper metric because it can violate the triangle inequality in general, even though this particular example happens to satisfy it.
Question 3: Natural Language Processing (Patent Classification) 18 marks
Part (a)[4 marks]
Justify case normalisation and stemming for patent classification. Provide examples.

Solution:

Case Normalisation (converting all text to lowercase):

  • Why: Without it, "Heating", "HEATING", and "heating" are three different features, splitting evidence across them. Normalisation unifies them into one feature, strengthening the signal.
  • Example: A patent about "Solar HEATING systems" and another about "Heating elements" both belong to Section F (Mechanics; Lighting; Heating). Without normalisation, the classifier sees different features; with it, both contribute to the same "heating" feature.

Stemming (reducing words to their root form):

  • Why: Reduces vocabulary size and groups related word forms. "heating", "heated", "heater", "heats" all stem to "heat", combining their evidence.
  • Example: Patents using "transport", "transportation", "transporting" all relate to Section B (Performing operations; Transporting). Stemming to "transport" unifies these features, improving classification.

Part (c)[4 marks]
A 97%-accuracy HMM POS tagger makes errors on patent text. Why? How to fix?

Solution:

Why it fails:

  • The tagger was trained on general-domain text (e.g., news articles, Wikipedia). Patent language uses highly specialised vocabulary and unusual sentence structures that the tagger never encountered.
  • Unknown words: Domain-specific terms like "photolithographic", "nucleotide", "thermoelectric" may not appear in the training vocabulary. The HMM's emission probabilities P(word|tag) are zero or estimated poorly for these words.

  • Domain adaptation: Manually annotate a set of patent texts with POS tags and either retrain or fine-tune the HMM on this domain-specific data.
  • Add patent vocabulary: Supplement the emission probability table with domain-specific words and their likely POS tags.

Part (e)[4 marks]
How to include POS tags in a BERT-based model (without ensemble)? How is it different from the BoW approach?

Solution:

Including POS in BERT:

  • Create a POS embedding layer (similar to BERT's existing token and position embeddings). Each POS tag (noun, verb, adjective, etc.) gets a learned embedding vector.
  • Add (or concatenate) the POS embedding to BERT's token embedding at the input layer: input = token_embedding + position_embedding + segment_embedding + POS_embedding
  • The POS information flows through all transformer layers and is integrated with BERT's contextual representations during fine-tuning.

How this differs from the BoW approach:

BoW + POSBERT + POS
POS tags are discrete, explicit features (e.g., a binary feature "word=light AND pos=NOUN") POS tags become learned continuous embeddings that interact with other representations through attention
POS is an independent feature dimension added to the feature vector POS is fused with contextual representations through transformer layers, allowing complex interactions between POS and word meaning
No word order; POS features are aggregated across the document POS features are position-aware and contribute to each token's contextual representation
