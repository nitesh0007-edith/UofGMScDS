Chapter 01

Spark Architecture vs. Hadoop

To understand why Spark dominates modern data engineering, we must first see how Hadoop MapReduce works and where its bottlenecks lie.

Fig 1.1 — Hadoop writes intermediate data to HDFS (disk) between every Map/Reduce stage
Fig 1.2 — Spark keeps intermediate data in RAM, eliminating costly disk cycles
Hadoop MapReduce

Traditional Approach

Job Tracker assigns Map and Reduce tasks to the cluster
Task Trackers execute individual Map and Reduce operations
HDFS — NameNode tracks locations; DataNodes hold blocks
All intermediate data written to disk after every stage
Apache Spark

Modern In-Memory Engine

Driver Program creates SparkContext and orchestrates the job
Cluster Manager (YARN, Mesos, K8s) allocates resources
Executors (JVMs) run tasks in thread pools — parallel execution
No mandatory persistent storage — data stays in memory (RAM)
AspectHadoopSpark
ProcessingDisk-based (read → process → write)In-memory (process in RAM)
Speed1× baselineUp to 100× faster
Task ModelOne task per JVM processMultiple tasks per Executor (thread pool)
Intermediate DataWritten to HDFS between stagesKept in RAM; disk spill only if needed
Fault ToleranceHDFS replicationRDD lineage (rebuilds lost partitions)
DriverJob Tracker stays runningDriver must stay alive for job duration
StorageBuilt-in HDFSPluggable (HDFS, S3, ADLS, etc.)

💡 Key Takeaway: Hadoop writes intermediate results to disk after every Map/Reduce step, creating I/O bottlenecks. Spark eliminates this by keeping data in memory, making iterative algorithms (like ML training) dramatically faster.

Chapter 02

Job Execution Anatomy

Spark's job lifecycle follows Lazy Evaluation — operations are not executed until absolutely necessary, allowing Spark to optimize the entire pipeline first.

Fig 2.1 — Spark builds a complete plan before executing, enabling global optimization
1

Initialization

The client creates a SparkContext — the entry point to Spark. It connects to the Cluster Manager and creates input RDDs (Resilient Distributed Datasets) from data sources like HDFS, S3, or local files.

2

DAG Construction (Lazy Phase)

You chain transformations like map(), filter(), groupByKey() — but nothing executes. Spark records each transformation in a DAG, seeing the full picture for optimization (e.g., fusing operations, pruning unnecessary steps).

3

Action Trigger

Computation begins only when an Action is called — collect(), count(), or saveAsTextFile(). This materializes the DAG into actual work.

4

Scheduling & Execution

The DAGScheduler breaks the graph into stages at shuffle boundaries. Each stage has tasks (one per partition). The TaskScheduler ships tasks to Executors, considering data locality and resource availability.

Fig 2.2 — A Spark Job splits into Stages at shuffle boundaries. Each Stage runs parallel Tasks.

🔍 Narrow vs. Wide: Narrow transformations (map, filter) keep data on the same partition — zero network cost. Wide transformations (groupByKey, reduceByKey) require a shuffle — data moves across the network. Shuffles are Spark's most expensive operation!

Code Example: Lazy Evaluation

# Nothing happens — Spark records the plan rdd = sc.textFile("hdfs://data/logs.txt") # RDD created filtered = rdd.filter(lambda line: "ERROR" in line) # Lazy mapped = filtered.map(lambda line: (line, 1)) # Lazy # THIS triggers execution result = mapped.count() # Action! → runs now print(result)

Until count() is called, Spark only builds a plan. It can fuse filter and map into a single pass — a massive optimization.

Chapter 03

Transformations & Actions

Every Spark operation is either a Transformation (lazy — builds the plan) or an Action (eager — executes the plan). This distinction is fundamental to efficient Spark code.

Fig 3.1 — Transformations build the plan; Actions materialize it into results

Common Transformations LAZY

map(func)
NARROW

Apply a function to every element, one output per input. e.g., [1,2,3] → [2,4,6]

filter(func)
NARROW

Keep only elements passing a condition. e.g., [1,2,3,4] → [3,4] (keep > 2)

flatMap(func)
NARROW

Like map, but each input can produce 0 or more outputs. e.g., splitting sentences into words.

groupByKey()
WIDE — SHUFFLE

Groups all values with the same key. Causes a full network shuffle.

reduceByKey(func)
WIDE — SHUFFLE

Aggregates values by key. Better than groupByKey — aggregates locally first, reducing shuffle volume.

Fig 3.2 — reduceByKey aggregates locally before shuffling, drastically reducing network I/O

Common Actions TRIGGER EXECUTION

collect()
ACTION

Returns all elements to the Driver. ⚠️ Huge datasets can crash the Driver with OutOfMemoryError.

count()
ACTION

Returns the number of elements. Safe and commonly used to check data volume.

saveAsTextFile(path)
ACTION

Writes RDD to external storage (HDFS, S3, etc.). Each partition becomes a separate file.

⚠️ collect() Warning: Never call collect() on a large dataset in production. Use take(n) to preview a small sample, or saveAsTextFile() to persist results to storage.

Chapter 04

Advantages & Disadvantages

Spark has become the de facto standard for large-scale data processing, but it's not a silver bullet. Understanding its strengths and limits helps you pick the right tool.

Speed: Spark vs. Hadoop MapReduce

Spark's in-memory processing delivers dramatic speed gains, especially for iterative algorithms like ML training that repeatedly scan the same data.

Hadoop
1×
Spark
Up to 100×

* Benchmarked on iterative ML workloads. Single-pass batch jobs typically see 10-30× improvement.

Advantages

Lightning Speed — Up to 100× faster than Hadoop; avoids costly disk I/O between stages
Unified Engine — Batch, SQL, streaming, ML (MLlib), and graph (GraphX) in one framework
Multi-Language — Written in Scala, with first-class APIs for Python (PySpark), Java, R, and SQL
Flexible Storage — Plugs into HDFS, S3, Azure ADLS, Delta Lake — no vendor lock-in
Rich Ecosystem — Powers Databricks; integrates with Kafka, Hive, and modern data platforms

Disadvantages

Not True Streaming — Spark Streaming uses micro-batches, not event-at-a-time. Apache Flink is better for sub-second latency
Memory Hungry — In-memory processing demands large RAM; insufficient memory causes disk spilling
Complexity — Tuning shuffle partitions, memory fractions, and executor sizing requires deep expertise
Driver Dependency — The Driver must remain alive; if it crashes, the entire job fails
Cost — RAM is more expensive than disk. Spark clusters can be costly for very large persistent workloads

✅ When to Choose Spark: Iterative ML training, interactive data exploration, complex ETL pipelines, SQL analytics over large datasets. For true real-time event-by-event processing, consider Apache Flink instead.

Batch vs. Streaming Processing

The final piece: understanding when to process data in large blocks (batch) versus handling it as it arrives (streaming). Each paradigm has distinct tradeoffs.

Stream Processing Challenges

Streaming introduces unique engineering challenges that batch processing doesn't face:

Variable Input Rates — Events may arrive in bursts or slow trickles. The engine must handle both without crashing or starving.
Buffering — When events arrive faster than they can be processed, the system must queue them without losing data.
State Management — Tracking running totals, counts, or windows of time requires maintaining state across events — and recovering it after failures.
Out-of-Order Events — Network delays mean events can arrive in the wrong order. The engine must decide when "enough" events have arrived to produce correct results.

🔍 Spark's Position: Spark Structured Streaming uses micro-batching — it collects small batches of events (e.g., every 100ms) and processes them as tiny batch jobs. This gives near-real-time latency with batch-like simplicity, but it cannot match the true event-at-a-time processing of Apache Flink.
