1. Introduction to Big Data & Distributed Computing

What is Big Data?

Big Data refers to datasets that are so large or complex that traditional data processing software cannot handle them efficiently. Think of processing:

  • Terabytes (TB): 1,000 Gigabytes - All books in a large library
  • Petabytes (PB): 1,000 Terabytes - All academic research papers ever written
  • Exabytes (EB): 1,000 Petabytes - All words ever spoken by humanity

Why Do We Need Distributed Computing?

Imagine you have 1 Terabyte (1,000 GB) of data to process:

Real-Time Calculation: Processing Time

Scenario: Processing 1 TB of web logs to count words

Single Machine: 1 TB at 100 MB/s = 10,000 seconds ≈ 2.8 hours
100 Machines: 1 TB distributed = 100 seconds ≈ 1.6 minutes

Speedup: 100x faster with 100 machines!

The Power of Parallel Processing

Vertical Scaling (Scale Up): Adding more power to a single machine (expensive, has limits)

Horizontal Scaling (Scale Out): Adding more machines (cheaper, unlimited scaling)

Challenge 1: Parallelism

Problem: Not All Tasks Can Run in Parallel

Some tasks must wait for others to complete. For example:

  • You can't aggregate data before collecting it
  • You can't sort data before reading it
  • Dependencies create bottlenecks

Real Example: Amdahl's Law

If 10% of your program must run sequentially, your maximum speedup is limited:

Max Speedup = 1 / (0.1 + 0.9/N)
Number of Processors (N) Maximum Speedup
10 5.3x
100 9.2x
1000 9.9x
∞ (Infinite) 10x (Maximum!)

Lesson: Even with infinite processors, 10% sequential code limits speedup to 10x!

Challenge 2: Load Balancing

Example: Uneven Data Distribution

Imagine processing social media data where:

  • Machine 1: Processes users A-M (80% of users)
  • Machine 2: Processes users N-Z (20% of users)

Result: Machine 2 finishes early and sits idle while Machine 1 is overloaded!

Solution: Split data evenly by quantity, not alphabetically

Challenge 3: Communication Overhead

Network Transfer Calculation

Scenario: Transferring intermediate results between stages

Transfer Time = Data Size / Network Speed

Example: 100 GB over 1 Gbps network

100 GB × 8 bits/byte ÷ 1,000,000,000 bits/sec = 800 seconds ≈ 13 minutes

Impact: Network transfer can take longer than computation!

Challenge 4: Fault Tolerance

Types of Failures in Distributed Systems

  • Worker Failure: A machine crashes mid-computation
  • Network Failure: Machines can't communicate
  • Disk Failure: Data becomes corrupted or lost
  • Slow Workers (Stragglers): One slow machine delays entire job

Real Calculation: Probability of Failure

If each machine has 99.9% uptime (0.1% failure rate):

Probability all 1000 machines work = 0.999^1000 = 36.8%

Shocking Reality: With 1000 machines, there's a 63% chance at least one fails!

Solution: Systems must handle failures gracefully

Challenge 5: Orchestration

Master Services Responsibilities

A coordinator (master) must handle:

  • Task Distribution: Assign work to workers
  • Progress Monitoring: Track which tasks are complete
  • Failure Detection: Identify failed workers
  • Task Reassignment: Restart failed tasks on healthy workers
  • Data Location: Tell workers where to find their data

3. MapReduce: The Solution

What is MapReduce?

MapReduce is a programming model developed by Google in 2004 that simplifies distributed computing by hiding complexity behind two simple functions: Map and Reduce.

The MapReduce Programming Model

MapReduce Data Flow

Input Data (Key1, Value1 pairs)
↓
Map Function: (Key1, Value1) → List[(Key2, Value2)]
↓
Sort & Shuffle: Group by Key2
↓
Reduce Function: (Key2, List[Value2]) → List[Value2]
↓
Output Results

Classic Example: Word Count

Step-by-Step Word Count Example

Input Document:

Hello World
Hello MapReduce
World of Big Data

Step 1: Map Phase

The Map function processes each line and emits (word, 1) pairs:

Map("Hello World"):
  emit("Hello", 1)
  emit("World", 1)

Map("Hello MapReduce"):
  emit("Hello", 1)
  emit("MapReduce", 1)

Map("World of Big Data"):
  emit("World", 1)
  emit("of", 1)
  emit("Big", 1)
  emit("Data", 1)

Step 2: Shuffle & Sort Phase

System automatically groups by key:

"Hello"     → [1, 1]
"World"     → [1, 1]
"MapReduce" → [1]
"of"        → [1]
"Big"       → [1]
"Data"      → [1]

Step 3: Reduce Phase

The Reduce function sums the counts:

Reduce("Hello", [1, 1]):
  emit("Hello", 2)

Reduce("World", [1, 1]):
  emit("World", 2)

Reduce("MapReduce", [1]):
  emit("MapReduce", 1)

...and so on

Final Output:

Hello: 2
World: 2
MapReduce: 1
of: 1
Big: 1
Data: 1

MapReduce Code Example (Pseudocode)

// Map Function
function map(String key, String value):
  // key: document name
  // value: document contents
  for each word in value:
    emit(word, 1)

// Reduce Function
function reduce(String key, Iterator values):
  // key: a word
  // values: list of counts
  int result = 0
  for each count in values:
    result += count
  emit(key, result)

MapReduce Execution Flow

Detailed Execution Architecture

1. Input Splitting: Data divided into M splits (typically 64MB each)
2. Master Assignment: Master assigns M map tasks and R reduce tasks to workers
3. Map Execution: Workers read input splits, apply map function, write to local disk
4. Shuffle: Reduce workers fetch intermediate data from mappers (network transfer)
5. Sort: Reduce workers sort intermediate data by key
6. Reduce Execution: Apply reduce function to each unique key
7. Output: R reduce workers produce R output files

MapReduce Optimizations

Performance Refinements

  • Combiner Function: Local reduce on mapper to reduce network traffic
  • Partitioning: Custom hash functions to balance reduce tasks
  • Data Locality: Schedule map tasks on machines that have the data
  • Backup Tasks: Run stragglers on multiple machines
  • Skip Bad Records: Auto-detect and skip corrupted data

Combiner Function Impact

Without Combiner (Word Count):

1 billion words → 1 billion (word, 1) pairs sent over network

With Combiner:

1 billion words → ~100,000 unique words with counts

Network Traffic Reduction: 10,000x less data!

Data Locality Optimization

Real Example: Processing 1 TB

Without Data Locality:

Transfer 1 TB over network at 1 Gbps = 2.2 hours

With Data Locality:

Read 1 TB from local disk at 100 MBps = 2.8 hours

Note: But with data locality, 100+ machines read in parallel!

100 machines reading in parallel = ~1.6 minutes

4. Hadoop: Open Source MapReduce

What is Hadoop?

Hadoop is an open-source implementation of Google's MapReduce and GFS (Google File System). Created by Doug Cutting in 2006, it's now the most popular big data framework.

Hadoop Architecture Overview

Hadoop Core Components

HDFS

Hadoop Distributed File System

Stores data across cluster

Handles data replication & fault tolerance

MapReduce

Processing Engine

Processes data in parallel

Handles job execution

YARN

Resource Manager

Manages cluster resources

Schedules jobs (Hadoop 2+)

HDFS Architecture

HDFS Components

NameNode (Master)
Stores metadata, manages namespace
↓ Coordinates ↓
DataNode 1
DataNode 2
DataNode 3
DataNode N
Store actual data blocks

HDFS Block Size & Replication

Default Configuration:

  • Block Size: 128 MB (Hadoop 2+) or 64 MB (Hadoop 1)
  • Replication Factor: 3 copies

Example Calculation: Storing 1 TB of data

Blocks = 1000 GB ÷ 0.128 GB = 7,813 blocks
Total Storage = 7,813 blocks × 3 replicas × 128 MB = 3 TB

Storage Overhead: 3x original data size

Benefit: 2 machines can fail without data loss!

Hadoop MapReduce Architecture

Hadoop 1.x Architecture

JobTracker (Master)
Schedules tasks, monitors progress
↓ Assigns Tasks ↓
TaskTracker 1
Map & Reduce Slots
TaskTracker 2
Map & Reduce Slots
TaskTracker 3
Map & Reduce Slots

Task Slots Explained

Each TaskTracker has a fixed number of "slots" for running tasks:

  • Map Slots: Can only run map tasks
  • Reduce Slots: Can only run reduce tasks

Problem: If all map tasks are done but reduce tasks haven't started, map slots sit idle!

YARN Architecture (Hadoop 2.x+)

YARN - Yet Another Resource Negotiator

ResourceManager
Global resource scheduler
↓ Allocates Resources ↓
NodeManager 1
Containers
NodeManager 2
Containers
NodeManager 3
Containers

YARN Improvements Over Hadoop 1.x

  • Generic Containers: No separate map/reduce slots - better utilization
  • Multiple Frameworks: Can run Spark, Flink, etc., not just MapReduce
  • Scalability: Supports 10,000+ nodes (vs 4,000 in Hadoop 1)
  • Per-Job ApplicationMaster: Better isolation and fault tolerance

Erasure Coding Savings (Hadoop 3.x)

Traditional Replication (3 copies):

1 PB data × 3 = 3 PB storage required

Erasure Coding (RS-6-3 scheme):

1 PB data × 1.5 = 1.5 PB storage required

Savings:

(3 PB - 1.5 PB) = 1.5 PB saved = 50% reduction!

Cost Impact: At $30/TB, saves $45,000 per PB!

Hadoop Fault Tolerance

How Hadoop Handles Failures

  • DataNode Failure: NameNode detects via heartbeat, re-replicates blocks from other DataNodes
  • TaskTracker Failure: JobTracker reassigns tasks to healthy nodes
  • NameNode Failure (Hadoop 1): Manual intervention required - single point of failure!
  • NameNode HA (Hadoop 2+): Automatic failover to standby NameNode
  • Speculative Execution: Slow tasks restarted on faster machines

6. Real-World Examples & Use Cases

Example 1: Web Index Creation (Like Google)

Building an Inverted Index for Search Engine

Scenario: Index 1 trillion web pages (100 PB of HTML)

Map Phase:

Input: (URL, HTML_content)
For each word in HTML_content:
  emit(word, URL)

Reduce Phase:

Input: (word, list_of_URLs)
Sort and deduplicate URLs
emit(word, sorted_URL_list)

Resource Calculation

Data: 100 PB = 100,000 TB
At 50 MB/s per machine: 100 PB ÷ 50 MB/s ÷ 10,000 machines ≈ 5.5 hours

Challenge: Shuffle phase might generate 200+ PB of intermediate data!

Solution: Use combiner to merge URLs locally before shuffle

Example 2: Log Analysis

Analyzing 1 TB of Web Server Logs

Goal: Find top 10 most accessed pages and average response time per page

Sample Log Entry:

192.168.1.1 - - [01/Jan/2024:00:00:01] "GET /index.html HTTP/1.1" 200 1234 0.045

Map Function:

function map(log_line):
  parse(log_line) → (IP, timestamp, URL, status, size, response_time)
  emit(URL, response_time)

Reduce Function:

function reduce(URL, list_of_response_times):
  count = length(list_of_response_times)
  avg_time = sum(list_of_response_times) / count
  emit(URL, (count, avg_time))

Processing Calculation

Assumptions:

  • Log size: 1 TB
  • Average log entry: 200 bytes
  • Cluster: 100 machines
Number of log entries: 1 TB ÷ 200 bytes = 5 billion entries
Each machine processes: 5 billion ÷ 100 = 50 million entries
Processing time at 10,000 entries/sec per machine: 5,000 seconds ≈ 83 minutes

With MapReduce on 100 machines: ~10-15 minutes

Example 3: Recommendation System

Collaborative Filtering with MapReduce

Goal: Find similar users based on movie ratings

Input Data: (UserID, MovieID, Rating)

Step 1 - Create User-Movie Matrix:

Map: (UserID, MovieID, Rating) → emit(UserID, (MovieID, Rating))
Reduce: Group all ratings by user

Step 2 - Calculate User Similarity:

Map: For each user pair, emit((User1, User2), similarity_score)
Reduce: Sum similarity scores across all movies

Computational Complexity

Scenario: 1 million users, 10,000 movies, 100 million ratings

User pairs to compare: C(1M, 2) = 499,999,500,000 pairs

Challenge: 500 billion comparisons!

Optimization: Only compare users who rated same movies

Average user rates 100 movies → ~10,000 pairs per user
Reduced comparisons: 1M users × 10K pairs = 10 billion (50x less!)

Example 4: Image Processing

Thumbnail Generation for 1 Billion Images

Task: Create thumbnails (100x100px) from original images

Map: (image_id, image_bytes) →
  thumbnail = resize(image_bytes, 100, 100)
  emit(image_id, thumbnail)

Reduce: Identity function (just write to output)

Processing Time Estimate

Assumptions:

  • 1 billion images, average 2 MB each
  • Resize time: 0.1 seconds per image

Single Machine:

1 billion images × 0.1 sec = 100 million seconds ≈ 3.2 years!

With 1,000 machines:

100 million sec ÷ 1,000 = 100,000 seconds ≈ 27.8 hours

With 10,000 machines: ~2.8 hours

Example 5: Machine Learning - Distributed Training

Training a Logistic Regression Model

Dataset: 100 GB of labeled training data

Map Phase (Gradient Calculation):

Input: (sample_id, features, label)
Calculate partial gradient for this sample
emit(1, partial_gradient)  // All to same reducer

Reduce Phase (Gradient Aggregation):

Input: (1, list_of_partial_gradients)
Sum all partial gradients
Update model weights
emit(iteration, updated_weights)

Iterative Process: Repeat for multiple epochs

When NOT to Use MapReduce

MapReduce Limitations

  • Small Data: Overhead not worth it for < 1 GB
  • Real-time Processing: Batch-oriented, minutes to hours latency
  • Iterative Algorithms: ML algorithms need many passes (use Spark instead)
  • Graph Processing: Better frameworks exist (GraphX, Pregel)
  • Random Access: Not a database, use HBase/Cassandra for lookups

7. Summary & Key Takeaways

Core Concepts Recap

  • Big Data Problem: Traditional systems can't handle TB-PB scale data efficiently
  • Distributed Computing: Horizontal scaling (many machines) beats vertical scaling
  • Challenges: Load balancing, fault tolerance, communication overhead, orchestration
  • MapReduce Solution: Simple programming model hides complexity
  • Two Functions: Map transforms data, Reduce aggregates it
  • Hadoop: Open-source implementation with HDFS + MapReduce + YARN
  • Performance: Linear scaling up to thousands of nodes
  • Real Impact: 80-90% cost reduction vs traditional warehouses

The MapReduce Philosophy

"Move computation to data, not data to computation. Let the framework handle failures, load balancing, and distribution. Developers just write Map and Reduce."
