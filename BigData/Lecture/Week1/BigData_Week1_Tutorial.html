📊 What is Big Data?

Definition: "Data whose capturing, storage, curation, management, and processing are NOT POSSIBLE with traditional tools within a tolerable amount of time"

Understanding the Definition

🎯 Big Data is Defined by Its USE

  • Capturing, Storage, Curation, Management: How we store and access the data
  • Processing: How we can use it to gain value

⚙️ Big Data is Defined by Software and Hardware

  • "Traditional tools": The software we have available
  • "Tolerable amount of time": What hardware we have to run that software

📈 The Big Data Explosion

Volume of Data Created, Captured, Copied, and Consumed Worldwide

The volume of data generated is projected to exceed 180 zettabytes by 2025

Why This Explosion?

✅ We are producing and storing more data than we ever have

✅ Broad acceptance that it's better to store data, even if you don't know what you'll use it for

✅ Massive increase in the number and usage time of internet-enabled devices

✅ Result: A LOT of Data Science jobs now available! 💼

🎯 The Five V's of Big Data

📦 Volume

The sheer amount of data generated every second. From gigabytes to petabytes and beyond.

🎨 Variety

Structured, semi-structured, and unstructured data: text, video, audio, photos, JSON, XML

⚡ Velocity

Near-real-time data processing. Speed at which data is generated and processed.

✔️ Veracity

Data quality and trustworthiness. Is it really the "true" data? Errors? Alterations?

📊 Variability

Data flows, peaks & valleys. Inconsistency in data patterns and flow rates.

🔗 Complexity

Entities, data hierarchies, links, and complex relationships between different data sources.

💡 Pro Tip: Pick any TWO of these dimensions and you are probably working in a Big Data space!

📊 Big Data is Relative

How "Big Data" Has Changed Over Time

📝 Note: What people agree on as 'Big Data' changes over time. This shift happens because our hardware and software have become much more powerful. However, we are approaching the limits of our current technology.

🖥️ Hardware Primer

One of the key factors that turn normal computing problems into Big Data problems is insufficient hardware. Or to put it another way: Can I solve this problem on a laptop?

Four Main Hardware Aspects

🔧

General Purpose Compute

How much normal work the machine can do in a fixed period of time (CPU performance)

⚡

Specialist Compute

How much hardware-accelerated work the machine can do (GPUs, Quantum cores)

💾

Storage Stack

How much of each type of storage the machine has (RAM, SSD, HDD)

🌐

Network

How quickly it can send or receive data (bandwidth and latency)

🔧 General Purpose Compute (CPU)

What it does: How quickly the CPU can complete a given task

How it improves: Engineers fit more transistors inside and optimize the internal architecture for common operations

⚠️ Important: Use representative benchmarks to evaluate compute capacity, NOT cores, clock speeds, or power consumption alone!

⚡ Specialist Compute

CPUs are designed for a wide range of tasks, so they're often not optimized enough for specific workloads. That's where specialist hardware comes in:

  • GPUs (Graphics Processing Units): Excellent for Deep Learning and parallel processing
  • Quantum Cores: For quantum computing applications
  • TPUs (Tensor Processing Units): Google's custom chips for machine learning

💾 The Storage Stack

Inside a typical computer, we have a wide range of tiered storage types:

Registers (CPU)
Cache L1-L3
Main Memory (RAM)
Solid State Drive (SSD)
Magnetic Hard Drive (HDD)
Optical Disks / Tape Storage

⏱️ Storage Access Times (Latency Comparison)

Storage Type Latency If CPU = 1 second Characteristics
CPU Cycle 0.3 ns 1 second ⚡ Fastest, extremely small
L1 Cache 0.9 ns 3 seconds 🚀 Very fast, CPU-direct
L2 Cache 2.8 ns 9 seconds ⚡ Fast, CPU-direct
L3 Cache 12.9 ns 43 seconds ✅ Fast, shared cache
Main Memory (RAM) 120 ns 6 minutes 📦 Fast, volatile
SSD 50-150 μs 2-6 days 💾 Medium, persistent
Hard Disk 1-10 ms 1-12 months 🗄️ Slow, cheap, large
Network (LAN) 1 ms 1 year 🌐 Variable latency

🔍 Real-World Example: Why This Matters

Imagine you need to process 1 TB of data:

  • From RAM: ~8 seconds
  • From SSD: ~3-10 minutes
  • From HDD: ~2-3 hours
  • Over Network: ~2-20 hours (depending on bandwidth)

💡 Key Insight: The storage hierarchy dramatically affects Big Data processing performance!

🌐 Network Considerations

Much of Big Data processing involves distributing work across multiple machines, which means network communication is critical.

  • LAN Transfer Latency: ~1 ms
  • Network transfers are nearly always slower than local storage (with exceptions in specialized server farms)
  • Bandwidth matters: 1 Gbps vs 10 Gbps vs 100 Gbps connections

💰 Cost Trends

In general, the cost of hardware has been decreasing over time. However:

  • 📉 The pace of cost reduction has slowed dramatically for compute and storage
  • ⚠️ We are approaching physical limits of current technology
  • 💡 This is why efficient algorithms and distributed systems are crucial

📈 Horizontal vs. Vertical Scaling

The solution to Big Data problems is always the same: leverage more or faster hardware. We call this "scaling-up" our storage or processing.

🔄 Two Approaches to Scaling

📊 Comparison Table

Aspect ⬆️ Vertical Scaling ↔️ Horizontal Scaling
Strategy Buy bigger computers (mainframes) Add more low-cost machines
Cost 💰💰💰 Very expensive 💰 Cost effective
Limits ⚠️ Physical limits (reticle limit, transistor size) ✅ Theoretically unlimited
Complexity Simple - no code changes needed Complex - requires specialized tools and programming
Failure Impact Single point of failure Fault tolerant (redundancy)
When Used 1990s - early 2000s 2000s - present (modern Big Data)
💡 Key Insight: This Big Data course is really about Horizontal Scaling of Compute and Storage, along with the tools that enable both!

🚧 Physical Limits of Vertical Scaling

1. Reticle Limit (Die Size)

We can only make CPUs and GPUs up to a set size defined by the reticle limit. We are already at that limit in some cases (around 800-1000 mm² for server chips).

2. Transistor Size

We can make transistors smaller, but:

  • Cost increases dramatically (5nm costs 5x more than 45nm per chip)
  • Physical limits: approaching atomic scale
  • Heat dissipation becomes a major challenge
  • Quantum effects start to matter

🔬 Real Example: Moore's Law Slowing Down

Moore's Law predicted transistor count doubling every 18-24 months. However:

  • 1970s-2000s: Law held true consistently
  • 2010s: Slowing down significantly
  • 2020s: More focus on specialized hardware (GPUs, TPUs) and distributed systems

Result: Horizontal scaling has become the primary solution for Big Data!

⚡ Compute Parallelization

Parallel Computing: Solving a (Big Data) problem by breaking down that problem into multiple tasks that can be solved simultaneously. These tasks can then be solved using horizontally scalable resources.

🎯 Where Can Parallelism Happen?

1️⃣ Within a CPU Core

Pipelining: Split the processor datapath into multiple successive stages (fetching → decoding → execution)

Superscalar: Multiple copies of certain components (e.g., multiple ALUs)

⚠️ Invisible to the programmer

2️⃣ Across CPU Cores (Same Machine)

Each CPU core can process different instructions

Exposed to programmers as "threads"

✅ Common in modern programming

3️⃣ Across Different Machines

Tasks compiled and sent to machines with data

This is horizontal scaling

🎯 Primary focus of Big Data systems

📚 Flynn's Taxonomy

A hardware classification system for CPUs based on instruction and data streams:

SISD

Single Instruction, Single Data

Classic von Neumann architecture - serial computer

📱 Example: Old single-core CPUs

SIMD

Single Instruction, Multiple Data

Data divided among processors, same instruction sequence

🎮 Examples: GPUs, AVX (Advanced Vector Extensions)

MISD

Multiple Instruction, Single Data

Very rare architecture

🔬 Example: Systolic arrays (special purpose)

MIMD ⭐

Multiple Instruction, Multiple Data

Autonomous processors executing independent programs

💻 Examples: Modern multi-core CPUs, distributed systems

🎯 For Big Data: We primarily work with MIMD architecture - collections of autonomous processors that can execute multiple independent programs, each with its own data stream!

📊 Measuring Parallelism Effectiveness

Speed-up(n, p) = Serial Time(n) / Parallel Time(n, p)


Where:

n = input size

p = number of parallel processors

Understanding Speed-up:

  • Range: 0 < Speed-up ≤ p
  • Linear Speed-up: Speed-up = p (ideal case - 100% efficiency)
  • Sub-linear Speed-up: Speed-up < p (common in practice)
  • Super-linear Speed-up: Speed-up > p (rare, usually due to cache effects)

🎯 Real-World Example: Data Processing Job

Scenario: Processing 1 TB of log files

  • Single Machine: 10 hours
  • 10 Machines: 1.2 hours
  • Speed-up: 10 / 1.2 = 8.33x
  • Efficiency: 8.33 / 10 = 83.3%

Why not 10x? Overhead from:

  • Network communication
  • Data shuffling between machines
  • Coordination and synchronization
  • Non-parallelizable parts (Amdahl's Law)

⚠️ Amdahl's Law

The theoretical speed-up is limited by the serial portion of the program:

Speed-up ≤ 1 / (S + (P / N))

Where:

  • S = Serial portion (cannot be parallelized)
  • P = Parallel portion (can be parallelized)
  • N = Number of processors

💡 Key Insight: Even if only 5% of your program is serial, your maximum speed-up is limited to 20x, regardless of how many processors you add!

🔢 Amdahl's Law Example

Program with 10% serial code (S=0.1, P=0.9):

  • 2 processors: Max speed-up = 1 / (0.1 + 0.9/2) = 1.82x
  • 4 processors: Max speed-up = 1 / (0.1 + 0.9/4) = 3.08x
  • 8 processors: Max speed-up = 1 / (0.1 + 0.9/8) = 4.71x
  • ∞ processors: Max speed-up = 1 / 0.1 = 10x

Conclusion: The 10% serial portion limits us to at most 10x speed-up!

🎮 Real-World Parallelization Example

Star Citizen: Server Meshing

Problem: The video game universe is too large to simulate on one server

Solution: Break the game world into regions and assign one server to manage each region

  • Players seamlessly move between servers as they traverse the world
  • Each server handles physics, AI, and player interactions for its region
  • Servers communicate to share boundary information
  • Scales horizontally - add more servers for more regions
