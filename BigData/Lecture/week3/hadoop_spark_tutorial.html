🔧 Hadoop Job Execution Anatomy

Let's understand how a MapReduce job executes in Hadoop. We'll follow a job from submission to completion through all 23 steps!

• Client: Submits the MapReduce job

📋 Complete Job Execution: 23 Steps

1

Run Job

The client submits a MapReduce program to the cluster. This includes:

  • JAR file containing map and reduce code
  • Job configuration (input/output paths, mapper/reducer classes)
  • Job parameters
// Example: Word Count job submission Job job = Job.getInstance(conf, "word count"); job.setJarByClass(WordCount.class); job.setMapperClass(TokenizerMapper.class); job.setReducerClass(IntSumReducer.class); job.waitForCompletion(true);
2

Get Job ID

The client contacts the Job Tracker to request a unique Job ID. This ID is used to track the job throughout its lifecycle.

Example Job ID: job_202402_0001
Format: job_[timestamp]_[sequence_number]
3

Copy Job Resources

The client copies necessary resources to a shared location (HDFS) so all worker nodes can access them:

  • Job JAR: Contains compiled code
  • Job Configuration XML: Job settings and parameters
  • Input Split Information: How to divide the input data
⚠️ Important: All worker nodes must be able to access these resources. That's why they're stored in HDFS, not local filesystem!
4

Submit Job

After resources are copied, the client formally submits the job to the Job Tracker. The Job Tracker adds the job to its internal queue.

Client
→
Job Submission
→
Job Tracker Queue
5

Initialize Job

The Job Tracker initializes the job by:

  • Creating internal data structures for tracking
  • Setting up job progress monitoring
  • Preparing for task creation
6

Retrieve Input Splits

The Job Tracker retrieves input split information from HDFS. Each split represents a portion of data that will be processed by a single map task.

🔍 Understanding Input Splits

Input splits are logical divisions of input data. By default:

  • Each split corresponds to one HDFS block (default 128 MB)
  • One map task is created per split
  • Splits are data-agnostic (they don't understand file format)

Example: File Split into 3 Blocks

Split 1: Bytes 0-128 MB → Map Task 1
Split 2: Bytes 128-256 MB → Map Task 2
Split 3: Bytes 256-320 MB → Map Task 3
✅ Best Practice: Splits should be at most one HDFS block size. This ensures data locality - the map task can run on the same machine where the data block is stored!
7

Register Task and Get Heartbeat Confirmation

Task Trackers send heartbeat messages to the Job Tracker every few seconds. These heartbeats serve two purposes:

  • Health Check: Confirms the Task Tracker is alive and functioning
  • Capacity Report: Indicates how many free slots are available for new tasks
🔄 Heartbeat Frequency: Typically every 3 seconds. If Job Tracker doesn't receive a heartbeat for 10 minutes, the Task Tracker is considered failed.

Task Assignment Strategy

The Job Tracker uses a scheduler to decide which tasks run where:

Scheduler Type Description Best For
FIFO First In First Out - jobs executed in submission order Simple, single-user environments
Fair Scheduler Gives equal share of resources to all jobs Multi-user environments with equal priority
Capacity Scheduler Allocates guaranteed capacity to different queues Organizations with different departments/teams

📍 Data Locality Optimization

The scheduler tries to run tasks where the data is located:

  1. Data-Local: Task runs on the same node as the data (BEST)
  2. Rack-Local: Task runs on a different node but same rack
  3. Off-Rack: Task runs on a different rack (requires network transfer)
8

Get Job Resources

Once a Task Tracker is assigned a task, it retrieves the job resources from HDFS:

  • Downloads the Job JAR file
  • Retrieves job configuration
  • Prepares local working directory
Network Traffic: This step involves network transfer if the JAR isn't cached locally. The distributed cache (covered later) helps optimize this.
9

Launch Task

The Task Tracker launches a new JVM (Java Virtual Machine) to execute the task in isolation.

🤔 Why a separate JVM?
• Isolation: Prevents one task from crashing others
• Security: Limits what tasks can access
• Resource Control: Can set memory limits per task
• Clean State: Each task starts fresh
10

Run Map Task

The map task begins execution! Here's what happens:

  1. InputFormat reads the input split
  2. Data is converted to key-value pairs
  3. Each pair is passed to the map() function
  4. Map function emits intermediate key-value pairs
// Example: Word Count Mapper public class TokenizerMapper extends Mapper<Object, Text, Text, IntWritable> { private final static IntWritable one = new IntWritable(1); private Text word = new Text(); public void map(Object key, Text value, Context context) { StringTokenizer itr = new StringTokenizer(value.toString()); while (itr.hasMoreTokens()) { word.set(itr.nextToken()); context.write(word, one); // Emit (word, 1) } } }
Example Output:
Input: "hello world hello"
Map Output: (hello, 1), (world, 1), (hello, 1)
11

Read Data and Execute Task

During map execution, the task reads data from the HDFS block. This is optimized for:

  • Streaming reads: Data is read sequentially
  • Local reads: Preferably from local disk (data locality)
  • Block-level operations: Efficient for large files

🔌 InputFormat Classes

InputFormat Usage
TextInputFormat Reads text files line by line (default)
KeyValueTextInputFormat Reads key-value pairs separated by delimiter
SequenceFileInputFormat Reads binary sequence files (efficient!)
NLineInputFormat Creates split every N lines
DBInputFormat Reads from SQL databases via JDBC
12

Intermediate Sort

After the map function emits key-value pairs, they are sorted by key in memory. This is crucial for the reduce phase!

🔄 Sort Process

  1. Buffer in Memory: Map outputs are first stored in a circular memory buffer (default 100 MB)
  2. Sort on Buffer Full: When buffer is 80% full, data is sorted by key
  3. Partition: Sorted data is divided into partitions (one per reducer)
  4. Spill to Disk: Each partition is written to local disk
  5. Merge Spills: Multiple spill files are merged into one sorted file

Memory Buffer Visualization

Buffer 80% Full → Trigger Sort & Spill
Available Space
💾 Why sort now? Sorting at this stage means the reduce phase receives sorted data, making it much more efficient. It's easier to sort smaller chunks of data than one massive dataset!
13

Write Map Output

The sorted and partitioned map output is written to the local disk of the Task Tracker node (NOT HDFS!).

⚠️ Important: Map output is written to LOCAL disk, not HDFS. Why?
• Speed: Local writes are faster
• Temporary: This data is deleted after the job completes
• Network: Reduces unnecessary HDFS network traffic

📤 OutputFormat Classes

OutputFormat Description
TextOutputFormat Writes plain text files (default)
SequenceFileOutputFormat Writes binary sequence files
MapFileOutputFormat Writes sorted MapFiles with index
NullOutputFormat Discards all output (for side effects only)
MultipleOutputs Writes to multiple files based on keys
14

Report Map Done

When the map task completes successfully, the Task Tracker notifies the Job Tracker through the next heartbeat:

  • Task completion status
  • Location of map output files
  • Task counters (records processed, bytes read, etc.)
✅ Job Tracker Updates:
• Marks the map task as complete
• Records output location for reducers
• Updates job progress (e.g., "50% map complete")
15

Retrieve Map Partitions

The Job Tracker informs reduce tasks where to find their input data. Each reducer needs to fetch its partition from EVERY completed mapper.

Shuffle Phase Begins

Map 1
Partition A, B, C
Map 2
Partition A, B, C
Map 3

⬇️

(gets partition A from all maps)

(gets partition C from all maps)
🔀 Shuffle: The process of transferring map outputs to reducers. This is often the most expensive part of a MapReduce job due to network transfer!
16

Register Reduce Task and Get Heartbeat Confirmation

Just like map tasks, reduce tasks are assigned to Task Trackers based on heartbeat communication and available resources.

17

Get Job Resources (for Reduce)

The Task Tracker retrieves job resources again (if not already cached locally) to execute the reduce task.

18

Launch Reduce Task

Similar to map tasks, a new JVM is launched for the reduce task to ensure isolation and clean execution environment.

19

Run Reduce Task

The reduce task executes! But first, it needs to gather all its input data...

20

Read Data and Execute Task

The reduce task performs these steps:

🔄 Copy Phase

  1. Fetch Map Outputs: Reducer copies its partition from each mapper over HTTP
  2. Store in Memory: Small amounts are kept in memory buffer
  3. Spill to Disk: Large amounts are written to disk

🔀 Merge Phase

  1. In-Memory Merge: Memory buffers are merged
  2. On-Disk Merge: Disk files are merged using recursive merging
  3. Final Merge: Creates a single sorted input stream

Reducer Input Merging

Memory: [a:1, a:1, b:2]
Disk File 1: [a:1, c:3]
Disk File 2: [b:1, c:2]
⬇️ Merge
Sorted Stream: [a:1, a:1, a:1, b:1, b:2, c:2, c:3]

⚙️ Reduce Phase

Finally, the reduce function is called for each unique key with all its values:

// Example: Word Count Reducer public class IntSumReducer extends Reducer<Text, IntWritable, Text, IntWritable> { private IntWritable result = new IntWritable(); public void reduce(Text key, Iterable<IntWritable> values, Context context) { int sum = 0; for (IntWritable val : values) { sum += val.get(); } result.set(sum); context.write(key, result); // Emit (word, total_count) } }
Example:
Input to Reducer: (hello, [1, 1, 1]), (world, [1])
Reduce Output: (hello, 3), (world, 1)
21

Write Reduce Output

Unlike map output, reduce output is written directly to HDFS - this is the final output of the job!

💾 HDFS Output:
• Output is replicated (default 3 copies)
• Each reducer writes to a separate file (part-r-00000, part-r-00001, etc.)
• Output is persistent and fault-tolerant
22

Report Reduce Done

The Task Tracker notifies the Job Tracker that the reduce task completed successfully, along with:

  • Final task status
  • Reduce counters
  • Output file locations
23

Report Job Complete

When all reduce tasks complete, the Job Tracker marks the job as SUCCESSFUL and notifies the client.

🎉 Job Complete!
• Client receives success notification
• Job statistics and counters are returned
• Temporary map output files are cleaned up
• Job is removed from Job Tracker's active jobs list
📊 Job Statistics Example:
• Total time: 245 seconds
• Map tasks: 150
• Reduce tasks: 10
• Input records: 10,000,000
• Output records: 50,000
• Data processed: 1.2 TB

🎯 Hadoop Advanced Concepts

Now that we understand the basic execution flow, let's explore advanced features that make Hadoop production-ready.

💾 Distributed Cache

The distributed cache is a mechanism to efficiently distribute read-only files to all nodes in the cluster.

🤔 Problem it solves:
Imagine you have a 100 MB lookup table needed by every map task. Without distributed cache, you'd package it with every task, sending the same file hundreds of times over the network!

How Distributed Cache Works:

  1. File is uploaded to HDFS once
  2. Job configuration references the cached file
  3. Each Task Tracker downloads the file once and caches it locally
  4. All tasks on that node share the same cached copy
// Using Distributed Cache Job job = new Job(conf); job.addCacheFile(new URI("/path/to/lookup/file.txt#alias")); // In mapper, access the cached file @Override protected void setup(Context context) { Path[] cacheFiles = context.getLocalCacheFiles(); // Read the cached file BufferedReader reader = new BufferedReader( new FileReader(cacheFiles[0].toString()) ); }
✅ Use Cases:
• Lookup tables
• Machine learning models
• Configuration files
• Small reference datasets
• JAR libraries

📊 Counters

Counters are a way to track statistics and metrics during job execution.

Built-in Counter Groups:

Counter Group Examples
MapReduce Task Counters MAP_INPUT_RECORDS, MAP_OUTPUT_RECORDS, REDUCE_INPUT_RECORDS
FileSystem Counters BYTES_READ, BYTES_WRITTEN, HDFS_BYTES_READ
Job Counters TOTAL_LAUNCHED_MAPS, DATA_LOCAL_MAPS, RACK_LOCAL_MAPS

Custom Counters:

// Define custom counter enum enum MyCounters { INVALID_RECORDS, VALID_RECORDS } // In mapper/reducer public void map(LongWritable key, Text value, Context context) { try { // Process record context.getCounter(MyCounters.VALID_RECORDS).increment(1); } catch (Exception e) { context.getCounter(MyCounters.INVALID_RECORDS).increment(1); } }
💡 Counter Benefits:
• Track data quality issues
• Monitor job progress
• Debug and optimize jobs
• No performance overhead (aggregated efficiently)

🗜️ Compression

Compression is crucial for Hadoop performance! It reduces:

  • Disk space required
  • Network bandwidth for shuffling
  • I/O operations

Compression Codec Comparison:

Codec Splittable? Compression Ratio Speed Best For
Gzip ❌ No High Medium Final output, cold storage
Bzip2 ✅ Yes Very High Slow High compression needed, archival
LZO ✅ Yes (with index) Medium Very Fast Intermediate data, hot data
Snappy ❌ No Low-Medium Very Fast Intermediate data, map output
LZ4 ❌ No Low-Medium Extremely Fast Real-time processing
⚠️ Splittable vs Non-Splittable:
• Splittable: Can create multiple map tasks for one compressed file (efficient parallelism)
• Non-Splittable: One compressed file = one map task (limits parallelism)
// Enable compression in Hadoop job conf.setBoolean("mapreduce.map.output.compress", true); conf.setClass("mapreduce.map.output.compress.codec", SnappyCodec.class, CompressionCodec.class); // For final output conf.setBoolean("mapreduce.output.fileoutputformat.compress", true); conf.setClass("mapreduce.output.fileoutputformat.compress.codec", GzipCodec.class, CompressionCodec.class);
✅ Recommended Strategy:
• Map Output: Use Snappy (fast, good for temporary data)
• Final Output: Use Gzip or Bzip2 (better compression for storage)
• Cold Storage: Use Bzip2 (highest compression)

🚨 Failure Handling

Hadoop is designed to handle failures gracefully. Let's understand how!

1️⃣ Task Failures

Detection Methods:

  • Runtime Exception: User code throws an exception
  • JVM Crash: Task Tracker detects child JVM died
  • Timeout: Task doesn't report progress for too long (default 10 minutes)

Recovery Process:

  1. Task Tracker detects failure and reports to Job Tracker
  2. Job Tracker decrements attempt count (default: 4 attempts allowed)
  3. Task is rescheduled on a different node (if possible)
  4. If all attempts fail, the entire job fails
🔄 Speculative Execution: If a task is running slowly (stragglers), Hadoop may launch a duplicate task on another node. Whichever finishes first wins!

2️⃣ Task Tracker Failures

Detection:

  • Job Tracker doesn't receive heartbeat for 10 minutes (configurable)
  • Too many task failures on the same node

Recovery Process:

  1. Job Tracker marks Task Tracker as failed
  2. All tasks (even successful ones) on that node are rescheduled
  3. Node is blacklisted temporarily
  4. After some time, node can be reconsidered (in case it was just temporary)
⚠️ Why reschedule successful map tasks? Because their output is stored on local disk of the failed node. If the node is down, reducers can't fetch the data!

3️⃣ Job Tracker Failure

Single Point of Failure: In Hadoop 1.x, if Job Tracker fails, all running jobs fail!

Solution in Hadoop 2.x (YARN):
• Job Tracker functionality split into ResourceManager and ApplicationMaster
• ResourceManager can have high-availability setup
• Each job has its own ApplicationMaster (isolated failures)

4️⃣ Data Node Failures

HDFS Replication to the Rescue:

Since HDFS replicates blocks (default 3 copies), losing one Data Node doesn't cause data loss!

  1. Name Node detects Data Node failure (no heartbeat)
  2. Marks all blocks on that node as under-replicated
  3. Triggers re-replication from other nodes that have copies
  4. System continues operating normally
✅ Result: Hadoop automatically maintains the replication factor and ensures fault tolerance!

⚠️ Hadoop Limitations and Challenges

While Hadoop was revolutionary, it has several limitations that led to the development of newer frameworks like Spark.

📉 Declining Popularity

Google Trends: Hadoop vs Spark

Hadoop peaked around 2015, while Spark has surpassed it in popularity since 2017. Why?

1️⃣ HDFS and Small Files Problem

🐘 The Problem:
HDFS is optimized for large files (hundreds of MB to GB). Small files create serious issues!

Why Small Files Are Problematic:

  • Memory Overhead: Every file/block needs metadata in Name Node (150 bytes per block). Million small files = massive memory usage!
  • Processing Overhead: Each file becomes a separate map task (task startup overhead)
  • Network Overhead: More blocks = more network operations
📊 Example Impact:
• 1 billion files × 150 bytes = 150 GB of Name Node memory just for metadata!
• 1 GB data in 1 KB files = 1 million map tasks (instead of ~8 with large files)

Workarounds (not ideal):

  • Merge small files into larger files (SequenceFiles)
  • Use HAR (Hadoop Archive) files - adds complexity
  • Use HBase instead of files

2️⃣ Speed and Latency Issues

🐢 Hadoop is Slow! Why?

The Disk Bottleneck:

Hadoop writes to disk after every single operation:

Map Output: Written to local disk ⏱️

Shuffle: Read from disk, transfer over network ⏱️⏱️

Reduce Input: Written to disk, then read ⏱️⏱️⏱️

Reduce Output: Written to HDFS ⏱️⏱️⏱️⏱️

📊 Performance Impact:
• Disk writes: ~100 MB/s
• Memory access: ~10 GB/s (100x faster!)
• Network: ~1 GB/s (10x faster than disk)

Why Disk-Heavy Design?

Hadoop was designed in 2006 when:

  • RAM was expensive (100x more than today!)
  • Commodity servers had 4-8 GB RAM
  • Disk was the most reliable storage for intermediate data

3️⃣ Cost of Chaining MapReduce Jobs

⛓️ Real-World Problem: Most algorithms need multiple MapReduce iterations!

Examples Requiring Multiple Iterations:

  • Machine Learning: Gradient descent, K-means clustering (10-100 iterations)
  • Graph Processing: PageRank, shortest path (iterative)
  • Data Pipelines: ETL with multiple transformation stages

The Iteration Penalty:

Iteration 1:

Read from HDFS → Map → Shuffle → Reduce → Write to HDFS

Iteration 2:

Read from HDFS (again!) → Map → Shuffle → Reduce → Write to HDFS

Iteration 3:

Read from HDFS (again!) → Map → Shuffle → Reduce → Write to HDFS

Each iteration pays full I/O cost! 💸

// Example: Logistic Regression requires iterating over same data for (int i = 0; i < 100; i++) { // 100 iterations Job job = new Job(conf); // Each iteration: // - Starts new JVMs // - Reads data from HDFS // - Writes results to HDFS // - Shuts down JVMs job.waitForCompletion(true); } // Total time: 100× single iteration cost!

4️⃣ Lack of Streaming and State Sharing

🚫 MapReduce Doesn't Support:

Real-Time Stream Processing:

  • MapReduce is batch-only (processes complete datasets)
  • Can't handle continuous data streams (sensor data, logs, etc.)
  • Minimum latency is several seconds to minutes

State Sharing Between Tasks:

  • Tasks are completely isolated (by design for fault tolerance)
  • No shared memory or state
  • Only communication is through disk
Use Cases Hadoop Can't Handle Well:
• Real-time analytics
• Interactive data exploration
• Graph processing
• Machine learning with large models
• Complex multi-stage pipelines

5️⃣ Ease of Use Concerns

👨‍💻 Developer Pain Points:

Verbose Code:

// Hadoop WordCount: ~60 lines of Java code public class WordCount { public static class TokenizerMapper extends Mapper<...> { // 20+ lines for mapper } public static class IntSumReducer extends Reducer<...> { // 20+ lines for reducer } public static void main(String[] args) { // 20+ lines for job configuration } } // vs Spark WordCount: 3 lines! val counts = sc.textFile("data.txt") .flatMap(line => line.split(" ")) .map(word => (word, 1)) .reduceByKey(_ + _)

Other Usability Issues:

  • Everything must fit Map → Reduce pattern
  • No interactive mode (must submit jobs)
  • Poor support for iterative algorithms
  • Complex debugging (distributed logs)
  • Long development cycles (compile → deploy → test)
✨ Enter Apache Spark: Designed to solve all these problems!

"RAM is scarce and expensive"

  • Use disk as primary storage
  • Materialize everything
  • Simple fault tolerance (recompute from disk)

⚡ Spark Philosophy

"RAM is abundant and cheap"

  • Use memory as primary storage
  • Lazy evaluation
  • Lineage-based fault tolerance

• RAM costs dropped 20x from 2006
• Commodity servers with 64-128 GB became affordable
• SSDs became available (faster than HDD but still slower than RAM)

Speed

Ease of Use

Combine SQL, streaming, ML, and graph processing

🔍 Apache Spark Deep Dive

Let's understand Spark's architecture and programming model in detail.

🎯 Resilient Distributed Datasets (RDDs)

RDD is the fundamental data structure in Spark. Think of it as an immutable distributed collection of objects.

📚 RDD Definition:
A Resilient Distributed Dataset is a read-only, partitioned collection of records that can be:
  • Resilient: Can be reconstructed if lost (lineage tracking)
  • Distributed: Partitioned across cluster nodes
  • Dataset: Collection of data objects

RDD Properties:

  • Immutable: Once created, cannot be changed (transformations create new RDDs)
  • Partitioned: Split into chunks for parallel processing
  • Lazy Evaluated: Transformations aren't computed until an action is called
  • Cacheable: Can be stored in memory for reuse
  • Fault Tolerant: Can be rebuilt from lineage information

Creating RDDs:

// 1. From a file val textFile = sc.textFile("hdfs://path/to/file.txt") // 2. From a collection val numbers = sc.parallelize(List(1, 2, 3, 4, 5)) // 3. By transforming an existing RDD val words = textFile.flatMap(line => line.split(" ")) // 4. By reading from external storage val jsonData = sc.read.json("s3://bucket/data.json")

⚙️ Transformations vs Actions

Spark operations are divided into two types:

🔄 Transformations

Lazy operations that define a new RDD

  • map()
  • flatMap()
  • filter()
  • reduceByKey()
  • groupByKey()
  • join()
Nothing is computed yet! Just builds execution plan.

▶️ Actions

Eager operations that trigger computation

  • count()
  • collect()
  • reduce()
  • take(n)
  • saveAsTextFile()
  • foreach()
Triggers execution! Returns results to driver.

Example: Understanding Lazy Evaluation

// These are transformations - nothing happens yet! val lines = sc.textFile("data.txt") // Transformation val words = lines.flatMap(_.split(" ")) // Transformation val pairs = words.map(w => (w, 1)) // Transformation val counts = pairs.reduceByKey(_ + _) // Transformation // This is an action - NOW everything executes! counts.saveAsTextFile("output") // Action // Execution happens here ↑ // Spark optimizes the entire pipeline before executing
💡 Why Lazy Evaluation?
• Spark can optimize the entire pipeline (not just individual operations)
• Avoid computing intermediate results that aren't needed
• Better pipelining (combine multiple operations)
• Minimize data shuffling

📊 Common Spark Operations

1. map() - Transform Each Element

// Input: [1, 2, 3, 4] val numbers = sc.parallelize(List(1, 2, 3, 4)) // Double each number val doubled = numbers.map(x => x * 2) // Result: [2, 4, 6, 8]

2. flatMap() - Transform and Flatten

// Input: ["hello world", "spark is fast"] val lines = sc.parallelize( List("hello world", "spark is fast") ) // Split each line into words val words = lines.flatMap(line => line.split(" ")) // Result: ["hello", "world", "spark", "is", "fast"]

3. filter() - Keep Matching Elements

// Input: [1, 2, 3, 4, 5, 6] val numbers = sc.parallelize(List(1, 2, 3, 4, 5, 6)) // Keep only even numbers val evens = numbers.filter(x => x % 2 == 0) // Result: [2, 4, 6]

4. reduceByKey() - Aggregate by Key

// Input: [("a", 1), ("b", 1), ("a", 1), ("c", 1), ("a", 1)] val pairs = sc.parallelize( List(("a", 1), ("b", 1), ("a", 1), ("c", 1), ("a", 1)) ) // Sum values by key val counts = pairs.reduceByKey((x, y) => x + y) // Result: [("a", 3), ("b", 1), ("c", 1)]

💾 Caching and Persistence

This is where Spark shines! You can cache RDDs in memory for reuse.

// Load data and cache it val data = sc.textFile("large_dataset.txt").cache() // First action - reads from disk val count1 = data.count() // Takes ~30 seconds // Second action - reads from memory! val count2 = data.count() // Takes ~0.5 seconds // Third action - still in memory val sample = data.take(10) // Takes ~0.3 seconds

Storage Levels:

Storage Level Description Use When
MEMORY_ONLY Store in memory as deserialized objects Default, fastest access
MEMORY_AND_DISK Spill to disk if doesn't fit in memory Large datasets, avoid recomputation
MEMORY_ONLY_SER Store as serialized (compressed) objects Memory-constrained, can afford CPU overhead
DISK_ONLY Store only on disk Rarely used, RDD is expensive to recompute
import org.apache.spark.storage.StorageLevel // Different ways to persist data.cache() // Same as MEMORY_ONLY data.persist(StorageLevel.MEMORY_AND_DISK) data.persist(StorageLevel.MEMORY_ONLY_SER) // Remove from cache when done data.unpersist()

🔗 Lineage-Based Fault Tolerance

Spark's secret sauce for fault tolerance without replication!

🤔 The Problem: What if a cached RDD partition is lost?
✅ The Solution: Recompute just that partition using lineage!

How It Works:

Step 1: Create RDDs with transformations

val lines = sc.textFile("data.txt") // RDD 1 val words = lines.flatMap(_.split(" ")) // RDD 2 (depends on RDD 1) val pairs = words.map(w => (w, 1)) // RDD 3 (depends on RDD 2) val counts = pairs.reduceByKey(_ + _) // RDD 4 (depends on RDD 3)

Step 2: Spark tracks the lineage (dependency graph)

RDD1 → RDD2 → RDD3 → RDD4

Step 3: If RDD3 partition 2 is lost...

❌ Partition 2 of RDD3 lost on node failure

Step 4: Recompute from lineage

✅ Read partition 2 of RDD1 → apply flatMap → apply map

Benefits of Lineage:
• No need to replicate in-memory data
• Only recompute lost partitions (not entire RDD)
• Can rebuild from any point in lineage
• Minimal overhead (just track dependencies)

🚀 Shared Variables

Spark provides two types of shared variables for efficient distributed programming:

1. Broadcast Variables

Efficiently distribute large read-only data to all nodes.

// Without broadcast - sent with every task (wasteful!) val lookup = Map( "US" -> "United States", "UK" -> "United Kingdom", ... ) data.map(row => lookup(row.country)) // lookup sent 1000s of times! // With broadcast - sent once per node val broadcastLookup = sc.broadcast(lookup) data.map(row => broadcastLookup.value(row.country)) // Efficient!
📊 Example: 100 MB lookup table, 1000 tasks, 10 nodes
• Without broadcast: 100 GB transferred (100 MB × 1000 tasks)
• With broadcast: 1 GB transferred (100 MB × 10 nodes)
100x reduction!

2. Accumulators

Variables that can only be added to (useful for counters and sums).

// Create accumulator val errorCount = sc.longAccumulator("Errors") val validCount = sc.longAccumulator("Valid Records") // Use in transformations data.foreach(record => { try { processRecord(record) validCount.add(1) } catch { case e: Exception => errorCount.add(1) } }) // Read value in driver println(s"Valid: ${validCount.value}, Errors: ${errorCount.value}")
✅ Use Cases:
• Debugging (count invalid records)
• Monitoring (track progress)
• Metrics (sum values across partitions)

📝 Complete Example: Word Count

// Hadoop: ~60 lines of Java // Spark: 3 lines of Scala! val counts = sc.textFile("data.txt") .flatMap(line => line.split(" ")) // Split into words .map(word => (word, 1)) // Create pairs .reduceByKey(_ + _) // Sum by word .collect() // Bring to driver // That's it! 🎉

📝 Complete Example: Logistic Regression

// Load and cache data (reused across iterations) val data = sc.textFile("data.txt") .map(parsePoint) .cache() // Keep in memory! // Initialize weights var w = Vector.random(D) // Iterative training for (i <- 1 to iterations) { // Compute gradient (data is in memory!) val gradient = data.map { point => val label = point.label val features = point.x val prediction = 1 / (1 + exp(-label * (w dot features))) label * (prediction - 1) * features }.reduce(_ + _) // Update weights w -= gradient } // Fast because data stays in memory across all iterations!
🚀 Performance:
• First iteration: ~6 seconds (load from disk)
• Subsequent iterations: ~0.6 seconds each (from memory)
• 10x faster per iteration!
• 100 iterations: Spark ~60s vs Hadoop ~12,700s (200x faster!)

📊 When to Use Each

🐘 Use Hadoop When:

  • Processing data larger than cluster memory
  • Simple one-pass batch processing
  • Cost-sensitive (less RAM needed)
  • Very large archival storage with processing
  • Existing Hadoop infrastructure

⚡ Use Spark When:

  • Iterative algorithms (ML, graph processing)
  • Interactive data analysis
  • Real-time stream processing
  • Complex multi-stage pipelines
  • Need for speed and development productivity

🎯 The Verdict

⚡ Spark is the clear winner for most modern use cases!

Hadoop MapReduce is still relevant for:
  • Very large batch jobs that don't fit in memory
  • Systems where Hadoop is already deployed
  • Cost-sensitive environments with limited RAM

However, for new projects, Spark is the recommended choice due to:
  • ✅ Significantly better performance
  • ✅ Much easier to develop with
  • ✅ Support for multiple workload types
  • ✅ Active development and community
