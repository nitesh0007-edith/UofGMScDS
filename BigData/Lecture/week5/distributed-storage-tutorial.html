Module 1

RAID — Redundant Array of Independent Disks

RAID combines multiple hard drives into one logical unit to improve performance, reliability, or both.

Real-World Analogy

RAID 0 splits homework across notebooks (fast, lose one = toast). RAID 1 writes the same in two notebooks (safe, double paper). RAID 5 splits AND writes "cheat notes" (parity) to reconstruct any lost one.

Figure 1 — RAID 0 vs RAID 1 vs RAID 5. Yellow blocks = parity data.

💡

Why XOR? XOR has a magical property: XOR any two of three values (Disk1, Disk2, Parity) to get the third. This is how RAID 5 reconstructs data from a failed disk.

Module 2

Distributed File Systems

When one machine isn't enough, DFS spreads data across a network — introducing challenges around consistency, availability, and performance.

Real-World Analogy

Imagine librarians in different buildings managing copies of the same books. A DFS is the rules for keeping every library in sync.

Figure 2 — Four core components of any DFS.

The Three Core Challenges

🔄 Consistency

Multiple copies must stay in sync. Machine A updates → Machine B must see it.

✅ Availability

System keeps working when hardware fails. Users access files without interruption.

⚡ Performance

Network reads are slower than local. Minimize latency, maximize throughput.

⚠️

OneDrive Sync Risks: Latency (read old version), Conflicts (two edit at once), Ordering (changes arrive out of order). Can cause data loss or corruption.

Module 3

NFS — Network File System

Created in 1984, NFS uses a simple client-server model. The server is stateless — it doesn't track who's connected.

Real-World Analogy

A shared filing cabinet with a sign-out sheet. Anyone can read, but to edit you sign out — nobody else can edit until you return it.

Figure 3 — NFS architecture: parallel reads, exclusive write locks.

Connection Steps

1. Contact Port Mapper

Client verifies the NFS service is alive.

2. Mount Request

Client receives a file handle — a reference to the remote filesystem.

3. File Operations

Read/write as if files are local. Server is stateless — crash & restart is seamless.

4. Write Locking

Lock acquired for writes. Persists until explicitly released.

🔑

Statelessness: Server stores no client info. Crash recovery is simple, but server can't notify clients of changes.

Module 4

HDFS — Hadoop Distributed File System

Storage backbone of Hadoop. Write-once-read-many on cheap commodity hardware.

Real-World Analogy

A warehouse: NameNode = manager with the inventory list (knows where everything is, never touches items). DataNodes = shelves storing items. Shelf breaks? Manager copies from other shelves with duplicates.

Figure 4 — HDFS: NameNode manages metadata; DataNodes store replicated blocks.

👑 NameNode

  • Role: Metadata — names, permissions, block locations
  • RAM-based for fast lookups
  • Recovery: Checkpoint + Journal (WAL)
  • Weakness: Single point of failure

📦 DataNode

  • Role: Store actual data blocks
  • Runs on cheap commodity hardware
  • 3x replication by default
  • Self-healing: Auto re-replicates on failure
Module 5

HDFS Reads & Writes

Clients read directly from DataNodes. Writes use a sequential pipeline for efficiency.

Figure 5 — Client asks NameNode, then reads directly from DataNodes.

Figure 6 — HDFS write pipeline.

1
Request DataNode list — NameNode selects nodes for the pipeline.
2
Pipeline formed — Client→DN1→DN2→DN3 chain established.
3
Stream data — 64KB packets flow through. Each node stores & forwards.
4
ACKs return — Last node confirms, ACKs bubble back to client.

Why a Pipeline?

❌ Broadcast

Send to ALL nodes at once. Saturates client bandwidth — 100Mbps / 3 = 33Mbps each.

❌ Sequential

Send to DN1, wait, DN2, wait... Extremely slow — total time = sum of all transfers.

✅ Pipeline

Client sends at full bandwidth. Each node forwards to next. Near-simultaneous receipt.

Module 6

HDFS Block States & Recovery

Blocks go through lifecycle states. HDFS uses time-limited leases instead of permanent locks.

Figure 7 — Block state lifecycle: RBW → failure → recovery → finalized → complete.

Committed vs Complete

  • Committed: ≥1 replica finalized. Safe but not fully replicated
  • Complete: ALL replicas finalized. Fully reliable

Leases vs Locks

  • NFS Locks: Persist until released. Crash = stuck forever!
  • HDFS Leases: Auto-expire. Crash = system self-heals
