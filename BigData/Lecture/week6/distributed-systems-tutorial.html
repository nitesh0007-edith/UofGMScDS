Section 01 — Hadoop Distributed File System

HDFS States & Recovery

How big data files are written, stored, and recovered when things go wrong

Imagine you're writing a huge 5 GB file across 10 different computers. What happens if one of those computers crashes in the middle of writing? HDFS — the Hadoop Distributed File System — has a very careful system of block states to handle exactly this problem.

💡
Real-World Analogy: Think of HDFS like writing a book across multiple notebooks. Each notebook page is a "block". If you spill coffee on one notebook while writing, you need a system to know exactly which pages were ruined and how to rewrite them from scratch.

Block Replica States (DataNode View)

From the perspective of each individual DataNode (the worker computers storing the data), a block can be in one of four states:

✍️
RBW
Replica Being Written
Data is actively being streamed to this block right now. Like a Word document that's currently open and being typed into.
⏳
RWR
Replica Waiting to be Recovered
A node crashed mid-write! This block was partially written and is now in limbo, waiting for someone to come fix it.
🔧
RUR
Replica Under Recovery
Recovery has started! The system is actively working to either complete or roll back the partial write on this block.
✅
Finalized
Complete & Stamped
The block is fully written, verified, and has received a generation stamp. Immutable and safe.

NameNode's View of Block States

The NameNode is the "master controller" that keeps track of where all blocks are. It has a different, higher-level view of block states:

NameNode State What It Means Example
Under Construction Blocks are currently being written (RBW state at DataNodes) File upload in progress
Under Recovery Block recovery is happening after a failure Recovering after DataNode crash
Committed At least 1 replica is Finalized, but not the required number yet Partially replicated block
Complete All required replicas are Finalized — safe! Default: 3 replicas all Finalized
🎯
Example: HDFS defaults to storing 3 copies of every block across 3 different DataNodes. A block is "Complete" only when all 3 copies reach "Finalized" state. If only 1 copy is finalized, it's "Committed" — available but not fully safe yet.

Generation Stamps — The Version Number

Every finalized block gets a Generation Stamp (GS) — a unique timestamp-like version number. Think of it like a version number on a document. When recovery happens, the GS is incremented, so the system always knows which copy is the "latest" valid version.

// Simplified mental model of a block with generation stamp Block { blockId: 12345, size: 134217728, // 128 MB generationStamp: 1007, // incremented each recovery state: "FINALIZED", locations: ["datanode-1", "datanode-4", "datanode-9"] }

Locks vs. Leases — A Critical Difference

Traditional file systems (like NFS — the Network File System) use file locks. HDFS takes a completely different approach with leases. Here's why it matters:

🔒 File Locks (NFS)
  • Lock held until client explicitly releases
  • If client crashes, lock stays FOREVER
  • Other clients are blocked indefinitely
  • Manual administrator intervention needed
  • Catastrophic in distributed systems
⏰ Leases (HDFS)
  • Lease has a built-in expiry time
  • If client crashes, lease expires automatically
  • System triggers Lease Recovery
  • Other clients can write after recovery
  • Safe, self-healing behavior
🍕
Pizza Analogy for Leases: Imagine ordering a pizza delivery slot (lease). If you don't pick it up within 30 minutes (expiry), the shop cancels and gives the slot to someone else. With a lock, you'd hold the slot forever — even if you fell asleep — blocking everyone else!

Lease Recovery Process

1
Client Crashes or Disconnects

A client was writing a file and suddenly goes offline. The Lease Manager detects no heartbeat.

2
Lease Expires

The soft limit (~60 seconds) and hard limit (~1 hour) timers tick. When exceeded, the Lease Manager flags the file for recovery.

3
Lease Recovery Triggers

The Lease Manager initiates Lease Recovery, which internally triggers Block Recovery for all affected blocks.

4
DataNodes Synchronize

A primary DataNode is elected. All replicas agree on a common valid length, truncating partial writes. Generation stamp is updated.

5
File Closed & Lease Released

The file is marked as closed in the NameNode's namespace. The lease is released and the file is now available to other clients.

🌍 Real-World Example

When Apache Spark writes results to HDFS and the executor process crashes mid-write, HDFS's lease recovery automatically cleans up the partial file within ~60 seconds. Your next Spark job can safely read the directory without manually deleting corrupt partial files. This is why HDFS-backed data lakes are self-healing.

Section 02 — Brewer's Theorem

The CAP Theorem

Why you can never have it all in a distributed system

Proposed by Eric Brewer in 2000, the CAP theorem states that any distributed data store can only guarantee two out of three fundamental properties. Understanding this helps explain every architectural decision in systems like Cassandra, MongoDB, and MySQL.

🏦
Real-World Analogy: Think of a bank with branches across cities. You can have: all branches showing the same balance (Consistency) + always being open (Availability) + working even when the internet is cut (Partition Tolerance) — but not all three at once. If branches can't communicate (network partition), they must either refuse transactions (sacrifice Availability) or risk showing different balances (sacrifice Consistency).

The Three Properties

🔄
Consistency (C)
Every node in the system returns the exact same, most up-to-date data at the same time. No stale reads. No diverged state. Like a single synchronized database.
📡
Availability (A)
Every request receives a response — success or failure. The system never says "I'm too busy" or goes completely offline. Every node that works will respond.
🌐
Partition Tolerance (P)
The system keeps functioning even when network partitions occur — i.e., some nodes can't talk to others. Messages can be dropped or delayed without crashing the system.

⚠️
NFS and the CAP Theorem: The Network File System (NFS) is a classic CA system. It has only one copy of data on one server. This gives you consistency (one source of truth) and availability (always responds), but if the network splits or the server goes down — game over. Zero partition tolerance.

The Real-World Impact

🌍 Shopping Cart Example (Amazon's Dynamo)

Amazon's shopping cart uses an AP system. If a network partition occurs, you might add an item on your phone AND on your laptop simultaneously while offline. When connectivity restores, both additions are kept (eventual consistency). Amazon decided it's better to show a slightly stale cart than to block you from adding items at all. This is a conscious business decision: "always available" beats "always consistent" for shopping carts.

Section 03 — Data Structure Design

Log-Structured Merge Trees

How Cassandra, HBase, and LevelDB handle millions of writes per second

Imagine you need to store 1 million sensor readings per second from IoT devices. A traditional B-Tree database would require random disk writes — extremely slow. LSM Trees solve the "index problem" by turning all writes into sequential appends, which are orders of magnitude faster.

📓
Analogy: A traditional database is like updating a sorted address book in-place (random writes). LSM Trees are like keeping a notepad where you just write new entries at the bottom (sequential writes), then periodically sorting and merging into a clean address book in the background.

The Five Core Components

📝 Write Ahead Log (WAL)

Purpose: Crash recovery. Every write is appended here first, before anything else. Sequential disk writes = blazing fast. Even if power cuts out, data is safe here.

🧠 MemTable

Purpose: In-memory sorted buffer. New writes go here for instant reads. Typically implemented as a Red-Black Tree or Skip List. Lives in RAM — super fast but not persistent.

💾 SSTables (Spill Files)

Purpose: Persistent, immutable, sorted files on disk. When MemTable fills up, it's "flushed" here. Each SSTable is sorted and never modified once written.

🔀 Compactor

Purpose: Background janitor. Merges multiple SSTables into fewer, larger ones. Removes deleted or overwritten records. Like defragmenting a hard drive.

🗂️ Index

Tracks which SSTable files contain which keys. Uses Bloom Filters (see Section 4) to skip files that definitely don't contain a key.

🔍 Bloom Filter

Probabilistic gate per SSTable. Answers "is key X possibly in this file?" with no false negatives. Eliminates most unnecessary disk reads.

The write is appended to the Write Ahead Log on disk immediately. This is a sequential write — extremely fast. Even if the machine crashes after this step, data is not lost.

2

Data is added to the in-memory MemTable (a sorted data structure like a Red-Black Tree). The client immediately gets an acknowledgment — the write is "done" from the client's perspective.

3

When the MemTable reaches its size limit (e.g., 128 MB), it's sorted and written to a new immutable SSTable file on disk. This is also a sequential write — fast!

4

The index is updated to record which keys are in the new SSTable. The WAL is cleared since the data is now safely on disk in the SSTable.

Deletes and Updates — The Tombstone Mechanism

LSM Trees can't modify existing SSTables (they're immutable!). So how do deletes and updates work?

🪦
Tombstones: Deletes in LSM Trees write a special tombstone marker — essentially a "this key is deleted" record. Updates write a new version of the key. The old value becomes orphaned. The Compactor later cleans up both the old value and the tombstone during compaction.
// Timeline of key "user:42" through LSM operations: t=1: Write user:42 → {name: "Alice", age: 30} // Written to MemTable → SSTable-1 t=2: Update user:42 → {name: "Alice", age: 31} // New entry in SSTable-2 (SSTable-1 untouched) t=3: Delete user:42 // Tombstone written to SSTable-3 // Read at t=3: checks SSTable-3, finds tombstone → "not found" // Compaction merges SSTable-1,2,3 → removes all "user:42" entries

🌍 Cassandra in Production

Netflix uses Cassandra (LSM Tree-based) to serve its recommendation engine. During peak hours they handle tens of millions of writes per second. The LSM architecture makes this possible — every write is just a sequential append to a WAL and an in-memory insert. The SSTable compaction happens quietly in the background during off-peak hours without affecting write performance.

Section 04 — Probabilistic Data Structures

Bloom Filters

Space-efficient membership testing with controlled uncertainty

A Bloom Filter answers the question: "Have I seen this item before?" — but it does so using a fraction of the memory a normal set would require. The trade-off? Occasional false positives. It might sometimes say "yes, I've seen this" when it hasn't. But it never has false negatives — if it says "never seen this", it's definitely right.

🎪
Real-World Analogy: Imagine a bouncer at a club using a rough list of banned people. They might mistakenly stop an innocent person who looks like someone on the list (false positive), but they will NEVER let in a definitely-banned person. That's a Bloom Filter — fast, space-efficient, with controlled false positive risk.

How It Works — The Mechanics

📊
Step 1: Bit Array
Start with an array of m bits, all set to 0. This is the Bloom Filter. Typical size: thousands to millions of bits (still tiny compared to actual data).
#️⃣
Step 2: Hash Functions
Choose k hash functions. Each function maps any key to a position in the bit array. Different functions → different positions for the same key.
✍️
Step 3: Insert
To insert key "Alice": run it through all k hash functions. Each function outputs a bit index. Set those bits to 1. Repeat for every key.
🔍
Step 4: Query
To check "is Bob in the set?": hash Bob with all k functions. If ANY resulting bit is 0 → definitely not in set. If ALL bits are 1 → probably in set.

Understanding False Positives

False positives happen because multiple keys can share bit positions (hash collisions). Here's a clear example:

// Insert both "Alice" and "Bob": hash1("Alice") = 2, hash2("Alice") = 7, hash3("Alice") = 13 hash1("Bob") = 5, hash2("Bob") = 9, hash3("Bob") = 2 // Bits set: 2, 5, 7, 9, 13 // Now check "Charlie" (never inserted): hash1("Charlie") = 5, hash2("Charlie") = 9, hash3("Charlie") = 2 // All three bits (5, 9, 2) happen to be 1 → FALSE POSITIVE! // Filter says "Charlie might be here" — but Charlie was never inserted!
⚠️
Key Insight: False positives cause unnecessary disk reads in LSM Trees (you read an SSTable file that doesn't actually have the key). But they never cause wrong data — you just do some extra work. False negatives are impossible because bits can only be set, never unset.

Mathematical Optimization

You can tune your Bloom Filter using these parameters to hit a target false positive rate:

optimal_k = (m/n) × ln(2)   |   pfp ≈ (1 - e^(-kn/m))^k
m = bits  |  n = elements  |  k = hash functions  |  pfp = false positive probability
n (elements)Desired pfpm (bits needed)k (hash functions)Memory

💡
Tracking 10 million keys with 1% false positive rate takes only 11.4 MB. A HashSet storing the same 10M strings might take 500 MB+. That's a 40x+ memory saving — and why Bloom Filters are used everywhere in large-scale systems!

🌍 Real Deployments of Bloom Filters

Google Chrome uses Bloom Filters to check if URLs are malicious (Safe Browsing API) — the local filter eliminates 99%+ of unnecessary network lookups. Cassandra uses them per SSTable to dramatically reduce read amplification. Bitcoin's SPV nodes use Bloom Filters to request only relevant transactions without revealing which addresses they're tracking. Medium uses them to avoid showing users articles they've already read.
