Question 1 — HDFS Read & Write Protocols 15 marks
Describe the read and write protocols, when accessing files in the Hadoop Distributed File System (HDFS).

The Write Protocol 8 marks

When a client wants to write a file to HDFS, it follows a pipeline-based protocol that ensures data is replicated across multiple DataNodes for fault tolerance. Think of it like sending a parcel through a chain of delivery points — each one passes it forward.

  1. Client notifies NameNode (NN): The client calls create() or opens a file for append. The NN checks permissions, ensures the file doesn't already exist, and records the new file in the namespace.
  2. NN allocates block & replicas: The NN creates a new blockID for the first block and decides which DataNodes (DNs) will host the replicas (typically 3 copies, placed on different racks).
  3. Client receives writing credentials: The NN returns a lease (a write lock — guaranteeing only 1 writer but many readers), the blockID, and the list of DN locations.
  4. Client builds a pipeline: The client arranges the DNs into a pipeline ordered by network proximity, minimising the total distance to the last DN. For example: Client → DN1 (same rack) → DN2 (different rack) → DN3.
  5. Data is packetised: The client breaks data into 64KB packets and pushes them into a local buffer.
  6. Buffer flushed to pipeline: When the buffer is full, packets flow through the pipeline. Each DN writes the data locally and forwards to the next DN.
  7. Repeat for every block: All steps above are repeated for each new 128MB block of the file.
  8. Asynchronous acknowledgment: Packets are sent asynchronously — the next packet is pushed before the previous one is fully acknowledged. The hflush() call can be used to guarantee all packets written before it are visible to readers.
Real-World Analogy

Imagine uploading a large video to YouTube. The system splits it into chunks, sends each chunk through a chain of storage servers, and each server confirms receipt. You don't wait for the first chunk to be fully confirmed before sending the second — that's the asynchronous pipeline in action.

The Read Protocol 7 marks

Reading is simpler than writing — the client communicates with the NN only for metadata, then reads data directly from DataNodes. This keeps the NN out of the data path, which is critical for scalability.

  1. Client contacts NN for metadata: Fetches two things — the list of blocks composing the file, and for each block, the list of DNs holding replicas (sorted by distance from the client).
  2. Direct DN contact: For each block, the client picks the closest DN and reads directly from it. If that DN fails, it transparently tries the next closest replica.
  3. DN sends data + checksum: The DN responds with the block's data file and its metadata (including a checksum). The client verifies the checksum to detect corruption.

Failure scenarios: A read can fail if the DN is down/unreachable, the block is corrupted (checksum mismatch), or the DN no longer holds that replica (it may have been moved during rebalancing).

Question 2 — NameNode Crash Recovery 6 marks
Assume the NameNode in HDFS crashes. What are the resources the NN uses for recovery and how?

Three Recovery Resources 2 marks each

The NameNode is the brain of HDFS — it holds all file-to-block mappings and the directory tree. If it crashes, we'd lose the entire filesystem's structure. HDFS uses three complementary resources to recover from such failures:

1. Image (FsImage)

An in-memory data structure holding the complete state of the NN — the entire filename space (directory tree, file-to-block mappings, permissions). Kept in RAM for fast access, but RAM is volatile — it's lost on crash.

2. Checkpoint

A periodic snapshot of the in-memory image dumped to stable storage (disk). Think of it as a "save game" — a known-good state we can return to. Created by the Secondary NameNode periodically.

3. Journal (EditLog / WAL)

Between checkpoints, every operation that changes the namespace (create file, delete file, rename, etc.) is recorded in a write-ahead log on disk. "Write-ahead" means the operation is logged before it's applied to the in-memory image. This ensures no mutation is lost even if the system crashes mid-operation.

Recovery Process

On restart, the NN: (1) loads the last checkpoint from disk → (2) replays all operations from the journal on top of it → (3) reconstructs the full, up-to-date image in memory. It's like loading your last save game and then replaying all your moves since that save.

Real-World Analogy

Think of a bank. The image is everyone's current balance in the computer. The checkpoint is a nightly printout of all balances. The journal is the list of all transactions since the last printout. If the computer crashes, the bank loads last night's printout and replays today's transactions to recover all balances.

Question 3 — HBase Logical Data Model 15 marks
Describe the HBase logical data model.

What is HBase?

HBase is a sparse, distributed, persistent, multidimensional, sorted map. It provides a table-like abstraction, but it is NOT a relational database. There are no transactions, no normalisation, and no SQL-style relational API. Instead, users define their own flexible, extensible schema.

Real-World Analogy

Imagine a massive filing cabinet where each drawer (row) is labelled and sorted alphabetically. Inside each drawer, folders (column families) contain individual documents (columns). Each document can have multiple dated versions. You can quickly find any drawer by its label, but you can't do complex queries across drawers like a relational database would.

Data Addressing — The 3D Map

Every piece of data in HBase is located using three coordinates:

DimensionDescriptionExample
Row Key (required)A string that uniquely identifies the row. Rows are sorted lexicographically."user_00123"
Column NameFormat: family:qualifier. Column families are defined at table creation."profile:email"
TimestampEach cell can have multiple versions, indexed by timestamp.1609459200

Each cell value is just an uninterpreted byte array — HBase doesn't know or care about the data type.

Key Properties

  • Row atomicity: Operations on a single row are atomic, but there's no atomicity across rows
  • Column Families (CFs): Columns are grouped into families. Each table has a small number of CFs (typically <10), but each CF can have millions of columns. Access control is set at the CF level.
  • Tablets (Regions): A table is horizontally partitioned into tablets. Each tablet holds all rows between a start and end row key. Each tablet is assigned to a tablet server.
  • Versioning: Cells can have multiple versions (each with a timestamp). Latest version returned by default. Users can request specific number of versions or all versions within a time period.

Basic Operations

Put(row_key, column_key, timestamp, value) → Insert/update a cell Get(row_key) → Returns all cells in a row Get(row_key, column_key, timestamp) → Returns a specific cell version Scan(start_row_key, end_row_key) → Returns all rows in a key range

In relational terms, these are basically selections and projections — no joins, no aggregations at the database level.

Question 4 — HBase System Entities & Roles 15 marks
What are the main entities and their roles in the HBase system?

Architecture Overview

HBase follows a master-slave architecture with three main components, all built on top of HDFS and SSTables. It also depends on a distributed coordination service (ZooKeeper) for scheduling, monitoring, locking, and failure detection.

1. The Master 6 marks

The Master is the coordinator of the entire HBase cluster — but crucially, it is NOT involved in actual data reads/writes. This is the only way to avoid scalability bottlenecks in master-based architectures.

  • Tablet management: Creates new tablets, deletes tablets, and merges two tablets together
  • Assignment: Assigns tablets to tablet servers (decides which server manages which data)
  • Health monitoring: Detects new or "expired" (crashed) tablet servers
  • Load balancing: Redistributes tablets across servers to keep workload even
  • Schema management: Handles table, column family, and column additions/deletions
Real-World Analogy

The Master is like a library director — they decide which librarian manages which shelves, hire/fire staff, and reorganise sections. But when a visitor wants a book, they go directly to the librarian at that shelf, never to the director.

2. Tablet Servers (Region Servers) 6 marks

These are the workhorses that actually store and serve data.

  • Storage: Each tablet server manages 10s to 1000s of tablets
  • Data serving: Handles all read and write requests for its own tablets directly from clients
  • Splitting: When a tablet grows too large (e.g., >100-200MB), the tablet server splits it into two smaller tablets
  • Dynamic scaling: Tablet servers can be added or removed from the cluster dynamically

Key distinction: Tablet servers split tablets, while the Master merges, creates, and deletes them.

3. HDFS & SSTables 3 marks

HBase doesn't have its own storage engine — it sits on top of HDFS (an open-source implementation of Google's GFS) for durable, distributed storage. Data within tablets is stored as SSTables — persistent, ordered, immutable maps of keys to values, indexed and stored in the local file system.

Quick Summary

Master = brain (coordinates, doesn't touch data) | Tablet Servers = muscles (store & serve data) | HDFS = bones (underlying storage). The Master stays out of the data path → scalable architecture.

Question 5 — Replication Read/Write Protocols 7 marks
Describe read/write protocols for a replicated big data system so that reads always see the latest version of a data item.

The Core Challenge

When data is replicated across multiple servers, we need protocols to ensure that a reader always sees the most recent write. There are three major approaches, each with different trade-offs between performance, availability, and consistency.

Protocol 1 — Primary Replica 2 marks

Read

All reads go to a known primary replica server.

Write

All writes go to the same primary replica server.

Drawback: The primary becomes a performance bottleneck and a single point of failure. If it goes down, neither reads nor writes work until failover.

Protocol 2 — Read Any, Write All 2 marks

Read

Reads can go to any replica (nearest one).

Write

Writes must go to ALL replicas. Uses two-phase commit: (1) all replicas tentatively apply write, (2) if all OK → commit. Writer logs commit decision to stable storage for crash recovery.

Trade-off: Reads are fast and highly available (any copy works). Writes are slow and less available (all replicas must be up).

Protocol 3 — Quorum (R + W > N) 3 marks (incl. advantages/disadvantages)

The most flexible approach. Instead of reading from any/all, a reader contacts R replicas and a writer contacts W replicas, where:

Quorum Rules

R + W > N (guarantees read-write overlap → readers always see latest write)

W + W > N (guarantees write-write overlap → detects conflicts)

Where N = total number of replicas

Real-World Example

With N=3 replicas, setting R=2 and W=2: every read overlaps with every write (2+2=4 > 3). This means at least one server in any read set will have the latest write. It's like asking 2 out of 3 friends — if one of them witnessed the event, you'll hear the latest news.

Key advantage: You can tune R and W per data item based on access patterns. Read-heavy data? Set R=1, W=3. Write-heavy? Set R=3, W=1.

Moodle Answer Strategy

For 7 marks, briefly describe all 3 protocols (2-3 sentences each) and discuss trade-offs. The quorum protocol is the most important — make sure to state the R+W>N rule with an example.
