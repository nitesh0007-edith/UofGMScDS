Question 1 — HDFS 20 marks
Q1a. Explain the significance of block size in HDFS. What are the trade-offs of choosing larger vs. smaller block sizes?

Q1a. HDFS Block Size 4 marks OFFICIAL

In HDFS, files are split into fixed-size blocks (default 128MB) that are distributed across the cluster. The block size is not just a storage parameter — it fundamentally affects how the entire system behaves, from the NameNode's memory consumption to MapReduce job performance. Think of it like choosing the size of shipping containers: too small and you drown in logistics overhead; too big and you lose flexibility.

FactorLarger Blocks (e.g., 256MB+)Smaller Blocks (e.g., 64MB)
NameNode Memory Fewer blocks to track = less NN memory. A 1TB file at 256MB = 4,096 blocks. The NN stores metadata for every block in RAM, so fewer blocks means the NN can manage more total data. Many more blocks to track = more NN memory pressure. Same 1TB file at 64MB = 16,384 blocks — 4x the metadata overhead.
Map Task Behavior Each map task processes one block, so larger blocks mean more data per mapper. This can cause data skew — one mapper might take much longer if its block has disproportionately complex data. More map tasks with less data each = finer-grained parallelism. Better load balancing across the cluster.
Overhead Less seek overhead relative to data transfer. The ratio of seek time to transfer time favors large blocks on HDDs. More block management overhead: more metadata RPCs, more block reports from DataNodes, more network connections.
Parallelism Less parallelism — fewer blocks means fewer tasks that can run concurrently. Some cluster nodes may sit idle. Higher parallelism — many small tasks can saturate all available slots in the cluster.
Real-World Analogy

Imagine you are moving house and packing books into boxes. If you use enormous boxes (large block size), you need fewer boxes and it is simpler to label and track them, but each box is extremely heavy and one person might struggle to carry it while others stand idle. If you use tiny boxes (small block size), everyone can help carry at the same time, but you end up with hundreds of boxes to label, stack, and track — the logistics become a nightmare. The 128MB default is the "Goldilocks" box size for most workloads.

Key Formula to Remember

Default block size = 128MB (was 64MB in Hadoop 1.x, doubled in Hadoop 2.x).

NN stores ~150 bytes of metadata per block in RAM. At 128MB blocks, 1PB of data = ~8 million blocks = ~1.2GB of NN memory. At 64MB blocks, same data = ~2.4GB. This is why block size directly constrains the maximum cluster capacity.

Q1b. Describe HDFS's replica placement strategy. What is the default replication factor and how are replicas distributed? What is the rationale behind this strategy?

Q1b. HDFS Replica Placement Strategy 8 marks OFFICIAL

HDFS replicates every block to protect against hardware failures. The default replication factor is 3, meaning three copies of each block exist in the cluster. However, where those copies are placed matters enormously — a naive strategy could leave all copies vulnerable to a single rack-level failure (power outage, network switch death, or even a fire).

  1. First replica — same node as the writer: If the client writing the data is running on a DataNode in the cluster, the first replica is placed on that same machine. This is the fastest option since no network transfer is needed. If the client is external (not on a cluster node), a random DataNode is chosen with consideration for available disk space and current load.
  2. Second replica — different rack: The second copy is placed on a DataNode in a completely different rack. This is the critical decision — it guarantees that even if an entire rack goes down (switch failure, power outage, cooling failure), at least one copy survives on another rack. This single cross-rack transfer is the "insurance premium" HDFS pays for fault tolerance.
  3. Third replica — same rack as the second, but a different node: The third copy is placed on another DataNode within the same rack as the second replica. This might seem counterintuitive — why not spread across three racks? The answer is write bandwidth optimization: only one cross-rack network transfer is needed in the write pipeline instead of two. Intra-rack bandwidth (typically 1-10 Gbps within a rack) is much higher than inter-rack bandwidth.
The Balancing Act

This strategy is a carefully calculated compromise between two competing concerns:

Reliability: Data survives a complete rack failure because replicas exist on 2 different racks.

Write performance: Only 1 cross-rack transfer in the write pipeline (not 2), since the 2nd and 3rd replicas share a rack. Inter-rack bandwidth is the most precious and congested resource in a data center.

Read performance: Having 2 replicas on one rack increases the chances of a reader finding a nearby copy.

Real-World Analogy

Imagine you have important documents that absolutely cannot be lost. You keep one copy in your desk drawer (same node — fast access). You send a second copy to a colleague on a completely different floor of the building (different rack — survives your entire floor flooding or losing power). You give a third copy to someone sitting near that colleague on the other floor (same rack as second, different node — quick to copy since they are close together, and now two copies survive even if your floor is destroyed). If you sent all three copies to three different floors, you would have maximum redundancy but would spend three times the effort on inter-floor mail. HDFS's strategy says: "One floor of separation is enough insurance; let's not waste bandwidth on more."

What survives what? Node failure: 2 replicas still available. Rack failure: at least 1 replica on the other rack survives. The only scenario that loses data is if both racks fail simultaneously, which is extremely unlikely in a well-managed data center.

Q1c. Describe the step-by-step process a client follows to read a file from HDFS.

Q1c. Client File Read Steps 8 marks OFFICIAL

Reading a file from HDFS is designed to be scalable — the NameNode is only consulted for metadata (block locations), while actual data flows directly between the client and DataNodes. This keeps the NameNode out of the data path, preventing it from becoming a bottleneck even when thousands of clients read simultaneously.

  1. Client calls open() on DistributedFileSystem: The client application invokes the HDFS client library, which creates a DistributedFileSystem object. This is the entry point — the client says "I want to read file /data/logs/2019.csv".
  2. DistributedFileSystem contacts the NameNode: The DFS makes an RPC call to the NameNode requesting the block locations for the file. The NameNode responds with a list of blocks composing the file and, for each block, a list of DataNodes holding replicas — sorted by network proximity to the client (same node > same rack > different rack). This proximity-based sorting is crucial for read performance.
  3. Client reads via FSDataInputStream: The DFS returns an FSDataInputStream to the client. When the client calls read(), the stream connects to the closest DataNode holding the first block. Data flows directly from that DataNode to the client — the NameNode is completely out of the loop at this point.
  4. Data is streamed and checksums verified: As data arrives, the client verifies checksums for each chunk. HDFS stores checksums alongside data blocks, so the client can detect silent data corruption (bit rot, disk errors). If a checksum fails, the client knows the data is corrupted and will not use it.
  5. Block boundary handling: When the client finishes reading the current block, the FSDataInputStream closes the connection to that DataNode and finds the best (closest) DataNode for the next block. This process repeats block-by-block until the entire file has been read. The client may also pre-fetch the next block's location to reduce latency.
  6. Transparent failure handling: If a DataNode fails mid-read (connection drops, timeout) or returns corrupted data (checksum mismatch), the client transparently retries with the next closest DataNode holding a replica of that block. The client also reports the bad block to the NameNode so it can schedule re-replication — ensuring the corrupted or unavailable replica is replaced elsewhere in the cluster.

Real-World Analogy

Think of reading a book that has been split into chapters stored in different libraries across a city. First, you call the central catalog office (NameNode) and ask: "Where are the chapters of 'War and Peace'?" The catalog tells you: "Chapter 1 is at Library A (5 min away), Library B (20 min away), and Library C (45 min away). Chapter 2 is at Library D (3 min away)..." You go to the closest library for each chapter, read it there, and move on. If Library A is closed when you arrive, you simply go to Library B instead. Block size (128MB) trades NN memory vs. parallelism. Replica placement (3 copies: node, different-rack, same-rack-as-2nd) balances reliability against write bandwidth. File reads go through NN only for metadata, then stream directly from closest DNs with checksum verification and transparent failover.

Question 2 — MapReduce & Spark 20 marks
Q2a. What factors should be considered when choosing the optimal number of reducers in a MapReduce job?

Q2a. Optimal Number of Reducers 4 marks OFFICIAL

Choosing the right number of reducers is one of the most impactful tuning decisions in MapReduce. It is not a simple "more is better" situation — there is a sweet spot that depends on your cluster size, data distribution, and downstream requirements.

Too Few Reducers

Overloaded reducers: Each reducer must process a massive amount of data, leading to memory pressure and potential out-of-memory errors. If you have 100TB of intermediate data and only 2 reducers, each must handle 50TB.

Underutilized cluster: Most nodes sit idle while a handful do all the reduce work. You are paying for a 1000-node cluster but only using 2 nodes for the reduce phase.

Long tail problem: One slow reducer becomes the bottleneck for the entire job.

Too Many Reducers

Excessive overhead: Each reducer incurs framework overhead — task startup, JVM initialization, shuffle connections, output file creation. With 100,000 reducers, this overhead dominates actual computation.

Too many small output files: Each reducer creates one output file. Thousands of tiny files flood the NameNode with metadata and make downstream processing inefficient (the "small files problem").

Shuffle bottleneck: Every mapper must connect to every reducer, creating O(M x R) network connections.

The Rule of Thumb

Optimal reducers = 0.95 x (nodes x max_containers_per_node) or 1.75 x (nodes x max_containers_per_node)

The 0.95 multiplier means almost all reduce slots are used in a single wave. The 1.75 multiplier allows for two waves, giving better load balancing — faster reducers finish and pick up work from the second wave.

Data skew is the hidden danger: Even with the "right" number of reducers, if some keys have vastly more values than others (e.g., in a word count, "the" appears millions of times while "zymurgy" appears once), a few reducers get hammered while others finish instantly. Solutions: custom partitioner, salting keys, or using a combiner to pre-aggregate.

Real-World Analogy

Imagine you are sorting mail into pigeonholes at a post office. If you have only 2 pigeonholes for all of London, each one overflows. If you have 10 million pigeonholes (one per person), you waste space and it takes ages to check them all. The sweet spot is somewhere like one per postal district — manageable, well-distributed, and efficient. But if one district is a corporate headquarters receiving 50% of all mail, that pigeonhole still overflows — that is data skew.

Q2b. Explain how a reduce-side join works in MapReduce. Why is it considered expensive?

Q2b. Reduce-Side Join 6 marks OFFICIAL

A reduce-side join is the most general (and most expensive) way to join two datasets in MapReduce. Unlike a relational database that can use indexes, hash joins, or sort-merge joins transparently, MapReduce has no built-in join operator — you must implement it yourself using the shuffle mechanism.

  1. Tag records in the mapper: Both datasets are fed into mappers. Each mapper reads records and emits key-value pairs where the key is the join column and the value is tagged with a source identifier. For example, if joining Orders (by customer_id) with Customers (by id), the mapper emits (customer_id, {"source": "orders", "data": ...}) and (customer_id, {"source": "customers", "data": ...}).
  2. Framework shuffles and groups by join key: The MapReduce framework sorts and partitions all emitted pairs by key, then groups them. All records with the same join key — from both datasets — end up at the same reducer.
  3. Reducer separates and cross-products: The reducer receives all values for a given key. It separates them by source tag into two lists (e.g., all orders for customer #42 and the customer #42 record). Then it performs a cross-product of the two lists to produce the joined output — every order paired with the customer details.
-- Dataset A: Orders -- -- Dataset B: Customers -- (C42, order_001) (C42, "Alice Smith") (C42, order_002) (C99, "Bob Jones") (C99, order_003) Mapper Output (tagged): (C42, [A, order_001]) (C42, [B, "Alice Smith"]) (C42, [A, order_002]) (C99, [B, "Bob Jones"]) (C99, [A, order_003]) After Shuffle (grouped by key): Reducer for C42: [A, order_001], [A, order_002], [B, "Alice Smith"] Reducer for C99: [A, order_003], [B, "Bob Jones"] Reducer Cross-Product: (C42, order_001, "Alice Smith") (C42, order_002, "Alice Smith") (C99, order_003, "Bob Jones")

Why is it expensive? ALL data from BOTH datasets must be shuffled across the network. Every single record is serialized, partitioned, transferred over the network, deserialized, sorted, and merged. For two 1TB datasets, that is 2TB of network I/O through the shuffle — the most expensive operation in MapReduce. Unlike a map-side join (which requires sorted, co-partitioned data), a reduce-side join imposes no preconditions on the data, which is why it is the "fallback" join strategy.

Real-World Analogy

Think of a matchmaking service. Both men and women submit their profiles (mapper tags each record with its source). The service sorts everyone by their preferred criteria — say, city (the join key). Then, in each city, the matchmaker (reducer) takes all the men and all the women and pairs them up. The expensive part? Every single person had to travel to a central sorting facility first (the shuffle), even if 99% of them will not match. A map-side join is like pre-sorting everyone into city groups beforehand — much cheaper, but requires advance preparation.

Q2c. Why does Spark use lazy evaluation for transformations? What advantages does this provide?

Q2c. Spark Lazy Evaluation 4 marks OFFICIAL

In Spark, transformations (like map, filter, join) are lazy — they do not execute immediately. Instead, they build up a DAG (Directed Acyclic Graph) of operations, which is a logical execution plan. Actual computation is deferred until an action is called (collect(), count(), saveAsTextFile()). This design is not laziness for laziness's sake — it enables powerful optimizations.

  • Query optimization: The Catalyst optimizer can inspect the entire DAG before execution. It can reorder operations (push filters before joins to reduce data early), combine adjacent maps into a single pass (pipelining), and eliminate unnecessary shuffles. Without seeing the full plan, none of these optimizations would be possible.
  • Minimize shuffles: Shuffles (data redistribution across the network) are the most expensive operation. By analyzing the full DAG, Spark can identify which operations truly require a shuffle and which can be pipelined within the same partition — a technique called stage fusion.
  • Pipelining within partitions: Multiple transformations can be fused into a single pass over the data. Instead of materializing intermediate results (map → write to disk → read → filter → write to disk → read → ...), Spark applies all transformations to each record in a single pipeline, drastically reducing I/O.
  • Fault recovery via lineage: Because the DAG records how each RDD was derived, Spark can recompute lost partitions by replaying the lineage from the parent RDD. No need for expensive checkpointing or replication of intermediate results — the DAG is the recovery plan.
Real-World Analogy

Lazy evaluation is like a GPS that plans your entire route before you start driving. If you told it "go to the supermarket, then the pharmacy, then home" one instruction at a time (eager evaluation), it might send you across town to the supermarket, then back across town to the pharmacy, then home — very inefficient. But if you give it all three destinations upfront (lazy evaluation), it can plan an optimal loop: supermarket first (it is on the way to the pharmacy), pharmacy next (it is near the supermarket), then straight home. It found shortcuts and avoided backtracking because it saw the whole plan before executing.

Lazy vs. Eager Execution

Lazy (Spark): Build plan → Optimize → Execute once. Result: fewer passes over data, fewer shuffles, minimal intermediate materialization.

Eager (classic MapReduce): Execute each step immediately, write to disk, read back. Each MR job is independent — no cross-job optimization. Result: many I/O round trips.

Q2d. Compare groupByKey and reduceByKey in Spark. When should you use each?

Q2d. groupByKey vs. reduceByKey 6 marks OFFICIAL

Both operations group data by key, but they differ dramatically in how much data crosses the network during the shuffle. This distinction is one of the most important performance considerations in Spark programming.

groupByKey

Mechanism: Shuffles every single value for each key across the network to a single executor. Only after all values arrive does it group them into a list.

Network cost: Extremely high — all raw data is transferred.

Memory risk: The receiving executor must hold ALL values for a key in memory simultaneously. If one key has billions of values, it causes an OutOfMemoryError.

When to use: Only when you genuinely need all raw values together (e.g., collecting all purchase items per customer for a recommendation engine).

reduceByKey

Mechanism: Performs a local pre-aggregation (combining) on each partition before shuffling. Only the partially reduced results are sent across the network.

Network cost: Dramatically lower — only aggregated values are transferred.

Memory safety: Each partition only holds one accumulated value per key, so memory usage is bounded by the number of distinct keys, not the number of values.

When to use: Whenever your operation is associative and commutative (sum, count, max, min, concatenation). This is the vast majority of real-world use cases.

// Word count example — 3 partitions, counting occurrences of "hello" // === groupByKey approach === // Partition 1 has: ("hello",1), ("hello",1), ("hello",1) → shuffles 3 values // Partition 2 has: ("hello",1), ("hello",1) → shuffles 2 values // Partition 3 has: ("hello",1), ("hello",1), ("hello",1) → shuffles 3 values // Network: 8 values shuffled → reducer receives [1,1,1,1,1,1,1,1] → sums to 8 // === reduceByKey approach === // Partition 1 has: ("hello",1), ("hello",1), ("hello",1) → locally reduces to ("hello",3) // Partition 2 has: ("hello",1), ("hello",1) → locally reduces to ("hello",2) // Partition 3 has: ("hello",1), ("hello",1), ("hello",1) → locally reduces to ("hello",3) // Network: 3 values shuffled → reducer receives [3,2,3] → sums to 8

In this simple example, reduceByKey shuffles 3 values instead of 8. In a real cluster with millions of records, the difference is enormous — potentially 100x less network I/O.

Real-World Analogy

Imagine a national election with voting booths across the country. groupByKey is like shipping every single ballot paper to a central counting office in London — trucks full of paper, traffic jams, warehouse needed to store millions of slips. reduceByKey is like counting at each local polling station first and then phoning in only the totals: "Candidate A: 4,231. Candidate B: 3,897." Obviously, sending a few numbers is vastly more efficient than shipping truckloads of paper. You would never ship all the ballots unless you needed to inspect each individual one (which is the groupByKey use case — when you truly need every raw value).

Question 3 — NoSQL & Distributed Systems 20 marks
Q3a. Apply the CAP theorem to two scenarios: (i) a chat application and (ii) a banking application. Justify your choices.

Q3a. CAP Theorem Application 6 marks OFFICIAL

The CAP theorem states that a distributed data store can simultaneously provide at most two out of three guarantees: Consistency (every read returns the most recent write), Availability (every request receives a response), and Partition tolerance (the system continues operating despite network splits between nodes).

The Real-World Constraint

Network partitions are inevitable in distributed systems — cables get cut, switches fail, data centers lose connectivity. Since you cannot avoid P, the real engineering choice is between C and A during a partition: Do you refuse to serve requests (sacrifice Availability to maintain Consistency)? Or do you serve potentially stale data (sacrifice Consistency to maintain Availability)?

(i) Chat Application — Choose AP (Availability + Partition Tolerance)

Justification: Users expect a chat app to always work. If WhatsApp showed "Service Unavailable" every time there was a minor network hiccup between data centers, users would switch to a competitor immediately. A slight delay or messages arriving slightly out of order is tolerable — humans are accustomed to this in real conversations (you sometimes receive a reply before the message it references). The key insight: eventual consistency is acceptable for chat. Messages will eventually be delivered and ordered correctly, but the app never goes down.

Real-life example: WhatsApp, Slack, and iMessage all prioritize AP. Messages might show one checkmark (sent but not confirmed) during a partition, but the app keeps working. You can still send and read messages. Once the partition heals, everything syncs up.

(ii) Banking Application — Choose CP (Consistency + Partition Tolerance)

Justification: A bank cannot show the wrong balance. If you have $100 and withdraw $80, the system must never show $100 to another terminal that then allows another $80 withdrawal — that would be a $60 loss for the bank. It is far better to say "Service temporarily unavailable, please try again" than to serve stale data that leads to double-spending. Users understand ATMs sometimes go offline — they do not understand losing money.

Real-life example: ATMs that say "Cannot connect to bank — transaction declined" during outages. Traditional banking systems (SWIFT, core banking platforms) all prioritize consistency. An ATM that always works but sometimes shows wrong balances would be a catastrophe.

Q3b. Explain how region splitting works in BigTable/HBase. Why is it necessary?

Q3b. BigTable/HBase Region Splitting 5 marks OFFICIAL

In HBase (and its predecessor Google BigTable), a table is horizontally partitioned into regions. Each region contains a contiguous range of row keys — for example, one region might hold all rows from "aardvark" to "elephant" and another from "elephant" to "zebra". Region splitting is the mechanism by which HBase automatically manages data growth.

  1. Region grows beyond threshold: As data is written, a region accumulates more and more rows. When it exceeds a configured size threshold (typically 256MB, configurable), the region server hosting it decides it is time to split.
  2. Split point determined: The region server identifies the middle row key — the key that divides the region's data roughly in half. This ensures the two resulting regions are approximately equal in size, preventing imbalanced splits.
  3. Two new child regions created: The original region is replaced by two non-overlapping daughter regions. If the original held rows [A, M), the daughters might be [A, G) and [G, M). Every row key in the original belongs to exactly one daughter — no data is lost or duplicated.
  4. Master notified and regions reassigned: The region server notifies the HBase Master of the split. The Master updates its metadata (the hbase:meta table) and may reassign the new regions to different region servers for load balancing. This is how HBase distributes hot data across the cluster.
  5. Transparent to clients: Clients are unaware of the split. Their next request is automatically routed to the correct new region via the updated metadata. Ongoing scans may need to be retried, but the client library handles this transparently.
Real-World Analogy

Imagine a single, massive phone book that covers an entire country. As the population grows, the book becomes too thick to handle — it falls apart, it is slow to search, and it takes up too much shelf space. So you split it: A-M goes into one volume, N-Z into another. Each volume goes to a different shelf (region server). If the A-M volume gets too thick again (maybe there are many people whose names start with "S" after a further split), you split again: A-F and G-M. This process continues recursively, automatically, and transparently. A person looking up "John Smith" does not need to know which volume he is in — the directory (Master/meta table) tells them.

Why is Splitting Necessary?

Performance: Smaller regions mean faster reads and writes — less data to scan within each region.

Parallelism: More regions can be distributed across more servers, enabling higher throughput.

Load balancing: Hot regions (receiving many reads/writes) can be split and distributed to reduce hotspots.

Scalability: Without splitting, a table would be limited to a single server's capacity. With splitting, it scales horizontally to thousands of servers.

Q3c. What uniquely identifies a cell in HBase?

Q3c. HBase Cell Identifier 3 marks OFFICIAL

Every cell in HBase is uniquely identified by a 3-tuple that acts as a coordinate in HBase's multidimensional sorted map:

Column (family:qualifier) A two-part identifier: the column family (defined at table creation, stored together on disk) and the qualifier (arbitrary, can be added dynamically). Together they name the column. "profile:email"

Real-World Analogy

Think of HBase as a massive archive of employee records. To find a specific piece of information, you need three things: the employee ID (row key), the document type and field (column family:qualifier, like "personal:phone_number"), and which version you want (timestamp — the employee may have changed phone numbers three times). Without all three coordinates, you cannot pinpoint a single piece of data. By default, you get the most recent version, just like asking HR for an employee's "current" phone number.

Q3d. How can Cassandra guarantee read-your-own-writes consistency? Explain the quorum approach with a concrete example.

Q3d. Cassandra Read-Your-Own-Writes 6 marks OFFICIAL

Read-your-own-writes means that after you write a value, any subsequent read you perform is guaranteed to see that write (or a later one). In Cassandra, which replicates data across multiple nodes, this is not guaranteed by default — you might write to nodes A and B, then read from node C, which has not received the update yet. The solution lies in quorum mathematics.

The Quorum Rule

R + W > N guarantees that any read quorum overlaps with any write quorum.

Where: N = total replicas, W = number of nodes that must acknowledge a write, R = number of nodes that must respond to a read.

If these sets overlap, at least one node in the read set must have the latest write. That node returns the newest timestamp, which "wins" over stale values from other nodes.

Concrete Example: N=3, W=2, R=2

Suppose a key is replicated to nodes {A, B, C} (N=3).

Write phase (W=2): You update the key's value to "42". Cassandra writes to 2 of the 3 replicas — say nodes A and B acknowledge. Node C has not been updated yet (it will be eventually, but not right now).

Read phase (R=2): You immediately read the key. Cassandra reads from 2 of the 3 replicas. No matter which 2 it picks, at least one must overlap with the write set {A, B}:

- Read from {A, B}: A has "42", B has "42" — returns "42".

- Read from {A, C}: A has "42", C has old value — "42" wins (newer timestamp).

- Read from {B, C}: B has "42", C has old value — "42" wins.

R + W = 2 + 2 = 4 > 3 = N. The overlap is mathematically guaranteed. There is no combination of 2 read nodes that avoids both write nodes.

N = 3 replicas: [A] [B] [C] Write W=2: [A✓] [B✓] [C ] ← 2 nodes have new value Read R=2, possible sets: {A, B} → overlap with write ✓ (both have new value) {A, C} → overlap with write ✓ (A has new value) {B, C} → overlap with write ✓ (B has new value) No R=2 set can avoid BOTH A and B → guaranteed overlap!

Alternative approach — Sticky sessions: Instead of quorum, route all requests from the same client to the same coordinator node. This coordinator always writes locally first, so it always has the latest value for that client's reads. Simpler, but creates potential hotspots and does not survive coordinator failure. The quorum approach is more robust and is the standard recommendation.

Real-World Analogy

Imagine passing a secret note in a classroom of 3 friends (N=3). You hand the note to 2 of them (W=2). Later, you ask 2 of the 3 to tell you the latest message (R=2). No matter which 2 you ask, at least 1 must be someone you gave the note to — because you can pick at most 1 person who did not receive it (there is only 1 such person, but you are asking 2). That overlapping friend will tell you the correct, latest message. If you only asked 1 friend (R=1), you might ask the one person who never got the note — and you would hear old, stale information.

Moodle Answer Strategy

For 6 marks, state the R + W > N rule clearly, work through a concrete numeric example (N=3, W=2, R=2), enumerate all possible read sets to prove overlap, and mention the alternative QUORUM or sticky session approach. A table showing different N/W/R configurations and their guarantees will impress the examiner.

Q3 Summary — NoSQL & Distributed Systems

CAP: partitions are inevitable, so choose AP (chat — always available, eventually consistent) or CP (banking — always correct, sometimes unavailable). HBase splits regions at middle row key when they exceed threshold — transparent, automatic, enables horizontal scaling. Cell = (row_key, family:qualifier, timestamp). Cassandra read-your-writes: ensure R + W > N so read and write quorums always overlap.
