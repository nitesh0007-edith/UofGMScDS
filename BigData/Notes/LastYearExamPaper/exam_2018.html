Question 1 — HDFS 25 marks
Q1a. Describe the HDFS Read Protocol step by step. 10 marks — 2 per step

HDFS Read Protocol — 5 Steps 10 marks

When a client wants to read a file from HDFS, the process is designed so that the NameNode (NN) only provides metadata — it never touches the actual data. This keeps the NN lightweight and prevents it from becoming a bottleneck, even when thousands of clients are reading simultaneously. The actual data flows directly from DataNodes (DNs) to the client, which is the key to HDFS's scalability.

  1. Client contacts NameNode: The client sends a read request to the NameNode, which looks up the file in its namespace. The NN responds with a complete list of all blocks that make up the file, along with the DataNode locations for each block and their replicas. This is purely a metadata operation — the NN knows where every block lives because DataNodes periodically send heartbeats and block reports. The NN sorts the DN locations by network distance from the client, so the client always knows which DN is closest. 2 marks
  2. Client selects closest DataNode per block: For each block, the client picks the nearest DataNode from the sorted list and opens a direct TCP connection to it. "Nearest" is determined by Hadoop's network topology awareness — same-node is closest, then same-rack, then same-datacenter, then cross-datacenter. The client contacts the DN directly, bypassing the NameNode entirely for data transfer. This is critical for performance: if the NN had to relay all data, it would become a massive bottleneck. 2 marks
  3. DataNode sends block data + metadata (checksum): The selected DataNode reads the block from its local disk and streams it to the client. Along with the raw data, the DN also sends checksum metadata — a pre-computed hash of the block data. This checksum was originally computed when the block was written and stored alongside the block on the DN's local filesystem. Sending the checksum alongside the data allows the client to verify data integrity end-to-end without a separate request. 2 marks
  4. Client verifies checksum — falls back to next replica on corruption: Upon receiving the block, the client independently computes a checksum of the received data and compares it against the checksum sent by the DN. If they match, the block is healthy and the client proceeds. If they do not match, the data is corrupt — the client reports the corrupted block to the NameNode (so it can schedule a re-replication), then seamlessly retries the read from the next closest replica in the list. This is completely transparent to the application using HDFS. 2 marks
  5. Process repeats for each block: The above steps are repeated for every block that constitutes the file. Blocks are read in order (block 0, then block 1, then block 2, etc.) to reconstruct the file sequentially. Each block may come from a different DataNode, which distributes the I/O load across the cluster and enables parallel reading when clients need it. The client assembles the blocks into the complete file. 2 marks
Real-World Analogy — Ordering Book Chapters from Warehouses

Imagine you want to read a 5-chapter book, but each chapter is stored in different Amazon warehouses across the country. You first call the central catalog office (the NameNode) and ask: "Where are all the chapters of this book?" The catalog office tells you: "Chapter 1 is in London, Manchester, and Edinburgh warehouses. Chapter 2 is in Bristol, Leeds, and Glasgow warehouses..." and so on. You then call the nearest warehouse for each chapter. When a chapter arrives, you check it against a photo of the original page (checksum). If the chapter is damaged, you call the backup warehouse. You never ask the catalog office to ship the chapters themselves — it only tells you where to find them.

Key Design Principle

Separation of metadata and data paths. The NameNode handles metadata (which blocks, which DNs) while DataNodes handle data transfer. This separation is what makes HDFS scale to thousands of simultaneous readers without the NN becoming a bottleneck.

Q1b. Describe the NameNode recovery mechanism, including all recovery resources. 15 marks — 5 per resource

The Problem: Why Recovery Matters

The NameNode is the single most critical component in HDFS. It holds the entire filesystem's directory tree, file-to-block mappings, and access permissions in memory. If the NameNode crashes and this state is lost, the entire cluster's data becomes inaccessible — the blocks still exist on DataNodes, but nobody knows which blocks belong to which files. HDFS solves this with three complementary recovery resources.

Three Recovery Resources 5 marks each

1. Image (In-Memory State)

The Image is the complete, live state of the NameNode held entirely in RAM. It contains the full filename namespace (directory tree), all file-to-block mappings, block-to-DataNode locations, replication counts, permissions, and quotas. It is kept in memory for lightning-fast lookups — every metadata operation (open file, list directory, check permissions) is served from this in-memory structure. However, RAM is volatile — if the NameNode process crashes or the machine loses power, the Image is completely lost. This is why the other two resources exist.

2. Journal (EditLog / Write-Ahead Log)

The Journal (also called the EditLog or WAL) is a write-ahead log stored on persistent disk storage. Every single operation that modifies the namespace — creating a file, deleting a directory, renaming, changing permissions, adding a block — is written to this log BEFORE it is applied to the in-memory Image. This "write-ahead" guarantee means that even if the NN crashes mid-operation, the journal on disk has a complete record of everything that happened. The journal grows continuously between checkpoints.

3. Checkpoint (FsImage)

The Checkpoint (stored as FsImage on disk) is a periodic, complete snapshot of the in-memory Image dumped to stable storage. Think of it as a "save game" — a frozen-in-time copy of the entire filesystem state at a specific moment. Checkpoints are created periodically to limit how much of the journal needs to be replayed during recovery. Without checkpoints, recovery would require replaying the entire journal from the very beginning of the filesystem's life, which could take hours or days on a large cluster.

Recovery Process

When the NameNode restarts after a crash, it performs the following recovery sequence:

  1. Load last checkpoint (FsImage) from disk into memory: This restores the filesystem state to the point when the checkpoint was created. It may be hours or even a day old, so it is not fully up-to-date.
  2. Replay all journal (EditLog) entries recorded after that checkpoint: Each journal entry represents a single namespace mutation. By replaying them in order on top of the checkpoint, the NameNode reconstructs exactly the state it had before the crash.
  3. Full state recovered: The in-memory Image is now identical to what it was before the crash. The NN can begin serving clients again.
Recovery Formula

Current State = Last Checkpoint + Replay(all Journal entries since that checkpoint)

This is the same principle used by databases everywhere — the checkpoint provides the baseline, and the journal provides the delta.

Secondary NameNode — The Checkpoint Helper

The Secondary NameNode is a separate process (often running on a different machine) whose sole job is to periodically merge the checkpoint (FsImage) with the journal (EditLog) to produce a new, up-to-date checkpoint. Here is the key nuance that examiners love to test:

The Secondary NameNode is NOT a hot standby. It does NOT take over when the primary NameNode fails. It cannot serve client requests. Its only purpose is housekeeping — by creating fresh checkpoints, it ensures that the journal stays short, which means recovery after a crash is fast (fewer entries to replay). Without the Secondary NameNode, the journal would grow indefinitely, and recovery could take a very long time.

Real-World Analogy — A Hospital Records System

Think of a hospital. The Image is the doctor's complete mental model of every patient's condition — fast to access but lost if the doctor suddenly forgets everything. The Checkpoint is the weekly patient status report that the hospital prints and files in a cabinet — a reliable snapshot, but potentially days out of date. The Journal is the daily logbook where every nurse writes down every medication change, every new symptom, every procedure performed — written down before anything is done to the patient. If the doctor loses their memory (crash), the hospital loads last week's printed report and replays this week's daily logs to reconstruct the full picture. The Secondary NameNode is like a medical secretary who, every few hours, takes the latest weekly report plus all the daily logs since then and produces a fresh combined report. The secretary doesn't treat patients — they just keep the paperwork manageable.

Q1 Summary

HDFS Read: Client asks NN for block locations, reads directly from nearest DN, verifies checksums, falls back to replicas on corruption. NameNode Recovery: Image (volatile RAM state) + Journal (write-ahead log on disk) + Checkpoint (periodic snapshot on disk). Recovery = load checkpoint + replay journal. Secondary NN merges checkpoint+journal into fresh checkpoints but is NOT a failover standby.

Question 2 — MapReduce 25 marks
Q2a. Given a web graph where each line is source_url → target_url1, target_url2, ..., write MapReduce pseudocode to count how many pages link TO each page (in-link count). 10 marks

Understanding the Problem

We have a dataset representing the link structure of the web. Each line describes one web page and the pages it links to. Our goal is to invert this relationship: instead of knowing "Page A links to Pages B, C, D", we want to know "Page B is linked to by 47 pages" — the in-link count (also called the in-degree in graph theory). This is exactly the kind of computation that drives Google's PageRank algorithm, which counts and weights incoming links to determine page importance.

Mapper 5 marks

The mapper's job is simple but crucial: for each line of input, it parses the source URL and all its target URLs, then emits each target as a key with the value 1. The source URL is deliberately ignored in the output because we do not care about who is doing the linking — we only care about who is being linked to.

map(key, value): // key = line offset (ignored) // value = "sourceURL\ttargetURL1, targetURL2, targetURL3, ..." parts = value.split("\t") source = parts[0] // we don't emit the source targets = parts[1].split(",") for each target in targets: emit(target.trim(), 1) // one "vote" for this target page

For example, if the input line is google.com → wikipedia.org, github.com, bbc.co.uk, the mapper emits three key-value pairs: (wikipedia.org, 1), (github.com, 1), and (bbc.co.uk, 1).

Reducer 5 marks

The MapReduce framework groups all values by key before handing them to the reducer. So the reducer receives a target page URL as the key and a list of 1s as the values — one 1 for every page that links to it. The reducer simply sums all the 1s to get the total in-link count.

reduce(key, values): // key = target page URL // values = [1, 1, 1, 1, ...] (one per in-link) total_inlinks = 0 for each v in values: total_inlinks += v emit(key, total_inlinks)

For example, if wikipedia.org appears as a target in 1,500 different source lines, the reducer receives (wikipedia.org, [1, 1, 1, ..., 1]) with 1,500 ones, and emits (wikipedia.org, 1500).

Real-World Analogy — Counting Social Media Followers

Think of Instagram. Each user's "following" list is one line of input. The mapper reads "Alice follows Bob, Charlie, Diana" and notes one follower-vote for each of Bob, Charlie, and Diana. After all users' following lists are processed, the reducer counts up all the votes for each person. Bob might end up with 500 follower-votes (meaning 500 people follow him). This is exactly the in-link count — the number of incoming edges in a directed graph.

Why This Works with MapReduce

The beauty of this approach is that it is embarrassingly parallel. Each mapper processes one chunk of the web graph independently. The framework handles grouping all the "votes" for the same target page together, regardless of which mapper emitted them. The reducer then performs a simple aggregation. No mapper needs to communicate with any other mapper — all coordination happens in the shuffle phase.

Q2b. How would you find the top-k most in-linked pages? 5 marks

Top-k Most In-Linked Pages 5 marks

After Q2a gives us the in-link count for every page, we need to find the k pages with the highest counts. There are two main approaches:

Approach 1: Chained MapReduce Job

Run a second MapReduce job that takes the output of Q2a as input. The mapper can invert key/value to emit (inlink_count, page_url). Use a single reducer (set number of reducers to 1) that maintains a sorted top-k list (e.g., a min-heap / priority queue of size k). In the cleanup() method, emit the final top-k entries.

Approach 2: Setup/Cleanup Pattern

Within a single MapReduce job, each mapper maintains a local top-k list using a priority queue. In cleanup(), each mapper emits its local top-k. A single reducer merges all local top-k lists into a global top-k. This is more efficient because it reduces the data flowing through the shuffle.

Real-World Analogy — Finding the Most-Followed Instagram Accounts

After counting all followers (Q2a), you want to find the top 10 most-followed accounts. Rather than sorting billions of accounts globally (expensive), each regional office (mapper) picks its local top 10, then a single headquarters (reducer) takes all the regional top-10 lists and finds the global top 10. This is much faster than a full global sort because only a tiny fraction of the data needs to travel to the reducer.

Q2c. Describe the data flow from Mapper to Reducer (the shuffle and sort phase) in detail. 10 marks

The Shuffle and Sort Phase — Detailed Data Flow 10 marks

The shuffle is the "magic middle" of MapReduce — it is where the framework takes the unordered output from many mappers and delivers it to reducers as sorted, grouped streams. It is often the most expensive phase of a MapReduce job because it involves disk I/O, network transfer, and sorting. Here is exactly what happens:

  1. Mapper writes to circular memory buffer (100MB default): As the map function emits key-value pairs, they are not written directly to disk. Instead, they go into an in-memory circular buffer, typically 100MB in size. This buffer acts as a staging area that batches output for efficiency. Using a circular buffer means the mapper does not need to allocate new memory for each output record — it just wraps around and reuses space. ~1 mark
  2. Spill triggered at 80% threshold: When the buffer is 80% full (the io.sort.spill.percent threshold), a background thread begins spilling the buffer contents to disk. The mapper can continue writing to the remaining 20% while the spill happens, avoiding a complete stall. If the mapper produces output faster than the spill thread can write, the mapper must wait — this is called back-pressure. ~1 mark
  3. Partition + sort before each spill: Before data is spilled to disk, it undergoes two critical transformations. First, each key-value pair is partitioned — a hash function is applied to the key, and the result modulo the number of reducers determines which reducer will receive this record. Second, within each partition, the records are sorted by key. This means each spill file on disk contains data already organized by destination reducer and sorted within each reducer's partition. ~2 marks
  4. Combiner runs on each spill (if configured): If the programmer has specified a combiner function (a local mini-reducer), it runs on each spill before the data is written to disk. The combiner performs partial aggregation — for example, in our in-link counting job, instead of spilling (wikipedia.org, 1), (wikipedia.org, 1), (wikipedia.org, 1), the combiner would collapse these into (wikipedia.org, 3). This dramatically reduces the amount of data written to disk and later transferred over the network. ~1 mark
  5. Multiple spills merged into a single sorted file per mapper: A mapper may produce many spill files over its lifetime (each time the buffer fills up). Once the mapper finishes, all spill files are merged into a single sorted file per partition using a multi-way merge sort. If there are many spill files, the merge may happen in multiple rounds. The combiner may run again during this merge. The result is one neatly organized file on the mapper's local disk, with sections for each reducer. ~1 mark
  6. Partitions transferred to reducers over HTTP (the shuffle): Each reducer contacts every mapper via HTTP and requests its partition of the data. This is the actual network transfer phase — the "shuffle" in the strictest sense. The framework uses a configurable number of parallel HTTP connections (default: 5) so that reducers can fetch data from multiple mappers simultaneously. Data transfer begins as soon as mappers finish, even while other mappers are still running. ~2 marks
  7. Reducer merges sorted runs from multiple mappers: Each reducer now has one sorted file from each mapper. It performs a multi-way merge sort to combine all these files into a single sorted stream. This is efficient because each input file is already sorted — the merge just picks the smallest key across all files at each step. If the combined data is too large for memory, intermediate merge files are spilled to disk. ~1 mark
  8. Final sorted input fed to reduce function: The fully merged, sorted stream is fed to the reduce() function. All values for the same key are guaranteed to be adjacent (because the stream is sorted), so the framework can efficiently group them and call the reducer with (key, [list of all values]). The reduce function processes each group and writes its output to HDFS. ~1 mark
Real-World Analogy — A National Postal Sorting System

Imagine a country's postal system. Each post office (mapper) collects outgoing letters (mapper output) into a mailbox (circular buffer). When the mailbox is nearly full, a postal worker empties it (spill), sorting letters by destination ZIP code (partitioning) and alphabetically within each ZIP code (sorting). If there is a local assistant (combiner), they consolidate duplicate packages before the truck arrives. Multiple batches from the day are merged into one neat stack per ZIP code. Trucks (HTTP shuffle) then deliver each ZIP code's stack to the regional sorting center (reducer). Each regional center receives stacks from every post office in the country and merges them all into one final alphabetical sequence (merge sort). The mail carrier (reduce function) then delivers each person's complete bundle of letters in order.

Performance insight: The shuffle phase is often the bottleneck of a MapReduce job because it involves both disk I/O (spills, merges) and network I/O (transfer to reducers). This is why combiners are so valuable — they reduce the data volume at the source, before it hits the network. A well-designed combiner can reduce shuffle data by 10x or more.

Q2 Summary

In-link counting: Mapper emits (target, 1) for each link; Reducer sums to get total in-links per page. Top-k: Chain a second job or use setup/cleanup pattern with a priority queue and a single reducer. Shuffle: Buffer (100MB) → spill at 80% → partition + sort → combiner → merge spills → HTTP transfer to reducers → merge sort → reduce function.

Question 3 — Storage Systems 25 marks
Q3a. Explain the LSM Tree and why it provides high write throughput. 8 marks

LSM Tree — Log-Structured Merge Tree 8 marks

The Log-Structured Merge Tree (LSM Tree) is a data structure designed for workloads with far more writes than reads. It is the storage engine behind HBase, Cassandra, LevelDB, RocksDB, and many other modern databases. The fundamental insight is one sentence long but worth 2 marks on its own:

The Key Insight

LSM Trees convert random writes into sequential writes. On spinning disks, sequential writes are 100-1000x faster than random writes. On SSDs, the difference is smaller but still significant (5-10x). By never updating data in place, LSM trees avoid the random I/O penalty that cripples B-tree-based storage engines under heavy write loads.

Here is how the write path works:

  1. Write to in-memory MemTable (fast): All incoming writes go directly into an in-memory sorted data structure called the MemTable (typically a red-black tree or skip list). Writing to memory is extremely fast — there is no disk seek involved. Simultaneously, the write is appended to a Write-Ahead Log (WAL) on disk for durability. The WAL is a simple append-only file — also sequential, so it is fast too. 2 marks
  2. MemTable flushed as sorted SSTable to disk: When the MemTable reaches a size threshold (e.g., 64MB), it is frozen and flushed to disk as an SSTable (Sorted String Table) — an immutable, sorted file. Because the MemTable is already sorted in memory, this flush is a single sequential write to disk, which is the fastest possible disk operation. A new empty MemTable is created to accept further writes. 2 marks
  3. No random disk I/O for writes: Unlike a B-tree, which must read a page from disk, modify it, and write it back (random read + random write for every update), the LSM tree never performs random disk I/O on the write path. Every disk operation is either an append (WAL) or a sequential write (SSTable flush). This is why LSM trees achieve dramatically higher write throughput than B-trees. 2 marks
  4. Background compaction keeps reads reasonable: Over time, many SSTables accumulate on disk. Reading becomes slower because a lookup may need to check multiple SSTables (since the same key could exist in several). To mitigate this, a background compaction process periodically merges multiple SSTables into fewer, larger ones, discarding deleted and overwritten entries. Compaction is essentially a merge sort of sorted files — it is sequential I/O and does not block writes. 2 marks

LSM Tree vs B-Tree — Trade-offs

AspectLSM TreeB-Tree
Write speedVery fast (sequential I/O only)Slower (random I/O to update pages)
Read speedPotentially slower (may check multiple SSTables)Fast (single tree traversal)
Write amplificationHigher (data written multiple times due to compaction)Lower (write once to correct page)
Space amplificationMay have temporary duplicates across SSTablesMinimal (in-place updates)
Best forWrite-heavy workloads (logging, time-series, IoT)Read-heavy workloads (OLTP, indexing)
Real-World Analogy — Two Note-Taking Systems

B-Tree approach: You have a massive, alphabetically organized notebook. Every time you need to update a fact, you find the right page (random seek), erase the old entry, and write the new one. Finding the right page takes time, especially if the notebook has thousands of pages.

LSM Tree approach: Instead, you jot every new fact on a fresh sticky note (MemTable) and toss it into an inbox. When the inbox is full, you sort all the sticky notes alphabetically and staple them into a neat packet (SSTable flush). You never search through old packets to update them — you just write a new note. Later, during a quiet period, you merge multiple packets into one clean, updated master packet (compaction). Writing is blazing fast because you never search — but reading may require checking several packets until you find the latest version of a fact.

Q3b. Describe the HBase on-disk storage format in detail. 9 marks

HBase On-Disk Storage Format 9 marks

HBase stores its data on HDFS using a file format called HFile, which is based on the SSTable (Sorted String Table) concept from Google's Bigtable paper. Understanding the HFile format is essential because it determines how HBase achieves fast lookups on top of an append-only distributed filesystem.

HFile Structure 3 marks

Each HFile is a self-contained, immutable file on HDFS with the following internal structure:

SectionPurposeLocation in File
Data BlocksContain the actual sorted key-value pairs. Each block is typically 64KB. Multiple blocks make up the data section.Beginning of file
Meta BlocksStore optional metadata, most importantly Bloom filters for fast negative lookups.After data blocks
File InfoGeneral file-level metadata: average key length, compression codec, comparator class, etc.After meta blocks
Data IndexAn index mapping key ranges to data block offsets. Enables binary search to jump directly to the right block.Near end of file
Meta IndexAn index for locating meta blocks (Bloom filters, etc.) within the file.Near end of file
TrailerA fixed-size block at the very end of the file containing pointers (offsets) to all the other sections. This is read first when opening an HFile.End of file

Key-Value Sort Order 2 marks

Within data blocks, key-value pairs are sorted by a composite key in the following order:

(row_key, column_family, column_qualifier, timestamp DESC)

The descending timestamp order means that the most recent version of a cell appears first, which makes "get latest version" queries very efficient — HBase just reads the first entry it finds for a given row+column combination.

Block Index — Efficient Lookups 1 mark

The Data Index records the first key of each data block. To find a specific key, HBase performs a binary search on the index to identify which data block contains the target key, then reads only that one block from disk. This means reading a single key from a multi-gigabyte HFile requires loading only the index (small, often cached in memory) and one 64KB data block — not the entire file.

Bloom Filters — Quick Negative Lookups 1 mark

Bloom filters are probabilistic data structures stored in the meta blocks of each HFile. Before performing an expensive disk read, HBase checks the Bloom filter to determine if the requested key could possibly exist in this HFile. If the Bloom filter says "no", the HFile is skipped entirely — saving a disk seek. Bloom filters can have false positives (saying "maybe" when the key is absent) but never false negatives (they never say "no" when the key is actually present).

Column Family Separation 1 mark

Each column family is stored in its own set of HFiles (called a "Store" in HBase terminology). This means accessing one column family never requires reading HFiles belonging to another column family. This is a major advantage for wide tables where queries typically access only a few column families — only the relevant HFiles are read, not the entire row's data.

Real-World Analogy — A Well-Organized Library

Think of each HFile as a book in a library. Each book (HFile) contains: a table of contents at the back (the Data Index) that tells you which chapter covers which topic; a quick-reference card (Bloom filter) on the front cover that lets the librarian instantly say "this book definitely does NOT contain information about Topic X" without opening it; and inside, the content (Data Blocks) is organized alphabetically within each genre section (Column Family). Books within the same genre are shelved together, and you never need to open a Science book when looking for History.

Q3c. Explain Cassandra's consistency levels and the trade-offs involved. 8 marks

Cassandra Consistency Levels 8 marks

Cassandra is a distributed database where data is replicated across multiple nodes for fault tolerance. The key question is: when a client reads or writes data, how many replicas must participate before the operation is considered successful? This is controlled by consistency levels, and Cassandra is unique in that it lets you set this per query — giving you fine-grained control over the consistency-availability trade-off.

The Quorum Formula 2 marks

Strong Consistency Guarantee

R + W > N

Where: N = replication factor (total number of replicas of each data item), R = number of replicas that must respond to a read, W = number of replicas that must acknowledge a write.

When R + W > N, the read set and write set must overlap — at least one node in any read will have seen the latest write. This guarantees the client always sees the most recent data.

Consistency Level Options 4 marks

LevelReplicas RequiredConsistencyAvailabilityLatency
ONE1 replica respondsWeakest — may read stale dataHighest — works even if N-1 replicas are downLowest
QUORUMMajority: floor(N/2) + 1Strong (when both R and W are QUORUM)Good — tolerates minority failuresModerate
ALLAll N replicas must respondStrongest — guaranteed latest dataLowest — fails if any single replica is downHighest
LOCAL_QUORUMQuorum within local datacenterStrong within datacenterGood for multi-DC deploymentsLow (no cross-DC latency)

With N=3 (the most common configuration): ONE requires 1 replica, QUORUM requires 2 replicas, ALL requires 3 replicas.

The Trade-off 2 marks

Every consistency level represents a point on the consistency-availability-latency spectrum:

  • Higher consistency (more replicas) = lower availability + higher latency. If you require ALL replicas to respond, the operation fails if even one replica is unreachable. And you must wait for the slowest replica (the "tail latency" problem).
  • Lower consistency (fewer replicas) = higher availability + lower latency. With ONE, the operation succeeds as long as any single replica is alive, and you get the response as fast as the quickest replica can answer. But you might read stale data.
  • QUORUM is the sweet spot for most applications — it provides strong consistency (when used for both reads and writes) while tolerating minority failures. With N=3 and QUORUM (R=2, W=2): 2+2=4 > 3, so strong consistency is guaranteed, and the system tolerates 1 node failure.
  • Tunable per query: The most powerful aspect is that Cassandra lets you choose different consistency levels for different operations. You might use QUORUM for financial transactions but ONE for social media feed reads where occasional staleness is acceptable.
Real-World Analogy — Group Decision Making

ONE is like asking one friend if a restaurant is good — you get an instant answer, but it might not reflect the latest reviews. QUORUM is like polling a majority of your friend group — "3 out of 5 of you say it's good? Let's go." It takes a bit longer but you get a reliable consensus. ALL is like requiring unanimous agreement from every single friend before going out — you get the most informed decision, but if even one friend's phone is off, you're stuck waiting. LOCAL_QUORUM is like only polling friends in your city — faster (no long-distance calls) but you miss input from friends in other cities.

CAP Theorem connection: Cassandra is fundamentally an AP system (Availability + Partition tolerance) — it defaults to being always available, even at the cost of consistency. However, by using QUORUM or ALL consistency levels where R + W > N, you can effectively make it behave like a CP system for specific queries. This tunability is Cassandra's key differentiator from systems like HBase (which is strictly CP).

Q3 Summary

LSM Tree: converts random writes to sequential — writes go to MemTable + WAL, flush as sorted SSTables, background compaction keeps reads fast. Trade-off: faster writes, potentially slower reads than B-trees. HBase HFiles: data blocks + meta blocks + file info + data index + meta index + trailer; sorted by (row, CF, qualifier, timestamp DESC); Bloom filters for negative lookups; column families in separate HFiles. Cassandra: R+W>N for strong consistency; ONE (fast/weak) vs QUORUM (balanced) vs ALL (slow/strong); tuneable per query; higher consistency = lower availability + higher latency.
