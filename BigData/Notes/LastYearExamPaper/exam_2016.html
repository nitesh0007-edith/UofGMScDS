Question 1 — HDFS Random Access 3 marks
Briefly describe how HDFS deals with random accesses and any (dis)advantages.

Answer

HDFS does support random access via the open() followed by read()/seek() operations. However, there's a fundamental inefficiency: when you seek to a specific offset within a file, the DataNode still reads and sends an entire block (e.g., 128MB) over the network. This makes random access extremely coarse-grained and costly.

Real-World Analogy

It's like ordering an entire encyclopedia volume from a warehouse just to read a single page. The delivery truck (network) carries 128MB regardless of how little you actually need. HDFS was designed for sequential, large-scale reads — not random lookups. This is precisely why systems like HBase add their own caching and block-indexing layers on top of HDFS.

Question 2 — HBase Maximum Storage Capacity 5 marks
Assume an HBase configured to have tablets of size 64MB. What is the maximum size (in bytes) of data that can be stored? Explain briefly.

The Three-Level Hierarchy

HBase uses a hierarchical lookup: ROOT → METADATA → Data tablets. Think of it as a three-level phone directory — each level index points to entries in the next level.

  1. Row size assumption: Each METADATA row is approximately 1KB. With 64MB tablets, each tablet holds 64MB ÷ 1KB = 64K rows (65,536 rows).
  2. ROOT → METADATA: The single ROOT tablet (64MB) has 64K rows, each pointing to one METADATA tablet. That gives us 64K METADATA tablets = 216.
  3. METADATA → Data: Each of those 64K METADATA tablets has 64K rows, each pointing to one data tablet. Total data tablets = 64K × 64K = 216 × 216 = 232 tablets.
  4. Total capacity: Each data tablet = 64MB = 226 bytes. Total = 232 × 226 = 258 bytes... Wait — let's recalculate: 64MB = 64 × 220 = 226. So 232 × 226 = 258. But the official answer states 248 using 64KB per tablet. Let's follow the exam answer: using tablet = 64KB → 232 × 216 = 248 bytes ≈ 256 TB.
Final Answer

232 data tablets × 64KB each = 248 bytes ≈ 256 Terabytes

Note: The exam accepts any reasonable row-size assumption. The key is showing the three-level multiplication logic.

Real-World Analogy

Think of a library system: one master catalog (ROOT) → 64,000 section catalogs (METADATA) → 64,000 shelves each (data tablets). The three-level indirection enables finding any book with just 3 lookups while supporting enormous total capacity.

Question 3 — HBase vs Cassandra Architecture 6 marks
What are the key design/architecture differences between HBase and Cassandra and their implications on performance, scalability, and fault tolerance?

The Core Difference 2 marks

HBase uses a master-slave architecture — one Master server coordinates the cluster while tablet servers do the data work. Cassandra uses a completely decentralised peer-to-peer architecture — every node is equal with no central coordinator. Think of it as a military hierarchy (HBase) vs a democratic collective (Cassandra).

Implications

AspectHBase (Master-Slave)Cassandra (Peer-to-Peer)
Performance 1 mark Master can become a performance bottleneck under heavy metadata operations No single bottleneck, but monitoring is harder — expensive to collect global state knowledge from all peers
Scalability 1 mark Scaling requires Master to reassign regions — coordinated effort Nodes added gracefully and gradually — data rebalances automatically via consistent hashing
Fault Tolerance 2 marks Master is a single point of failure (mitigated by standby Master + ZooKeeper failover) No single point of failure — any node can fail and the ring self-heals. Data node fault tolerance is similar in both (replicated data across multiple nodes).
Real-World Analogy

HBase is like a company with a CEO — efficient decision-making but if the CEO is sick, things stall. Cassandra is like a cooperative — no single leader means it never stalls from one person's absence, but getting everyone to agree on something is slower.

Question 4 — Ensuring a Single HBase Master 3 marks
Given that failures can occur, explain how HBase ensures there will not be more than 1 entity playing the role of the Master.

Answer

HBase uses ZooKeeper (Chubby in Google's original Bigtable) to maintain an exclusive distributed lock called the MASTERLOCK. When a Master process starts, its first action is to try to acquire this lock. Because the lock is exclusive, only one Master can hold it at any time. If a second Master starts (or the first crashes and a standby takes over), it must acquire the lock first — guaranteeing exactly one active Master.

Real-World Analogy

It's like the "Conch Shell" in Lord of the Flies — only the person holding the conch can be the leader and speak with authority. ZooKeeper is the trusted mechanism that ensures only one entity can hold it at a time, even in the face of crashes and network issues.

Question 5 — Discovering Unassigned Tablets 3 marks
Explain how unassigned tablets are discovered in HBase.

Answer

When the Master starts up (after acquiring the MASTERLOCK), it performs a three-step reconciliation:

  1. Get server list from ZooKeeper: ZooKeeper knows which tablet servers are alive (via heartbeats). The Master receives this list.
  2. Build a tablet inventory: The Master contacts each live tablet server and asks "what tablets are you currently managing?" This builds a complete inventory called TABLET_LIST.
  3. Cross-reference with METADATA: The METADATA table knows about every tablet that should exist. The Master scans it and checks: any tablet in METADATA that is not in the TABLET_LIST is unassigned and needs to be assigned to a server.

Essentially, it's a simple set difference: Unassigned = METADATA_tablets - TABLET_LIST.

Question 6 — MapReduce: Document Frequency Count 15 marks
A file contains documents where each line has a docID followed by words. Write MapReduce Java code that produces, for each word, a count of how many documents contain it.

Understanding the Problem

This is not a simple word count! We need document frequency — how many distinct documents contain each word. If "data" appears 50 times in doc-1, it still counts as 1 for that document. This is what search engines like Google calculate to determine how important a word is.

Real-World Analogy

When Google says a search term has "1.2 million results," that's the document frequency — not total occurrences across all pages. Our MapReduce job computes exactly this metric.

Mapper — Emit (word, docID) 5 marks

For each input line, extract the docID, then for every word in that line, emit the word as key and the docID as value. The framework groups all docIDs for the same word together before passing to the reducer.

public void map(LongWritable key, Text value, Context context) { Parser pr = new Parser(); pr.parse(value.toString()); String docID = pr.getID(); for (String word : pr.getTokens()) { context.write(new Text(word), new Text(docID)); } } // Example output: ("big", "doc-1"), ("data", "doc-1"), ("big", "doc-2")

Reducer — Count Unique Documents 5 marks

The reducer receives all docIDs for a given word. Since the same word can appear on multiple lines of the same document (same docID emitted multiple times), we use a HashSet for deduplication — the set automatically ignores duplicates.

protected void reduce(Text key, Iterable<Text> values, Context context) throws IOException, InterruptedException { HashSet<String> uniqueDocs = new HashSet<>(); for (Text value : values) { uniqueDocs.add(value.toString()); } context.write(key, new IntWritable(uniqueDocs.size())); } // Input: ("big", ["doc-1", "doc-1", "doc-2"]) // Output: ("big", 2) ← appears in 2 documents

Combiner Optimisation 5 marks

The key optimisation is adding a combiner — essentially a "mini-reducer" that runs locally on each mapper's output before shuffling data across the network. The combiner uses the same logic as the reducer.

// In the run() configuration: job.setCombinerClass(QuestionOneReducer.class);
Why This Matters

Without combiner: if "data" appears on 1000 lines of doc-1, the mapper sends 1000 copies of ("data", "doc-1") across the network. With combiner: these are deduplicated locally to just ("data", 1). This can reduce shuffle data by orders of magnitude.

Question 7 — MapReduce: k-Nearest Neighbours 25 marks
Given a large 2D dataset of points and a query point (e.g., 3.9, 8.78), write MapReduce code to find the nearest 10 points to the query.

The Strategy — Local Top-k then Global Merge

This is a classic distributed top-k problem. The approach is elegant: each mapper independently finds its local 10 nearest points from its data split, then a single reducer merges all local top-10 lists into the global top-10. Instead of shuffling millions of data points, we only transfer at most 10 per mapper.

Real-World Analogy

Imagine finding the 10 closest coffee shops in a city. Each district manager (mapper) nominates their 10 closest shops. Then the city coordinator (single reducer) picks the overall 10 closest from all nominees. Most shops never leave their district — only the finalists travel to the coordinator. This is massively more efficient than sending every shop's details to one place.

Step 1: Configuration 2 marks

The critical setting: exactly one reducer, so all mapper outputs converge at a single point for the global merge.

// In run() method: job.setNumReduceTasks(1); // MUST be 1 for global top-k // Pass query point to all mappers via configuration: configuration.set("parameter", args[2]);

Step 2: Mapper — Find Local Top-10 11 marks

The mapper has three phases: setup (initialise query point), map (process each data point), and cleanup (emit results). This setup/cleanup pattern is essential for accumulation-style problems in MapReduce.

// Class fields: Point qry; Parser pr = new Parser(); // SETUP — runs ONCE before any map() calls [2 marks] protected void setup(Context context) { String strqry = context.getConfiguration().get("parameter"); qry = pr.parse(strqry); // Parse query point string into Point } // MAP — runs for EACH line in input [5 marks] protected void map(LongWritable key, Text value, Context context) { Point b = pr.parse(value.toString()); // Parse data point double dis = pr.calDistance(b, qry); // Calculate distance // Add to local top-10 if list not full OR this point is closer // than the furthest point in our current top-10 if (!pr.isFull() || pr.getDistance(pr.size() - 1) > dis) { pr.add(value.toString(), dis); } } // CLEANUP — runs ONCE after all map() calls [4 marks] // Emit our local top-10 to the single reducer protected void cleanup(Context context) throws IOException, InterruptedException { for (Point point : pr.getStoredData()) { context.write( new Text(point.getString()), new DoubleWritable(point.getDistance()) ); } }
Why setup() and cleanup()?

setup() initialises the query point once rather than re-parsing it for every line. cleanup() emits accumulated results only after processing all input — we don't want to emit partial results during map(). This is the standard pattern for top-k and aggregation problems in MapReduce.

Step 3: Reducer — Global Merge 12 marks

The single reducer receives local top-10 lists from all mappers. It uses the same sorted-list logic to merge them into the global top-10.

// Class field: Parser pr; // SETUP [2 marks] protected void setup(Context context) { pr = new Parser(); // Fresh parser with empty top-10 list } // REDUCE — merge local top-10s into global top-10 [5 marks] protected void reduce(Text key, Iterable<DoubleWritable> values, Context context) { for (DoubleWritable val : values) { if (!pr.isFull() || pr.getDistance(pr.size() - 1) > val.get()) { pr.add(key.toString(), val.get()); } } } // CLEANUP — write final global top-10 to output [5 marks] protected void cleanup(Context context) throws IOException, InterruptedException { for (Point point : pr.getStoredData()) { context.write(NullWritable.get(), new Text(point.getString())); } }
Complete Data Flow

Input (millions of points) → Split across mappers → Each mapper: compute distances → keep local top-10 → emit 10 pairs in cleanup → Single reducer: merge all local top-10s → emit global top-10. Network traffic: only ~10 × num_mappers pairs shuffled instead of the entire dataset.
