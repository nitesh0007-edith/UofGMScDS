Question 1 — HDFS Client Read Protocol & Concepts 20 marks
(a) Describe the steps involved in the HDFS client read protocol. (b) Identify which statements about core HDFS concepts are true or false.

Part (a): HDFS Client Read Protocol 10 marks

When a client application wants to read a file stored in HDFS, it follows a carefully designed protocol that keeps the NameNode out of the data transfer path entirely. This separation of metadata operations (handled by NameNode) from data operations (handled by DataNodes) is the single most important architectural decision in HDFS — without it, the NameNode would become a crippling bottleneck as soon as you scale to hundreds of clients reading simultaneously.

  1. Client calls open() on DistributedFileSystem: The client application initiates a read by calling the open() method on the DistributedFileSystem object. This is the entry point — like picking up the phone to call the library's front desk before visiting.
  2. DistributedFileSystem contacts NameNode via RPC: Under the hood, the DFS object makes a Remote Procedure Call (RPC) to the NameNode, requesting the locations of all blocks that make up the target file. The NameNode looks up its in-memory metadata and returns the block IDs along with the list of DataNodes hosting each block's replicas.
  3. Block locations sorted by proximity (rack-awareness): The NameNode doesn't just return random DataNode addresses — it sorts them by network distance from the client. If the client is on Rack A, a DataNode on Rack A comes first, then one on Rack B (same data center), then one in a different data center. This is HDFS's rack-aware optimization that minimizes network hops.
  4. Client receives FSDataInputStream wrapping DFSInputStream: The client gets back a stream object. The FSDataInputStream is the public API wrapper, while the DFSInputStream underneath manages the actual block-level communication with DataNodes. The client doesn't need to know about blocks at all — it just reads bytes.
  5. DFSInputStream connects to closest DataNode for first block: The stream automatically connects to the nearest DataNode for Block 0 of the file. This happens transparently — the client just calls read() and data arrives.
  6. Data streams directly from DataNode to client: Data flows DataNode → DFSInputStream → client application. The NameNode is completely uninvolved at this stage. This is critical: if 1000 clients are reading files simultaneously, they each talk to different DataNodes directly, and the NameNode is not stressed at all.
  7. End of block — close connection, move to next block: When all bytes of the current block have been read, the DFSInputStream closes its connection to that DataNode and opens a new connection to the closest DataNode for the next block. This block-hopping is invisible to the client application.
  8. Transparent failure handling: If a DataNode becomes unreachable mid-read, the DFSInputStream silently switches to the next closest replica of that block. The client never sees an error — it just continues reading without interruption.
  9. Checksum verification at the client: After receiving block data, the client verifies it against the stored checksum. If there's a mismatch (data corruption), the DFSInputStream reports the bad block to the NameNode and retries with the next replica. This means HDFS can self-heal from silent data corruption.
  10. Client calls close() on FSDataInputStream: When done reading, the client closes the stream, which releases all DataNode connections.
Real-World Analogy: The Library Catalog

Think of HDFS read like visiting a massive university library. You walk to the information desk (NameNode) and ask "Where is the book 'Distributed Systems'?" The librarian checks the catalog (metadata) and tells you: "Chapter 1 is on Shelf 3A, Row B. Chapter 2 is on Shelf 7C, Row D. Chapter 3 is on Shelf 2A, Row A — and that's the closest shelf to where you're standing." You then walk to each shelf yourself and pick up the chapters directly. The librarian never leaves the desk, never carries a single page for you — they only pointed you in the right direction. If Shelf 3A is locked (DataNode failure), the librarian also told you "there's a backup copy on Shelf 5B" — so you just go there instead.

Why This Architecture Matters

NameNode is NOT in the data path. This is the single most exam-critical point. If the NameNode had to relay every byte of data between DataNodes and clients, it would become a bottleneck at ~100 clients. By only serving lightweight metadata (block locations = a few kilobytes), the NameNode can serve thousands of clients per second. The actual heavy lifting (terabytes of data transfer) happens peer-to-peer between clients and DataNodes.

Common Exam Pitfall: Students sometimes write that "the NameNode sends data to the client" — this is WRONG and will cost marks. The NameNode only sends metadata (block locations). All actual data flows directly from DataNodes to the client.

Part (b): HDFS Core Concepts — True/False 10 marks

These statements test your understanding of how HDFS components work together. Each one touches on a fundamental design principle that makes HDFS capable of storing petabytes of data reliably across thousands of commodity machines.

StatementVerdictExplanation
NameNode stores ALL metadata in memory TRUE The entire namespace (directory tree, file-to-block mappings, permissions) lives in RAM for sub-millisecond lookups. This is why NameNode needs a beefy machine with lots of RAM — a cluster with 100 million files might need 60+ GB of heap.
DataNodes store actual data blocks on local disks TRUE Each DataNode stores blocks as regular files on its local Linux filesystem (typically ext4 or xfs). A block = one file on the DN's disk. DataNodes are cheap commodity machines with lots of disk space but modest RAM.
HDFS files are write-once by default TRUE The original HDFS design only supported create-and-write-once-read-many. Append support was added later (HDFS-265), but random writes are still not supported. This simplifies replication enormously — you never have to worry about two writers modifying the same block simultaneously.
Default replication factor = 3 TRUE Each block is stored on 3 different DataNodes by default. The placement policy: 1st replica on the writer's node (or a random node), 2nd replica on a different rack, 3rd replica on the same rack as the 2nd but a different node. This survives both node failures AND entire rack failures.
NameNode is a single point of failure TRUE If the NameNode dies, the entire filesystem is inaccessible (you can't find any blocks without the metadata). Mitigated by HA mode: a Standby NameNode keeps a synchronized copy and takes over within seconds using ZooKeeper for leader election.
DataNodes send heartbeats every 3 seconds TRUE Every 3 seconds, each DN sends a heartbeat to the NN saying "I'm alive." If the NN misses 10 consecutive heartbeats (~30 seconds), it declares that DN dead and begins re-replicating its blocks to other DNs to restore the replication factor.
Block reports are sent periodically to NameNode TRUE Every ~6 hours (configurable), each DN sends a full block report — a list of every block ID stored on that DN. The NN uses this to verify that its in-memory metadata matches reality. This is also how the NN rebuilds its block-to-DN mapping after a restart.
NameNode makes all replication decisions TRUE The NN alone decides when to replicate (under-replicated block), where to place replicas (rack-aware placement), and when to delete excess replicas (over-replicated). DataNodes just follow orders — they never initiate replication on their own.
Data never flows through the NameNode TRUE This is the golden rule of HDFS architecture. The NN only handles metadata. All data transfer happens directly between clients and DataNodes (for reads) or through DataNode pipelines (for writes). This separation is what makes HDFS scalable to thousands of concurrent clients.
Analogy: The Airport Control Tower

The NameNode is like an airport control tower. It knows where every plane (block) is, which runway (DataNode) it's on, and coordinates takeoffs and landings (replication decisions). But the control tower never carries passengers or cargo itself — that's the planes' job. If the control tower goes dark, no plane can land or take off (single point of failure), but the planes already in the air keep flying (data already on DataNodes is safe, just inaccessible until the NN recovers).

Question 2 — MapReduce Job Design for DISTINCT 20 marks
(a) Design a MapReduce job that implements the DISTINCT operation on a specific column of a table/file. (b) Design a MapReduce job that performs aggregation with DISTINCT — grouping by one column and counting distinct values of another column.

Part (a): MapReduce for DISTINCT 10 marks

The DISTINCT operation removes duplicate values from a dataset, returning only unique entries. In SQL, this is SELECT DISTINCT column FROM table. The beauty of implementing this in MapReduce is that the framework's shuffle and sort phase already groups identical keys together — so deduplication comes almost for free if you choose your key wisely.

The Core Insight

Make the target column value the MAP KEY. Since MapReduce guarantees that all values sharing the same key are routed to the same reducer, and the reducer is called exactly once per unique key, simply emitting the column value as the key automatically deduplicates it. The reducer just outputs the key and ignores the values entirely.

Mapper (5 marks): Parse each input line, extract the target column value, and emit it as the key. The value is irrelevant — we use NullWritable to save bandwidth.

// Mapper: emit column value as key, null as value map(key, line): fields = line.split(",") // or whatever delimiter target_value = fields[TARGET_COL] // extract the column we want DISTINCT on emit(target_value, NullWritable) // key = column value, value = nothing

Reducer (5 marks): The reducer receives each unique key exactly once (with a list of null values). It simply outputs the key — that IS the distinct value.

// Reducer: output each unique key once reduce(key, values): emit(key, NullWritable) // just write the key — it's already unique // We completely ignore 'values' — they're all NullWritable anyway
Real-World Analogy: Sorting Mail by ZIP Code

Imagine a post office needs to know which ZIP codes received mail today. Workers (mappers) read each letter's ZIP code and toss it into a labeled bin. Each bin represents a unique ZIP. At the end, a supervisor (reducer) walks past the bins and writes down each label — that's the list of distinct ZIPs. They don't count letters, don't open them — just note which bins exist. The physical sorting (shuffle & sort) did the deduplication automatically.

Walkthrough example: Suppose we have a file of customer orders and want distinct cities:

Input LineMapper Output (key, value)
Order1, Alice, London("London", null)
Order2, Bob, Paris("Paris", null)
Order3, Carol, London("London", null)
Order4, Dave, Tokyo("Tokyo", null)
Order5, Eve, Paris("Paris", null)

After shuffle & sort, the reducer sees: ("London", [null, null]), ("Paris", [null, null]), ("Tokyo", [null]). Output: London, Paris, Tokyo. Three distinct cities — duplicates eliminated.

Why not use a Combiner? You can add a combiner (identical to the reducer) to reduce network traffic during the shuffle phase. Each mapper node would locally deduplicate before sending data to reducers. This is an optimization, not a requirement — but mentioning it in the exam shows deeper understanding.

Part (b): MapReduce for GROUP BY with COUNT DISTINCT 10 marks

This is the more complex variant: SELECT group_col, COUNT(DISTINCT value_col) FROM table GROUP BY group_col. We need to group rows by one column and, within each group, count how many unique values exist in another column. This requires the reducer to actively track uniqueness using a data structure.

Mapper: Emit the grouping column as the key and the column we want to count distinct values of as the value.

// Mapper: key = group column, value = column to deduplicate map(key, line): fields = line.split(",") group_value = fields[GROUP_COL] // e.g., "department" dedup_value = fields[VALUE_COL] // e.g., "employee_city" emit(group_value, dedup_value)

Reducer: For each group key, collect all values into a HashSet (which automatically deduplicates), then output the set's size as the count.

// Reducer: use HashSet for deduplication, output count reduce(group_key, values): HashSet<String> unique = new HashSet<>() for each v in values: unique.add(v) // duplicates are silently ignored by the set emit(group_key, unique.size()) // count of distinct values
Real-World Analogy: Teacher Counting Unique Students

Imagine a school where multiple teachers (mappers) grade homework. Each teacher writes a slip: "Subject: Maths, Student: Alice", "Subject: Maths, Student: Bob", "Subject: Maths, Student: Alice" (Alice submitted twice). The slips are sorted into piles by subject (shuffle). The head teacher (reducer) goes through the Maths pile, keeping a checklist of unique names (HashSet). Alice appears twice but only gets one tick. Final count: 2 unique students in Maths.

Walkthrough example: Count distinct cities per department:

InputMapper Output
Alice, Sales, London("Sales", "London")
Bob, Sales, Paris("Sales", "Paris")
Carol, Sales, London("Sales", "London")
Dave, Engineering, Tokyo("Engineering", "Tokyo")
Eve, Engineering, Tokyo("Engineering", "Tokyo")

After shuffle: ("Sales", ["London", "Paris", "London"]) and ("Engineering", ["Tokyo", "Tokyo"]).

Reducer for Sales: HashSet = {London, Paris} → emit("Sales", 2). Reducer for Engineering: HashSet = {Tokyo} → emit("Engineering", 1).

Memory Consideration: The HashSet approach works well when the number of distinct values per group fits in memory. If a single group has billions of distinct values, you'd need approximate algorithms (like HyperLogLog) or a secondary sort / multi-pass approach. For the exam, the HashSet solution is the expected answer.

Why This Works — MapReduce Guarantees

MapReduce guarantees that all values for the same key are sent to the same reducer. This means all rows belonging to "Sales" end up at one reducer, all "Engineering" rows at another. Within each reducer call, we see every value for that group — so using a HashSet gives us a complete picture of unique values. No data is split across reducers for the same key.

For Part (a): Write the pseudocode for mapper and reducer (5 lines total). State clearly: "The shuffle phase groups identical keys, so each unique value appears as a reducer key exactly once." For Part (b): Emphasize the HashSet in the reducer. A walkthrough example with 3-4 rows is worth extra marks — it proves you understand the data flow end-to-end.

Question 3 — NoSQL & Distributed Systems Concepts 20 marks
(a) Evaluate statements about LSM Trees. (b) Explain BigTable's storage model — row store vs column store. (c) Compare standard and order-preserving consistent hashing. (d) Discuss quorum replication and linearizability.

Part (a): LSM Tree Statements — True/False Analysis 5 marks

Log-Structured Merge Trees (LSM Trees) are the storage engine behind Cassandra, HBase, LevelDB, RocksDB, and many modern databases. Their fundamental innovation is turning expensive random disk writes into cheap sequential ones — at the cost of slightly more complex reads.

StatementVerdictDetailed Explanation
LSM trees convert random writes to sequential writes TRUE This is the core advantage. A B-tree must seek to the right page and update in place (random I/O). An LSM tree writes to an in-memory buffer (MemTable) and periodically flushes it as a sorted, immutable file (SSTable) in one sequential disk write. Sequential writes are 100-1000x faster than random writes on HDDs.
Writes go to MemTable first, then flushed as SSTable TRUE Every write is first inserted into the MemTable (a sorted in-memory structure, typically a red-black tree or skip list). When the MemTable reaches a size threshold (e.g., 64MB), it's frozen and written to disk as an SSTable — a sorted, immutable file. A new empty MemTable takes its place.
LSM reads can be slower than B-tree reads TRUE To read a key, LSM must check: (1) the current MemTable, (2) the frozen MemTable being flushed, (3) multiple SSTables on disk from newest to oldest. In the worst case, this means checking many files. B-trees find the key in O(log n) with a single index lookup. Bloom filters mitigate this by quickly ruling out SSTables that definitely don't contain the key.
Compaction merges SSTables to improve read performance TRUE Over time, many SSTables accumulate on disk. Compaction is a background process that merges multiple SSTables into a single, larger SSTable — deduplicating keys and discarding deleted entries (tombstones). Fewer SSTables = fewer files to check during reads = better read performance.
Bloom filters help avoid unnecessary disk reads TRUE Each SSTable has an associated Bloom filter — a probabilistic data structure that can say "this key is definitely NOT in this SSTable" or "this key might be in this SSTable." If the Bloom filter says no, we skip that SSTable entirely, saving a disk read. False positive rate is typically <1%.
Real-World Analogy: The Sticky Note Strategy

Imagine you're taking notes during a busy day. The B-tree approach is like finding the exact right page in your thick, organized notebook and inserting a note in alphabetical order every time — you spend most of your time flipping pages (random seeking). The LSM approach is like scribbling on a sticky note (MemTable) and slapping it on your desk. When your desk is full of stickies, you sort them all at once and staple them into a neat packet (flush to SSTable). Finding a specific note later means checking multiple packets (slower reads), but the writing itself was blazing fast because you never had to search for the right position.

LSM Write Path Summary

Write → WAL (write-ahead log) → MemTable → [threshold reached] → Flush to SSTable → [background] Compaction merges SSTables. The WAL ensures durability — if the machine crashes before flushing, the MemTable can be rebuilt from the WAL on restart.

Part (b): BigTable — Row Store vs Column Store 5 marks

This is one of the most commonly confused topics. BigTable (and its open-source counterpart HBase) is neither a pure row store nor a pure column store — it's a column-family store, which is a hybrid that takes the best of both worlds.

Pure Row Store (e.g., MySQL)

All columns of a row are stored contiguously on disk. Reading one column requires loading the entire row. Great for OLTP (fetching complete records), wasteful for analytics (scanning one column across millions of rows).

Pure Column Store (e.g., Parquet)

Each individual column is stored in its own file/segment. Scanning one column across millions of rows is extremely fast. But reconstructing a full row requires reading from many separate files.

BigTable / HBase: Column-Family Store (The Hybrid)

Columns are grouped into column families (declared at table creation time). Each column family is stored in its own set of HFiles/SSTables on disk. Within a column family, data is stored row-oriented — all columns in that family for a given row are contiguous. Across column families, data is stored separately — reading one family does NOT require loading others.

Real-World Analogy: The Filing Cabinet

Think of a big filing cabinet for employee records. Each drawer is a column family: one drawer for "personal info" (name, address, phone), another for "payroll" (salary, tax, bank details), another for "medical" (health records). When HR needs someone's payroll info, they only open the "payroll" drawer — they never touch the "medical" drawer. But within that drawer, all payroll fields for one person are clipped together (row-oriented within the family). This is far more efficient than either (a) storing everything in one massive folder per person (pure row store — must scan medical records to find salary) or (b) having separate drawers for every single field like "first name", "last name", "salary" individually (pure column store — too fragmented for record-level access).

Part (c): Consistent Hashing — Standard vs Order-Preserving 5 marks

Both techniques distribute data across nodes arranged in a logical ring, but they differ fundamentally in how keys are mapped to positions on that ring — and this difference has profound implications for what queries are efficient.

Standard Consistent Hashing

How it works: Apply a hash function (e.g., Murmur3, MD5) to the key, producing a hash value. This hash value determines the position on the ring. The key is assigned to the first node clockwise from that position.

Key property: The hash function destroys the natural ordering of keys. Keys "user_1000" and "user_1001" will likely land on completely different nodes, potentially on opposite sides of the ring.

Advantage: Excellent uniform distribution regardless of key patterns. Even if all keys start with "A", the hash spreads them evenly.

Disadvantage: Range queries are impossible. "Give me all users with IDs 1000-2000" requires contacting ALL nodes, since those keys are scattered everywhere.

Used by: Cassandra (with Murmur3Partitioner), DynamoDB, Riak.

Order-Preserving Consistent Hashing

How it works: The key maps DIRECTLY to a position on the ring without hashing — or uses an order-preserving hash. Keys that are numerically/lexicographically close stay close on the ring.

Key property: The natural ordering of keys is preserved. Adjacent keys land on the same node or neighboring nodes.

Advantage: Range queries are efficient! "All users 1000-2000" maps to a contiguous arc on the ring, hitting only 1-2 nodes instead of all nodes.

Disadvantage: Potential hotspots. If keys are timestamps, all recent writes cluster on one node (the one responsible for "now"). If keys have a skewed distribution (e.g., most start with "S"), some nodes get overloaded.

Used by: BigTable/HBase (rows sorted by key), Cassandra (with ByteOrderedPartitioner — rarely used).

Real-World Analogy: Student Dormitory Assignment

Standard hashing is like assigning students to dorms by hashing their name — "Alice" goes to Dorm 7, "Bob" to Dorm 2, "Carol" to Dorm 7, "Dave" to Dorm 5. The distribution is even, but friends whose names are close alphabetically might be in completely different buildings. If you want to find "all students from A-C," you have to check every single dorm. Order-preserving is like assigning alphabetically — A-F in Dorm 1, G-L in Dorm 2, etc. Now you can easily find "all students from M-P" by going to Dorm 3. But if 80% of students have last names starting with S-Z, Dorm 5 is wildly overcrowded while Dorm 1 is nearly empty.

Part (d): Quorum Replication & Linearizability 5 marks

Quorum-based replication is the most flexible approach to balancing consistency and availability in distributed systems. But there's a subtle and critical distinction: quorum overlap guarantees freshness, but NOT linearizability.

Quorum Rules Recap

R + W > N — Read quorum + Write quorum > Total replicas. Guarantees every read set overlaps with every write set, so at least one node in any read has the latest write.

W + W > N — Two write quorums always overlap. Guarantees write-write conflict detection — no two writes can succeed without at least one node seeing both.

Example: N=3, R=2, W=2. Any 2 nodes you read from must include at least 1 node that participated in the last write (since 2+2=4 > 3).

Linearizability is a stronger guarantee: every operation appears to take effect instantaneously at some point between its start and end. In other words, once a write completes, ALL subsequent reads (even from different clients) must see it. It provides the illusion of a single copy of the data, even though it's replicated.

Critical Insight: Quorum ALONE does NOT guarantee linearizability. Here's why: two concurrent writes W1 and W2 might reach different subsets of nodes in different orders. Client A reads from nodes that saw W2 last, Client B reads from nodes that saw W1 last — both satisfy the quorum rule (R+W>N), but they see different "latest" values. This violates linearizability because the system appears to go backward in time for one of the clients.

Real-World Analogy: The Jury Problem

Quorum replication is like requiring majority agreement in a jury system. With 5 jurors (N=5), you need 3 to agree on a verdict (W=3 for writes) and ask 3 for the result (R=3 for reads). Since 3+3=6 > 5, at least one juror in any "read panel" participated in the last "write decision." BUT — imagine two trials happening simultaneously for the same case. Jury A convicts at 2:00 PM, Jury B acquits at 2:01 PM. Depending on which 3 jurors you ask, you might get different verdicts. To be truly linearizable, you need a single supreme court (consensus protocol) that serializes all decisions into one definitive order.

What's needed beyond quorum for linearizability?

  • Version vectors / Lamport timestamps: Attach a logical clock to each write so nodes can determine causal ordering. This detects conflicts but doesn't resolve them automatically.
  • Consensus protocols (Paxos, Raft): All writes go through a leader elected by consensus. The leader serializes writes into a total order. This provides true linearizability but at the cost of latency and availability (requires majority of nodes to be reachable).
  • Last-Write-Wins (LWW): Cassandra's approach — attach a wall-clock timestamp to each write, always keep the latest timestamp. This provides eventual consistency (all nodes converge to the same value eventually) but NOT linearizability, because wall clocks can drift and concurrent writes with close timestamps may resolve differently on different nodes.

Q3 Key Takeaways

(a) LSM Trees trade read speed for write speed — sequential writes to SSTables, Bloom filters and compaction mitigate read cost. (b) BigTable is a column-family store — a hybrid between row and column stores, offering selective access at the family level. (c) Standard hashing gives uniform distribution but no range queries; order-preserving enables range queries but risks hotspots. (d) Quorum ensures freshness (R+W>N) but linearizability requires additional mechanisms like consensus protocols or version vectors.
