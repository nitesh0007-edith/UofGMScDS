Question 1 — Drawbacks of RDBMS for Big Data 10 marks
Discuss the drawbacks of using a traditional RDBMS approach for Big Data applications. Address both normalization and consistency/transactional concerns.

a) The Normalization Problem 5 marks

Relational databases are fundamentally built on the principle of normalization — the practice of decomposing data into many smaller tables, each representing a single entity or relationship, linked together by foreign keys. In a traditional setting with modest data sizes, this is elegant and eliminates redundancy. However, when data grows to big data scale, normalization becomes a serious liability.

The core issue is that big data applications typically need to retrieve attributes that are spread across many normalized tables. To reassemble a complete picture, the database must perform expensive multi-way JOIN operations. JOINs are notoriously expensive and resource-hungry operations even on moderately sized tables — they require comparing rows across tables, often with complex indexing and temporary storage. When tables grow to billions of rows, the cost of these JOINs becomes utterly intolerable.

Real-World Analogy: The E-Commerce Product Page

Consider an e-commerce platform like Amazon. In a normalized RDBMS, you would have separate tables for Users, Orders, Products, Reviews, and Shipping. To render a single product page that shows the product details, all its reviews, buyer information for each review, and related order stats, the database must perform a 5-way JOIN across all these tables. At Amazon's scale (hundreds of millions of products, billions of reviews), this JOIN would take minutes or even hours — completely unacceptable for a page that needs to load in milliseconds. This is why Amazon built DynamoDB (a NoSQL database) instead of relying on an RDBMS.

Why NoSQL Avoids This

NoSQL databases like HBase, Cassandra, and MongoDB use denormalized data models where related data is stored together in a single row or document. Instead of JOINing 5 tables, all the data for a product page lives in one place — a single read retrieves everything. The trade-off is data redundancy, but at big data scale, storage is cheap while computation (JOINs) is expensive.

b) The Consistency / ACID Transaction Problem 5 marks

The second major drawback of RDBMS is its insistence on strong consistency through ACID transactions (Atomicity, Consistency, Isolation, Durability). ACID guarantees that every transaction is processed reliably — either all operations in a transaction succeed, or none do. While this sounds desirable, enforcing ACID in a distributed environment requires two extremely expensive mechanisms:

  • Distributed concurrency control: Guards against conflicts when multiple users access the same data simultaneously. Requires locks, timestamps, or multi-version protocols across all nodes — all of which add significant overhead.
  • Distributed commit protocols: Guards against partial failures using protocols like Two-Phase Commit (2PC). Every participating node must agree before a transaction commits. If any node fails or is slow, the entire transaction is blocked.

The critical insight is that many big data applications simply do not need transaction grouping. They can tolerate slight inconsistencies for a brief period in exchange for much higher performance and availability.

ACID Model (RDBMS)

Atomicity: All or nothing.
Consistency: Always valid state.
Isolation: Transactions don't interfere.
Durability: Committed = permanent.
Best for: Banking, financial systems.

BASE Model (NoSQL)

Basically Available: System always responds.
Soft state: State may change over time.
Eventually consistent: Will converge.

Best for: Social media, streaming, analytics.

Real-World Analogy: Netflix vs Banking

When Netflix updates your "Continue Watching" list, it does not need ACID transactions. If your watch history takes 2 seconds to propagate across all servers, nobody notices or cares — eventual consistency is perfectly fine. But when you transfer $500 from savings to checking at a bank, the system absolutely must guarantee that the money leaves one account and arrives in the other atomically. There is no acceptable "eventually consistent" state where the money is in neither account. This is why banks use RDBMS with ACID, while Netflix uses NoSQL with BASE.

Common exam pitfall: Do not say RDBMS is "bad" — it is excellent for applications that genuinely need ACID. The drawback is that it enforces this expensive guarantee even for applications that do not need it, wasting resources at big data scale.

Exam Strategy

For 10 marks, spend roughly equal time on both drawbacks (~5 marks each). For normalization: mention normalized tables, foreign keys, multi-way JOINs, and that JOINs are expensive at scale. For consistency: mention ACID, distributed concurrency control, distributed commit protocols, and the BASE alternative. Give a concrete example for each.

Question 2 — HDFS Global and Local Metadata 10 marks
Describe the global and local metadata that HDFS maintains per file. Who maintains each type?

Overview

HDFS maintains two distinct levels of metadata for every file in the system. The global metadata is maintained centrally by the NameNode and provides the "big picture" — where everything is and who can access it. The local metadata is maintained independently by each DataNode and provides the "ground truth" — the actual condition and size of the data blocks physically stored on that node. Together, these two levels allow HDFS to efficiently manage, locate, and verify files across a cluster of thousands of machines.

a) Global Metadata — Maintained by the NameNode 5 marks

The NameNode is the central authority of the HDFS cluster, and it maintains three critical categories of global metadata:

  1. Access restrictions: The NameNode stores the permissions and access control information for each file and directory. When a client requests to read, write, or delete a file, the NameNode checks these restrictions first to honour or deny the request. This is analogous to the security policy in a traditional file system.
  2. Location information: For every file, the NameNode knows exactly which DataNodes store which blocks and where the replicas of each block are located. This is the mapping that makes it possible for a client to read data — without it, the client would have no idea which machine in the cluster holds the data it needs.
  3. Namespace state: The NameNode maintains the complete filename tree — every directory, every file, and the hierarchical structure that connects them. This is the equivalent of a traditional filesystem's directory structure, but for a distributed system spanning thousands of nodes.
Key Insight: Why Global Metadata is In-Memory

The NameNode keeps all of this metadata in RAM for fast access. This means every file operation (open, create, delete, list) is served from memory without disk I/O. The downside is that this limits the total number of files HDFS can manage to what fits in the NameNode's memory — typically a few hundred million files.

b) Local Metadata — Maintained by Each DataNode 5 marks

Each DataNode independently tracks metadata about the blocks it physically stores on its local disks:

Local MetadataPurposeWho Uses It
File sizes The actual size (in bytes) of each block stored locally. DataNodes periodically communicate this to the NameNode. The NameNode uses aggregated size information for global space management — knowing how much total capacity is used and available across the cluster.
File checksums A cryptographic checksum computed for each stored block. This is a fingerprint that uniquely identifies the correct content of the block. Used for corruption detection and recovery. When a client reads a block, it can verify the checksum to ensure the data has not been silently corrupted on disk. If corruption is detected, the system can retrieve a healthy replica from another DataNode.
Real-World Analogy: The Warehouse Chain

Think of HDFS as a national warehouse chain. The NameNode is the headquarters (HQ) — it knows the complete catalog of all products (namespace), which warehouse stores which products and where the backup copies are (location info), and who is authorized to access what (access restrictions). Each DataNode is a local warehouse — it tracks the weight/size of each crate it stores (file sizes) and performs quality checks on each item (checksums). The warehouses report their local stats back to HQ periodically so the head office can manage the global supply chain effectively.

Important distinction: The NameNode does NOT store the actual data blocks — it only stores metadata about them. The DataNodes store the actual data. If the NameNode crashes, you lose the map to the data (recoverable from checkpoints/journals), not the data itself.

Exam Strategy

Clearly separate your answer into global (NameNode) and local (DataNode) sections. For global: name all three types (access restrictions, location information, namespace state). For local: name both types (file sizes, file checksums) and explain why each matters. Mention that sizes are communicated to NN for space management and checksums are for corruption detection.

Question 3 — MapReduce2 (YARN) Entities and Roles 10 marks
Describe the main entities involved in a MapReduce2 (YARN) execution and their roles.

Overview: Why YARN Exists

MapReduce2 (also known as YARN — Yet Another Resource Negotiator) was introduced to solve a fundamental architectural flaw in MapReduce1: the JobTracker was responsible for both resource management and job execution, making it a bottleneck and single point of failure. YARN decouples these two concerns into separate entities, enabling better scalability and the ability to run frameworks beyond just MapReduce.

The Five Key Entities 2 marks each

EntityScopePrimary Role
ClientExternalSubmits MapReduce jobs to the cluster
Resource Manager (RM)Cluster-wide (1 per cluster)Allocates resources (containers), schedules execution
Node Manager (NM)Per-node (1 per machine)Launches and monitors containers on its node
Application Master (AM)Per-job (1 per job)Manages the job, coordinates tasks, negotiates for resources
HDFSCluster-wideProvides distributed I/O for job input and output

Detailed Role Breakdown

The Client is the entry point. A user or application packages a MapReduce job (specifying the Mapper class, Reducer class, input/output paths) and submits it to the Resource Manager. The client then monitors the job's progress and retrieves results upon completion.

The Resource Manager (RM) is the ultimate authority for allocating cluster resources. It maintains a global view of all available resources (CPU, memory) across all nodes and decides how to distribute containers — the fundamental unit of resource allocation in YARN. A container is essentially a reserved slice of a node's CPU and memory. The RM has two main components: a Scheduler (which decides how to allocate resources) and an Applications Manager (which accepts job submissions and launches the first container for each job's Application Master).

Node Managers (NMs) are the per-machine agents responsible for the physical work. Each NM manages the resources on its specific node — it launches containers as instructed by the RM, monitors their resource usage (CPU, memory), and reports back to the RM. If a container exceeds its allocated resources, the NM can kill it.

The Application Master (AM) is the most interesting entity in YARN. There is one AM per job, and it acts as the job-specific coordinator. The AM negotiates with the Resource Manager to obtain containers for its map and reduce tasks, then works with the Node Managers to launch and monitor those tasks. If a task fails, the AM can request a new container and retry. The AM essentially takes over the job-management role that was previously handled by the monolithic JobTracker in MR1.

HDFS provides the distributed storage layer. Map tasks read their input splits from HDFS, intermediate shuffle data is stored locally, and reduce tasks write their final output back to HDFS.

Real-World Analogy: Film Production

Think of running a YARN job like producing a film. The Client is the producer who commissions the film and says "I want this movie made." The Resource Manager is the film studio — it controls all the sound stages, cameras, and equipment, and decides which production gets what resources. The Node Managers are the stage managers at each sound stage — they set up the equipment and make sure each stage is running properly. The Application Master is the director — they manage the creative process, coordinate the actors and crew, and negotiate with the studio for more resources when needed. HDFS is the film archive where raw footage is stored and final edits are saved.

The Key Architectural Insight

The fundamental principle of YARN is separation of concerns. Resource management (RM + NM) is completely decoupled from job execution (AM + YarnChild). This means the RM never needs to understand the semantics of MapReduce — it just allocates containers. This is why YARN can run Spark, Tez, Flink, and other frameworks alongside MapReduce, all sharing the same cluster resources.

Exam Strategy

For 10 marks with 5 entities at 2 marks each, name each entity clearly and give its role in 1-2 sentences. The AM is the most important to explain well — emphasize it is per-job and takes over the JT's job management role. Mention that the RM/NM pair handles resource management while AM handles job execution.

Question 4 — Recoverability: WAL, Journals, Checkpoints, Backup Nodes 10 marks
Explain the principles of recoverability in distributed systems, covering write-ahead logging, journals, checkpoints, and backup nodes.

The Recoverability Problem

In any distributed system, critical state (like the HDFS namespace or an HBase region's metadata) is kept in memory for fast access. But memory is volatile — a power failure, crash, or hardware fault can destroy it instantly. The recoverability problem is: how do we ensure that no data is permanently lost when this happens? The solution uses three complementary resources that work together to make recovery possible.

The Three Recovery Resources 2 marks each

1. Image (In-Memory State)

The Image is the live, in-memory data structure holding the current state of the system. In HDFS, this is the complete namespace — every file, directory, block mapping, and permission. It is kept in RAM because memory access is orders of magnitude faster than disk access. However, RAM is volatile: if the machine crashes or loses power, the Image is destroyed completely.

2. Checkpoint (Periodic Snapshot)

A Checkpoint is a periodic dump of the entire in-memory Image to stable storage (disk). Think of it as a "save game" — a complete snapshot of the system's state at a specific moment in time. Checkpoints are created at regular intervals (e.g., every hour). They provide a known-good state that the system can return to after a crash.

3. Journal / WAL (Write-Ahead Log)

The Journal (also called the EditLog or Write-Ahead Log) records every single mutation that occurs between checkpoints. Crucially, operations are saved to the Journal BEFORE they are applied to the in-memory Image — this is the "write-ahead" principle. This ordering guarantee is critical: it means that even if the system crashes mid-operation, the Journal on disk contains a record of what was about to happen. Every file creation, deletion, rename, or permission change is logged here before execution.

The WAL principle is non-negotiable: Operations are ALWAYS written to the Journal on stable storage FIRST, and only THEN applied to the in-memory Image. If you reverse this order, a crash after applying but before logging means the operation is lost — violating recoverability.

The Recovery Process 2 marks

When the system crashes and restarts, recovery follows a simple but powerful two-step process:

  1. Load the last Checkpoint: Read the most recent checkpoint file from disk into memory. This restores the system to the state it was in when that checkpoint was created — which might be minutes or hours old.
  2. Replay the Journal: Read the Journal file and re-apply every recorded operation on top of the loaded checkpoint, in order. This "fast-forwards" the state from the checkpoint time to the exact moment of the crash.

After these two steps, the in-memory Image is fully reconstructed to its pre-crash state, and the system can resume normal operations.

Why This Works

Because the Journal records every mutation (thanks to the write-ahead principle), and the Checkpoint captures a complete snapshot, the combination of the two is equivalent to the full state. No operation is ever lost, even if the crash happens between checkpoints.

Backup Node 2 marks

A Backup Node provides an additional layer of safety by storing replicas of the Image, Journal, and/or Checkpoints on a completely separate machine. If the primary node's disk is destroyed (not just a crash, but actual hardware failure), the system can still recover from the Backup Node's copies. The Backup Node receives a continuous stream of Journal entries from the primary node and maintains its own up-to-date copy of the Image, so it can take over quickly if needed.

Real-World Analogy: The Chef's Recipe System

Imagine a head chef at a restaurant who keeps all recipes in their head (Image — volatile memory). Every month, they print out a complete recipe book (Checkpoint — periodic snapshot to stable storage). Every day, they write down any recipe changes or new recipes in a logbook (Journal — write-ahead log). If the chef gets amnesia (crash!), recovery is straightforward: read the last printed recipe book (load checkpoint), then read through the daily logbook to apply all changes since the book was printed (replay journal). The Backup Node is like giving a copy of everything to a sous-chef at a different restaurant — if the original restaurant burns down, the sous-chef has everything needed to continue.

Quick Reference

Image = volatile in-memory state (fast but fragile) | Checkpoint = periodic disk snapshot (the safety net) | Journal/WAL = log of all changes between checkpoints (write BEFORE apply) | Recovery = load checkpoint + replay journal | Backup Node = remote replica of everything for disaster recovery.

Question 5 — MapReduce1 vs MapReduce2: Differences and Similarities 10 marks
Compare and contrast MapReduce1 and MapReduce2 (YARN) architectures, highlighting key differences and similarities.

MapReduce1 Architecture 3 marks

MapReduce1 was the original Hadoop execution framework with a relatively simple architecture consisting of four entities:

  • Client: Submits MapReduce jobs to the cluster.
  • JobTracker (JT): The central daemon that handles both resource management and job execution. It decides which TaskTracker gets which task, monitors progress, and handles failures.
  • TaskTrackers (TT): Run on each cluster node. They execute map and reduce tasks as instructed by the JobTracker and report status back.
  • HDFS: Provides distributed storage for input and output data.

The fundamental problem with MR1 is that the JobTracker is a monolithic component that conflates two very different responsibilities: managing cluster resources (which nodes have capacity) and managing individual jobs (which tasks should run where). This makes the JT a bottleneck — it must track every task of every job, limiting scalability to around 4,000 nodes. It also means the cluster can only run MapReduce workloads.

MapReduce2 (YARN) Architecture 4 marks

YARN fundamentally restructures the system by decoupling resource management from job execution:

  • Resource Manager (RM): Takes over the resource management role from the JobTracker. It maintains a global view of cluster resources and allocates containers (units of CPU + memory) to applications.
  • Node Managers (NM): Replace TaskTrackers. Each NM manages resources on its specific node — launching containers, monitoring usage, and reporting to the RM.
  • Application Master (AM): This is the key innovation. Each job gets its own AM, which takes over the job management role from the old JobTracker. The AM negotiates with the RM for containers, works with NMs to launch tasks, and monitors task progress. If a task fails, the AM handles the retry.
  • YarnChild: The actual processes running inside containers, executing map or reduce tasks.

Side-by-Side Comparison

AspectMapReduce1MapReduce2 (YARN)
Resource ManagementJobTracker (centralized)Resource Manager + Node Managers
Job ManagementJobTracker (same entity!)Application Master (per-job)

Key Similarities

Despite the architectural differences, MR2 preserves important continuity with MR1:

  • Both use HDFS as the underlying distributed storage layer.
  • Both support the same MapReduce programming model (map phase, shuffle, reduce phase).
  • MR2 is fully backwards compatible — existing MR1 applications can run on YARN without modification.
  • Both rely on data locality — scheduling tasks on nodes where the input data physically resides.
Real-World Analogy: School Administration

MR1 is like a school where one teacher does both class management AND teaching — they assign classrooms, manage timetables, AND teach every subject. When the school grows, this single teacher becomes overwhelmed. MR2 is like hiring a separate administrator who handles room bookings and timetables (Resource Manager), while individual teachers (Application Masters) focus purely on teaching their specific subject. The school can now also offer music, art, and drama (Spark, Tez, Flink) — not just maths (MapReduce). The administrator does not need to understand each subject; they just allocate rooms and equipment.

One-Line Summary

MR1 = monolithic JobTracker (resource + job management combined, MR only). MR2/YARN = decoupled RM/NM for resources + per-job AM for execution, supporting any framework, fully backward-compatible.

Question 6 — HBase Data Item Location Procedure 10 marks
Describe the procedure by which a client locates a specific data item in HBase, including all components involved and the RegionServer-HDFS interaction.

Architecture Components 2 marks

The HBase data location procedure involves five key components working together:

  • Client: The application that wants to read or write a specific row in an HBase table.
  • ZooKeeper (ZK): A distributed coordination service that acts as the starting point for all data lookups — it knows where the META table lives.
  • META Table: A special HBase table that maps every region (a contiguous range of row keys) to the RegionServer currently hosting it. It is the master directory of the entire HBase cluster.
  • HBase Master: Manages the cluster (assigns regions to RegionServers, handles failover), but is NOT involved in the data path for reads/writes.
  • RegionServers (RS): The workhorses that actually store and serve data. Each RS hosts multiple regions.
  • HDFS: The underlying distributed file system where HBase data is physically stored as HFiles.

The Location Procedure 6 marks

When a client wants to find a specific data item (identified by a row key), it follows a multi-step lookup process that progressively narrows down the location:

  1. Client contacts ZooKeeper: The client first queries ZooKeeper to discover which RegionServer is currently hosting the hbase:meta table. ZooKeeper maintains this information as a small piece of coordination data. This is a lightweight lookup — ZooKeeper is designed for exactly this kind of fast metadata query.
  2. Client queries the META table: Armed with the META table's location, the client contacts that RegionServer and scans the META table to find which RegionServer hosts the specific region containing the target row key. The META table is organized so that each row maps a region's start/end key range to a RegionServer address.
  3. Client contacts the target RegionServer: The client now goes directly to the identified RegionServer and requests the data for the target row key. The RS looks up the data in its in-memory MemStore and/or on-disk HFiles and returns the result.
  4. Client caches the location: To avoid repeating this three-step lookup for every subsequent request, the client caches the region-to-RegionServer mapping locally. Future reads for row keys in the same region skip ZooKeeper and META entirely and go straight to the RS. The cache is invalidated when a region is moved or split.
Why Three Steps Instead of One?

You might wonder: why not just ask ZooKeeper directly where a specific row key lives? The answer is scale. ZooKeeper is designed to store tiny amounts of coordination data, not millions of region mappings. The META table can be large (one row per region, potentially millions of regions), so it lives on a RegionServer just like any other HBase table. ZooKeeper only stores the single pointer to META's location — a clean separation of concerns.

RegionServer-HDFS Interaction 2 marks

Once the client reaches the correct RegionServer, the RS must actually retrieve the data from HDFS. The RegionServer acts as a mediator between the client and the underlying HDFS storage:

  • Region-to-HDFS-block mapping: The RegionServer maintains mappings from its hosted regions to the corresponding HDFS blocks (HFiles). It knows exactly which HDFS files contain the data for each region.
  • Data retrieval: The RS reads the required data blocks from HDFS DataNodes (acting as an HDFS client itself), caches frequently accessed blocks in its block cache (an in-memory LRU cache), and returns the data to the HBase client.
  • Block cache optimization: On subsequent reads for the same or nearby data, the RS can serve directly from its block cache without touching HDFS at all, significantly reducing latency.
Real-World Analogy: University Library System

Imagine you need to find a specific book in a large university library system spread across multiple buildings. You first go to the reception desk (ZooKeeper) and ask: "Where is the master catalog?" The receptionist tells you it is in Building A, Room 201. You go to that room and check the master catalog (META table), which tells you that books with call numbers in your range are on Floor 3 of Building C. You go directly to that floor and shelf (RegionServer) and find your book. You make a mental note (cache) that books in this call number range are on Floor 3 of Building C — next time you need a book from the same range, you skip the reception desk and catalog and go straight to the shelf.

Common misconception: The HBase Master is NOT involved in the data location procedure. Clients never contact the Master to find data — the Master only handles administrative tasks like region assignment and failover. Data lookups go through ZK, META, and RS only.

Location Flow Summary

Client → ZooKeeper (get META location) → META table on RS (get target region's RS) → Target RegionServer (get data) → Cache result. The RS mediates between client and HDFS: it maps regions to HDFS blocks, reads from DataNodes, caches in block cache, and returns data to client.
