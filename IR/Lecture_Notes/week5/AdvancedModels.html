Chapter 0

The Big Picture

At its core, Information Retrieval (IR) is about finding the right documents for a user's query. Think of it as the brain behind every search engine — the part that decides which results to show and in what order.

THE RETRIEVAL PIPELINE

The retrieval model is the relevance decision mechanism — it scores and ranks documents.

What Exactly Is a "Model"?

An IR model defines three things: how documents are represented, how queries are represented, and how we match them. Over decades, researchers have developed increasingly sophisticated models:

Relevance: The Central Challenge

Relevance is subjective — people often disagree on whether a document is relevant. Retrieval models simplify this by making assumptions:

Topical Relevance

Query and document are about the same topic. This is what most models focus on.

User Relevance

Takes into account novelty, freshness, language preference, user context. Much harder to model!

Chapter 1

Boolean Model

The simplest retrieval model. Documents are either relevant or not relevant — there's no in-between and no ranking.

How It Works

Documents are represented as a set of keywords (terms).

Queries use Boolean operators: AND, OR, NOT.

Matching: A document is relevant if its terms satisfy the logical expression.

Boolean Relevance Decision
$$R(D, Q) = D \models Q$$

A document D is relevant iff it satisfies the query Q.

Setup: We have three documents and three terms A, B, C:

d₁ = {A, B}    d₂ = {B, C}    d₃ = {A, B, C}

Query: A AND B AND NOT C

Results:

• d₁ = {A, B} → has A ✓, has B ✓, no C ✓ → RETRIEVED ✓

• d₂ = {B, C} → missing A ✗ → NOT retrieved

• d₃ = {A, B, C} → has C ✗ (violates NOT C) → NOT retrieved

✓ Advantages

  • Simple to understand and implement
  • Exact matching — predictable results
  • Full Boolean Algebra available

✗ Disadvantages

  • No ranking — results are an unordered set
  • Complex queries confuse users
  • Either "nothing found" or "information overload"
Chapter 2

Vector Space Model

Instead of binary yes/no, the Vector Space Model (VSM) represents documents and queries as vectors in a high-dimensional space, where each dimension is a term.

The Intuition

Imagine each term as an axis. A document's position in this space depends on which terms it contains and how important they are. The query is also a point. Documents closer to the query are more relevant.

2D VECTOR SPACE (terms: "cat" and "dog")

D₁ is closer to Q (smaller angle θ), so D₁ ranks higher.

Cosine Similarity
$$\text{sim}(D, Q) = \cos(\theta) = \frac{\vec{D} \cdot \vec{Q}}{|\vec{D}| \times |\vec{Q}|} = \frac{\sum_{i=1}^{n} d_i \cdot q_i}{\sqrt{\sum_{i=1}^{n} d_i^2} \cdot \sqrt{\sum_{i=1}^{n} q_i^2}}$$

Numerical Example:

Document D = (3, 2, 0, 1) and Query Q = (1, 0, 0, 2) with terms {cat, dog, fish, bird}.

$$\vec{D} \cdot \vec{Q} = (3)(1) + (2)(0) + (0)(0) + (1)(2) = 3 + 0 + 0 + 2 = 5$$

$$|\vec{D}| = \sqrt{9+4+0+1} = \sqrt{14} \approx 3.74$$

$$|\vec{Q}| = \sqrt{1+0+0+4} = \sqrt{5} \approx 2.24$$

$$\cos(\theta) = \frac{5}{3.74 \times 2.24} = \frac{5}{8.37} \approx 0.597$$

The similarity score is 0.597 (on a 0–1 scale).

Drawbacks: The similarity metric (cosine, dice, etc.) is chosen ad-hoc with no theoretical basis. There's no guidance on when to stop ranking, and term weights (like TF-IDF) lack probabilistic justification.
Chapter 3

Probabilistic Models

Why probabilities? Because IR is full of uncertainty — uncertain representations, uncertain queries, uncertain relevance. Probability theory is the natural language for reasoning about uncertainty.

The Probability Ranking Principle (PRP)

"Rank documents in order of decreasing probability of relevance to the user."

— Robertson (1977)

Ranking Criterion
$$\text{Rank by } P(R \mid D)$$

Probability that document D is relevant (R) given the query.

IR as Classification

Think of it as a two-class classification problem for each document:

Document D
→
P(R|D) > P(NR|D)?
→
Yes → Relevant
Document D
→
P(R|D) > P(NR|D)?
→
No → Not Relevant

Bayes' Rule in IR

We use Bayes' Rule to flip the probabilities into something we can estimate:

Applying Bayes' Rule
$$P(R \mid D) = \frac{P(D \mid R) \cdot P(R)}{P(D)}$$

Classify as relevant if the likelihood ratio exceeds the prior ratio:

$$\frac{P(D \mid R)}{P(D \mid NR)} > \frac{P(NR)}{P(R)}$$

Binary Independence Model (BIM)

The foundational probabilistic model makes two key assumptions:

Binary representation: Each document is a vector of 0s and 1s (term present or absent). No term frequency!
Independence: Terms occur independently in documents. So \(P(A \cap B) = P(A) \cdot P(B)\).

Define: \(p_i = P(d_i=1 \mid R)\) = probability term \(i\) appears in relevant docs, and \(s_i = P(d_i=1 \mid NR)\) = probability it appears in non-relevant docs.

BIM Retrieval Status Value (RSV)
$$RSV(D) = \sum_{i: d_i=1} \log \frac{p_i (1 - s_i)}{s_i (1 - p_i)}$$

Sum only over terms present in the document.

Deriving IDF from BIM

Here's the beautiful part: without relevance information, if we set \(p_i = 0.5\) and estimate \(s_i \approx n_i/N\) (collection frequency), we get:

IDF from Probabilistic Foundations
$$RSV = \sum_{i: d_i=1} \log \frac{N - n_i}{n_i}$$

This is exactly IDF (Inverse Document Frequency)! A theoretical justification for a widely-used heuristic.

Numerical Example — IDF Calculation:

Collection of N = 10,000 documents. Term "retrieval" appears in n = 50 docs.

$$\text{IDF}(\text{retrieval}) = \log \frac{10000 - 50}{50} = \log \frac{9950}{50} = \log(199) \approx 5.29$$

Term "the" appears in n = 9,800 docs.

$$\text{IDF}(\text{the}) = \log \frac{10000 - 9800}{9800} = \log \frac{200}{9800} = \log(0.0204) \approx -3.89$$

Rare terms get high IDF → more informative. Common terms get low/negative IDF → less useful.

With Relevance Feedback

If we do know which documents are relevant (from user feedback), we can refine:

BIM with Relevance Information
$$RSV = \sum_{i: d_i=1} \log \frac{r_i / (R - r_i)}{(n_i - r_i) / (N - n_i - R + r_i)}$$
SymbolMeaning
\(r_i\)Number of relevant documents containing term \(i\)
\(R\)Total number of relevant documents
\(n_i\)Total documents containing term \(i\)
\(N\)Total documents in collection
Limitation: BIM uses binary (0/1) representation — it ignores term frequency entirely. Effectiveness drops by up to 50% without TF. This motivates BM25!
Chapter 4

BM25 — Best Match 25

BM25 is arguably the most important ranking function in IR history. It extends BIM by adding term frequency and document length, making it dramatically more effective. It's still used in production search engines today!

The BM25 Formula
$$R(D, Q) = \sum_{i \in Q} \underbrace{\log \frac{(r_i + 0.5)/(R - r_i + 0.5)}{(n_i - r_i + 0.5)/(N - n_i - R + r_i + 0.5)}}_{\text{IDF component}} \cdot \underbrace{\frac{(k_1 + 1) \cdot tf_i}{K + tf_i}}_{\text{TF saturation}} \cdot \underbrace{\frac{(k_2 + 1) \cdot qf_i}{k_2 + qf_i}}_{\text{Query TF}}$$

where \( K = k_1 \cdot \left( (1-b) + b \cdot \frac{dl}{avgdl} \right) \)

Breaking Down BM25

Let's understand each component piece by piece:

IDF — Inverse Document Frequency

Without relevance information (\(R=0, r_i=0\)):

$$\log \frac{(0.5)/(0.5)}{(n_i + 0.5)/(N - n_i + 0.5)} = \log \frac{N - n_i + 0.5}{n_i + 0.5}$$

This gives rare terms high weight and common terms low weight. The 0.5 smoothing prevents log(0) issues.

TF Saturation — Diminishing Returns

$$\frac{(k_1 + 1) \cdot tf}{K + tf}$$

This is a saturation function. Unlike raw TF, it doesn't grow linearly. After a point, more occurrences barely increase the score.

Intuition: A document mentioning "python" 50 times isn't 50× more relevant than one mentioning it once — there are diminishing returns!

Document Length Normalisation

$$K = k_1 \cdot \left( (1 - b) + b \cdot \frac{dl}{avgdl} \right)$$

The parameter b controls how much document length affects scoring:

• b = 0: No length normalisation — long and short docs treated equally.

• b = 1: Full normalisation — a term in a long doc is less important than in a short doc.

• b = 0.75 is the typical sweet spot (TREC experiments).

Parameter Values

ParameterTypical ValueControls
\(k_1\)1.2TF saturation speed. Higher = slower saturation.
\(k_2\)0 – 1000Query TF saturation. Often 0 for short queries.
\(b\)0.75Document length normalisation strength.

Worked BM25 Example:

Collection: N = 10,000 docs, avgdl = 200 words. No relevance info (R=0, r=0).

Query: "information retrieval". Document D has dl = 250 words.

Term "information": tf = 5, appears in n₁ = 3000 docs.
Term "retrieval": tf = 3, appears in n₂ = 100 docs.

Using k₁ = 1.2, b = 0.75:

$$K = 1.2 \times ((1 - 0.75) + 0.75 \times \frac{250}{200}) = 1.2 \times (0.25 + 0.9375) = 1.425$$

IDF for "information":

$$\log \frac{10000 - 3000 + 0.5}{3000 + 0.5} = \log \frac{7000.5}{3000.5} \approx \log(2.333) \approx 0.847$$

IDF for "retrieval":

$$\log \frac{10000 - 100 + 0.5}{100 + 0.5} \approx \log(98.51) \approx 4.59$$

TF component for "information" (tf=5):

$$\frac{(1.2 + 1) \times 5}{1.425 + 5} = \frac{11}{6.425} \approx 1.712$$

TF component for "retrieval" (tf=3):

$$\frac{2.2 \times 3}{1.425 + 3} = \frac{6.6}{4.425} \approx 1.492$$

Final Score:

$$\text{Score} = (0.847 \times 1.712) + (4.59 \times 1.492) = 1.449 + 6.848 = \boxed{8.297}$$

Notice: "retrieval" contributes much more because it's rarer (higher IDF)!

Why BM25 is legendary: It's theoretically grounded (from BIM), includes TF saturation, handles document length, and has only 2–3 tunable parameters. It's still one of the best performing unsupervised ranking models — used by Elasticsearch, Solr, Lucene, and more!
Chapter 5

Language Models for IR

Important Distinction: "Language models" in classical IR are simple probability distributions over words (unigram multinomials) — NOT the neural LLMs (like GPT/BERT) used in modern NLP. Don't confuse them!

The Core Idea

Imagine each document generates words randomly according to its own probability distribution. Given a query, we ask: "Which document was most likely to have generated this query?"

THE GENERATIVE STORY

User thinks of ideal doc
→
Picks keywords as query
→
System finds matching doc

Unigram Language Model

A unigram model treats each word independently — like drawing words from a bag:

Query Likelihood
$$P(Q \mid D) = \prod_{q_i \in Q} P(q_i \mid D)$$

Rank documents by how likely each one is to generate the query.

The Maximum Likelihood Estimate (MLE) is simply:

Maximum Likelihood Estimate
$$P(q_i \mid D) = \frac{f_{q_i, D}}{|D|}$$

Term frequency in the document divided by document length.

Example: Document D = "weather in Glasgow is sunny Glasgow weather" (6 words total).

Query Q = "Glasgow weather"

$$P(\text{Glasgow} \mid D) = \frac{2}{6} = 0.333$$

$$P(\text{weather} \mid D) = \frac{2}{6} = 0.333$$

$$P(Q \mid D) = 0.333 \times 0.333 = 0.111$$

The Zero Probability Problem

What if a query term is missing from the document? Then \(P(q_i \mid D) = 0\), and the entire product becomes zero! This is too harsh — a document about "weather forecasting in Glasgow" shouldn't score zero just because it doesn't contain "forecast".

Solution: Smoothing! We "borrow" probability from seen words and give it to unseen words, using the collection as a background model.

Jelinek-Mercer Smoothing

Mix the document model with the collection model using a parameter λ:

Jelinek-Mercer Smoothed LM
$$P(q_i \mid D) = (1 - \lambda) \cdot \frac{f_{q_i, D}}{|D|} + \lambda \cdot \frac{c_{q_i}}{|C|}$$

λ = 0 → pure document model. λ = 1 → pure collection model.

Effect of λ on Behaviour

λ ValueBehaviourBest For
Small (≈ 0.1)Acts like Boolean AND — all terms must be presentShort queries (2–3 words)
Large (≈ 0.7)Acts like Boolean OR — coordination-level matchLong/verbose queries

Smoothing in Action:

Same doc D (6 words). Query: "Glasgow forecast". Collection C has 1,000,000 words total.

"forecast" has 0 occurrences in D, but cforecast = 500 in collection. Using λ = 0.1:

$$P(\text{Glasgow} \mid D) = 0.9 \times \frac{2}{6} + 0.1 \times \frac{c_{\text{Glasgow}}}{10^6}$$

$$P(\text{forecast} \mid D) = \underbrace{0.9 \times \frac{0}{6}}_0 + 0.1 \times \frac{500}{10^6} = 0.00005$$

Now the score is very small but not zero! The document still has a chance.

Dirichlet Smoothing

An alternative that adapts the amount of smoothing based on document length:

Dirichlet Smoothed LM
$$P(q_i \mid D) = \frac{f_{q_i, D} + \mu \cdot \frac{c_{q_i}}{|C|}}{|D| + \mu}$$

Typical μ: 1000 – 2000. Short docs get more smoothing, long docs less.

LM vs. BM25

Language Models don't explicitly model "relevance" — they model query generation. BM25 is generally considered more robust across different collections, while LM performance can vary. Both remain highly effective baselines.

Chapter 6

Divergence From Randomness (DFR)

DFR is an elegant information-theoretic framework developed at the University of Glasgow. The key intuition is beautifully simple:

"The more the divergence of the within-document term frequency from its frequency within the collection, the more information carried by the word in the document."

In other words: if a term appears in a document much more than you'd expect by random chance, that term is informative for that document.

Three Components of DFR

DFR Weighting Formula
$$w_{d,q} = \sum_{t \in q \cap d} (-\log P_1) \cdot (1 - P_2)$$
Randomness Model (P₁): How surprising is the term frequency? Measures divergence from a random baseline using distributions like Binomial or Poisson. The -log(P₁) gives high scores when the occurrence is unlikely under randomness (= informative).
Information Gain / After-effect (P₂): Controls TF absorption — repeated occurrences of a term should give diminishing returns. The Laplace model uses: \(\frac{tf}{tf + 1}\).
Document Length Normalisation: Normalises tf relative to average document length. Normalisation 2: \(tf_n = tf \cdot \log_2\left(1 + c \cdot \frac{\text{avglen}}{\text{len}(d)}\right)\)

PL2: A Concrete DFR Model

Poisson randomness + Laplace after-effect + Normalisation 2:

PL2 Scoring Function
$$\text{PL2}(d,q) = \frac{1}{tf_n + 1} \cdot \left( tf_n \cdot \log_2 \frac{tf_n}{\lambda} + (\lambda - tf_n) \cdot \log_2 e + 0.5 \cdot \log_2(2\pi \cdot tf_n) \right)$$

where \(\lambda = \frac{TF}{N}\) (average term frequency across documents)

Why DFR is special: It's not a single model but a framework for generating many models. Some (like DPH) are completely parameter-free — no tuning required! DPH performs excellently as a "one-size-fits-all" weighting model.

DFR Intuition with a Card Analogy:

Imagine a deck of cards representing all words in the collection. Each "ace" is an occurrence of term t. You draw l(d) cards (document length).

If you pull out far more aces than expected, something non-random is happening — that document is really about term t! DFR formalises this with the Binomial/Poisson distribution.

Chapter 7

Field-Based Models

Real documents aren't flat text — they have structure. A word in the title means something different than the same word buried in the body. Field-based models exploit this.

DOCUMENT FIELDS EXAMPLE

TITLE
Spanish Steps - Wikipedia
BODY
The Spanish steps is a set of stairs in Rome…
ANCHOR
Spanish Steps, Piazza de Spagna
URL
en.wikipedia.org/Piazza_de_Spagna

Two Approaches to Field Combination

Score Combination

Compute a separate score per field, then combine: \(\text{score} = \sum_f w_f \cdot \text{score}_f\)

⚠ Breaks TF absorption!

Frequency Combination ✓

Combine TFs first, then score: \(tf = \sum_f w_f \cdot tf_f\)

✓ Preserves TF absorption!

BM25F

Robertson et al. (2004) proposed combining per-field normalised term frequencies:

BM25F — Per-Field Normalisation
$$\widetilde{tf} = \sum_f w_f \cdot \text{Norm}_f(tf_f)$$

Each field gets its own weight \(w_f\) and its own length normalisation.

PL2F — DFR with Fields

Similarly, the DFR framework extends to fields with Normalisation 2F:

PL2F Normalised TF
$$tf_n = \sum_f w_f \cdot tf_f \cdot \log_2\left(1 + c_f \cdot \frac{\text{avglen}_f}{l_f(d)}\right)$$
Why field length matters differently: Increasing body length may dilute a term's importance, but increasing anchor text length means more inlinks — a sign of authority! Each field needs separate normalisation.
Chapter 8

Proximity Models

Unigram models treat documents as "bags of words" — word order doesn't matter. But it does! Searching for "University of Glasgow" shouldn't rank pages about "Glasgow Caledonian University" equally.

Markov Random Field (MRF) Model

Metzler & Croft (2005) formalized proximity using three types of dependencies:

Full Independence

Normal bag-of-words. Each term scored alone.

Sequential Dependence

Adjacent pairs: "white house", "house rose".

Full Dependence

All pairs: "white garden", "house rose", etc.

MRF Combined Score
$$\text{score}(d, Q) = \lambda_1 \sum_{t \in Q} w(t, d) + \lambda_2 \sum_{c \in Q_o} w(c, d) + \lambda_3 \sum_{c \in Q_u} w(c, d)$$

Qo = ordered pairs, Qu = unordered pairs. Typically λ₁ = 0.8, λ₂ = λ₃ = 0.1

Query: "white house rose garden"

Full Independence (λ₁ = 0.8):

0.8 × [w(white,d) + w(house,d) + w(rose,d) + w(garden,d)]

Sequential Dependence (λ₂ = 0.1):

+ 0.1 × [w("white house",d) + w("house rose",d) + w("rose garden",d)]

Full Dependence (λ₃ = 0.1):

+ 0.1 × [w("white rose",d) + w("white garden",d) + w("house garden",d)]

Documents where "white house" appears as a phrase get a significant boost!

Key Takeaways

Boolean is simple but inflexible — no ranking, no partial matches.
Vector Space introduced ranking but lacks theoretical foundations.
BIM gave us the probabilistic interpretation of IDF but ignores term frequency.
BM25 combines the best of probabilistic theory with TF saturation and length normalisation. Still a gold standard.
Language Models offer an intuitive "query generation" perspective with effective smoothing techniques.
DFR provides an information-theoretic framework with parameter-free models.
Field-based and Proximity models add structure and word-order awareness for real-world effectiveness.

The Modern Landscape

Today, these classical models form the backbone of first-stage retrieval. They're combined with machine learning (Learning to Rank), neural models (BERT-based re-rankers), and many other signals (PageRank, spam scores, user behavior) to build modern search systems. Understanding them is essential — they're the foundation everything else builds upon.
