01 Part 1

Relevance Estimation Models

Given a query and a document — how do we score how relevant they are to each other?

The Big Idea: Think of searching as matchmaking. You have a question, and millions of documents. Each model below is a different strategy for figuring out which documents are the best "match" for your question.
Aa

Lexical Models (Word Matching)

The simplest approach. These count how many words your query and a document share. If you search for apple pie recipe, it literally looks for those exact words.

Common algorithms: TF-IDF (term frequency × rarity) and BM25 (a smarter version that handles long documents better).

⚠️ Problem: Searching "grand prix" won't match a document that says "gp"
+

Learned Sparse Retrieval (LSR)

To fix the synonym problem, we use neural language models like BERT to read documents and expand them with related terms. A doc about a "collision" might get "crash", "accident", "mercedes" added behind the scenes.

Still sparse (word-based), but now much smarter about connecting related concepts.

Examples: SPLADE, DeepCT
→

Single-Vector Dense Models (Bi-Encoders)

Instead of individual words, these embed the entire query and entire document each into one big numeric vector (typically 500–2000 numbers). Then we measure how "close" they are in that vector space using cosine similarity or dot product.

Query → [0.3, 0.8, ...] ⟷ Doc → [0.2, 0.7, ...] = Score ✓
Example: DPR (Dense Passage Retrieval)
⊞

Multi-Vector / Interaction Models

Rather than crushing a whole document into a single vector, these keep a separate vector for every token. Then they use a trick called MaxSim — for each query word, find the most similar document word, then sum those scores.

More nuanced than a single vector, because fine-grained word-level matching is preserved.

Example: ColBERT
⚡

Mono Models (Cross-Encoders)

The heavyweight champion. These concatenate the query and document into one single input, then run it through every Transformer layer with full cross-attention. Every word in the query can "see" every word in the document.

Gives the best relevance scores — but is extremely expensive. Cost scales quadratically O(d²), so you can't run it over millions of documents.

Examples: MonoBERT, MonoT5

Duo+ Models

Like Cross-Encoders, but they evaluate multiple documents at once against the query and each other. Great for fine-grained re-ranking.

DuoBERT, RankGPT

Quality Estimators

These evaluate a document on its own — no query needed. They just ask: "Is this document high-quality enough to ever be relevant?" Think of it as spam filtering.

02 Part 2

Training

How do we actually teach these neural models what "relevant" means?

Key Insight: All these models learn from labeled examples — pairs of queries and documents that humans have judged as relevant or not relevant. The training strategy matters a lot.

Pointwise Training

Looks at one document at a time. Asks: "On a scale of 0 to 1, how relevant is this doc?" Optimizes using Mean Squared Error to push the score toward the label.

Simple, but doesn't learn relative ordering.

Pairwise Training

Uses triples: (Query, Good Doc, Bad Doc). Optimizes so the good doc always scores higher than the bad one. Uses Cross Entropy Loss.

Learns ranking — which is what search actually needs.

Finding Negative Examples

Datasets usually tell us which docs ARE relevant. But the model also needs to see non-relevant docs. Where do we find them?

🎲

In-Batch Negatives

The lazy but effective trick. When training in a batch of 32 queries, each query's relevant doc becomes a negative for the other 31 queries. Free negatives with zero extra computation.

🎯

Hard Negatives

Run a search engine and grab the top-ranked documents that are actually wrong. These are the "tricky" negatives — they look relevant but aren't. Forces the model to learn much finer distinctions.

⚠️ Risk: You might accidentally pick a doc that IS relevant (false negative)

Advanced Training Tricks

🎓 Distillation

Have a powerful but slow "teacher" model (like a Cross-Encoder) score documents. Then train a fast "student" model to mimic those scores. You get speed AND quality.

🤖 Synthetic Data

No real queries? Use an LLM to read your documents and generate fake but realistic queries. Surprisingly effective for bootstrapping training data.

03 Part 3

Retrieval Engines

You have a model that can score relevance. Now how do you search millions of documents quickly?

The Challenge: A Cross-Encoder gives amazing scores, but scoring every document in a billion-document corpus would take hours. We need smart data structures to make search fast.
🔨

Brute Force (Forward Index)

The naive approach. Pre-compute and store the representation for every document, then at search time, score every single one. Works on small datasets. Completely impractical at scale.

📖

Inverted Index (Sparse Retrieval)

The classic search engine trick. Build a lookup table: for each word, store which documents contain it. When someone searches "machine learning", only check documents that have those words. Massively scalable.

"machine" → [Doc 3, Doc 17, Doc 42]
"learning" → [Doc 3, Doc 8, Doc 42]
∩ Match: Doc 3, Doc 42
Only works with sparse (word-based) models
🌐

Approximate Nearest Neighbor (ANN)

For dense vector models, we build a graph where similar documents are connected as neighbors. At search time, we "walk" through the graph toward vectors close to our query. Fast, but approximate — we might miss the absolute best result.

Popular algorithm: HNSW (Hierarchical Navigable Small World) — builds a multi-layer graph for efficient navigation.

Used for: Bi-Encoder and other dense models
⭐

Re-Ranking (The Cascading Approach)

The most popular strategy in production. It combines the best of both worlds in two stages:

1

First Stage

Fast engine (Inverted Index or ANN) grabs top ~1000 candidates

2

Re-Rank

Expensive model (Cross-Encoder) precisely re-orders just those candidates

3

Result

Scalability of fast search + accuracy of deep models

⚠️ Limited by what the first stage retrieves — if a great doc is missed early, re-ranking can't save it

04 Bonus

Retrieval Augmented Generation (RAG)

Where retrieval meets generative AI — the architecture powering modern AI assistants.

The Shift: Search results used to be shown directly to humans as ranked lists. Now, an LLM sits in the middle — it reads the retrieved documents and generates a natural language answer for you.
1

User Query

"What causes northern lights?"

2

Retrieve

Search engine finds top relevant docs

3

Read

LLM reads retrieved documents

4

Generate

LLM writes a grounded answer

✓ Reduces Hallucinations

By grounding the LLM's output in actual retrieved documents, it's far less likely to make things up. The facts come from real sources.

✓ Enables Attribution

Because we know which documents were retrieved, we can trace every claim in the generated answer back to a specific source.

Recap

The Full Picture

User Query → Fast Retrieval
Inverted Index / ANN → Re-Rank
Cross-Encoder → LLM Reads Docs
RAG → Answer ✨
That's the modern search stack.
Sparse retrieval for speed → Dense re-ranking for precision → Generative AI for natural answers.
Each layer builds on the one before it.
