1. What is Information Retrieval?

Information Retrieval (IR) is the science of searching for information in documents, searching for documents themselves, searching for metadata that describes documents, or searching within databases.

Key Definition: Information Retrieval is finding material (usually documents) of an unstructured nature (usually text) that satisfies an information need from within large collections (usually stored on computers).

Real-Life Examples

Example 1: Google Search

When you search for "best pizza near me," Google's IR system:

  • Processes your query (tokenizes, removes stopwords like "me")
  • Searches billions of indexed web pages
  • Ranks results based on relevance, location, ratings, and other factors
  • Returns the most relevant pizza places in seconds

Example 2: Amazon Product Search

Searching for "wireless headphones under $100" involves:

  • Understanding intent: finding products (not articles about headphones)
  • Filtering by price constraint ($100)
  • Ranking by relevance, reviews, sales, and personalization
  • Handling synonyms (wireless = Bluetooth)

Example 3: Email Search (Gmail)

Finding "emails from John about meeting" demonstrates:

  • Structured + unstructured search (sender field + content)
  • Temporal relevance (recent emails ranked higher)
  • Context understanding (meeting = appointment, discussion, etc.)

IR vs. Traditional Databases

Aspect Information Retrieval Database Systems
Data Type Unstructured text Structured data
Query Type Natural language, keywords SQL, precise queries
Results Ranked by relevance Exact matches
Match Best match (approximate) Exact match

Component Details

Offline Processing (Indexing Time): Documents are crawled, processed, and indexed before any user queries arrive. This happens continuously as new content is added.
Online Processing (Query Time): When a user submits a query, the system processes it, retrieves matching documents, ranks them, and returns results - all within milliseconds.

Real-World: Netflix Search Architecture

When you search for "action movies with Tom Cruise":

  • Text Processing: Extracts ["action", "movies", "tom", "cruise"]
  • Index Lookup: Finds all titles matching these terms
  • Ranking: Considers: your viewing history, popularity, ratings, release date
  • Personalization: Prioritizes genres you watch frequently
  • Results: "Top Gun: Maverick" ranks higher than older films

Final Processed Terms for Indexing

1. Tokenization

Breaking text into individual units (tokens), typically words.

Challenge: "New York City"

Should this be 3 tokens ["New", "York", "City"] or 1 token ["New York City"]?

Solution: IR systems often use both approaches - create individual tokens AND recognize common phrases (n-grams).

2. Stopword Removal

Removing common words that carry little meaning: "the", "a", "an", "is", "are", etc.

Trade-off: Stopwords reduce index size but can affect phrase queries. "to be or not to be" becomes meaningless after stopword removal!

3. Stemming

Reducing words to their root form: "running", "runs", "ran" → "run"

Porter Stemmer Examples

  • connection, connections, connective → connect
  • computing, computer, computation → comput
  • excellence, excellent → excel

4. Indexing: The Inverted Index

The inverted index is the core data structure in IR systems, enabling fast retrieval.

What is an Inverted Index?

Core Idea: Instead of storing documents and searching through them sequentially, we create a dictionary mapping each term to the list of documents containing it.
Inverted Index Structure
Document Collection:
─────────────────────────────────────────────────────
Doc1: "Machine learning is transforming technology"
Doc2: "Deep learning and machine learning are related"
Doc3: "Technology advances with artificial intelligence"
Doc4: "Machine learning algorithms improve over time"

After Text Processing:
─────────────────────────────────────────────────────
Doc1: [machine, learn, transform, technolog]
Doc2: [deep, learn, machine, learn, relat]
Doc3: [technolog, advanc, artifici, intellig]
Doc4: [machine, learn, algorithm, improv, time]

Inverted Index:
─────────────────────────────────────────────────────
Term          │ Postings List (DocID, Frequency, Positions)
──────────────┼─────────────────────────────────────────────
advanc        │ → (Doc3, f:1, pos:[2])
algorithm     │ → (Doc4, f:1, pos:[3])
artifici      │ → (Doc3, f:1, pos:[3])
deep          │ → (Doc2, f:1, pos:[1])
improv        │ → (Doc4, f:1, pos:[4])
intellig      │ → (Doc3, f:1, pos:[4])
learn         │ → (Doc1, f:1, pos:[2])
              │   (Doc2, f:2, pos:[2,4])
              │   (Doc4, f:1, pos:[2])
machine       │ → (Doc1, f:1, pos:[1])
              │   (Doc2, f:1, pos:[3])
              │   (Doc4, f:1, pos:[1])
relat         │ → (Doc2, f:1, pos:[5])
technolog     │ → (Doc1, f:1, pos:[4])
              │   (Doc3, f:1, pos:[1])
time          │ → (Doc4, f:1, pos:[5])
transform     │ → (Doc1, f:1, pos:[3])

Why Inverted Index?

Performance Comparison

Without Index (Sequential Search):

  • Search for "machine learning" in 1 million documents
  • Must scan ALL documents: ~1,000,000 comparisons
  • Time: Several seconds ⏱️

With Inverted Index:

  • Look up "machine" in dictionary: 1 comparison
  • Look up "learning" in dictionary: 1 comparison
  • Intersect posting lists: ~100 comparisons
  • Time: Milliseconds ⚡

Index Construction

Index Building Process
Step 1: Parse documents and create (Term, DocID, Position) tuples
────────────────────────────────────────────────────────────────
Doc1: "search engine"     → (search, 1, 1), (engine, 1, 2)
Doc2: "engine optimization" → (engine, 2, 1), (optim, 2, 2)
Doc3: "search results"    → (search, 3, 1), (result, 3, 2)

Step 2: Sort by term (then by DocID)
────────────────────────────────────────────────────────────────
(engine, 1, 2)
(engine, 2, 1)
(optim, 2, 2)
(result, 3, 2)
(search, 1, 1)
(search, 3, 1)

Step 3: Group by term and create postings
────────────────────────────────────────────────────────────────
engine  → [(Doc1, pos:2), (Doc2, pos:1)]
optim   → [(Doc2, pos:2)]
result  → [(Doc3, pos:2)]
search  → [(Doc1, pos:1), (Doc3, pos:1)]

Step 4: Write to disk with compression
────────────────────────────────────────────────────────────────
[Dictionary] ← [Postings Lists stored efficiently]

5. Retrieval Models

Retrieval models determine HOW we match queries to documents and calculate relevance scores.

1. Boolean Retrieval Model

Concept: Documents either match or don't match - binary decision. Uses AND, OR, NOT operators.

Boolean Query Example

Query: "machine AND learning NOT deep"

Process:

  • Get postings for "machine": {Doc1, Doc2, Doc4}
  • Get postings for "learning": {Doc1, Doc2, Doc4}
  • Get postings for "deep": {Doc2}
  • Intersect machine AND learning: {Doc1, Doc2, Doc4}
  • Subtract deep: {Doc1, Doc4}

Result: Doc1 and Doc4 (no ranking, just matching)

2. Vector Space Model (VSM)

Concept: Represent documents and queries as vectors in a multi-dimensional space. Similarity = angle between vectors.
Vector Space Model Visualization
Terms: [machine, learning, deep, algorithm]

Document Vectors (based on term weights):
─────────────────────────────────────────────────────
Doc1: [0.8, 0.6, 0.0, 0.3]  (machine-heavy, no "deep")
Doc2: [0.5, 0.7, 0.9, 0.1]  (deep learning focus)
Doc3: [0.2, 0.1, 0.0, 0.8]  (algorithm focus)

Query: "machine learning"
Query Vector: [0.7, 0.7, 0.0, 0.0]

Similarity Calculation (Cosine Similarity):
─────────────────────────────────────────────────────
cos(Query, Doc1) = 0.92  ← Highest similarity
cos(Query, Doc2) = 0.78
cos(Query, Doc3) = 0.21  ← Lowest similarity

Ranking: Doc1 > Doc2 > Doc3

3. Probabilistic Models (BM25)

BM25 (Best Match 25) is the most widely used ranking function in modern search engines.

BM25 considers:
  • Term Frequency (TF): How often does the query term appear in the document?
  • Inverse Document Frequency (IDF): How rare is the term across all documents?
  • Document Length: Longer documents are normalized to avoid bias

Real-World: Elasticsearch Search

When searching GitHub code for "sort algorithm":

  • TF: Files with multiple occurrences of "sort" rank higher
  • IDF: "algorithm" is common (low IDF), "quicksort" is rarer (high IDF)
  • Length Normalization: A 50-line file with 3 mentions beats a 5000-line file with 5 mentions
  • Field Boosting: Matches in filename/title ranked higher than in comments

7. Evaluation of IR Systems

How do we measure if a search engine is working well?

Key Metrics

Precision: Of the results returned, how many are relevant?
Precision = (Relevant Results Retrieved) / (Total Results Retrieved)
Recall: Of all relevant documents in the collection, how many did we find?
Recall = (Relevant Results Retrieved) / (Total Relevant Documents)
Precision & Recall Visualization
Document Collection (100 documents total)
20 are actually relevant to the query

Search Engine Returns 15 Results:
┌─────────────────────────────────────────┐
│  Retrieved Results (15)                 │
│  ┌───────────────────────────────┐      │
│  │ Relevant & Retrieved (12) ✓✓✓│      │  ← True Positives
│  │                               │      │
│  └───────────────────────────────┘      │
│  Not Relevant but Retrieved (3) ✗✗✗    │  ← False Positives
└─────────────────────────────────────────┘

Not Retrieved (85):
┌─────────────────────────────────────────┐
│  Relevant but NOT Retrieved (8) ✓✓✓    │  ← False Negatives
│  Not Relevant & Not Retrieved (77)      │  ← True Negatives
└─────────────────────────────────────────┘

Calculations:
─────────────────────────────────────────
Precision = 12 / 15 = 0.80 (80%)
  → "80% of returned results are relevant"

Recall = 12 / 20 = 0.60 (60%)
  → "Found 60% of all relevant documents"

F1-Score = 2 × (0.80 × 0.60) / (0.80 + 0.60) = 0.686
  → Harmonic mean balancing precision & recall

Additional Metrics

Mean Average Precision (MAP)

Evaluates ranking quality across multiple queries:

  • Calculate Average Precision for each query
  • Take the mean across all queries
  • Higher MAP = better ranking overall

Normalized Discounted Cumulative Gain (NDCG)

Accounts for position and graded relevance:

  • Highly relevant results at top = high score
  • Relevant results buried deep = lower score
  • Allows graded relevance (not just binary relevant/not relevant)
  • Example: Perfect (5), Excellent (4), Good (3), Fair (2), Bad (0)

Real-World: A/B Testing at Google

Google runs thousands of experiments yearly:

  • Treatment: New ranking algorithm
  • Control: Current algorithm
  • Metrics:
    • Click-Through Rate (CTR) on top results
    • Time to successful click (faster = better)
    • Reformulation rate (do users re-search?)
    • Long-click rate (did user find what they need?)
  • Decision: Deploy if treatment shows 1-2% improvement
