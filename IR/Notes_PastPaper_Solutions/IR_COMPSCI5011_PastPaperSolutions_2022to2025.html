✦ Complete Solution
r1
rewriter1 — Pseudo-Relevance Feedback (PRF):
Information used: The top-3 documents retrieved by BM25 (bm25 % 3 means "take top 3"). These are assumed relevant without user input.

Operation: Extracts high-scoring terms from the top-3 documents to expand the original query. Terms selected have high TF in those docs but are relatively rare in the collection (high IDF).

  • Bo1 (Bose-Einstein 1) — Terrier's default PRF model. Uses DFR framework to weight expansion terms.
  • RM3 (Relevance Model 3) — Uses language model interpolation. Re-estimates P(t|relevance) from top-k docs.
  • KL-Divergence expansion — Selects terms that differentiate the top-k docs from the general collection.
r2
rewriter2 — Query-only expansion (no retrieved documents):
Information used: Only the original query text — no retrieved documents involved. The rewriter2 >> bm25 pipeline means rewriting happens before any retrieval.

Example approaches:
  • Thesaurus-based expansion — Adds synonyms/related terms from a manually constructed thesaurus (e.g., WordNet). "car" → adds "automobile, vehicle".
  • Word embedding expansion — Uses Word2Vec or GloVe to find nearest-neighbour terms in embedding space. Semantically similar terms added to query.
  • Spelling correction — Detects and corrects misspellings before retrieval. "iphone" ← "iphon".
  • Query classification — Detects navigational vs informational intent to reformulate query structure.

Q2c4 marks4 assumptions made by IR models that don't hold
✦ Model Answer (1 mark each)
1
Term Independence (Bag of Words): Models assume terms contribute independently to relevance. In reality, "New York" as a phrase has a completely different meaning to "New" and "York" separately. A query for "New York Times" might retrieve documents about "new times in York, England".
2
Document Independence: Each document is scored independently of others. In reality, retrieving 10 nearly-identical documents about the same sub-topic is less useful than 10 diverse relevant documents. Users care about novelty — a document already seen is not useful even if highly relevant.
3
Static Relevance: Relevance is treated as a fixed property. In reality, a news article about "COVID vaccines" from March 2020 might be highly relevant then but misleading or outdated in 2026. Temporal context matters.
4
Topical Relevance = User Relevance: Models rank by topical match, but users also require credibility, readability, and authority. A highly relevant but peer-reviewed medical paper may be useless to a patient who cannot understand it. A Wikipedia article about "cancer treatment" might be more useful than the most topically-relevant oncology journal paper.
Q2d5 marksTAAT vs DAAT traversal steps and memory requirements
term1 → [id=2, tf=2], [id=5, tf=1], [id=6, tf=2]
term2 → [id=2, tf=4], [id=3, tf=5], [id=5, tf=5]
Retrieve K results from corpus of N documents (K < N).
✦ Complete Solution
TAAT
Term-At-A-Time — process one complete posting list before the next:
  1. Initialise score accumulator array A[1..N] = 0
  2. Process term1's full list: A[2] += w(2,2), A[5] += w(5,1), A[6] += w(6,2)
  3. Process term2's full list: A[2] += w(2,4), A[3] += w(3,5), A[5] += w(5,5)
  4. Sort all non-zero accumulators; return top K
Memory requirement: O(N) — one accumulator per document in the entire corpus. For N=1 billion docs this is impractical.
DAAT
Document-At-A-Time — process all terms for one doc before moving to next doc (sorted by docID):
  1. Maintain pointer into each posting list. Maintain top-K min-heap.
  2. Next candidate docID = 2 (lowest across all lists)
  3. Doc 2: present in term1 (tf=2) AND term2 (tf=4) → compute score → add to heap if qualifies
  4. Doc 3: term1 not present, term2 (tf=5) → compute score → check heap
  5. Doc 5: term1 (tf=1), term2 (tf=5) → compute score → check heap
  6. Doc 6: term1 (tf=2), term2 not present → compute score → check heap
  7. Return K documents from heap
Memory requirement: O(K) for the heap + O(|query terms|) for current posting list pointers. For K=10, this is essentially O(1) regardless of N.
✔ Key Difference
TAAT: O(N) memory — must hold an accumulator for every document. For large corpora this is prohibitive.
DAAT: O(K) memory — only maintains a K-element heap and a few posting list pointers. Scales to any corpus size.

Q3a4 marks4 key components shared by most advanced retrieval models
✦ Model Answer (1 mark each)
1
Term Frequency (TF) component: Captures how often a query term appears in the document. More occurrences = stronger topical signal. BM25 uses a saturating TF function; PL2 models TF as a Poisson process. Important because term repetition indicates the document discusses the topic in depth.
2
Inverse Document Frequency (IDF) / Informativeness component: Captures how rare a term is across the collection. Common terms (appearing in most docs) discriminate nothing; rare terms carry high information. BM25 uses \(\log\frac{N-df+0.5}{df+0.5}\); PL2 uses informativeness from DFR framework.
3
Document Length Normalisation: Long documents naturally contain more terms, giving them artificially high TF scores. A 10,000-word document mentioning "python" 20 times is not necessarily more relevant than a focused 200-word document mentioning it 4 times. BM25 uses parameter b (0=none, 1=full); PL2 uses normalisation relative to average document length.
4
Query Term Weighting: Not all query terms are equally important. IDF of query terms themselves can weight contributions — rare query terms are more discriminating. For long queries (e.g., from relevance feedback), query TF also matters. BM25 uses IDF of query terms; PL2 applies query frequency normalisation.
Q3b(i)2 marksWhy does searching "i-phone" fail? Propose a fix.
✦ Model Answer

Problem: The standard tokeniser splits "i-phone" at the hyphen into two tokens: "i" and "phone". The token "i" is typically a stopword (or has near-zero IDF) and is removed. The query effectively becomes just "phone", losing the brand-specific signal. Documents indexed under the token "iphone" (without hyphen) will NOT match "phone" — so genuine iPhone product pages are missed.


Solution (lexical, no dense retrieval): Modify the tokeniser to handle hyphens specifically for compound terms: either (a) treat "i-phone" as a single token "iphone" (merge the hyphen), or (b) index both the hyphenated form AND the merged form at the same positions. This ensures "i-phone" queries match "iphone" documents and vice versa. Optionally, add product-specific synonym rules (e.g., "iphone" ↔ "i-phone").

Q3b(ii)2 marksLTR changes to promote higher-margin brands
✦ Model Answer

Add a query-independent feature called profit_margin_score (or brand_revenue_weight) to the LTR feature vector. This feature maps each product/brand to a numeric value representing its profit contribution.

At training time, LambdaMART or another LTR algorithm will learn the weight for this feature alongside all relevance-based features (BM25 score, title match, PageRank, etc.). The model will automatically discover the optimal trade-off between relevance and profitability from training data.

The training labels must include both relevance judgements (to maintain result quality) and business objective signals, or a custom loss function can be defined that jointly optimises NDCG and revenue-weighted ranking.

Q1a(i–iii)4 marksWord variants in IR
✦ Complete Solution
i
What are word variants? Different surface forms of the same underlying concept. Includes: inflectional forms (run/running/ran), derivational forms (nation/national/nationality), spelling variants (colour/color, organise/organize), hyphenation (e-mail/email), abbreviations (USA/United States).

Example not in lectures: "utilise" vs "utilize" — British vs American spelling. A system without variant handling would fail to match a UK document about "utilise" to a US query "utilize".
ii
Why variants affect IR performance: Different variant forms are indexed as completely distinct terms. A document containing "organise" will NOT be retrieved for a query containing "organize", even though they represent the same concept. This reduces effectiveness because relevant documents become unreachable.
iii
Mainly affects Recall, not Precision. Recall is reduced because relevant documents using alternative variant forms of the query terms will not be retrieved — they're simply absent from the result set. Precision may actually improve slightly (only exact matches are retrieved), but the system misses too many relevant documents overall. Stemming/lemmatisation is the standard solution — it conflates variants to a common root so "organise" and "organize" both → "organis".

Q3c(i)5 marksWhy is Recall@1000 p-value NaN for reranker?
✦ Model Answer

The pipeline is: reranker = first_stage >> pt.text.get_text(dataset, "text") >> MonoT5()

The MonoT5 cross-encoder re-ranker only re-scores and re-orders documents already retrieved by first_stage. It cannot retrieve new documents — it is applied to a fixed candidate set of 1000 documents provided by BM25.

Therefore, Recall@1000 for reranker is identical to Recall@1000 for first_stage on every single query. The set of 1000 documents in both cases is exactly the same; only their ordering changes.

When the paired t-test computes the per-query differences: Recall@1000(reranker) − Recall@1000(first_stage) = 0 for ALL queries. The variance of these differences is zero. A t-test requires dividing by standard deviation — dividing by 0 produces an undefined (NaN) result. The test statistic is undefined because there is literally no variance to test.

Q3b4 marksWhy upper bounds facilitate search efficiency
✦ Model Answer

For each term t in a posting list, we precompute UB(t) = the maximum possible score contribution of t to any document in its posting list (the highest tf×idf or BM25 partial score across all documents).

During DAAT query processing, we maintain a threshold θ = the score of the Kth-best document found so far. For candidate document D currently being processed, we know its accumulated partial score from terms already evaluated. The maximum total score D could possibly achieve = current partial score + sum of UB(t) for all remaining unprocessed query terms.

If this maximum is less than θ, document D cannot enter the top-K regardless of its remaining term scores. We can skip D entirely without evaluating the remaining terms.

Concrete example: Query terms t1, t2, t3. UB(t1)=5, UB(t2)=3, UB(t3)=1. Current threshold θ=7. After processing t1, doc D has score 2. Max possible = 2+3+1=6 < 7 → skip D. We never look up t2 and t3 for D. This is MaxScore. WAND extends this by also advancing posting list pointers to jump over entire blocks of skippable documents at once.

Q3d7 marksDense retrieval, long documents, passage aggregation
✦ Model Answer
why
Why dense retrieval can't handle long documents:
Dense bi-encoders (e.g., BERT-based) compress an entire document into a single fixed-size dense vector (e.g., 768 dimensions from the [CLS] token). For long documents, this single vector cannot capture all the information — early content dominates, later content is "forgotten". Furthermore, BERT has a hard token limit (~512 tokens), so long documents physically cannot be processed in one pass without truncation, causing loss of potentially relevant content later in the document.
sol
Solution: Split documents into short overlapping passages (e.g., 100-word windows with 50-word overlap). Encode each passage independently. At query time, retrieve relevant passages and aggregate passage scores to produce a document-level score.
M1
MaxPassage: Document score = score of its single highest-scoring passage.
Intuition: "A document is only as relevant as its most relevant section." One strong passage is sufficient evidence of relevance.
For a document with relevant content spread throughout: MaxPassage is suboptimal — it finds the single best passage but ignores the cumulative evidence from multiple relevant sections. A document with 5 moderately relevant passages might score lower than one with 1 highly relevant passage, even though the first is more comprehensively on-topic.
M2
SumPassage / AveragePassage: Document score = sum (or average) of all passage scores.
Intuition: "Multiple relevant passages provide stronger cumulative evidence." Each relevant passage contributes to the final score.
For spread relevant content: SumPassage is much better — it accumulates evidence across all relevant passages, rewarding documents where relevant information is distributed throughout rather than concentrated. However, it can bias towards longer documents (more passages = higher sum), which is why normalisation (average) may be preferred.

Q1b5 marksRaw TF problem and solutions
✦ Model Answer

(i) Problem with raw TF in web search: Raw TF is proportional to document length — a 10,000-word document containing "python" 20 times is ranked above a focused 200-word document containing it 4 times, even though the shorter document may be more relevant. Spammers can also artificially inflate TF by keyword stuffing ("python python python python...") to manipulate rankings without genuine relevance.

(ii) Solutions:

  • Log TF: \(w = 1 + \log(tf)\) if tf>0. Ten occurrences gives weight 1+log(10)=2, but 100 occurrences only gives 1+log(100)=3 — not 10×. Diminishing returns.
  • BM25 TF saturation (k₁ parameter): \(\frac{tf \times (k_1+1)}{tf + k_1 \times (\ldots)}\). As tf→∞, this approaches (k₁+1) — a hard ceiling. With k₁=1.2: tf=1 → 1.09, tf=5 → 1.75, tf=100 → 2.18, tf→∞ → 2.2. The contribution saturates and cannot be gamed by keyword stuffing.
  • Length normalisation (BM25 b parameter): Divides TF by a function of document length relative to average. Long documents pay a penalty, short documents get a boost.
Q1c5 marksEvaluate a new neural web ranking approach
✦ Model Answer
1
Define test collection: Use an existing standard test collection (e.g., TREC Web Track, MS MARCO) with a fixed document corpus, query set (topics), and relevance judgements (qrels). This ensures fair comparison on identical data.
2
Run both systems: Apply the original ranking approach and the new neural approach to all queries in the test set. Collect the full ranked lists (top 1000 per query) for each system.
3
Compute evaluation metrics: Calculate MAP, NDCG@10, P@10, Recall@1000 for each system on each query using the relevance judgements. These capture different aspects of retrieval quality (early precision vs overall ranking quality).
4
Statistical significance test: Apply paired t-test or Wilcoxon signed-rank test on per-query metric values (e.g., per-query AP for MAP comparison). If p<0.05 AND the new system has a higher mean metric value, we conclude the improvement is statistically significant and not due to random variation.
5
Sanity check: Run an A/A test (compare system vs itself) to verify the evaluation pipeline produces p≈1.0 (no spurious differences). Report both the metric improvement and p-value.

doc1 should rank higher. Here's why: w2 is rare (P(w2|C)=0.1 vs P(w1|C)=0.8), making it the more informative query term. doc1 has 3 occurrences of w2 vs doc2's 1 occurrence — doc1 is much stronger on the rare, informative term. The smoothed result (doc1>doc2) correctly reflects this: JM smoothing's collection term P(w1|C)=0.8 is so dominant that after smoothing, both docs have similar w1 probabilities (~0.74 vs 0.79), while the difference in w2 (0.12 vs 0.10) drives the final ranking in doc1's favour.

Q3b4 marksWhy positive feedback is more useful than negative
✦ Model Answer

Positive feedback (marking documents relevant) directly identifies what the user is looking for. In Rocchio, the positive centroid \(\frac{\beta}{|D_R|}\sum_{D_R}\vec{d}\) moves the query towards relevant documents — this is a clear, targeted, well-defined direction in vector space. Each relevant example adds new discriminating terms that characterise the topic of interest.

Negative feedback is less useful because: (1) The space of non-relevant documents is enormous and diffuse — there are countless ways to be irrelevant. Moving the query "away from non-relevant" has no single coherent direction. (2) Negative examples may share terms with positive examples, causing the feedback to inadvertently suppress genuinely useful terms. (3) In Rocchio, γ (negative weight) is set much smaller than β (0.15 vs 0.75) precisely because the negative feedback direction is noisy.

Example: A user searching for "python programming tutorials". If they mark a document about "python the snake" as non-relevant, the negative feedback would suppress the term "python" — which is also the most important query term for relevant programming documents! Positive feedback on actual programming tutorials would reinforce terms like "code", "syntax", "function" without this problem.

Q3c5 marksNeural re-ranking limitation and solution
✦ Model Answer

Key limitation — the Recall Ceiling Problem: Neural re-rankers only re-score documents already retrieved by BM25 in the top-K. If a relevant document falls below rank K in BM25's output (e.g., because it uses different terminology than the query), the neural model never sees it and can never recover it. The re-ranker is strictly bounded by first-stage recall — it can improve the ordering of K candidates but cannot expand the candidate set.

Solution — Dense Retrieval as First Stage: Replace or augment BM25 with a dense bi-encoder retrieval model (e.g., DPR, ANCE). Dense models encode queries and documents in a shared semantic vector space and retrieve documents by approximate nearest-neighbour search. They can find relevant documents that have no lexical overlap with the query (e.g., a document about "automobiles" for a query about "cars"). Key steps:

  1. Fine-tune a bi-encoder (e.g., BERT-based) on relevance-labelled (query, positive doc, negative doc) triples
  2. Encode all documents offline → store dense vectors in ANN index (e.g., FAISS)
  3. At query time: encode query → retrieve top-K by cosine/dot product similarity → pass to cross-encoder re-ranker
  4. Alternatively, use hybrid (BM25 + dense) to get the benefits of both lexical and semantic retrieval

Single HITS iteration starting from h=a=(1,1,1,1):

Update authority a(p) = Σh(q) for q→p:
a(1): 2→1 ✓, 4→1 ✓ → a(1) = h(2)+h(4) = 1+1 = 2

Normalise by max=2: a = (1, 0, 1, 0.5)

Update hub h(p) = Σa(q) for p→q:
h(1): 1→3 → h(1) = a(3) = 2
h(2): 2→1, 2→3 → h(2) = a(1)+a(3) = 2+2 = 4
h(3): 3→4 → h(3) = a(4) = 1
h(4): 4→1 → h(4) = a(1) = 2
Normalise by max=4: h = (0.5, 1, 0.25, 0.5)
iii
Matrix-based alternative:
Let A be the adjacency matrix above. Then:
• Authority scores = principal eigenvector of M_a = ATA
• Hub scores = principal eigenvector of M_h = AAT

Steps:
1. Compute AT (transpose of A)
2. Compute M_a = ATA (4×4 matrix)
3. Compute M_h = AAT (4×4 matrix)
4. Apply power iteration to M_a: start with uniform vector v=(1,1,1,1), repeatedly compute v' = M_a·v, normalise v' by its L2 norm, until convergence. The converged vector is the authority score vector.
5. Similarly apply power iteration to M_h for hub scores.
The eigenvectors correspond to the same converged values as iterative HITS.
