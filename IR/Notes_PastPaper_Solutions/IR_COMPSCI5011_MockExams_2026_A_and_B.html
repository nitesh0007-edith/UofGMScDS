The log table on the cribsheet is log₁₀ but DCG uses log₂ — the question will provide specific log₂ values (e.g. "use log₂(3)=1.585, log₂(5)=2.322").

Q2 (~20 marks): Weighting and Relevance Feedback. Heap's Law estimation + Rocchio revised query vector + JM smoothing ranking + one conceptual question (IDF scenarios, assumptions, or positive vs negative feedback).

Key cribsheet insight
The log table is base-10. The IDF formula on the cribsheet says log(N/Dk) — no base specified. In past papers, log₂ was used. For closed book, expect the question to specify the base or provide the value. For Rocchio, the formula is given exactly — memorise which terms get α (query), β (relevant), γ (non-relevant). For JM smoothing, the formula is a product over all query terms — don't forget the product notation.

Q2e2 marksWhy is positive relevance feedback generally more useful than negative feedback in Rocchio? Give a specific example to support your answer.
✦ Model Answer

Positive feedback moves the query directly towards relevant documents — there is a single, well-defined direction to move in vector space. The relevant centroid represents what the user wants, so adding β×(relevant centroid) is a targeted, coherent update.

Negative feedback is less useful because: (1) The space of non-relevant documents is enormous and incoherent — there are infinitely many ways to be irrelevant. Moving "away" from non-relevant has no single clear direction. (2) Terms in non-relevant documents may overlap with relevant ones.

Example: Query "python programming". A user marks a document about "python (the snake)" as non-relevant. The negative feedback would suppress the term "python" — the most important term for the actual relevant programming documents! Positive feedback on a relevant programming tutorial would instead add terms like "code", "function", "syntax" without this risk.

This is why γ (negative weight) is set much smaller than β in practice (e.g. Rocchio defaults: α=1, β=0.75, γ=0.15).

Question 3 — Modern IR Systems (20 marks)
Q3a4 marksTAAT vs DAAT — memory complexity for retrieving top-K results from N documents.
Posting lists for query terms: term1 → [doc2, doc5, doc9], term2 → [doc2, doc3, doc7, doc9]. Retrieve top K=2 results from N=1,000,000 documents.
(i) Describe TAAT processing steps. (ii) Describe DAAT processing steps. (iii) Compare memory requirements.
✦ Model Answer
TAAT
Term-At-A-Time:
1. Initialise score accumulator array A[1..N] = 0 for all N=1,000,000 documents
2. Process term1's full posting list: A[doc2] += score, A[doc5] += score, A[doc9] += score
3. Process term2's full posting list: A[doc2] += score, A[doc3] += score, A[doc7] += score, A[doc9] += score
4. Scan all N accumulators and extract top K=2
Memory: O(N) — must maintain one accumulator per document in the entire corpus
DAAT
Document-At-A-Time:
1. Maintain pointer into each posting list. Maintain a K=2 min-heap.
2. Next candidate = doc2 (smallest docID across all lists). Accumulate term1 score + term2 score for doc2. Update heap.
3. Next candidate = doc3. Only term2 contributes. Update heap.
4. Next = doc5 (term1 only). Next = doc7 (term2 only). Next = doc9 (term1+term2).
5. Return top-2 from heap.
Memory: O(K) — only a K-element heap plus pointers for each query term
iii
TAAT requires O(N) = O(1,000,000) memory — prohibitive for web-scale corpora. DAAT requires O(K) = O(2) memory regardless of collection size — scales to billions of documents. DAAT is the dominant approach in production systems.
Q3b4 marksState four assumptions made by IR models that do not hold in practice. For each, briefly explain why it fails and give a consequence.
✦ Model Answer (1 mark each)
1
Term Independence (Bag of Words): Models assume terms contribute independently. Fails because word order and co-occurrence matter: "New York" ≠ "New" + "York". Consequence: phrase queries are not correctly scored — a document with "New" and "York" in different sentences scores the same as one with "New York" as a named entity.
2
Document Independence: Each document is scored independently of others in the result set. Fails because users care about diversity — ten documents on the same sub-topic are less useful than ten covering different aspects. Consequence: result sets can be redundant; a cluster of near-duplicate documents may fill the top-10.
3
Static Relevance: Relevance is treated as a fixed property of a (query, document) pair. Fails because relevance is temporal: a news article about "vaccine safety" from 2020 may contradict current medical consensus. Consequence: outdated but topically-matching documents rank alongside current, accurate documents.
4
Topical Relevance = User Relevance: Models rank by topical match, assuming this equals what the user needs. Fails because users also need credibility, readability, and authority. Consequence: a peer-reviewed paper on "blood pressure medication" may rank below a tabloid article that happens to have higher TF-IDF, even though the peer-reviewed source is more useful.
Q3c5 marksNeural IR architectures — bi-encoder vs cross-encoder trade-offs.
(i) Why can a bi-encoder be used as a first-stage retriever but a cross-encoder cannot? [2 marks]
(ii) What does MaxSim measure, and which architecture does it approximate? [2 marks]
(iii) Compute MaxSim for the query and document below [1 mark]. Use: MaxSim = Σ_{qi∈Q} max_{dj∈D} sim(qi, dj).
d1d2d3
q10.90.30.5
q20.10.80.4
✦ Model Answer
i
Bi-encoder: Encodes query and document independently into single dense vectors (e.g., BERT [CLS] token). Document vectors can be computed offline and stored in an ANN (approximate nearest-neighbour) index. At query time, only the query needs encoding. Retrieval is O(1) dot-product search — scales to billions of documents.

Cross-encoder: Encodes (query, document) pairs jointly using full token-to-token attention. This requires encoding every candidate (query, document) pair at query time — O(|d|²) cost per candidate. For a corpus of millions of documents this is computationally infeasible as a first stage.
ii
MaxSim measures the relevance between a query and a document using late interaction. For each query token, it finds the most similar document token (max over all doc tokens), then sums these per-query-token maxima. It approximates a bi-encoder with token-level granularity (ColBERT architecture) — more expressive than a single vector bi-encoder but far cheaper than a full cross-encoder, since document token embeddings can be pre-computed offline.
iii
MaxSim(q1) = max(0.9, 0.3, 0.5) = 0.9    [best match: d1]

Q2b3 marksWhat problem does smoothing solve in language models? Explain with a worked example using the JM formula.
✦ Model Answer

The zero probability problem: The maximum likelihood estimate P(w|d) = tf(w,d)/|d| assigns probability 0 to any word not seen in document d. With the product formula P(q|d) = ∏P(w|d), if any query term has tf=0 in a document, the entire document scores zero — even if it contains all other query terms and is highly relevant. This is clearly incorrect.

JM smoothing solution: Interpolate between the document-specific probability and the collection-level probability. Even if tf(w,d)=0:

P(w|d) = (1−λ)×0/|d| + λ×P(w|C) = λ×P(w|C) > 0

Example: Query term "python", tf(python,d)=0, |d|=100, P(python|C)=0.005, λ=0.5.
Without smoothing: P(python|d)=0 → document score=0 regardless of other terms.
With JM: P(python|d)=0.5×0+0.5×0.005=0.0025 → document can still contribute to ranking.

Question 3 — Modern IR: Neural Models and Online Evaluation (20 marks)
Q3a5 marksDense retrieval and long documents.
(i) Explain why dense bi-encoder retrieval struggles with long documents [2 marks].
(ii) Define MaxPassage and SumPassage aggregation. For documents where relevant information is spread evenly throughout, which aggregation method is more appropriate and why? [3 marks]
✦ Model Answer
i
Dense bi-encoders (e.g., BERT-based) encode an entire document into a single fixed-size vector (e.g., 768-dim [CLS] embedding). For long documents this single vector cannot capture all the information — early content tends to dominate; later relevant content is effectively lost. Additionally, BERT has a hard token limit (~512 tokens), so long documents must be truncated, discarding potentially relevant content in later sections entirely.
ii
Solution: Split each document into short overlapping passages. Encode each passage independently. At query time, retrieve passages and aggregate passage scores to produce a document-level score.

MaxPassage: Document score = score of its single highest-scoring passage. "A document is as relevant as its best section."

SumPassage: Document score = sum (or average) of all passage scores. "Cumulative evidence across all relevant passages."

For evenly-distributed relevance → SumPassage is more appropriate. When relevant information is spread throughout (e.g., a comprehensive review article where each section contributes), summing passage scores accumulates evidence from all relevant sections — a document with 10 moderately-relevant passages scores higher than one with 1 highly-relevant passage. MaxPassage would only capture the single best section and miss the cumulative signal from the remaining relevant content.

Q3c5 marksNeural IR efficiency-effectiveness trade-off.
(i) Explain why the p-value for Recall@1000 is NaN when comparing a BM25 pipeline against a pipeline that adds a MonoT5 re-ranker on top of BM25 [2 marks].
(ii) Propose three ways to improve Recall@1000 using neural components. For each, explain the mechanism [3 marks].
✦ Model Answer
i
The MonoT5 re-ranker only re-scores documents already retrieved by BM25. It cannot retrieve new documents — the candidate set of 1000 is fixed by BM25. Therefore, Recall@1000 is identical for both systems on every query. The paired t-test computes differences that are all exactly zero — the standard deviation of differences = 0. Dividing by zero standard deviation produces NaN (undefined). This is not a bug but a mathematically correct result indicating zero variance in the comparison.
ii
1. Dense bi-encoder as first stage: Replace or augment BM25 with a dense retriever (e.g., DPR, ANCE). Encodes query and documents in semantic vector space — retrieves semantically similar documents even with zero lexical overlap (vocabulary mismatch). Directly expands the set of retrievable relevant documents.

2. Hybrid retrieval (BM25 + dense, union): Take the union of BM25's top-1000 and dense retriever's top-1000. Each method recovers different relevant documents (BM25: exact lexical match; dense: semantic match). The combined pool has higher recall than either alone.

3. Learned sparse (SPLADE): Uses a neural model to expand both query and document with semantically related terms before standard inverted-index retrieval. The query "car" also retrieves documents about "automobile", "vehicle", "sedan" without any change to the retrieval infrastructure. Increases recall by expanding the term-matching space.

The MaxSim computation is then a simple dot-product lookup — no BERT forward pass needed per document. This reduces online latency from O(N×model_forward) to O(N×|q|×|d|×dim) — fast matrix operations over pre-stored embeddings.
