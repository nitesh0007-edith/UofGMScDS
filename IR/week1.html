🎯 What is Information Retrieval?

Week 1 Core Concept

Definition

Information Retrieval is the science of search engines.

It focuses on how to best address the information needs of users by:

  • Effectively: Get the right information to a user!
  • Efficiently: Get it to users quickly!

Real-World Examples

🌐 Web Search

Google, Bing processing billions of queries daily

  • Search the entire web
  • Rank billions of pages
  • Return results in milliseconds

📧 Email Search

Finding messages in Gmail, Outlook

  • Search your personal emails
  • Filter by date, sender, attachments
  • Full-text search across content

📚 Academic Search

ACM Digital Library, IEEE Xplore

  • Search scholarly papers
  • Filter by field, year, citations
  • Find relevant research

🤖 AI Assistants

ChatGPT, Claude using RAG (Retrieval-Augmented Generation)

  • Combine IR with LLMs
  • Retrieve relevant documents
  • Generate informed responses

🔄 Information Retrieval vs. Databases

Week 1

📚 Reference: L01-Intro.pdf - Pages 7, 13-14

Topics: Documents vs. Database Records, Imprecision in IR

The Fundamental Difference

💾 Database Query

  • Structured Query: SQL, precise syntax
  • Structured Data: Well-defined fields
  • Exact Match: Find all records meeting criteria
  • Accuracy: Results are verified and complete
  • Example: SELECT * FROM trains WHERE destination='Edinburgh'
VS

🔍 Information Retrieval

  • Natural Language: Keywords, phrases
  • Unstructured Data: Free text, documents
  • Best Match: Ranked list of relevant docs
  • Heuristics: Estimate relevance
  • Example: "Glasgow buildings architecture"

Database Records vs. Documents

Aspect Database Records Documents (IR)
Structure Well-defined fields (attributes) Unstructured text
Example Bank record: account number, balance, name, address News article, email, web page
Matching Easy to compare fields Complex - need to understand content
Result "Right" answer exists Relevance is subjective

💡 IR Techniques are Essentially Heuristics

Most Computer Science algorithms have a "right" answer. A heuristic tries to guess something close to the right answer.

Consider these problems:

  • ✓ Sort 10 integers - Has a correct answer
  • ✓ Find the highest integer - Has a correct answer
  • ✓ SELECT beers WHERE maker='X' - Has a correct answer
  • ❓ Find documents most relevant to "hippos in the zoo" - No single "right" answer!

Reference: L01-Intro.pdf, Page 14

What Makes a Document Relevant?

🤔 Many Factors to Consider

  • It contains the query terms?
  • It contains ALL of the query terms?
  • It contains the query terms many times?
  • It contains the query terms in a short document?
  • It contains the query terms close together?
  • It is fresh/recent?
  • It contains terms similar to the query terms?
  • It is authoritative (has many links)?
  • It has been clicked by many other people for the same query?

ALL of these are heuristics. They are not guaranteed to get a correct, relevant document for all users.

2️⃣ Indexing

Purpose: Store documents for fast retrieval

  • Inverted Index: Term → Document mapping
  • Dictionary: List of all unique terms
  • Postings: Documents containing each term
  • Speed: Query billions of docs in milliseconds

3️⃣ Retrieval

Purpose: Match queries to relevant documents

  • Query Processing: Apply same text processing
  • Matching: Find documents with query terms
  • Ranking: Order by relevance score
  • Output: Sorted list of documents

📝 Text Processing (Part 1)

Week 2

📚 Reference: L02-Architecture.pdf - Pages 7-19

Section: PART 1: TEXT PROCESSING

The "Bag of Words" Model

Fundamental Assumption

We treat all words in a document as independent index terms, disregarding:

  • ❌ Order of words
  • ❌ Structure (sentences, paragraphs)
  • ❌ Meaning and context
  • ❌ Relationships between words

Why? It's simple, yet surprisingly effective!

Reference: L02-Architecture.pdf, Page 8

Step 1: Tokenization (Lexical Analysis)

Breaking Text into Words

Input: "McDonald's slims down spuds. Fast-food chain to reduce certain types of fat."

Output: [McDonald's, slims, down, spuds, Fast-food, chain, to, reduce, certain, types, of, fat]

Challenges:

  • Numbers: 1999 vs. 510B.C
  • Hyphens: state-of-the-art vs. B-49
  • Punctuation: 510B.C vs. list.id
  • Case: Bank vs. bank

Small decisions can have major impact on effectiveness!

Reference: L02-Architecture.pdf, Pages 11-12

Step 2: Stopword Removal

Removing Common Words

Stopwords are words that are too frequent and don't help discriminate between documents.

Examples: the, am, and, is, are, of, at, in, to

Strategies:

  • List lookup: Negative dictionary/stopword list
  • Frequency analysis: Remove terms in >80% of documents
  • Collection-based: Use frequency info from other collections

⚠️ Caution: Some phrases need stopwords: "to be or not to be"

Reference: L02-Architecture.pdf, Page 13

Step 3: Stemming

Reducing Words to Root Form

Goal: Conflate word variants into a single form

Examples:

  • connect → connect (root)
  • connected → connect
  • connecting → connect
  • connections → connect

Benefits:

  • ✓ Increases recall - find more relevant documents
  • ✓ Reduces index size by 10-50%
  • ✓ Makes system more robust to user query variants

Problems:

  • ❌ GRAVITY has two meanings
  • ❌ GRAVITATION → GRAVITY (loses meaning distinction)

Reference: L02-Architecture.pdf, Pages 14-17

📚 Indexing (Part 2)

Week 2

📚 Reference: L02-Architecture.pdf - Pages 20-32

Section: PART 2: INDEXING

The Challenge

⚡ Speed is Critical

The English web has billions of documents. When a query is received:

  • ❌ Can't fetch all billions of pages and search them
  • ❌ Too slow to scan each document sequentially
  • ✅ Need a specialized data structure: The Index

Reference: L02-Architecture.pdf, Page 21

The Inverted Index

Main Idea

Invert the 'Document → Term' relationship to create a 'Term → Document' index

Components:

  • Lexicon (Dictionary): All unique terms in the collection
  • Postings List: For each term, list of documents containing it
  • Document Index: Metadata about each document

Benefits:

  • ✓ Maximize retrieval speed - only examine relevant documents
  • ✓ Enable ranking - store term frequencies

Costs:

  • ❌ Substantial storage overhead
  • ❌ Processing overhead to build and maintain

Reference: L02-Architecture.pdf, Pages 24-25

🎯 Retrieval Process (Part 3)

Week 2

📚 Reference: L02-Architecture.pdf - Pages 33-41

Section: PART 3: RETRIEVAL

The Retrieval Algorithm

Basic Search Process

  1. Lexicon Search: Find query terms in dictionary
  2. Fetch Occurrences: Pull out postings lists
  3. Manipulate Occurrences: Compute relevance scores
  4. Rank Documents: Sort by score (highest first)
  5. Return Results: Show top-ranked documents to user

Reference: L02-Architecture.pdf, Page 36

Best-Match Retrieval

Simple Ranking Algorithm

For each document I, Score(I) = 0
For each query term:
  Search the lexicon list
  Pull out the postings list
  For each document J in the list:
    Score(J) = Score(J) + 1

This simple algorithm:

  • Counts how many query terms appear in each document
  • Documents with more query terms rank higher
  • Better than binary AND (which requires ALL terms)

Reference: L02-Architecture.pdf, Pages 39-40

Comparison: Binary AND vs. Best-Match

Binary AND Search

  • Requires ALL query terms
  • All terms equally important
  • Returns: documents that match exactly
  • Problem: May return nothing!

Best-Match Ranking

  • Matches any query terms
  • More terms = higher score
  • Returns: ranked list of documents
  • Always returns something!

💡 The Power of Ranking

Modern search engines rank documents by estimated relevance. The output is a ranked list where:

  • Top documents are more likely to be relevant
  • Users typically only look at top 10 results
  • Ranking algorithms use many features (we'll learn more later!)

✨ Relevance & Effectiveness

Core Concept

📚 Reference: L02-Architecture.pdf - Pages 3-4, 23

Also: L01-Intro.pdf - Pages 15, 23

What is Relevance?

Task Statement

Build a system that retrieves documents that users are likely to find relevant to their queries.

Simple Definition:

A relevant document contains the information that a person was looking for when they submitted a query to the search engine.

Types of Relevance:

  • Topical Relevance: Document is about the same topic
  • User Relevance: Everything else (task, context, novelty, authority)

Reference: L02-Architecture.pdf, Page 3

Retrieval Models Define Relevance

How Systems Estimate Relevance

Ranking algorithms are based on retrieval models that:

  • Use statistical properties of text (not linguistic analysis)
  • Count simple features like words (not parse sentences)
  • Focus on topical relevance primarily

We'll cover these models in later weeks:

  • Vector Space Model (TF-IDF)
  • Probabilistic Models (BM25)
  • Language Models
  • Learning to Rank
  • Neural IR Models

Reference: L02-Architecture.pdf, Page 4
