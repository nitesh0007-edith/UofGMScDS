🔬 Keyboard Latency Experiment Analysis

Complete Critique with Variables, Design Flaws, and Solutions

📋 Experimental Scenario

Research Goal: Evaluate the impact of latency on keyboard usability

Conditions:

  • Condition 1: 5ms added latency
  • Condition 2: 25ms added latency
  • Condition 3: 50ms added latency

Procedure:

  • Participants experience all three conditions sequentially
  • Order: ALL participants do Condition 1 → 2 → 3
  • Same sentences typed in same order for each condition
  • Measure: Typing speed and error count

Planned Analysis:

  • Calculate total number of errors for each condition
  • Conclude: Condition with lowest errors = best performance

(a) Independent and Dependent Variables [3 marks]

✓ Model Answer

Independent Variable (IV) [1 mark]:

Keyboard latency (amount of added delay)

Levels: 3 levels

  • 5 milliseconds
  • 25 milliseconds
  • 50 milliseconds

Dependent Variables (DVs) [2 marks - 1 mark each]:

1. Typing speed (measured in words per minute or characters per minute)

2. Number of errors (count of typing mistakes per condition)

💡 Explanation:

  • The IV is what the researcher deliberately manipulates (the latency level)
  • The DVs are what the researcher measures to see the effect (speed and errors)
  • The hypothesis is that latency (IV) will affect typing performance (DVs)

(b) Critique and Improvements [15 marks]

⚠️ Major Problems Identified

This experimental design has SEVEN critical flaws that severely compromise the validity of the results:

❌ Problem 1: No Counterbalancing (CRITICAL FLAW)

🔴 Issue:

All participants experience conditions in the same fixed order (1→2→3). This creates severe order effects and confounding.

📉 Impact on Validity:

  • Practice Effects: Participants improve with practice as they progress through conditions. Condition 3 appears better simply because participants have practiced more, NOT because 50ms latency is actually better.
  • Fatigue Effects: Participants get tired. Condition 3 might appear worse due to fatigue, not latency.
  • Learning Effects: Participants learn the sentences. Later conditions benefit from memory.
  • Confounding: Cannot separate latency effects from order effects - results are completely invalid!

✅ Solution: Implement Counterbalancing

Method: Use a Latin Square design to balance order effects across participants.

Participant Group First Second Third
Group 1 (n=10) Condition 1 (5ms) Condition 2 (25ms) Condition 3 (50ms)
Group 2 (n=10) Condition 2 (25ms) Condition 3 (50ms) Condition 1 (5ms)
Group 3 (n=10) Condition 3 (50ms) Condition 1 (5ms) Condition 2 (25ms)

Benefit: Order effects are distributed equally across all conditions. Practice and fatigue effects average out.

❌ Problem 2: Same Sentences in Same Order

🔴 Issue:

Participants type the exact same sentences for each condition, creating massive learning effects.

📉 Impact on Validity:

  • Memorization: By Condition 2 and 3, participants have memorized the sentences
  • Muscle Memory: Finger movements become automatic
  • Unfair Comparison: Later conditions benefit from familiarity with the content
  • Reduced Sensitivity: Can't detect true latency effects because memorization masks the impact

✅ Solution: Use Different Equivalent Sentences

  • Create 3 sets of sentences that are equivalent in:
    • Length (same word count)
    • Complexity (similar vocabulary difficulty)
    • Character distribution (similar letter frequencies)
  • Randomly assign sentence sets to conditions for each participant
  • Example:
    • Set A: "The quick brown fox jumps over the lazy dog near the old barn"
    • Set B: "A small grey cat walks under the wide gate by the new house"
    • Set C: "The tall blue bird flies above the soft cloud through the warm air"

Benefit: Eliminates memorization effects while maintaining comparable difficulty.

❌ Problem 3: No Control/Baseline Condition

🔴 Issue:

No 0ms latency condition (normal keyboard with no added delay).

📉 Impact on Validity:

  • No Reference Point: Cannot determine if 5ms is already affecting performance or if it's acceptable
  • Limited Interpretation: Can only say which of the bad conditions is "least bad," not whether any are actually usable
  • Incomplete Story: Don't know baseline performance to compare against

✅ Solution: Add Baseline Condition

  • Condition 0: Normal keyboard with 0ms added latency (baseline)
  • Condition 1: 5ms added latency
  • Condition 2: 25ms added latency
  • Condition 3: 50ms added latency

Benefit: Can determine:

  • Whether 5ms is already degrading performance
  • At what latency level performance starts to degrade
  • Magnitude of degradation compared to optimal performance
❌ Problem 4: Fundamentally Flawed Analysis Methodology

🔴 Issue 4a: Using Total Errors Instead of Average

Planning to use "total number of errors" across all participants.

📉 Why This is Wrong:

  • Sample Size Sensitivity: If one condition has more/fewer participants (e.g., dropouts), totals are meaningless
  • No Individual Differences: Doesn't account for within-subject variance
  • Misleading: One very error-prone participant could skew entire condition's total

🔴 Issue 4b: Ignoring Typing Speed

Only considering errors, completely ignoring typing speed DV.

📉 Why This is Wrong:

  • Incomplete Picture: Participants might type slower to avoid errors - high latency could cause both slow typing AND errors
  • Speed-Accuracy Tradeoff: Cannot understand if participants sacrificed speed for accuracy
  • Missing Critical Data: Speed is often MORE sensitive to latency than errors

🔴 Issue 4c: No Statistical Testing

Planning to just "conclude lowest = best" without statistical tests.

📉 Why This is Wrong:

  • No Significance Testing: Differences could be due to random chance
  • No Confidence Intervals: Don't know reliability of results
  • Unreplicable: Cannot determine if findings are real or noise
  • Unpublishable: Any journal would reject this analysis

✅ Solution: Proper Statistical Analysis

1. Use Appropriate Descriptive Statistics:

  • Calculate mean (average) errors per participant for each condition
  • Calculate mean typing speed for each condition
  • Calculate standard deviations to show variability
  • Create graphs showing means with error bars (95% confidence intervals)

2. Use Appropriate Inferential Statistics:

Statistical Test: Repeated-Measures ANOVA (or Friedman if non-normal)

Why: Within-subjects design with 3+ conditions

Hypotheses:
• H₀: No difference in errors across latency conditions
• H₁: At least one latency condition differs in error rate

Significance Level: α = 0.05

Post-hoc Tests: If ANOVA significant, use Bonferroni-corrected pairwise comparisons to determine which conditions differ

3. Analyze BOTH Dependent Variables:

  • Run separate repeated-measures ANOVA for errors
  • Run separate repeated-measures ANOVA for typing speed
  • Consider creating a composite measure (e.g., errors per word typed) to capture speed-accuracy tradeoff

4. Report Effect Sizes:

  • Calculate partial η² (eta-squared) to show practical significance
  • Small effect: η² = 0.01
  • Medium effect: η² = 0.06
  • Large effect: η² = 0.14
❌ Problem 5: Within-Subjects Design May Not Be Optimal

🔴 Issue:

While within-subjects design has benefits, in this context it creates problems.

📉 Concerns:

  • Carryover Effects: Experience with one latency level affects perception of next
  • Contrast Effects: 50ms might seem worse after experiencing 5ms
  • Fatigue: Typing multiple conditions is tiring

✅ Alternative: Consider Between-Subjects Design

Design: Different participants for each condition

Group Condition Sample Size
Group 1 5ms latency 30 participants
Group 2 25ms latency 30 participants
Group 3 50ms latency 30 participants

Advantages:

  • No order effects
  • No learning/practice effects
  • No fatigue issues
  • Each participant fresh for their condition

Disadvantages:

  • Requires more participants (~90 vs ~30)
  • Individual differences between groups could affect results
  • Need random assignment to balance participant characteristics

Analysis: Use one-way ANOVA (or Kruskal-Wallis) for between-subjects comparison

❌ Problem 6: No Mention of Control Variables

🔴 Issue:

No specification of what will be kept constant across conditions.

📉 Potential Confounds:

  • Different keyboards used
  • Different times of day
  • Different environments (noise, lighting)
  • Different keyboards layouts/types
  • Different software implementations

✅ Solution: Specify Control Variables

Must Keep Constant:

  • Hardware: Same physical keyboard for all conditions (only latency changes via software)
  • Software: Same text entry application
  • Environment: Same quiet, well-lit testing room
  • Time Limits: Same duration for each typing task
  • Instructions: Standardized instructions for all participants
  • Seating: Same desk height, chair, screen distance
❌ Problem 7: No Sample Size Specified

🔴 Issue:

No mention of how many participants will be recruited.

📉 Concerns:

  • Underpowered: Too few participants = cannot detect real effects
  • Unreliable: Results may not replicate
  • Waste of Resources: If too small, study is pointless

✅ Solution: Conduct Power Analysis

Recommended Approach:

  • Conduct a priori power analysis to determine required sample size
  • Assumptions:
    • Power (1-β) = 0.80 (80% chance of detecting real effect)
    • Significance level (α) = 0.05
    • Expected effect size = medium (f = 0.25) based on literature
  • Result: Approximately 30 participants needed for within-subjects design
  • With dropout buffer: Recruit 35 participants

✅ COMPLETE IMPROVED EXPERIMENTAL DESIGN

Research Design:

  • Type: Within-subjects repeated-measures design with counterbalancing
  • Participants: 30 participants (recruit 35 for dropout)
  • Selection: University students, aged 18-30, regular computer users, no motor impairments

Conditions (4 total):

Condition Latency Purpose
0 (Baseline) 0ms Control - normal performance
1 5ms Low latency
2 25ms Medium latency
3 50ms High latency

Counterbalancing:

  • Use Latin Square design to vary condition order
  • 6 participants per ordering (5 possible orderings for 4 conditions)

Materials:

  • Sentences: 4 equivalent sets of 10 sentences each (same length, complexity)
  • Keyboard: Standard QWERTY keyboard with software-induced latency
  • Software: Custom text entry application with latency control and automatic logging

Procedure:

  1. Consent & Briefing: Obtain informed consent, explain study (5 min)
  2. Practice Session: 2 minutes to familiarize with typing task (0ms latency) (5 min)
  3. For Each Condition (order counterbalanced):
    • Set latency level (participant unaware of specific value)
    • Present sentence set (randomly assigned)
    • Participant types 10 sentences
    • 5-minute break between conditions
  4. Post-Task Questionnaire: Subjective ratings of difficulty, frustration, perceived delay (5 min)
  5. Debriefing: Explain latency manipulation (5 min)

Dependent Variables:

  • Typing Speed: Words per minute (WPM)
  • Error Rate: Number of errors per sentence
  • Error Types: Categorize as insertion, deletion, substitution, transposition
  • Subjective Measures: Perceived difficulty (1-7 Likert scale)

Control Variables:

  • Same keyboard hardware
  • Same quiet testing room
  • Same lighting conditions
  • Same text entry software
  • Same instructions
  • Same time limits per condition

Analysis Plan:

  1. Descriptive Statistics: Calculate means and SDs for each DV per condition
  2. Check Assumptions: Test normality (Shapiro-Wilk) and sphericity (Mauchly's)
  3. Inferential Tests:
    • Repeated-measures ANOVA for typing speed
    • Repeated-measures ANOVA for error rate
    • If sphericity violated, use Greenhouse-Geisser correction
  4. Post-hoc: Bonferroni-corrected pairwise comparisons if main effect significant
  5. Effect Sizes: Calculate partial η² for practical significance
  6. Visualization: Line graphs showing performance across latency levels with error bars

Hypotheses:

  • H₀: Keyboard latency has no effect on typing speed or error rate
  • H₁: Increased keyboard latency will decrease typing speed and/or increase error rate
  • Prediction: Performance will degrade progressively as latency increases from 0→5→25→50ms

(c) Inductive vs Deductive Inference [4 marks likely]

📚 Background: Understanding Inference Types

Deductive Inference

Direction: General → Specific

Logic: If premises are true, conclusion MUST be true

Example:

  • All keyboards with latency reduce typing speed (general rule)
  • This keyboard has latency (specific case)
  • Therefore, this keyboard reduces typing speed (certain conclusion)

Certainty: Absolute (if premises true)

Inductive Inference

Direction: Specific → General

Logic: Conclusion is probable based on evidence

Example:

  • Observed 30 participants type slower with latency (specific observations)
  • Therefore, latency probably reduces typing speed for all people (general conclusion)

Certainty: Probabilistic (based on sample)

✓ Two Reasons Why Inductive Inference is Used

Reason 1: Cannot Test Entire Population [2 marks]

Explanation:

  • We cannot test every human being who uses keyboards - this would be impossible (billions of people, impractical, too expensive, takes too long)
  • Instead, we test a sample (e.g., 30 university students) and use inductive inference to generalize findings to the broader population
  • We inductively infer: "If latency affects typing in our sample, it probably affects typing in the general population"
  • This is fundamentally probabilistic - we can never be 100% certain our sample perfectly represents everyone

Why not deductive?

  • Deductive inference requires testing ALL cases or having a universal law
  • We don't have a universal law stating "latency ALWAYS affects ALL people equally"
  • We're building knowledge from observations (bottom-up), not applying existing laws (top-down)

Reason 2: Natural Variability in Human Performance [2 marks]

Explanation:

  • Human behavior has inherent variability - people differ in typing ability, experience, adaptation, motor control, etc.
  • Even the same person shows variability across trials (fatigue, motivation, attention)
  • Because of this uncertainty, we cannot make absolute deductive statements like "Person X typing on 50ms latency keyboard WILL make exactly Y errors"
  • We must use statistical inference (a form of inductive reasoning) to determine:
    • Whether observed differences are real or due to random variation
    • What the likely range of effects is (confidence intervals)
    • How confident we can be in our conclusions (p-values)

Why not deductive?

  • Deductive inference assumes certainty and deterministic relationships
  • Human performance is probabilistic, not deterministic
  • We need to account for measurement error, individual differences, and random noise
  • Inductive statistical methods (t-tests, ANOVA, regression) handle this uncertainty

💡 Summary

Aspect Why Inductive Inference is Necessary
Population vs Sample We sample a subset and inductively generalize to the whole population
Uncertainty Human variability means we can only make probabilistic claims, not certain deductions
Direction of Reasoning We go from specific observations → general conclusions (inductive), not from general laws → specific predictions (deductive)
Statistical Methods Statistical inference (hypothesis testing, confidence intervals) is fundamentally inductive
