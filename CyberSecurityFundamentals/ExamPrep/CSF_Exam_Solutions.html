Q1 Travelsafe website with URL session IDs — vulnerability + defence [5 marks] ★ 5 pts ▼
A company called Travelsafe uses URL session IDs for authentication. Visitors can search for holiday locations. Mention and describe a potential vulnerability and corresponding defence mechanisms.
Crib Sheet Available
Crib Sheet Reference
No direct formula on crib sheet, but the Web Security concepts (HTTPS vs HTTP, session management) and the DAC/Access control context help frame this answer.
Primary Vulnerability: Session Hijacking (URL Session ID Exposure)

Because the session ID is embedded in the URL (e.g., travelsafe.com/search?session=12345abc), it is exposed in multiple dangerous ways:

  • Referer header leakage — if a user clicks an external link, the full URL (including the session ID) is sent in the HTTP Referer header to the external server.
  • Browser history / bookmarks — the session ID is stored in plain text in browser history, making it accessible to anyone with physical or malware access to the machine.
  • Log file exposure — web server logs record full URLs, so session IDs appear in access logs in plaintext.
  • Network interception — if not using HTTPS, any attacker on the network (e.g., a man-in-the-middle or packet sniffer) can trivially extract the session ID from request URLs.

Additionally, with a search box present, the site is also vulnerable to Cross-Site Scripting (XSS): if user input is not sanitised, an attacker can inject JavaScript that steals the session cookie/ID from other users.

Defence Mechanisms
DefenceHow It Mitigates
Use cookies instead of URL paramsSession IDs stored in cookies are not exposed in URLs, Referer headers, or browser history. Use HttpOnly and Secure flags.
Enforce HTTPS (TLS)Encrypts the entire request/response, preventing network-level eavesdropping of session tokens.
Session expiry & regenerationShort-lived sessions reduce the attack window. Regenerate session IDs after login to prevent fixation attacks.
Input sanitisation against XSSValidate and escape all user inputs in the search box to prevent injection of malicious scripts.
For full marks: name the vulnerability clearly (session hijacking via URL exposure), explain WHY it's a problem (URL is visible in logs, history, Referer), then give at least 2 specific defences.

(c) Why ECB is insecure + better alternative [7 marks]
Why ECB is insecure

Core problem: ECB encrypts each plaintext block independently with the same key. Therefore, identical plaintext blocks always produce identical ciphertext blocks.

  • Pattern leakage: Eve observes repeated ciphertext blocks → infers the plaintext structure without knowing the key. E.g., if "TRANSFER £1000" appears many times, the same ciphertext block repeats, revealing the pattern.
  • No diffusion across blocks: changing one plaintext block only changes the corresponding ciphertext block — no avalanche effect across the message.
  • Block replay attacks: an attacker can cut and paste ciphertext blocks (e.g., swap an account number block) without detection, because there's no chaining or integrity check.
Classic example: ECB-encrypted bitmap images retain the shape/structure of the original image — visually demonstrating ECB's pattern leakage.
Better Mode: CBC (Cipher Block Chaining)

How CBC mitigates:

  • XOR chaining: each plaintext block is XORed with the previous ciphertext block before encryption: C_i = EK(P_i ⊕ C_{i-1}). This means identical plaintext blocks produce different ciphertext blocks (because C_{i-1} differs each time).
  • IV (Initialization Vector): a random IV for the first block ensures that the same message encrypted twice produces different ciphertexts, preventing pattern analysis across sessions.
  • Error propagation: a change in one ciphertext block affects decryption of subsequent blocks, making replay/cut-and-paste attacks detectable.

Q2 Man-in-the-Middle attack (cafe Wi-Fi) + RSN defence [10 marks] ★ 10 pts ▼
(a) How can Eve conduct a MitM attack on Alice in the café? [5] (b) Discuss RSN's role in preventing MitM attacks and why it may not apply to public Wi-Fi. [5]
(a) MitM Attack Mechanism [5 marks]
  1. Rogue Access Point: Eve creates a Wi-Fi hotspot with the same SSID as the café's network (e.g., "CafeFreeWifi"). Alice's device auto-connects or she connects manually, not noticing it's fake.
  2. Traffic interception: All Alice's traffic now routes through Eve's device. Eve can read HTTP traffic in plaintext.
  3. SSL Stripping (for HTTPS): Eve forwards requests to the real server over HTTPS, but downgrades Alice's connection to HTTP. Alice sees the site loads but doesn't notice the missing padlock.
  4. Credential harvesting: Alice's login credentials to the company cloud service are now sent in plaintext to Eve's AP before being forwarded.
  5. Data modification: Eve can alter responses — e.g., inject malicious scripts into web pages, redirect Alice to phishing pages, or modify data in transit.
(b) RSN (Robust Security Network) in Wi-Fi [5 marks]

What RSN does: RSN (IEEE 802.11i / WPA2) establishes mutual authentication between the client and the access point using a 4-way handshake. It prevents rogue APs because:

  • Both client and AP must prove knowledge of the Pre-Shared Key (PSK) or use 802.1X enterprise authentication. A rogue AP without the correct PSK cannot complete the handshake.
  • RSN uses CCMP (AES-based) encryption to encrypt all traffic, so even if intercepted, data is unreadable.
  • The 4-way handshake derives fresh session keys (PTK) each session, preventing replay attacks.

Why RSN doesn't protect in public Wi-Fi:

  • Public café Wi-Fi is typically Open (no password) or uses WPA2-Personal with a shared, publicly known password. Eve knows the PSK too, so she can run a rogue AP with the same PSK.
  • RSN's mutual authentication only works when the PSK is secret. With a shared public password, RSN cannot distinguish the legitimate AP from Eve's rogue AP.
  • Enterprise RSN (802.1X) would fix this but requires certificate infrastructure — not feasible for a café.
Q3 SSO advantages + Kerberos 5-step protocol [12 marks] ★ 12 pts ▼
(a) SSO advantages + which CIAAN properties it provides [7]. (b) Kerberos 5-step process, focus on Ticket construction and usage [5].
Crib Sheet Available
Crib Sheet — Kerberos Process
Kerberos is the canonical SSO protocol. Key components: KDC (Key Distribution Centre), TGS (Ticket-Granting Server), AS (Authentication Server), Ticket (contains session key + identity, encrypted with service's key).
(a) SSO Advantages + CIAAN [7 marks]

Advantages over traditional auth: Single credential to manage → less password fatigue → users choose stronger passwords. Centralised security policy enforcement. Streamlined user experience.

PropertyProvided?Explanation
ConfidentialityYesCredentials transmitted once, over encrypted channel. Reduces exposure of sensitive authentication data.
IntegrityYesTickets are cryptographically signed/encrypted — cannot be modified without detection.
AvailabilityRiskSSO is a single point of failure — if the IdP goes down, access to ALL services is lost.
AuthenticityYesIdentity verified once by trusted authority; services trust the issued token as proof of identity.
AccountabilityYesCentralised logging of all authentication events; easier to audit who accessed what.
Non-repudiationYesAuthenticated session is tied to verified identity via the SSO provider — harder to deny actions.
(b) Kerberos 5 Steps [5 marks]
  1. User → AS (Authentication Server): User sends username in plaintext to the AS requesting a Ticket-Granting Ticket (TGT).
  2. AS → User (TGT): AS issues a TGT encrypted with the TGS's secret key (user cannot decrypt it), plus a session key encrypted with the user's password-derived key. The TGT contains: user identity, TGS session key, timestamp, validity period.
  3. User → TGS (Ticket-Granting Server): User presents the TGT + an Authenticator (timestamp encrypted with the TGS session key). The TGS decrypts the TGT to verify identity and checks the authenticator to prevent replay attacks.
  4. TGS → User (Service Ticket): TGS issues a Service Ticket encrypted with the target service's secret key. Contains: user identity, service session key, validity period. User also receives the service session key encrypted with the TGS session key.
  5. User → Service: User presents the Service Ticket + Authenticator to the target service. Service decrypts the ticket with its own key, verifies identity and timestamp. Access granted — no password was ever sent to the service.
Key insight: the user's password is never sent across the network. Tickets prove identity without revealing credentials. Each ticket is encrypted so only the intended recipient can read it.

Q8 RC4 stream cipher — known plaintext attack, KSA, invariance weakness [7 marks] ★ 7 pts ▼
(a) Known plaintext attack on RC4 (key=5 bytes, known fragment=3 bytes) [2]. (b) K[i]=i+1 for i=0-4, perform first 3 iterations of KSA [3]. (c) RC4 Invariance Weakness [2].
Crib Sheet Available
Crib Sheet — RC4 KSA (Key Scheduling Algorithm)
Step 1 (Init): S[i] = i; T[i] = K[i mod keylen]
Step 2 (KSA): j = (j + S[i] + T[i]) mod 256; Swap(S[i], S[j])
Step 3 (PRGA): i=(i+1)%256; j=(j+S[i])%256; Swap(S[i],S[j]); k=S[(S[i]+S[j])%256]
(a) Known Plaintext Attack [2 marks]

If an attacker knows 3 bytes of plaintext and has the corresponding ciphertext: keystream_byte = ciphertext_byte XOR plaintext_byte. The attacker now has 3 bytes of the RC4 keystream. Since RC4 is a stream cipher, the keystream is generated from the initial key. With a 5-byte key (40-bit keyspace = 2^40 possibilities), and knowing 3 keystream bytes, the attacker can: try all 2^40 keys, generate the first 3 keystream bytes for each, compare with the known 3 bytes, and identify the correct key. This is a known-plaintext exhaustive search — feasible given the short key length.

(b) First 3 KSA Iterations — K[i]=i+1 [3 marks]
Initial state: S[0..4] = {0,1,2,3,4}, T[i] = K[i mod 5] = i+1 T[0]=1, T[1]=2, T[2]=3, T[3]=4, T[4]=5 Initial S: [0, 1, 2, 3, 4] j = 0 Iteration i=0: j = (j + S[0] + T[0]) mod 256 = (0 + 0 + 1) mod 256 = 1 Swap(S[0], S[1]): swap S[0]=0 and S[1]=1 S = [1, 0, 2, 3, 4] Iteration i=1: j = (j + S[1] + T[1]) mod 256 = (1 + 0 + 2) mod 256 = 3 Swap(S[1], S[3]): swap S[1]=0 and S[3]=3 S = [1, 3, 2, 0, 4] Iteration i=2: j = (j + S[2] + T[2]) mod 256 = (3 + 2 + 3) mod 256 = 8 mod 5... Wait — S has 5 elements but mod 256 applies to full RC4 j = (3 + 2 + 3) mod 256 = 8 Swap(S[2], S[8]): but S only has indices 0-4 in this mini example Note: full RC4 uses 256-element array. In exam context with keylen=5: j = 8 mod 5 = 3 (if treating as circular 5-element) Swap(S[2], S[3]): swap S[2]=2 and S[3]=0 S = [1, 3, 0, 2, 4] State after 3 iterations: S = [1, 3, 0, 2, 4]
(c) RC4 Invariance Weakness [2 marks]

The Invariance Weakness refers to a property of RC4's KSA where certain initial key configurations produce initial states with predictable structures. Specifically, if the key has certain byte patterns, a significant fraction of the initial S-box values remain close to their initialized positions (i.e., S[i] ≈ i for many i). This means:

  • The first few bytes of keystream are statistically biased (not uniformly random).
  • The initial keystream bytes are correlated with the key — an attacker observing enough ciphertexts encrypted with related keys (same key prefix) can statistically recover key bytes.
  • This was exploited in WEP (802.11 Wi-Fi) encryption, breaking it entirely by analysing IV-prepended key patterns across packets.
Q9 Simple CBC encryption — encrypt P=1101 0110 1001, then change IV [6 marks] ★ 6 pts ▼
Block size: 4 bits. IV=1010. P=1101 0110 1001. E(Block,Key)=Block XOR Key, Key=0011. (a) Encrypt using CBC [3]. (b) How does changing IV to 0010 affect ciphertext? [3].
Crib Sheet Available
Crib Sheet — CBC Encryption Formula
C_i = E_K(P_i ⊕ C_{i-1}), where C_0 = IV
Here E(Block, Key) = Block XOR Key = Block XOR 0011
(a) CBC Encryption — IV=1010, Key=0011 [3 marks]
P1=1101, P2=0110, P3=1001 | IV=1010, Key=0011 Block 1: P1 ⊕ IV = 1101 ⊕ 1010 = 0111 C1 = E(0111) = 0111 ⊕ 0011 = 0100 Block 2: P2 ⊕ C1 = 0110 ⊕ 0100 = 0010 C2 = E(0010) = 0010 ⊕ 0011 = 0001 Block 3: P3 ⊕ C2 = 1001 ⊕ 0001 = 1000 C3 = E(1000) = 1000 ⊕ 0011 = 1011 Ciphertext = 0100 0001 1011
(b) Effect of changing IV=1010 → 0010 [3 marks]
New IV=0010, same P and Key: Block 1 changes: P1 ⊕ IV = 1101 ⊕ 0010 = 1111 C1_new = 1111 ⊕ 0011 = 1100 (was 0100) Block 2 changes (uses new C1): P2 ⊕ C1_new = 0110 ⊕ 1100 = 1010 C2_new = 1010 ⊕ 0011 = 1001 (was 0001) Block 3 changes (uses new C2): P3 ⊕ C2_new = 1001 ⊕ 1001 = 0000 C3_new = 0000 ⊕ 0011 = 0011 (was 1011) New Ciphertext = 1100 1001 0011

Key insight: Changing the IV changes ALL blocks of the ciphertext (due to chaining). This is the key security property of CBC — same plaintext, different IV → completely different ciphertext. This is why a fresh random IV should be used for every encryption session.

Q8 Feistel cipher — prove round transformation is 1-to-1 [5 marks] ★ 5 pts ▼
Crib Sheet Available
Crib Sheet — Feistel Structure Diagram
L_{i+1} = R_i
R_{i+1} = L_i ⊕ f(K_i, R_i)
Crib has the full visual diagram.
Proof: Feistel Round is Invertible (1-to-1)

Given: New state (L_{i+1}, R_{i+1}) = (R_i, L_i ⊕ f(K_i, R_i))

To prove: We can recover (L_i, R_i) from (L_{i+1}, R_{i+1}) and key K_i.

Given output: (L_{i+1}, R_{i+1}) Step 1: Recover R_i directly: L_{i+1} = R_i → R_i = L_{i+1} Step 2: Recover L_i: R_{i+1} = L_i ⊕ f(K_i, R_i) → L_i = R_{i+1} ⊕ f(K_i, R_i) → L_i = R_{i+1} ⊕ f(K_i, L_{i+1}) (we can compute f because we know K_i and R_i = L_{i+1}) Therefore (L_i, R_i) is uniquely determined from (L_{i+1}, R_{i+1}, K_i) ∎

Why this works: The key insight is that the left half is simply copied (L_{i+1} = R_i), so we always get R_i back for free. The XOR operation used in R_{i+1} is also its own inverse — XORing again with the same value undoes it. Crucially, this works regardless of what f is, because f is applied to R_i which we already recovered.

(a) Breaking the Ciphers [15 marks]
Cipher 1: Caesar Cipher (shift 2 to right)

A Caesar cipher shifts each letter by a fixed amount. "2 to the right" means each plaintext letter was moved 2 positions forward. To decrypt: shift each letter 2 positions backwards.

Example decryption rule: A=Z-1, so shift back 2 C → A, D → B, E → C, etc. To decrypt a specific message shifted by +2: Each cipher letter: subtract 2 (mod 26) Example: Z → X, A → Y, B → Z, C → A... The question doesn't provide the actual ciphertext for cipher 1 — you would apply shift -2 to each letter of the given ciphertext.
Cipher 2: Vigenère Cipher (keyword: passwd, ciphertext: ZCGNFJXQX)

Vigenère uses a keyword repeated to match plaintext length. Each letter of the keyword shifts the corresponding plaintext letter.

Keyword: passwd (length 6, repeats) Applied to 9 chars: p a s s w d p a s Ciphertext: Z C G N F J X Q X Decryption: P_i = (C_i − K_i) mod 26 Key values (a=0, b=1 ... z=25): p=15, a=0, s=18, s=18, w=22, d=3, p=15, a=0, s=18 Z(25) − p(15) = 10 = K C(2) − a(0) = 2 = C G(6) − s(18) = -12 mod 26 = 14 = O N(13) − s(18) = -5 mod 26 = 21 = V F(5) − w(22) = -17 mod 26 = 9 = J (check: could be I/J ambiguity) J(9) − d(3) = 6 = G X(23) − p(15) = 8 = I Q(16) − a(0) = 16 = Q X(23) − s(18) = 5 = F Plaintext: K C O V I G I Q F → "KCOVIGIQF" (variations possible depending on exact key letter interpretation)
