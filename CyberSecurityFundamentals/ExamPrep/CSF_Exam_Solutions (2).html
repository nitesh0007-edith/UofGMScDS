Q1 Travelsafe website with URL session IDs — vulnerability + defence [5 marks] ★ 5 pts ▼
A company called Travelsafe uses URL session IDs for authentication. Visitors can search for holiday locations. Mention and describe a potential vulnerability and corresponding defence mechanisms.
Crib Sheet Available
Crib Sheet Reference
No direct formula on crib sheet, but the Web Security concepts (HTTPS vs HTTP, session management) and the DAC/Access control context help frame this answer.
Vulnerability — Session Hijacking via URL-embedded Session ID

Because the session ID is embedded in the URL (e.g., travelsafe.com/search?session=12345abc), it is exposed in several dangerous ways:

  1. Referer header leakage: when a user clicks any external link, the browser sends the full URL (including session ID) in the HTTP Referer header to the destination server — no attacker needed.
  2. Browser history / bookmarks: session IDs are stored in plaintext in browser history, accessible to anyone with physical or malware access to the machine.
  3. Server log exposure: web server access logs record the full URL — anyone who can read logs can steal active sessions.
  4. Network interception: without HTTPS, any MitM or packet sniffer can read the session ID from URLs in transit.

Additionally, the search box makes the site vulnerable to Cross-Site Scripting (XSS) — unsanitised input lets an attacker inject JavaScript that steals session tokens from other users.

Defence Mechanisms
DefenceHow It Mitigates
Use cookies instead of URL paramsSession IDs in HttpOnly + Secure cookies are not exposed in URLs, Referer headers, or browser history. JavaScript cannot read them (HttpOnly).
Enforce HTTPS (TLS)Encrypts the entire request including URLs and cookies — prevents network-level interception.
Session expiry & regenerationShort-lived sessions reduce the attack window. Regenerate session ID after login to prevent fixation attacks.
Input sanitisation (XSS prevention)Validate and escape all search box input — prevents injected scripts from stealing session tokens.
For full marks: name the vulnerability (URL session ID exposure → session hijacking), explain 2–3 specific exposure routes, then give 2+ defences with reasoning.

(c) Why ECB is insecure + better alternative [7 marks]
Why ECB is insecure

Core problem: ECB encrypts each plaintext block independently with the same key. Therefore, identical plaintext blocks always produce identical ciphertext blocks.

  • Pattern leakage: Eve observes repeated ciphertext blocks → infers the plaintext structure without knowing the key. E.g., if "TRANSFER £1000" appears many times, the same ciphertext block repeats, revealing the pattern.
  • No diffusion across blocks: changing one plaintext block only changes the corresponding ciphertext block — no avalanche effect across the message.
  • Block replay attacks: an attacker can cut and paste ciphertext blocks (e.g., swap an account number block) without detection, because there's no chaining or integrity check.
Classic example: ECB-encrypted bitmap images retain the shape/structure of the original image — visually demonstrating ECB's pattern leakage.
Better Mode: CBC (Cipher Block Chaining)

How CBC mitigates:

  • XOR chaining: each plaintext block is XORed with the previous ciphertext block before encryption: C_i = EK(P_i ⊕ C_{i-1}). This means identical plaintext blocks produce different ciphertext blocks (because C_{i-1} differs each time).
  • IV (Initialization Vector): a random IV for the first block ensures that the same message encrypted twice produces different ciphertexts, preventing pattern analysis across sessions.
  • Error propagation: a change in one ciphertext block affects decryption of subsequent blocks, making replay/cut-and-paste attacks detectable.

Q2 Man-in-the-Middle attack (cafe Wi-Fi) + RSN defence [10 marks] ★ 10 pts ▼
(a) How can Eve conduct a MitM attack on Alice in the café? [5] (b) Discuss RSN's role in preventing MitM attacks and why it may not apply to public Wi-Fi. [5]
(a) MitM Attack Mechanism [5 marks]
Step 1 — Set up the rogue access point

Eve creates a Wi-Fi hotspot with the exact same SSID as the café's network (e.g., "CafeWifi"). Alice's device either auto-connects (if it's seen this SSID before) or she connects manually, with no way to tell it's fake.

Step 2 — Intercept all traffic

All Alice's traffic now flows through Eve's device before reaching the real internet. Eve acts as an invisible relay. Any unencrypted HTTP traffic is fully readable — login forms, session tokens, page content.

Step 3 — Defeat HTTPS via SSL Stripping

Eve forwards requests to the real cloud server over HTTPS (server thinks it's secure), but downgrades Alice's side to HTTP. Alice sees the page load normally but without the padlock — credentials are now sent in plaintext to Eve.

Step 4 — Harvest credentials + modify data

Alice's username/password are captured. Eve can also alter responses — inject malicious JavaScript, redirect Alice to a phishing page, or silently change data (e.g., a bank transfer destination).

(b) RSN (Robust Security Network) in Wi-Fi [5 marks]
What RSN does (why it helps on private networks)
  1. Mutual authentication via 4-way handshake: Both the client and AP must prove knowledge of the PSK. A rogue AP that doesn't know the real PSK cannot complete the handshake — Alice's device rejects it.
  2. Per-session encryption keys (PTK): Each session generates a unique Pairwise Transient Key via the handshake. Even if traffic is captured, replaying old packets won't work.
  3. CCMP/AES encryption: All data traffic is encrypted end-to-end — eavesdropping gives only ciphertext.
Why RSN fails on public café Wi-Fi
  1. Shared PSK is public: The café password is written on the wall. Eve knows it too — so she can set up a rogue AP with the same SSID and the same PSK. The 4-way handshake completes successfully because Eve has the correct key.
  2. RSN can't distinguish APs: WPA2-Personal has no way to verify which physical device is the "real" AP — only that the PSK matches. With a known PSK, the rogue AP passes all checks.
  3. Fix would require 802.1X Enterprise: Individual user certificates would prevent this, but this needs a RADIUS server — impractical for a café.
Q3 SSO advantages + Kerberos 5-step protocol [12 marks] ★ 12 pts ▼
(a) SSO advantages + which CIAAN properties it provides [7]. (b) Kerberos 5-step process, focus on Ticket construction and usage [5].
Crib Sheet Available
Crib Sheet — Kerberos Process
Kerberos is the canonical SSO protocol. Key components: KDC (Key Distribution Centre), TGS (Ticket-Granting Server), AS (Authentication Server), Ticket (contains session key + identity, encrypted with service's key).
(a) SSO Advantages + CIAAN [7 marks]
Advantages over traditional per-service authentication
  1. Single credential to manage: Users only remember one password, so they choose a stronger one and are less likely to reuse it across services.
  2. Reduced attack surface: Fewer credentials in transit means fewer interception opportunities. One strong authentication event replaces many weak ones.
  3. Centralised policy enforcement: Password complexity, MFA, and account lockout rules are enforced in one place — consistent across all connected services.
CIAAN properties Crib: Kerberos ticket = integrity + authenticity
PropertyProvided?Explanation
ConfidentialityYesCredentials transmitted once over encrypted channel (TLS). Reducing transmission frequency reduces interception risk.
IntegrityYesTickets are cryptographically signed/encrypted — any modification makes the ticket invalid and authentication fails.
AvailabilityRiskSSO is a single point of failure — if the IdP/KDC goes down, users cannot access ANY service.
AuthenticityYesIdentity is verified once by a trusted authority (KDC/IdP). All services trust the issued token as proof of identity.
AccountabilityYesAll authentication events are logged centrally — complete audit trail of who accessed what and when.
Non-repudiationYesThe authenticated session is tied to a verified identity. Users cannot deny actions performed under their session.
(b) Kerberos 5 Steps [5 marks] Crib: Kerberos components
Step 1 — User → Authentication Server (AS)

User sends their username in plaintext to the AS, requesting a Ticket-Granting Ticket (TGT). No password is sent at this stage.

Step 2 — AS → User (issues TGT)

AS responds with two things: (1) a TGT encrypted with the TGS's secret key (user cannot read it), and (2) a session key encrypted with the user's password-derived key. The TGT contains: user identity, TGS session key, timestamp, expiry. User decrypts the session key using their password — this is the only time the password is used.

Step 3 — User → Ticket-Granting Server (TGS)

User presents the TGT (opaque to them) + an Authenticator (current timestamp encrypted with the TGS session key). TGS decrypts the TGT, extracts the session key, and verifies the Authenticator — this prevents replay attacks.

Step 4 — TGS → User (issues Service Ticket)

TGS issues a Service Ticket encrypted with the target service's secret key (user cannot read it). Contains: user identity, service session key, validity period. User also gets the service session key encrypted with the TGS session key.

Step 5 — User → Service

User presents the Service Ticket + a new Authenticator to the target service. The service decrypts the ticket with its own secret key, verifies timestamp, and grants access. The user's password was never transmitted to the service.

Key exam insight: tickets are always encrypted with the recipient's secret key — the user carries the ticket but cannot forge or read it. This is what makes the system secure.
Q4 Digital Forensics — investigation steps + evidence collection [8 marks] ★ 8 pts ▼
Employee suspected of unauthorized access and data theft. (a) 4 steps a digital forensics investigator would take [4]. (b) Key digital evidence to collect [4].
(a) Four Investigation Steps [4 marks]
Step 1 — Identification & Preservation

Immediately isolate the computer from the network (unplug cable — do NOT shut down, as shutdown destroys volatile RAM data). Create a forensic bit-for-bit image of all storage media. Hash the image (SHA-256) to prove it hasn't been altered. Document chain of custody: who touched it, when, and why.

Step 2 — Collection (volatile data first)

Capture volatile data before powering down: running processes, active network connections, RAM contents (may contain encryption keys or passwords). Then collect disk images, log files, email archives, and USB connection history.

Step 3 — Examination & Analysis

Work only on forensic copies, never originals. Use tools like FTK or Autopsy to: recover deleted files from unallocated disk space, analyse file system timestamps, examine browser history, and correlate events with the off-hours timestamps the admin noticed.

Step 4 — Reporting

Compile a formal, legally admissible report with: timeline of events, evidence found, tools and versions used, hash values of all images, and a maintained chain of custody log for court admissibility.

Q6 Session state across HTTPS + Why HTTPS is more secure than HTTP [12 marks] ★ 12 pts ▼
(a) 2 mechanisms for session state in HTTPS + 2 disadvantages each [8]. (b) Why HTTPS is more secure than HTTP — 2 reasons with explanation [4].
(a) Session State Mechanisms [8 marks]
Mechanism 1 — Cookies

Server issues a session token stored as a browser cookie; sent automatically with every subsequent request via the Cookie header.

DisadvantageExplanation
CSRF (Cross-Site Request Forgery)Cookies are attached automatically to every matching request. A malicious site can trick the browser into firing an authenticated request to the target site — the victim's session cookie goes with it, authorising a forged action.
XSS Cookie TheftIf the site has an XSS vulnerability, an attacker injects JavaScript that reads document.cookie and sends the session token to a remote server. The attacker then uses it to impersonate the user.
Mechanism 2 — URL Session Tokens

Session ID embedded in the URL query string (e.g., site.com/page?session=abc123), passed in every link and redirect.

DisadvantageExplanation
Referer Header LeakageThe full URL — including the session ID — is sent to external sites via the HTTP Referer header when the user clicks a link. Any third-party analytics or ad service on the page receives the session ID.
Browser History & Log ExposureSession IDs are stored in plaintext in browser history, bookmarks, proxy logs, and server access logs. Anyone with access to these (local attacker, sysadmin, log breach) can steal the session.
(b) HTTPS vs HTTP [4 marks]
Reason 1 — Encryption (TLS)

HTTP sends all data as plaintext — anyone on the same network (Wireshark, Wi-Fi sniffer, ISP, MitM) can read passwords, session tokens, and credit card numbers verbatim. HTTPS wraps the connection in TLS, which negotiates a shared symmetric key (AES) via asymmetric key exchange. All request/response content is encrypted — intercepted traffic is unreadable ciphertext.

Reason 2 — Server Authentication (Certificate Verification)

HTTP has no server identity check — a MitM can silently proxy your connection with no indication. HTTPS requires the server to present a digital certificate signed by a trusted CA (Certificate Authority). The browser verifies the certificate chain before the connection proceeds, ensuring you're talking to the genuine server, not an impersonator.

Q7 Hospital wireless network APT breach — implications + security strategy [5 marks] ★ 5 pts ▼

(a) Implications [2 marks]
Patient Privacy (Confidentiality breach)

Patient records — diagnoses, medications, personal details — may be exposed. This triggers mandatory GDPR breach notification, potential fines (up to 4% of annual turnover under UK GDPR), reputational damage, and loss of patient trust. Confidentiality is fundamentally violated.

Operational Disruption (Safety risk)

Medical devices on the wireless network (infusion pumps, patient monitors, imaging equipment) could be disabled or manipulated, directly endangering patient lives. (b) Multi-layered Security Strategy [3 marks]
Layer 1 — Network segmentation

Technical: Isolate medical devices on a separate VLAN from general staff Wi-Fi. Implement WPA3 Enterprise with 802.1X — each device authenticates individually. Procedural: Quarterly network access reviews; only registered devices allowed on the medical VLAN.

Layer 2 — Detection & Monitoring

Technical: Deploy IDS/IPS to flag anomalous traffic (e.g., unexpected large transfers, lateral movement). Procedural: 24/7 SOC monitoring with a documented incident response plan specific to healthcare scenarios (DSPT compliance).

Layer 3 — Access Control

Technical: MFA for all staff accessing patient records; privileged access management (PAM) for system administrators. Procedural: Least privilege — clinical staff access only their own patients' records. Mandatory annual security awareness training covering phishing and social engineering.

Step 2 (KSA): j = (j + S[i] + T[i]) mod 256; Swap(S[i], S[j])
Step 3 (PRGA): i=(i+1)%256; j=(j+S[i])%256; Swap(S[i],S[j]); k=S[(S[i]+S[j])%256]
(a) Known Plaintext Attack on RC4 [2 marks]
Step 1 — Recover keystream bytes

Since RC4 is a stream cipher: Ciphertext = Plaintext XOR Keystream. Therefore: Keystream = Ciphertext XOR Plaintext. With 3 known plaintext/ciphertext pairs, the attacker recovers the first 3 bytes of the keystream directly — no key needed for this step.

Step 2 — Brute-force the 5-byte key

The 5-byte key has a keyspace of 240 ≈ 1 trillion possibilities. For each candidate key: run the RC4 KSA + generate the first 3 PRGA bytes, compare against the known keystream bytes. A match identifies the real key. This is feasible on modern hardware — the short key length is the fundamental weakness.

(c) RC4 Invariance Weakness [2 marks]
What the weakness is

Certain key byte patterns cause the KSA to leave many S-box values near their initialised positions (S[i] ≈ i). This means the initial keystream bytes are statistically biased — not uniformly random — and are correlated with portions of the key.

Why it matters — the WEP attack

An attacker who collects many ciphertexts encrypted with related keys (same key prefix, different IV prepended — exactly how WEP worked) can use statistical analysis of the first keystream byte across packets to recover key bytes one at a time. This is how WEP was completely broken: by collecting ~50,000–100,000 packets and running this statistical attack, the full WEP key could be recovered in minutes.

(b) Mobile Phone Data Recovery Steps [6 marks]
Step 1 — Secure & Isolate

Place the phone in a Faraday bag immediately to block cellular, Wi-Fi, and Bluetooth signals — this prevents remote wipe commands from reaching the device. Note the screen state, battery level, and any visible content.

Step 2 — Document & Photograph

Photograph the device before handling. Record IMEI, serial number, and model. Start the chain of custody log: who touched it, when, and under what authority.

Step 3 — Acquisition

Attach a write blocker to prevent any modification. Use specialist tools (Cellebrite UFED, Oxygen Forensic Detective) to extract data. Methods: logical extraction (standard backup), physical extraction (bit-for-bit NAND copy), or chip-off (for locked/damaged devices).

Generate a SHA-256 hash of the acquired image. Store the hash securely. Any future analysis copy must produce the same hash — proving the evidence has not been tampered with.

Step 5 — Analysis

Examine the image (never the device): call logs, SMS/iMessage, app data, GPS history, photos (EXIF metadata), deleted files in unallocated space, browser history, cloud app credentials.

Step 6 — Report

Compile a formal, legally admissible report: hash values, tools and version numbers used, timeline of findings, and a complete chain of custody record for court submission.

Q4 HTTP vs HTTPS — security differences + payment recommendation [8 marks] ★ 8 pts ▼
Step 1 — Core difference: Encryption

HTTP transmits everything as plaintext — anyone on the network (packet sniffer, ISP, Wi-Fi eavesdropper) can read all data including passwords and session tokens verbatim. HTTPS wraps the connection in TLS (Transport Layer Security): a handshake negotiates a shared symmetric key (typically AES-256), and all subsequent traffic is encrypted. Intercepted packets are unreadable ciphertext.

Step 2 — Server authentication

HTTP provides no server identity verification — a MitM attacker can silently relay your connection without any sign. HTTPS requires the server to present a digital certificate signed by a trusted CA. The browser verifies the full chain before connecting — you know you're talking to the genuine server.

Step 3 — Data integrity

HTTP allows in-transit modification — attackers can inject ads, malware scripts, or alter form fields silently. HTTPS uses a MAC (Message Authentication Code) in TLS — any tampering with ciphertext causes the MAC check to fail and the connection is dropped immediately.

Q5 Three wireless network threats + corresponding measures [3 marks] ★ 3 pts ▼
Threat 1 — Eavesdropping (passive) Crib: RSN/802.11i

Attacker captures Wi-Fi packets in range using a passive sniffer (e.g., Wireshark in monitor mode). On open or WEP networks, all traffic is readable. Defence: Enable WPA3/WPA2 (RSN — IEEE 802.11i) — all wireless frames are encrypted using CCMP/AES, making captured packets unreadable.

Threat 2 — Rogue Access Point (MitM setup)

Attacker deploys a fake AP with the same SSID as the legitimate network. Clients connect to the rogue AP, routing all traffic through the attacker. Defence: IEEE 802.1X Enterprise authentication — each AP must authenticate to the network controller using certificates; a rogue AP without valid credentials is refused. Wireless Intrusion Detection Systems (WIDS) can also detect rogue APs by scanning for unexpected SSIDs.

Threat 3 — Deauthentication (DoS) Attack

Attacker sends spoofed deauthentication frames (using victim's MAC address) forcing clients off the network. Because pre-802.11w management frames were unauthenticated, any device could send them. Defence: IEEE 802.11w (Management Frame Protection) — authenticates management frames cryptographically. Spoofed deauth frames are rejected.

Q8 Feistel cipher — prove round transformation is 1-to-1 [5 marks] ★ 5 pts ▼
Crib Sheet Available
Crib Sheet — Feistel Structure (visual diagram provided)
Li+1 = Ri  |  Ri+1 = Li ⊕ f(Ki, Ri)
The crib sheet has the full Feistel diagram — use it to check which half goes where. The key insight is that L is just copied across, so it's always recoverable.
Proof overview

Given: From the forward round we get the new state (Li+1, Ri+1) and we know key Ki. We need to recover the old state (Li, Ri).

Step 1 — Recover Ri for free
Forward rule: L_{i+1} = R_i Therefore: R_i = L_{i+1} ← read directly from output, no computation needed
Step 2 — Recover Li using XOR's self-inverse property
Forward rule: R_{i+1} = L_i ⊕ f(K_i, R_i) Rearrange: L_i = R_{i+1} ⊕ f(K_i, R_i) We already know R_i = L_{i+1} (from Step 1), so we can compute f(K_i, L_{i+1}) using the known key K_i. Therefore: L_i = R_{i+1} ⊕ f(K_i, L_{i+1}) ← fully determined
Conclusion
(L_{i+1}, R_{i+1}) → R_i = L_{i+1} → L_i = R_{i+1} ⊕ f(K_i, L_{i+1}) Both L_i and R_i are uniquely determined from (L_{i+1}, R_{i+1}, K_i). The round is 1-to-1 regardless of what f is. ∎
Why XOR works here: if A ⊕ B = C, then A = C ⊕ B (XOR is its own inverse). We XOR Ri+1 with f(…) to undo the original XOR. This is why f doesn't need to be invertible itself — only the XOR operation needs to be.

(b) Two Attack Scenarios [4 marks]
Scenario 1 — SSH Brute Force (Port 22)

The campus SSH server is reachable from the internet (needed for remote access). An automated bot discovers port 22 is open and runs a credential stuffing/brute-force attack against the student's username. If the student uses a weak or reused password, the attacker gains a full shell on the campus machine — with access to code, university network resources, and the student's home directory.

Mitigation: Use SSH key authentication (disable password auth), and use fail2ban to block IPs after repeated failures.

Scenario 2 — Email Phishing + Credential Interception

If unencrypted SMTP (port 25) or IMAP (port 143) is used, an attacker with access to the home network (e.g., compromised router) can capture login credentials in transit. More practically: a phishing email mimicking the university IT department prompts the student to "verify their account" on a fake portal — credentials captured there give the attacker access to SFTP (file server), SSH (campus machines), and email.

Mitigation: Always use SMTPS/IMAPS (encrypted). Be suspicious of unsolicited emails requesting credential re-entry.
