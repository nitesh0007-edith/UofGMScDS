🔍 What is Digital Forensics?

💡

Definition: Digital Forensics is a digital investigation that focuses on a digital device (computer, router, switch, cell-phone, SIM-card) involved in an incident or crime, using an adaptation of the scientific method — establish hypotheses, then test them against evidence.

Devices Investigated

  • Computers & laptops
  • Routers & switches
  • Cell-phones & SIM cards
  • USB drives & external storage
  • Cloud accounts

Crimes Involved

  • Computer intrusion / hacking
  • Internet fraud & scams
  • Child exploitation
  • Bomb threats via cell-phone
  • Insider threats
  • Financial fraud

Investigation Goals

🛡️ Prevent Further Intrusions

Reconstruct the intruder's modus operandi to predict and block future attacks.

📋 Assess Damage

Certify whether the system is safe for continued use after an incident.

⚖️ Reconstruct Incident

Build a factual record for criminal or internal disciplinary proceedings.

Evidence — Two Notions

Procedural Evidence

Anything on which our findings are based — what the investigator observes and records.

Legal Evidence

Defined by the "rules of evidence" — differs by legislation. Hear-say is procedural evidence but often excluded as legal evidence.

Who Needs to Know Digital Forensics?

Legal Professionals

  • Judges, prosecutors, attorneys
  • Law enforcement officers
  • Expert witnesses

IT/Security Professionals

  • Systems & network administrators
  • Security officers
  • Managers writing procedures

🗂️ Types of Digital Forensics

💻 Computer Forensics

Acquiring a snapshot of a computer's internal state — cloning the hard drive and/or RAM — then analysing the copy. Most common branch.

Key action: Bit-by-bit disk image

🌐 Network Forensics

Capturing and analysing network traffic. Helps with intrusion detection and reconstructing communication between hosts.

Key action: Packet capture (pcap)

📱 Mobile Forensics

Recovering data from mobile devices — call logs, messages, app data, GPS coordinates, even deleted content.

Key action: Logical/physical extraction

📊 Forensic Data Analysis (FDA)

Structured data analysis focused on financial crimes — e-mails, databases, spreadsheets to detect fraud.

Key action: Transaction analysis

⚠️

Most of the time these practices are used in digital crime investigations with the goal of leading to successful prosecution. Evidence must be gathered correctly or it may be inadmissible in court.

Golden Rule: Always work on a copy — never the original evidence. Any modification to original evidence can render it inadmissible.

🚔 Step 1: Seizure

🎯

Goal: Prevent digital devices from being used and data from being changed before a forensic copy is made.

Arriving at the Crime Scene

Photograph everything — Take pictures and screenshots of the scene, all devices, cables, and screen state before touching anything.
Inspect and label all hardware — Every device must be catalogued in a formal register: serial numbers, make, model, condition.
Bag all equipment in anti-static bags — Protects against electrostatic discharge and tampering. Be alert for hidden devices (e.g., USB hidden in a plug socket — "tower bomb").
Check BIOS time (F10 or similar) — Incorrect BIOS time can misdirect the investigation timeline. Record it carefully.
Record hard drive details — Model, serial number, capacity, interface type.

Powered-OFF vs Powered-ON Decision

Desktop is POWERED OFF

Remove the hard drive and place it in a safe, anti-static box. This preserves the disk state exactly.

Low risk

Desktop is POWERED ON

Decision: proceed with live forensics or power off? Live forensics captures volatile memory (RAM) but risks log-writing. Consider:

  • RAM contains running processes, encryption keys, open network sockets
  • Powering off destroys volatile data
  • Pulling the plug abruptly prevents graceful shutdown writes

High complexity

Registry — A Gold Mine of Evidence

The Windows registry contains a wide variety of information about users, software, hardware, and network configuration.

# Auto-start programs (malware persistence)
HKEY_LOCAL_MACHINE\Software\Microsoft\Windows\CurrentVersion\Run
  Value: Skype → "C:\Program Files\Skype\Skype.exe" /minimized

# User-specific auto-start (suspicious entry)
HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Explorer\Run
  Value: a → "notepad.exe"   <-- suspicious: why would notepad auto-run?

# DNS server (network configuration)
HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters
  Value: DhcpNameServer → 192.168.1.1 192.168.1.2
⚠️

Why BIOS time matters: If BIOS is set wrongly, timestamps on all files will be incorrect — pointing the investigation in the wrong direction. Always record and reconcile BIOS time with real-world time.

💾 Step 2: Acquisition

🎯

Goal: Create a forensically-sound, bit-for-bit copy of the suspect drive — then work only from that copy.

The Acquisition Process

Connect Write Blocker — Hardware device placed between the suspect drive and the forensic workstation. Ensures nothing can be written back to the evidence drive.
Create forensic image — Use FTK Imager or dd to create a bit-by-bit copy. Common formats: .dd (raw) or .E01 (EnCase, includes metadata).
Generate and store hash value — Compute MD5 / SHA-256 of both the original and the copy. Store securely. This proves the copy is identical and has not been tampered with.
Make a working copy — Work on a second copy of the image. If anything goes wrong you don't need to re-image the evidence drive.

Forensic Image Formats

FormatExtensionFeaturesUse Case
Raw.ddBit-for-bit copy, no metadataMaximum compatibility
EnCase.E01Includes metadata, compression, hashLaw enforcement standard
AFF.affOpen format, extensibleAcademic/research

Creating a Forensic Image with dd

# Basic dd image (raw format)
dd if=/dev/sda of=/mnt/evidence/suspect.dd bs=4096 conv=noerror,sync

# dc3dd — forensic version with progress, hashing
dc3dd if=/dev/sda of=/mnt/evidence/suspect.dd hash=md5 log=/mnt/evidence/hash.log

# Verify integrity after copy
md5sum suspect.dd
# Output: 3d9f5a... suspect.dd  (must match original hash)
🔒

Write Blockers are critical. Without a write blocker, mounting a drive can automatically update file access times (atime), modify the MBR, or trigger OS auto-run — all of which contaminate evidence.

#️⃣ Hashing in Digital Forensics

Hash functions are the backbone of evidence integrity verification. They provide a unique fingerprint of any data.

Key Properties of Cryptographic Hash Functions

Deterministic

Same input → always same output. Any bit change completely changes the hash.

One-Way

Cannot reverse the hash to recover the original data (pre-image resistance).

Collision Resistant

Computationally infeasible to find two different inputs with the same hash.

Hash Function Mathematics

\[ H: \{0,1\}^* \rightarrow \{0,1\}^n \]
\[ \text{If } D_1 \neq D_2 \text{, then } H(D_1) \neq H(D_2) \text{ (with overwhelming probability)} \]
\[ \text{Change 1 bit of input} \Rightarrow \text{approx. 50\% of output bits flip (avalanche effect)} \]

Common Algorithms Used in Forensics

AlgorithmOutput SizeStatusForensic Use
MD5128 bits (32 hex)Weak (collisions found)Still widely used for evidence integrity
SHA-1160 bits (40 hex)DeprecatedLegacy systems
SHA-256256 bits (64 hex)RecommendedModern investigations
SHA-512512 bits (128 hex)RecommendedHigh-security investigations

Whitelist Hashing

To find suspicious files, investigators compare a hash of every file on the suspect drive against a whitelist of known-good hashes (from a clean OS install).

# Create MD5 hash list of all files on suspect drive
md5deep -r /mnt/suspect/ > suspect_hashes.txt

# Create whitelist from known-clean Windows XP image
md5deep -r /mnt/winxp_clean/ > clean_hashes.txt

# Find files on suspect drive NOT in whitelist (potential malware/evidence)
comm -23 <(sort suspect_hashes.txt) <(sort clean_hashes.txt)
✅

Chain of Custody via Hash: Hash(original) → store → Hash(copy) → confirm equal → analyse copy. At any point, re-hash the copy to prove nothing has changed.

🔬 Step 3: Analysis

Analysis involves searching the forensic copy for evidence using multiple complementary techniques.

3a. Physical Searching with Autopsy

Autopsy Workflow

Hash check — Verify the image hash before starting analysis.
Create a case — Autopsy organises all findings under named cases with timestamps.
Ingest modules — Run keyword search, file type sorting, hash lookup, EXIF extraction.
String & grep search — Search for keywords (names, URLs, email addresses) across unallocated space too.
File carving with Foremost — Recover deleted files by their magic bytes (file signatures), not file system entries.

File Carving — How Foremost Works

Even deleted files leave their raw bytes on disk until overwritten. Foremost reads file magic bytes (headers and footers) to recover them.

File TypeMagic Header (hex)Magic Footer (hex)
JPEGFF D8 FFFF D9
PNG89 50 4E 4749 45 4E 44 AE 42 60 82
PDF25 50 44 46 (%PDF)%%EOF
ZIP/DOCX50 4B 03 0450 4B 05 06
# Extract all recoverable files from forensic image
foremost -i suspect.dd -o /output/carved/ -t jpg,pdf,doc,zip

# Search for keyword "password" in entire image (including unallocated)
grep -a -i "password" suspect.dd

# Extract EXIF metadata from recovered images
exiftool /output/carved/jpg/*.jpg | grep -E "Date|GPS|Camera"

3b. Registry Examination

Tool: RegViewer — opens raw registry hive files from the forensic copy.

Key Registry Hives

  • NTUSER.DAT — Per-user settings, recent files, typed URLs
  • SYSTEM — Hardware, services, USB history
  • SOFTWARE — Installed applications
  • SAM — User accounts and password hashes
  • SECURITY — Security policies, audit settings

What Registry Reveals

  • All users and their last login times
  • All USB devices ever connected (with timestamps)
  • Recently opened files (RecentDocs)
  • Malware persistence (Run keys)
  • Network shares and mapped drives
  • Installed/uninstalled software
# Copy registry hive from forensic image (mounted)
cp /mnt/suspect/Windows/System32/config/SYSTEM ./
cp /mnt/suspect/Windows/System32/config/SOFTWARE ./
cp /mnt/suspect/Users/johndoe/NTUSER.DAT ./

# Load into regripper for automated analysis
regripper -r SYSTEM -f system > system_analysis.txt
regripper -r NTUSER.DAT -f ntuser > user_analysis.txt

3c. Browser Analysis

Browser history is stored in structured databases. Autopsy can automatically extract and display them.

What Browser Analysis Reveals

  • Websites visited (with timestamps)
  • Downloads — what was downloaded, from where, when
  • Search queries typed into search engines
  • Bookmarks and saved passwords
  • Web-based email access
  • Local files accessed via browser (e.g., file:///C:/Users/johndoe/photos/)
# Example browser history entries found in investigation:
http://downloadmozilla.org/?product=firefox&os=win&lang=en-GB
http://windowsupdate.microsoft.com

3e. Timeline Reconstruction

Timeline is one of the most crucial parts — it establishes who did what, when.

Create .fls file — List all files and directories with timestamps from the image.
Generate body file — Convert .fls output to timeline format.
Import into Zeitline or Autopsy — Visual timeline of all file system events: created, modified, accessed, changed (CMAC).
Filter and correlate — Cross-reference with browser history, registry events, and log files.
# Sleuthkit: create file listing with timestamps
fls -r -m / suspect.dd > suspect.fls

# Create timeline
mactime -b suspect.fls -d > timeline.csv

# Filter for specific date range
grep "2024-03-15" timeline.csv
🕐

CMAC Timestamps: Every file has four timestamps — Created, Modified, Accessed, Changed (metadata). Anti-forensic tools can modify MAC times, but Changed time is harder to fake.

📄 Step 4: Report

⚖️

Purpose: The report must be objective, comprehensive, and prepared for potential court testimony. It contains no personal opinions — only factual findings.

Required Report Sections

📋 Job Description & Instructions

What was the investigator tasked with? Who commissioned the investigation? What was the legal authority?

🖥️ Description of Examined Items

Full list of all seized devices with serial numbers, hash values of images, condition on receipt.

🔍 Analysis Methodology

Which tools were used? Which techniques? In what order? All steps must be reproducible.

📑 Production List & Findings

All evidence found — files, emails, browser history, registry entries — with timestamps and references.

🚫

Avoid: Personal opinions, speculation, or unsupported conclusions. Every claim must be backed by recovered data with timestamps and provenance.

⚖️ Legal Framework

📚

Understanding the legal context is essential — evidence gathered outside legal authority is inadmissible. Investigators must know which legislation applies.

Computer Misuse Act 1990 (CMA)

The main UK legislation covering offences against computer systems. Deliberately does not define "computer" to allow for technological development.

Section 1 — Unauthorised Access

Accessing a computer without right. Must have knowledge that access is unauthorised (recklessness not sufficient). Includes "hacking".

Section 2 — Unauthorised Access with Intent

Section 1 offence committed as a precursor to another crime (e.g., accessing a bank account to commit fraud).

Section 3 — Unauthorised Acts (Impair Systems)

Recklessly impairing computer operation — covers DDoS attacks, ransomware, data destruction.

Section 3ZA — Serious Damage

Acts risking serious damage to human welfare, environment, economy, or national security. Targets critical infrastructure attacks.

Section 3A — Making/Supplying Malware

Creating or distributing malicious tools (malware, exploit kits) for use in CMA offences.

Other Relevant Legislation

Regulation of Investigatory Powers Act 2000 (RIPA)

Section 1(1)(b): Offence to intentionally intercept communications on a public telecom system without lawful authority.

Section 1(2): Offence to intercept communications on a private telecom system.

Key distinction: RIPA → interception during transmission; CMA → unauthorised access to stored data.

Data Protection Act 1998 (DPA)

Section 55 creates criminal offences for:

  • Obtaining or disclosing personal data
  • Procuring disclosure of personal data
  • Selling or offering to sell personal data

Often charged alongside CMA offences — e.g., Trojans stealing personal data.

Copyright, Designs and Patents Act 1988 (CDPA)

Covers digital piracy, software cracking, circumvention of copy-protection. Can apply when pirated software is found on a suspect machine.

England vs Scotland Evidence

England: Single-source evidence can be sufficient for conviction.

Scotland: Requires corroboration — evidence from more than one source. A significant practical difference for digital investigations.

Hot Topics

Trojan Defence

Defendant claims malware (Trojan, virus, bot) — not them — was responsible for the criminal act on their device.

Use cases:

  • Illegal content found on device → "I didn't put it there, a Trojan did"
  • Cybercrime committed via defendant's IP → "My machine was compromised"
  • Also used to claim lack of criminal intent even if technically responsible

Forensic counter: Investigate whether the malware actually exists and had capability for the alleged act.

Wireless Broadband Scenario

What if someone used your open WiFi for illegal activities?

  • Police trace the IP to your address
  • You are innocent but must prove it
  • Investigation: Test WiFi coverage range, check router logs, interview neighbours, seek other suspects
  • Protects yourself: Use WPA3, strong passwords, keep router logs enabled

Question 1 — Session ID Vulnerability [5 marks]

May 2025

A company called Travelsafe operates through a website that uses URL session IDs (unique numbers assigned to a user for a specific session after authentication). The website has a search box for holiday locations. Mention and describe a potential vulnerability and the corresponding defence mechanisms that can be employed to mitigate it.

Vulnerability: Session Hijacking via URL Session ID

What it is: When session IDs are embedded in URLs (e.g., travelsafe.com/search?sessionid=abc123&q=Paris), they are exposed in:

  • Browser history (stored on the device)
  • Server access logs
  • Referrer headers (sent to third-party sites linked from the page)
  • Shoulder surfing / screen capture
  • Bookmarked URLs accidentally shared

An attacker who obtains a valid session ID can replay it to impersonate the authenticated user — this is session hijacking.

Additional vulnerability: XSS via Search Box — If the search box reflects user input without sanitisation, attackers can inject JavaScript to steal session cookies or IDs.

Defence Mechanisms

  • Use HTTP-only cookies instead of URL parameters to store session IDs — cookies don't appear in URLs or referrer headers
  • Set Secure flag on cookies — transmitted only over HTTPS
  • Set HttpOnly flag — prevents JavaScript from reading the cookie (mitigates XSS-based theft)
  • Session timeout — invalidate sessions after inactivity (e.g., 15 minutes)
  • Regenerate session ID after login to prevent session fixation
  • Bind session to IP/User-Agent — detect anomalous session use
  • Input sanitisation on search box — prevent XSS (escape <, >, ", use Content-Security-Policy)
  • HTTPS everywhere — prevents man-in-the-middle interception of session IDs

Question 2 — SSO Security [7 marks]

May 2025

Some companies including Google and Amazon use SSO for user authentication. (1) Argue advantages and disadvantages of SSO over traditional approaches. (2) Explain what SSO means for Confidentiality, Integrity, Authenticity, Accountability, and Non-repudiation.

Part 1: Advantages vs Disadvantages

Advantages of SSODisadvantages of SSO
One set of credentials → reduced password fatigue → stronger passwords chosenSingle point of failure: compromise of SSO provider = access to ALL linked services
Fewer passwords to forget → fewer helpdesk password resetsIf SSO provider goes down, users locked out of all services
Centralised security policy enforcement (MFA, lockout, audit)Users less aware of individual application security — over-reliance on SSO
Reduced attack surface — one system to secure instead of manyPrivacy concern: SSO provider can track which services are accessed and when
Seamless user experience — improves productivityVendor lock-in risk if using proprietary SSO provider

Part 2: CIA+ Security Properties

Confidentiality

SSO centralises authentication → credentials transmitted fewer times, reducing interception opportunities. BUT: a single credential breach exposes all linked accounts.

Integrity

Centralised identity provider can enforce password policies and MFA, reducing likelihood of weak/compromised credentials altering data. Tokens (SAML/OAuth) are signed to prevent tampering.

Authenticity

SSO uses standardised protocols (SAML, OAuth 2.0, OIDC) with cryptographic assertions — strong guarantee that authenticated identity is genuine.

Accountability

All authentication events logged centrally — clear audit trail of who logged in when and from where across all services. Easier to detect anomalous login patterns.

Non-repudiation

Centralised log of authentication events with timestamps and IP addresses. User cannot deny logging in at a specific time — strong evidence in incident investigations.

Question 3 — Block Cipher ECB/CBC [13 marks]

May 2025

A block cipher on 4-bit blocks (hex digits). Key K implements permutation:

m0123456789abcdef
EK(m)1b5c7e2a493d0f68

(a) Decrypt ECB: 187b06  |  (b) Decrypt CBC with IV=1: 301be  |  (c) Explain ECB security issue [7 marks]

Step 0: Build the Inverse Permutation (Decryption Table)

To decrypt, we need DK = EK-1 — swap rows of the encryption table:

c0123456789abcdef
DK(c)c06a82e4f9713b5d

Derivation: EK(0)=1 → DK(1)=0; EK(1)=b → DK(b)=1; EK(2)=5 → DK(5)=2; EK(3)=c → DK(c)=3; EK(4)=7 → DK(7)=4; EK(5)=e → DK(e)=5; EK(6)=2 → DK(2)=6; EK(7)=a → DK(a)=7; EK(8)=4 → DK(4)=8; EK(9)=9 → DK(9)=9; EK(a)=3 → DK(3)=a; EK(b)=d → DK(d)=b; EK(c)=0 → DK(0)=c; EK(d)=f → DK(f)=d; EK(e)=6 → DK(6)=e; EK(f)=8 → DK(8)=f.

Part (a): ECB Decryption of 187b06 [2 marks]

\[ \text{Ciphertext blocks: } 1, 8, 7, b, 0, 6 \] \[ D_K(1) = 0 \] \[ D_K(8) = f \] \[ D_K(7) = 4 \] \[ D_K(b) = 1 \] \[ D_K(0) = c \] \[ D_K(6) = e \] \[ \boxed{\text{Plaintext} = 0f41ce} \]

ECB decryption: Each block decrypted independently — just look up each ciphertext nibble in DK.

Part (b): CBC Decryption of 301be, IV=1 [4 marks]

CBC Decryption formula:

\[ P_i = D_K(C_i) \oplus C_{i-1} \quad \text{where } C_0 = IV \]

Ciphertext blocks: C₁=3, C₂=0, C₃=1, C₄=b, C₅=e   IV = C₀ = 1

\[ P_1 = D_K(3) \oplus C_0 = a \oplus 1 \] \[ \text{In hex: } a = 1010_2, \; 1 = 0001_2 \Rightarrow a \oplus 1 = 1011_2 = b \] \[ P_1 = b \]
\[ P_2 = D_K(0) \oplus C_1 = c \oplus 3 \] \[ c = 1100_2, \; 3 = 0011_2 \Rightarrow c \oplus 3 = 1111_2 = f \] \[ P_2 = f \]
\[ P_3 = D_K(1) \oplus C_2 = 0 \oplus 0 = 0 \] \[ P_3 = 0 \]
\[ P_4 = D_K(b) \oplus C_3 = 1 \oplus 1 = 0 \] \[ P_4 = 0 \]
\[ P_5 = D_K(e) \oplus C_4 = 5 \oplus b \] \[ 5 = 0101_2, \; b = 1011_2 \Rightarrow 5 \oplus b = 1110_2 = e \] \[ P_5 = e \]
\[ \boxed{\text{Plaintext} = bf00e} \]

Part (c): ECB Security Issue [7 marks]

Why ECB is insecure:

  • ECB encrypts each block independently with the same key — same plaintext block always produces the same ciphertext block
  • This means an attacker observing ciphertext can detect repeating patterns that reveal structure of the plaintext
  • Classic example: encrypting a bitmap image in ECB — the encrypted image still shows the outline/shape of the original (the "ECB penguin" demonstration)
  • Eve doesn't need the key — she only needs to notice that block X appears in position Y across different messages to infer structural information
🔴

Eve's attack in the scenario: She notices that certain ciphertext blocks repeat across Alice's messages. Since each plaintext block always encrypts to the same ciphertext block, repeated blocks in ciphertext → repeated blocks in plaintext. She can infer "this message has the same header as that one" or build a codebook of known plaintext-ciphertext pairs.

More Secure Alternative: CBC (Cipher Block Chaining)

\[ C_i = E_K(P_i \oplus C_{i-1}), \quad C_0 = IV \]
  • Each plaintext block is XORed with the previous ciphertext block before encryption
  • Even if P₁ = P₂, C₁ ≠ C₂ (because they are XORed with different preceding blocks)
  • A random, unpredictable IV ensures that two encryptions of the same message with the same key produce completely different ciphertexts
  • Patterns in plaintext are destroyed — Eve cannot infer structural information

Even better: CTR, GCM modes — CTR turns a block cipher into a stream cipher (parallelisable); GCM adds authenticated encryption (also detects tampering).

Question 4 — RSA + Brute Force [15 marks]

May 2025

(a) Given public key e=17, n=1763, find the private key d. Show all steps. [10 marks]
(b) Brute force: 2⁸ encryptions/sec, 16-bit key. How long? What if key doubles? [5 marks]

Part (a): RSA Private Key Derivation

Factor n = 1763
\[ 1763 = 41 \times 43 \] \[ \text{Check: } 41 \times 43 = 41 \times 40 + 41 \times 3 = 1640 + 123 = 1763 \checkmark \] \[ \therefore \; p = 41, \; q = 43 \]
Compute Euler's Totient φ(n)
\[ \phi(n) = (p-1)(q-1) = 40 \times 42 = 1680 \]
Verify gcd(e, φ(n)) = 1
\[ \gcd(17, 1680) = ? \] \[ 1680 = 98 \times 17 + 14 \] \[ 17 = 1 \times 14 + 3 \] \[ 14 = 4 \times 3 + 2 \] \[ 3 = 1 \times 2 + 1 \] \[ 2 = 2 \times 1 \] \[ \therefore \gcd(17, 1680) = 1 \checkmark \text{ (e and φ(n) are coprime)} \]
Find d = e⁻¹ mod φ(n) using Extended Euclidean Algorithm

We need: 17d ≡ 1 (mod 1680)

\[ \text{Back-substituting from Euclidean steps:} \] \[ 1 = 3 - 1 \times 2 \] \[ = 3 - 1 \times (14 - 4 \times 3) = 5 \times 3 - 14 \] \[ = 5 \times (17 - 14) - 14 = 5 \times 17 - 6 \times 14 \] \[ = 5 \times 17 - 6 \times (1680 - 98 \times 17) \] \[ = 5 \times 17 - 6 \times 1680 + 588 \times 17 \] \[ = 593 \times 17 - 6 \times 1680 \] \[ \therefore \; 593 \times 17 \equiv 1 \pmod{1680} \] \[ \boxed{d = 593} \]

Verification: 17 × 593 = 10081 = 6 × 1680 + 1 = 10080 + 1 ✓

🔑

Result: Public key = (e=17, n=1763)  |  Private key = (d=593, n=1763)

Part (b): Brute Force Analysis

\[ \text{Key space} = 2^{16} = 65,536 \text{ keys} \] \[ \text{Speed} = 2^8 = 256 \text{ enc/sec} \] \[ \text{Worst-case time} = \frac{2^{16}}{2^8} = 2^8 = 256 \text{ seconds} \approx 4.27 \text{ minutes} \] \[ \text{Average time} = \frac{256}{2} = 128 \text{ seconds} \approx 2 \text{ minutes} \]

Practical scenario:

  • Practical — Encrypting a single short message (e.g., a bank PIN) where the adversary knows the plaintext structure. 2 minutes is feasible for a targeted offline attack.
  • Not Practical — Real-time interception where the session key changes every few seconds. By the time you crack it, the session is over.

Doubling the key size to 32 bits:

\[ \text{Key space} = 2^{32} = 4,294,967,296 \text{ keys} \] \[ \text{Time} = \frac{2^{32}}{2^8} = 2^{24} \approx 16.7 \text{ million seconds} \approx 194 \text{ days} \]

Doubling the key size squares the key space — it doesn't just double the time, it exponentially increases it. This is why modern symmetric keys are 128-256 bits.

Extra Q1 — Forensic Process Steps [8 marks]

A cyber investigator arrives at a scene where a desktop PC is suspected of being used in an internet fraud case. The PC is powered on. Describe the four main steps of a computer forensic investigation, explaining what actions should be taken at each step and why order matters. [8 marks]

Step 1: Seizure

The investigator must first prevent further changes to the digital evidence. Since the PC is powered on, a decision must be made: if RAM is critical (e.g., encryption keys may be in memory), perform live forensics first to dump RAM before powering off. Otherwise, carefully power off (pull plug to avoid graceful shutdown writes). Actions:

  • Photograph the scene and screen state
  • Label and register all hardware in a formal evidence log
  • Record BIOS time and hard drive details
  • Place in anti-static bags
  • Check for hidden storage devices (USB in plug sockets, etc.)

Why order matters: The scene must be preserved before anything is touched — any change, however small, can compromise admissibility.

Step 2: Acquisition

Create a forensically-sound bit-for-bit copy of the suspect drive using a write blocker and FTK Imager or dd. Generate an MD5/SHA-256 hash of both original and copy and store it securely. Create a working copy for analysis.

Why order matters: All analysis must be done on the copy — if acquisition is skipped or done incorrectly, any subsequent analysis taints the evidence.

Step 3: Analysis

Analyse the forensic copy using multiple techniques:

  • Physical search — Autopsy, keyword search, file carving with Foremost
  • Whitelist comparison — md5deep to identify non-OS files
  • Registry examination — installed software, USB history, auto-start entries
  • Browser analysis — visited URLs, downloads, cached pages
  • Timeline reconstruction — Zeitline/Autopsy to correlate events chronologically

For a fraud case: look for communication with fraudulent websites, downloaded tools, financial records, contact lists.

Step 4: Report

Produce a comprehensive, objective report: job description, list of examined items with hashes, methodology, and production list with findings. The report must contain no personal opinions and be suitable for use as evidence in court proceedings.

Extra Q2 — Chain of Custody [5 marks]

Explain what the "chain of custody" is in digital forensics and why it is critical for legal proceedings. What role does hashing play in maintaining chain of custody? [5 marks]

Chain of Custody is the chronological documentation of the seizure, custody, control, transfer, analysis, and disposition of physical or electronic evidence. It records:

  • Who collected the evidence
  • When and where it was collected
  • Who handled it at each stage
  • What was done to it at each stage
  • Where it is stored and how it is secured

Why critical: Courts require that evidence not have been tampered with between collection and presentation. Any break in the chain of custody can cause evidence to be ruled inadmissible — even if the evidence itself is genuine. Defence attorneys routinely challenge chain of custody.

Role of Hashing:

  • A hash (MD5 or SHA-256) of the original evidence drive is computed at the moment of acquisition
  • The same hash is recomputed from the forensic copy — if identical, the copy is proven to be an exact duplicate
  • Before analysis, the working copy's hash is recomputed — if identical, nothing has changed since acquisition
  • At court, the hash proves mathematically that the evidence presented is identical to what was seized
\[ \text{Integrity check:} \; H(\text{original}) = H(\text{copy}) \Rightarrow \text{copy is unmodified} \]

Extra Q3 — Trojan Defence [6 marks]

Describe the "Trojan Defence" and explain how a forensic investigator would attempt to rebut it. What evidence would they look for? [6 marks]

Trojan Defence: The defendant claims that malware (a Trojan horse, virus, or bot) was present on their device and was responsible for the alleged criminal activity — not the defendant themselves. The defendant may:

  • Deny knowledge of illegal content placed by the malware
  • Deny committing crimes performed via their compromised device
  • Admit technical responsibility but deny criminal intent due to malware involvement

How a forensic investigator rebuts the Trojan Defence:

  • Prove malware doesn't exist: Analyse the forensic image for any malware. If none is found, the defence claim is unsupported.
  • Prove malware lacked capability: Even if malware exists, demonstrate it could not have performed the specific act (e.g., it's a keylogger, not a file uploader).
  • Timeline analysis: Show that the criminal activity occurred when the defendant was known to be using the device (using login events, browser activity, file access times).
  • Browser and search history: Deliberately searched terms, bookmarks, or downloads that are consistent with intent — hard to attribute to a Trojan.
  • Registry and user activity: User interaction with specific files or applications at the time of the offence.
  • Anti-forensic evidence: Presence of file deletion tools, history-wiping tools suggests deliberate concealment, not Trojan activity.

Extra Q4 — Write Blockers & Evidence Contamination [4 marks]

Why must a write blocker be used during evidence acquisition? What specific risks arise if evidence is mounted without a write blocker? [4 marks]

A write blocker is a hardware device placed between the evidence drive and the forensic workstation that allows read commands to pass through but blocks any write commands.

Without a write blocker, mounting a drive may:

  • Update file access times (atime) — any file the OS reads will have its last-accessed timestamp changed, altering potential evidence
  • Modify the Master Boot Record (MBR) — some OS may write to the MBR during mounting
  • Trigger auto-run or auto-play — executing content on the suspect drive, potentially destroying volatile artefacts or triggering malware
  • Create swap/temp files — OS may write temporary data to the suspect volume
  • Write journaling data — file systems like NTFS/ext4 may update journal logs on mount

Any write to the evidence drive changes its hash value, breaking the chain of custody and potentially making all evidence from that drive inadmissible in court.

Extra Q5 — Computer Misuse Act Scenarios [6 marks]

For each scenario below, identify the most appropriate CMA 1990 offence section and justify your choice:
(a) A hacker accesses a company's server without permission to read customer data.
(b) A criminal launches a DDoS attack against a hospital, taking its systems offline.
(c) A developer sells a script designed to exploit CMA s.1 weaknesses.

(a) Section 1 — Unauthorised Access to Computer Material

The hacker accessed the server without authorisation and knew the access was unauthorised. There is clear intention to access data held in a computer. S.1 requires: knowledge of unauthorised access + intention to access a program or data. If the hacker accessed data to facilitate fraud, S.2 would also apply.

(b) Section 3 — Unauthorised Acts with Intent to Impair

A DDoS attack is explicitly covered under S.3, which addresses reckless or intentional acts that impair, prevent access to, or hinder computer operations. A hospital DDoS may additionally engage S.3ZA if it risks serious damage to human welfare (patient safety). S.3ZA is aimed specifically at attacks on critical national infrastructure.

(c) Section 3A — Making/Supplying Articles for Use in CMA Offences

The developer is making and selling tools specifically designed to facilitate S.1 offences. S.3A targets those who create or distribute malware, exploit kits, or scripts intended for use in S.1, S.3, or S.3ZA offences. The criminal intent is embedded in the purpose of the tool.

Extra Q6 — Phishing Email Investigation [6 marks]

An employee receives a suspicious email urging them to log in and download a "salary summary" PDF. (a) What should the employee check? (b) What can the information security team identify from the email and how?

Part (a): What the Employee Should Check

  • Sender email address — Does it match the legitimate domain? Attackers spoof display names but the actual address (e.g., hr@c0mpany.com vs hr@company.com) differs.
  • Grammar and language — Phishing emails often contain spelling mistakes, unusual phrasing, or awkward language.
  • Urgency/pressure — Legitimate HR emails don't demand immediate action or threaten consequences.
  • Links — Hover over links (don't click!) to see actual destination URL. Does it match what's displayed?
  • Attachment — Was a salary PDF expected? Does the filename match what was promised?
  • Good practice next step: Report to the IT/security team without opening the attachment. Inform colleagues as a heads-up.

Part (b): What the Security Team Can Identify

  • Sender IP address — Examine full email headers (View Source) to find the originating IP address, bypassing display name spoofing.
  • Sender domain reputation — Look up the originating IP/domain using tools like IPVoid, VirusTotal, or MX Toolbox to check if it is associated with known phishing campaigns.
  • Malicious attachment analysis — Submit the PDF to a sandbox (e.g., Any.Run, Cuckoo) or VirusTotal to check if it contains exploit code, macros, or malware without opening it on a real device.
  • Bonus: DKIM/SPF/DMARC header analysis to confirm whether the email domain is authenticated or spoofed.

Extra Q7 — ECB vs CBC Illustrated [4 marks]

Alice encrypts a 12-nibble message "aaaaaa bbbbbb" (where aaaaaa and bbbbbb are two identical 6-nibble blocks) using ECB mode, then CBC mode. Explain what the attacker observes in each case and what information they can infer.

ECB mode:

Since the two plaintext blocks are identical and ECB applies the same key independently to each: EK(aaaaaa) = X, EK(aaaaaa) = X. The ciphertext is XX — two identical blocks. An attacker immediately sees: "This message contains a repeated block of plaintext" — they know the message has repetitive structure even without knowing the key.

CBC mode:

\[ C_1 = E_K(P_1 \oplus IV) \] \[ C_2 = E_K(P_2 \oplus C_1) \]

Even though P₁ = P₂, since C₁ ≠ IV (generally), C₁ ≠ C₂. The ciphertext appears as two different blocks. The attacker sees no repetition and cannot infer that the plaintext blocks were identical.

Conclusion: ECB leaks plaintext patterns; CBC destroys them through chaining. This is why CBC (or better, GCM) should always be preferred over ECB for real messages.

Extra Q8 — File Carving Concept [4 marks]

Explain how Foremost can recover a deleted JPEG file from a forensic disk image when the file system entry for that file no longer exists.

When a file is "deleted" in most file systems (NTFS, FAT), the OS simply marks the directory entry as available — the actual file data remains on disk until overwritten by new data.

Foremost's file carving process:

  1. Read the raw disk image byte-by-byte (ignoring the file system entirely)
  2. Scan for known file signatures (magic bytes): JPEG files begin with FF D8 FF and end with FF D9
  3. When a JPEG header is found, start extracting bytes
  4. Continue until the JPEG footer FF D9 is found
  5. Save the extracted bytes as a recovered JPEG file

This works even without file system metadata because the actual data bytes are still present on disk. Limitations: if data has been overwritten, it cannot be recovered. Fragmented files (stored non-contiguously) are harder to recover correctly.

\[ \text{JPEG signature: } \underbrace{FF\;D8\;FF}_{\text{header}} \;\ldots\; \underbrace{FF\;D9}_{\text{footer}} \]

Extra Q9 — RIPA vs CMA Distinction [4 marks]

A hacker intercepts emails while they are being transmitted, and also breaks into the email server to read stored messages. Which legislation covers each action and why?

Intercepting emails in transmission → RIPA 2000

Section 1(1)(b) RIPA makes it an offence to intentionally and without lawful authority intercept communications during their transmission on a public telecommunications system. The key element is that the data is in motion on the network — this is RIPA's domain. Prosecutors would typically charge under RIPA here, though CMA may also be considered.

Breaking into the email server → CMA 1990

Accessing the email server without authorisation to read stored messages is a Section 1 CMA offence (unauthorised access to computer material). The data is at rest (stored), not in transit — this is CMA's domain. If the hacker then used the stored data to commit further crimes (fraud, blackmail), S.2 CMA would also apply.

Key distinction: RIPA = data in transit (interception); CMA = data at rest (unauthorised access). Both can be charged simultaneously if both actions occurred.
