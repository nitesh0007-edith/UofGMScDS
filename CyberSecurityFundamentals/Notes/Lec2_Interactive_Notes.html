💥 DoS & DDoS Attacks

Denial of Service (DoS)

Compromises availability by exhausting critical resources:

  • Network bandwidth — Flood with traffic
  • System resources — CPU, RAM exhaustion
  • Application resources — Connection pools, threads

TCP SYN Spoofing (Classic DoS)

Attacker sends many SYN packets with spoofed source IPs. Server allocates memory for half-open connections, becomes overwhelmed.

Attacker (spoofed IPs)          Server
    SYN (IP=1.2.3.4)  ──────►  Allocates slot, sends SYN+ACK
    SYN (IP=5.6.7.8)  ──────►  Allocates slot, sends SYN+ACK
    SYN (IP=9.0.1.2)  ──────►  Allocates slot...
    ...×10000         ──────►  ⚠️ Memory exhausted! Legit users denied

Normal TCP Three-Way Handshake

Step 1SYN — Client sends synchronize request to server
Step 2SYN+ACK — Server acknowledges and responds with its own SYN
Step 3ACK — Client acknowledges server response. Connection established!
Purpose: Reliable connection establishment. Used by TCP only (not UDP).

DDoS — Distributed DoS

Volumetric Attacks

Saturate bandwidth with massive traffic (UDP flood, ICMP flood)

Measured in: Gbps

Protocol Attacks

Exploit Layer 3/4 weaknesses (SYN flood, Smurf attack)

Measured in: PPS (packets/sec)

Application Layer

Target Layer 7 — HTTP GET/POST floods, Slowloris

Measured in: RPS (requests/sec)

Key Exam Point: DoS/DDoS attacks cannot be entirely prevented. You can only mitigate, detect, and respond.

DDoS Defence Strategies

Build a Response Plan ▼

Document roles, responsibilities, escalation paths. Have runbooks ready before an attack occurs.

Protect Network Infrastructure ▼

Use firewalls, IDS/IPS, rate limiting, and traffic scrubbing centers (e.g., Cloudflare, Akamai).

Attack Prevention / Preemption ▼

SYN cookies, connection limits, ingress/egress filtering (BCP38).

Detection and Filtering ▼

Anomaly detection, rate-based triggers, blackholing suspicious IPs.

Traceback and Identification ▼

IP traceback, network forensics, co-operation with ISPs to trace attack origin.

🔒 Security Protocols

Part A — Keeping a Secret

Protocol 1: Memorize

Suitable for Short secrets (passwords, keys)

Limitation Not for long secrets

Protocol 2: Paper

Threat Physical access by attacker

Risk Loss of paper → loss of secret

Mitigation Backups (but need secure tracking)

Protocol 3: Computer File

Vulnerabilities

  • Periodic backups made automatically
  • Easy file copying
  • Deleted files can be recovered

Protocol 4: File Encryption

Encrypt plaintext → ciphertext. Must erase original plaintext including all copies and backups

Time-limited vulnerability While plaintext is in filesystem

Protocol 5: Secret Algorithm

Threat Algorithm discovery

Problem Algorithms are computer programs — cannot be encrypted

Risk Designers at physical risk (coercion)

Part B — Communicating a Secret (Alice → Bob, Eve eavesdropping)

The fundamental problem: Alice wants to send secret info to Bob without Eve finding out.

Protocol 1: Secure Transmission Medium

Meet in secure room. Electronic transmissions easily intercepted. Email can be stored.

Impractical Cannot always meet physically

Protocol 2: Secure Preparation

Alice encrypts in secure area → sends via insecure medium → Bob decrypts in secure area

Assumption Eve can intercept AND replace messages

Protocol 3: Secret Algorithm

Keep algorithm details secret

Problems No peer review → more flaws. Designers at risk.

Protocol 4: Public Algorithm, Secret Key

Algorithm is public. Parameter (key) is secret.

Knowledge of algorithm is useless without the key.

Also called One-key / Symmetric Encryption

Block Cipher Modes

A block cipher encrypts fixed-size blocks (e.g., 128-bit for AES). The mode of operation determines how multiple blocks are chained together.

ECB — Electronic Codebook

Each block encrypted independently with the same key

C_i = E_K(P_i)

⚠️ INSECURE: Identical plaintext blocks → identical ciphertext blocks

CBC — Cipher Block Chaining

Each block XORed with previous ciphertext before encryption

C_i = E_K(P_i ⊕ C_{i-1})
C_0 = IV (Initialization Vector)

✅ Identical plaintexts → different ciphertexts (due to IV)

2025 Exam — Q3(c) 7 marks

Why is ECB mode insecure? If Eve observes repeating ciphertext blocks, what can she infer? Suggest a secure alternative and explain.

Why ECB is Insecure:

In ECB mode, each plaintext block is encrypted independently with the same key:

C_i = E_K(P_i)

This means: identical plaintext blocks → identical ciphertext blocks

What Eve Can Infer:

  • If Eve sees repeated ciphertext blocks, she knows the corresponding plaintext blocks are identical
  • She can learn the structure and patterns of the plaintext without knowing the key
  • Classic example: Encrypting a bitmap image in ECB reveals the image's visual structure
  • Eve can perform frequency analysis on blocks, just like classical cryptanalysis

Secure Alternative — CBC Mode:

C_i = E_K(P_i ⊕ C_{i-1}),  C_0 = IV
  • Each plaintext block is XORed with the previous ciphertext block before encryption
  • Identical plaintext blocks produce different ciphertext blocks
  • A random IV ensures even the first block is unpredictable
  • Patterns in plaintext are hidden
CBC is the secure alternative; CTR and GCM modes are also acceptable answers.

CBC Error Propagation

If an error occurs in ciphertext block Ci during transmission:

  • Plaintext block Pi is corrupted (decryption of Ci itself fails)
  • Plaintext block Pi+1 is corrupted (because Ci is used as XOR input)
  • Plaintext block Pi+2 onwards is fine (self-synchronizing after 2 blocks)
Self-synchronizing property: error affects exactly 2 blocks, then recovery. ECB only corrupts the single affected block.

🔑 RSA Cryptography

RSA is an asymmetric encryption algorithm based on the mathematical difficulty of factoring large numbers.

RSA Key Generation Algorithm

Choose two distinct primes p and q

They must be kept secret

Compute n = p × q

n is the modulus (public)

\[ n = p \times q \]
Compute Euler's Totient φ(n) = (p−1)(q−1)
\[ \phi(n) = (p-1)(q-1) \]
Choose public exponent e

Must satisfy: 1 < e < φ(n) and gcd(e, φ(n)) = 1

\[ \gcd(e, \phi(n)) = 1 \]
Find private key d = e⁻¹ mod φ(n)

Using Extended Euclidean Algorithm

\[ d \times e \equiv 1 \pmod{\phi(n)} \]
Public Key: (e, n)  |  Private Key: (d, n)
2025 Exam — Q4(a) 10 marks

Given public key e = 17, n = 1763, find the private key d. Show all steps.

Step 1: Factorise n = 1763

We need to find primes p and q such that p × q = 1763

Test divisibility:
1763 ÷ 41 = 43  ✓
So p = 41, q = 43
p = 41, q = 43 (both prime ✓)

Step 2: Compute φ(n)

\[ \phi(n) = (p-1)(q-1) = (41-1)(43-1) = 40 \times 42 = 1680 \]

Step 3: Find d = e⁻¹ mod φ(n)

We need d such that 17d ≡ 1 (mod 1680)

Use the Extended Euclidean Algorithm:

Apply extended Euclidean to gcd(17, 1680):

1680 = 98 × 17 + 14      →  14 = 1680 − 98×17
17   = 1  × 14 + 3       →   3 = 17 − 1×14
14   = 4  × 3  + 2       →   2 = 14 − 4×3
3    = 1  × 2  + 1       →   1 = 3 − 1×2
2    = 2  × 1  + 0       → GCD = 1 ✓

Back-substitution:
1 = 3 − 1×2
  = 3 − 1×(14 − 4×3)            = 5×3 − 1×14
  = 5×(17 − 1×14) − 1×14        = 5×17 − 6×14
  = 5×17 − 6×(1680 − 98×17)     = 5×17 − 6×1680 + 588×17
  = 593×17 − 6×1680
\[ d \equiv 593 \pmod{1680} \]
Private key d = 593
Verification: 17 × 593 = 10081 = 6 × 1680 + 1 ✓
2025 Exam — Q4(b) 5 marks

Assume 28 encryptions/second and 16-bit key size. How long for a brute-force attack? What happens if key size is doubled?

Time to Brute-Force 16-bit Key:

\[ \text{Keys to try} = 2^{16} = 65{,}536 \]
\[ \text{Time} = \frac{2^{16}}{2^8} = 2^{16-8} = 2^8 = 256 \text{ seconds} \approx 4.3 \text{ minutes} \]

Worst case vs Average case:

  • Worst case: 256 seconds (try all 65,536 keys)
  • Average case: 128 seconds (find key halfway through)

Practical vs Not Practical:

ScenarioPractical?
Encrypting a low-value message (e.g., a tweet)NOT worth brute-forcing (message has no value after 4 min)
Encrypting a time-sensitive bank transferPRACTICAL to attack (256 sec is fast enough)

Doubling Key Size (16-bit → 32-bit):

\[ \text{Keys} = 2^{32} = 4{,}294{,}967{,}296 \]
\[ \text{Time} = \frac{2^{32}}{2^8} = 2^{24} \approx 16.7 \text{ million seconds} \approx 194 \text{ days} \]
Doubling key size squares the search space. This is why modern systems use 128-bit or 256-bit keys — brute force becomes computationally infeasible.

URL Session ID Vulnerability (Exam Q1)

2025 Exam — Q1 5 marks

Travelsafe website stores session IDs in URLs (e.g., website.com/page?sessionid=abc123). What is the potential vulnerability and defence mechanisms?

Vulnerability: Session Hijacking via URL Exposure

  • Browser history: URL with session ID stored in browser history — accessible to anyone using the device
  • Server logs: Web server logs record full URLs including session ID
  • Referrer header: If user clicks a link to external site, session ID sent in HTTP Referer header
  • Shoulder surfing: Anyone watching screen can read session ID from address bar
  • Bookmarks: User may bookmark URL containing session ID
  • Session fixation: Attacker can craft a URL with known session ID and trick user into using it

Defence Mechanisms:

  • Store session IDs in cookies (not URLs) with HttpOnly and Secure flags
  • Use short-lived session IDs with automatic expiry
  • Implement session regeneration on authentication
  • Use HTTPS to prevent interception in transit
  • Validate session against IP address and User-Agent

Short Answer / Essay Questions

Tutorial W2 Q2 Essay

Explain, with examples, the terms Confidentiality, Integrity and Availability (CIA triad).

Confidentiality: Only authorised parties can access information. Example: NHS patient records accessible only to authorised doctors and administrators — not accessible to random staff or public.

Integrity: Information cannot be modified without authorisation. Example: NHS patient records cannot be changed without permission from an authorised clinician — ensures data accuracy and trustworthiness.

Availability: Authorised users can access information when needed. Example: Doctors must be able to access patient records 24/7, so servers and network infrastructure are designed for high availability and redundancy.

Tutorial W3 Q1 Essay

What is the difference between Threat, Vulnerability, and Risk? Provide examples.

Threat: An entity or event with the potential to cause harm. Example: A hacker launching a phishing campaign. Threats are identified, not calculated.

Vulnerability: A weakness in a system that a threat can exploit, making harm more likely or severe. Example: Unpatched SQL injection vulnerability in a web application. Vulnerabilities increase the probability or impact of a threat.

Risk: The potential for harm when a threat successfully exploits a vulnerability. Calculated as: Risk = Threat Probability × Impact. Example: If a phishing attack has 30% success probability and would cause £10,000 damage, Risk = £3,000 expected loss.

Relationship: Risk can be reduced by eliminating threats (hard), patching vulnerabilities (practical), or reducing impact through controls.

Tutorial W3 Q3 Essay

What is a Man-in-the-Middle attack? How can it be prevented?

MITM Definition: An attacker secretly interposes themselves between two communicating parties, intercepting, monitoring, and/or modifying the communication without either party's knowledge.

How it works: Alice believes she is communicating directly with Bob, but all messages pass through Eve, who can read them, modify them, or inject new messages.

Prevention methods:

  • Stronger WAP encryption (WPA3) on wireless access points
  • VPN with key-based encryption creates an encrypted tunnel
  • Public key authentication (PKI/certificates) verifies server identity
  • HTTPS with certificate pinning prevents SSL stripping
  • HSTS (HTTP Strict Transport Security) forces HTTPS connections

Additional Practice Questions

Practice Q1 5 marks

Explain why a system using the "best encryption techniques" can still be insecure. Give at least three examples.

Encryption addresses only one aspect of security (confidentiality in transit). Many other attack vectors exist:

  • Weak passwords: If users choose "password123", encryption of that password over the network is irrelevant — it will be guessed or dictionary-attacked
  • Key management failures: If the encryption key is stored unprotected in a plaintext file, the attacker bypasses encryption entirely
  • Endpoint compromise: If the user's device has malware/keylogger, the plaintext is captured before encryption
  • Social engineering/phishing: Users tricked into revealing credentials — no encryption prevents this
  • Misconfiguration: Correct algorithm, but cipher suite allows downgrade attacks (e.g., BEAST, POODLE on SSL)
  • Physical access: An attacker with physical access to device can bypass all software encryption

Security is a holistic discipline — all layers must be secured. Strong encryption is necessary but not sufficient.
