What is Cyber Security?

Cyber Security is the practice of protecting computers, networks, programs, and data from digital attacks, damage, or unauthorised access. It involves technical, organisational, and human-centric solutions.

Security = f(Confidentiality, Integrity, Availability, Authenticity, Accountability, Non-repudiation)

Real-World Attack Examples (from lecture)

Phishing

  • Deceptive emails mimicking legitimate entities
  • Goal: steal credentials / install malware
  • Countermeasure: user education, email filtering, MFA

Ransomware

  • Encrypts victim's files, demands ransom
  • Example: WannaCry (2017), attacked NHS
  • Countermeasure: backups, patching, EDR tools

Data Breach

  • Unauthorised access to sensitive data
  • Example: British Library ransomware attack
  • Countermeasure: encryption at rest, access control

Insider Threat

  • Malicious/negligent insiders cause damage
  • Hard to detect — legitimate access used
  • Countermeasure: least privilege, UEBA, logging

Threats, Vulnerabilities & Risk

Risk = Threat × Vulnerability × Asset Value

Threat

A potential event that could cause harm. Has a source (actor) and a motivation.

Examples: hacker, natural disaster, insider, nation-state

Vulnerability

A weakness that can be exploited. Not all vulnerabilities are exploited.

Examples: unpatched OS, default password, XSS flaw

Risk

The potential for loss when a threat exploits a vulnerability.

Mitigated by: controls, patches, policies

DDoS Attack — How It Works

[Attacker C&C Server] │ ┌────┴────┐ │ Botnet │ (thousands of compromised machines) └──┬──┬──┘ │ │ │ (Millions of requests per second) ▼ ▼ ▼ ┌─────────────────┐ │ Target Server │ ← Overwhelmed, legitimate users cannot connect └─────────────────┘ AVAILABILITY VIOLATED ❌

DDoS Mitigation Strategies

  • Rate limiting — restrict requests per IP per second
  • Traffic scrubbing — filter malicious traffic via CDN (Cloudflare)
  • Anycast diffusion — distribute attack traffic across global nodes
  • Blackholing — drop all traffic to targeted IP (last resort)
  • SYN cookies — protect against SYN flood without state table exhaustion

Man-in-the-Middle (MITM) Attack

Alice ──── (encrypts with Bob's fake key) ──→ ATTACKER ──── (re-encrypts) ──→ Bob ↑ │ │ └──── (attacker reads/modifies everything) ──────┘ │ INTEGRITY + CONFIDENTIALITY VIOLATED ❌

MITM Prevention

  • PKI / Certificate Authorities — verify public keys are genuine
  • Certificate Pinning — app only accepts known certificates
  • Mutual TLS (mTLS) — both parties verify each other
  • HSTS — forces HTTPS, prevents SSL stripping

Three-Way TCP Handshake (& SYN Flood Attack)

CLIENT SERVER │ │ │──── SYN (seq=x) ───────────────→│ Step 1: Client initiates │ │ │←─── SYN-ACK (seq=y, ack=x+1) ──│ Step 2: Server acknowledges │ │ │──── ACK (ack=y+1) ─────────────→│ Step 3: Connection established │ │ └─────── Data Transfer ───────────┘ SYN FLOOD ATTACK: Attacker sends thousands of SYN packets with FAKE source IPs Server allocates state for each → runs out of memory → Availability violated ❌ Fix: SYN cookies — server doesn't allocate state until ACK received

Cryptography Fundamentals

Cryptography provides confidentiality (encryption), integrity (hashing/MAC), and non-repudiation (digital signatures).

Symmetric Encryption — Block Ciphers

Same key used for encryption and decryption. Fast and suitable for large data.

ECB Mode (Electronic Codebook) — INSECURE

C_i = E_K(P_i)     (each block encrypted independently)
Plaintext: [P₁] [P₂] [P₃] [P₄] │ │ │ │ [E_K][E_K][E_K][E_K] ← same key, independent encryption │ │ │ │ Ciphertext: [C₁] [C₂] [C₃] [C₄] ⚠️ If P₁ = P₃, then C₁ = C₃ → PATTERNS VISIBLE → Information leak!

ECB Decryption (Tutorial Example)

-- Given key table, find plaintext from ciphertext --
Key mapping (substitution):
  0 → E,  1 → K,  2 → N,  3 → O,  4 → L,  5 → I,  6 → R,  7 → H,
  8 → D,  9 → A, 10 → T, 11 → S, 12 → C, 13 → B, 14 → M, 15 → P

Ciphertext: 7 9 1 1 0 6 11
             H A K K E R S    ← "HAKKERS" (plaintext)

CBC Mode (Cipher Block Chaining) — SECURE

C_i = E_K(P_i ⊕ C_{i-1})     (each block XORed with previous ciphertext)
P_i = D_K(C_i) ⊕ C_{i-1}     (decryption)
IV ──→ ⊕ ←── P₁ C₁ ──→ ⊕ ←── P₂ C₂ ──→ ⊕ ←── P₃ │ │ │ [E_K] [E_K] [E_K] │ │ │ C₁ ────────────────────→ C₂ ─────────────────→ C₃ ✅ Identical plaintext blocks → different ciphertext (patterns hidden) ✅ Error in C_i affects decryption of P_i AND P_{i+1} only

CBC vs ECB Comparison

PropertyECBCBC
Pattern Hiding❌ No✅ Yes
Parallel Encryption✅ Yes❌ No (sequential)
Error PropagationOnly that block2 blocks affected
Random Access✅ Yes❌ No

Security❌ Insecure✅ Secure

CBC Error Propagation (Tutorial Q)

If bit error occurs in ciphertext block C_i during transmission:

  • Decryption of P_i is completely garbled (all bits affected)
  • Decryption of P_{i+1} has exactly the same bit error (XOR propagation)
  • All subsequent blocks P_{i+2}, P_{i+3}, ... are correctly decrypted
Error in C_i → Garbled P_i, 1-bit error in P_{i+1}, P_{i+2}+ correct

RSA Asymmetric Encryption — Full Walkthrough

Choose two large primes p, q
Must be large (2048+ bits in practice). Example: p = 7, q = 11
Compute n = p × q
n = 7 × 11 = 77. This is the modulus (public).
Compute φ(n) = (p-1)(q-1)
φ(77) = 6 × 10 = 60. Euler's totient function.
Choose e: gcd(e, φ(n)) = 1, 1 < e < φ(n)
Common choice: e = 13 (coprime with 60). Public exponent.
Compute d = e⁻¹ mod φ(n)
Find d such that e×d ≡ 1 (mod φ(n)). Use Extended Euclidean Algorithm.
13×d ≡ 1 (mod 60) → d = 37 (since 13×37 = 481 = 8×60 + 1)
Public Key
(e, n) = (13, 77)
shared with everyone
Private Key
(d, n) = (37, 77)
kept secret
Encryption: C = M^e mod n     Decryption: M = C^d mod n

Worked Example — Encrypt M=2

-- Encryption --
M = 2, e = 13, n = 77
C = 2^13 mod 77
  = 8192 mod 77
  = 8192 - 106×77 = 8192 - 8162 = 30

-- Decryption --
C = 30, d = 37, n = 77
M = 30^37 mod 77
  = ... (use modular exponentiation)
  = 2  ✓

Why RSA is Secure

  • Security relies on difficulty of integer factorisation — given n, finding p and q is computationally infeasible for large n
  • No known polynomial-time algorithm for factorisation on classical computers
  • Breaking 2048-bit RSA would take longer than age of universe with current hardware

⚠️ Vulnerability: Character-by-Character RSA

If RSA encrypts each character independently (without padding):

  • Attacker builds a lookup table: encrypt all 128 ASCII chars with public key
  • Compare ciphertext characters to table → recover plaintext in O(n) time
  • This is why OAEP padding is mandatory in practice (randomises encryption)

Message Digests & Digital Signatures

Hash Function Properties

  • Pre-image resistance: Cannot find M from H(M)
  • Second pre-image resistance: Cannot find M' ≠ M where H(M') = H(M)
  • Collision resistance: Cannot find any M, M' where H(M) = H(M')
  • Avalanche effect: Small input change → completely different hash
SHA-256: "hello" → a665a45920422f...

Digital Signature Process

Sender signs
Sign = Encrypt(Hash(M), private_key)
Send M + Sign
Message + signature transmitted
Receiver verifies
Decrypt(Sign, public_key) == Hash(M)?

Provides: Integrity + Non-repudiation + Authenticity

Access Control Models

Access control ensures only authorised subjects can perform specified operations on objects.

Access Control = {Subject, Object, Operation} triple

Discretionary Access Control (DAC)

The owner of a resource decides who can access it. Represented as an access matrix or directed graph.

Access Matrix

Subject \ ObjectFile AFile BFile C
AliceRead, WriteRead—
BobReadRead, WriteRead
Charlie——Read, Write

Converting Matrix → Directed Graph

Alice ──Read,Write──→ File A Alice ──Read──────→ File B Bob ──Read──────→ File A Bob ──Read,Write──→ File B Bob ──Read──────→ File C Charlie ──Read,Write──→ File C (Each arrow = an access right)

From access matrix: Count rights per (subject, object) pair. Total edges in graph = total non-empty cells in matrix.

Role-Based Access Control (RBAC)

Permissions assigned to roles, users assigned to roles. Reduces admin overhead.

Users ──→ Roles ──→ Permissions ──→ Objects e.g.: Alice, Bob ──→ [Developer Role] ──→ {Read/Write src/, Read docs/} Charlie ──→ [Admin Role] ──→ {Read/Write everything, Deploy} Dave ──→ [Viewer Role] ──→ {Read docs/ only}

RBAC vs DAC

PropertyDACRBAC
Who decides access?Resource ownerSystem administrator
ScalabilityPoor (O(users×objects))Good (O(users+roles))
Least privilegeManualBuilt-in via role design
AuditPer-userPer-role (simpler)

Counting Edges: DAC vs RBAC (Tutorial Example)

-- 4 users, 10 objects in DAC --
Max edges = 4 × 10 = 40 (one edge per (user, object, right) triple)

-- Same system with RBAC (e.g., 3 roles) --
User-Role assignments: 4 edges
Role-Object assignments: 3 × 10 = 30 max
Total: 34 edges (and roles can be shared)

-- Practical benefit --
New employee → assign to role → instantly gets all role permissions
No need to set permissions on each object individually

UNIX File Permissions

rwxrwxrwx  →  [owner][group][others]
-- Permission bits --
r = read   (4)
w = write  (2)
x = execute(1)

-- Examples --
-rwxr-xr--   = 754  (owner: all, group: rx, others: r)
-rw-r--r--   = 644  (owner: rw, group: r, others: r) ← typical file
drwxr-xr-x   = 755  (directory, executable=traversable)

-- Tutorial Problem --
Parent dir: rwxr-xr-x (755)
File in dir: rw------- (600)
Can "others" read the file?
→ They CAN enter the directory (x on others)
→ But CANNOT read the file (--- on others)
→ Answer: No, but they know the file exists

Common Vulnerability: Overly Permissive Umask

  • Default umask 022 → files created as 644, dirs as 755
  • Umask 000 → files created as 666 (world-writable!) — dangerous
  • Principle of Least Privilege: grant minimum permissions needed

Password Storage — bcrypt/Argon2

-- WRONG: Store plaintext --
users_table: { username: "alice", password: "hunter2" }  ❌

-- WRONG: Unsalted hash --
hash = SHA256("hunter2") = "f52fbd32b2b3b86f..."  ❌
-- Rainbow table can reverse this! --

-- CORRECT: Salted hash (bcrypt) --
salt = random_bytes(16)   ← unique per user
hash = bcrypt(password + salt, cost=12)
store: (username, salt, hash)

-- Verification --
retrieved_hash = bcrypt(entered_password + stored_salt, cost=12)
if retrieved_hash == stored_hash: AUTHENTICATED ✅

-- Why bcrypt? --
-- Cost factor makes it deliberately slow (prevents brute force) --
-- Each hash takes ~100ms → 10 guesses/sec max --

Single Sign-On (SSO) — Exam Question 2025

Advantages

  • Users remember one password → less password reuse
  • Centralised security policies and auditing
  • Faster login experience
  • Easier MFA enforcement (one place)
  • Reduced credential theft surface area

Disadvantages / Risks

  • Single Point of Failure: SSO breach = all services compromised
  • Availability Risk: SSO outage = no access to anything
  • Trust dependencies across service providers
  • Complex implementation (SAML, OAuth, OIDC)
  • Session token theft more damaging
SSO Flow (SAML): User ──→ Service Provider (SP) SP ──→ Identity Provider (IdP): "Is this user who they claim?" IdP authenticates user (password + MFA) IdP ──→ SP: SAML Assertion (digitally signed) SP grants access based on assertion User ←── SP: Access granted ✅

Q1. Travelsafe session ID in URL vulnerability [2025 Exam]
5 marks

The Travelsafe website uses a URL of the form: https://www.travelsafe.uk/account?session=XYZ&user=alice

Identify TWO security vulnerabilities from storing the session ID in the URL, and for each suggest a countermeasure.

Model Answer

Vulnerability 1: Session ID in Browser History

  • The URL (including session ID) is stored in the browser's history and bookmarks. If an attacker gains physical/remote access to the device, they can hijack the session.
  • Countermeasure: Store session token in an HttpOnly; Secure cookie instead of the URL. Cookies are not logged in browser history.

Vulnerability 2: Session ID in Server Logs / Referrer Header

  • Web server access logs record the full URL including the session ID. System administrators or anyone with log access can see and steal session IDs. Additionally, if the page contains third-party resources (ads, analytics), the session ID is leaked in the HTTP Referrer header.
  • Countermeasure: Use cookie-based sessions. Implement short session timeouts and regenerate session IDs after login.

Additional valid points: URL shared accidentally, session fixation attacks

Q2. Single Sign-On — advantages, disadvantages, security aspects [2025 Exam]
7 marks

(a) State TWO advantages and TWO disadvantages of SSO. (4 marks)

(b) Which of the 6 security aspects does SSO directly address? Justify for each. (3 marks)

Model Answer (a)

Advantages:

  • Users maintain one password → less password reuse across services → lower credential theft risk
  • Centralised enforcement of security policies (MFA, password strength) applied once, benefits all services

Disadvantages:

  • Single Point of Failure: If the SSO identity provider is breached, ALL connected services are immediately compromised
  • Availability Risk: If SSO goes down, users cannot authenticate to ANY connected service

Model Answer (b)

  • Authenticity: SSO centrally verifies identity using strong authentication before granting tokens
  • Confidentiality: Reduces the number of places where credentials are stored, reducing exposure
  • Accountability: Centralised logging of all authentication events enables comprehensive audit trails
Q3. Block Cipher ECB/CBC — Decryption + Security Analysis [2025 Exam]
13 marks

(a) Decrypt the ciphertext using ECB mode with the given key table. (5 marks)

(b) Decrypt using CBC mode with IV given. (5 marks)

(c) Explain why CBC is more secure than ECB. (3 marks)

Model Answer (a) — ECB Decryption

In ECB mode: P_i = D_K(C_i). Each block is decrypted independently using the key table lookup.

For each ciphertext value, look up corresponding plaintext in key table.
Example: C = [7, 9, 1, 1, 0, 6, 11] with key table:
0→E, 1→K, 2→N, 3→O, 4→L, 5→I, 6→R, 7→H, 8→D, 9→A, 10→T, 11→S
→ P = H A K K E R S

Model Answer (b) — CBC Decryption

In CBC mode: P_i = D_K(C_i) ⊕ C_{i-1} (P_1 uses IV in place of C_0)

Step 1: D_K(C_i) — decrypt each block using key table
Step 2: XOR with previous ciphertext block (or IV for first block)
Example: If D_K(C_1) = 5, IV = 3, then P_1 = 5 ⊕ 3 = 6 → 'R'

Model Answer (c) — CBC vs ECB Security

  • Pattern hiding: In ECB, identical plaintext blocks produce identical ciphertext blocks, revealing patterns. In CBC, each block is XORed with the previous ciphertext, so identical plaintext blocks produce different ciphertext.
  • Semantic security: CBC with a random IV is semantically secure — an attacker cannot distinguish encryptions of the same message. ECB is not semantically secure.
  • Error propagation: CBC errors self-synchronise after 2 blocks; this is a side effect but not a security weakness of CBC over ECB.
Q4. RSA Key Generation + Brute Force Analysis [2025 Exam]
15 marks

(a) Given p=7, q=11, e=13: compute n, φ(n), d (private key). (8 marks)

(b) Encrypt message M=5 and verify decryption. (4 marks)

(c) Why is encrypting each character separately insecure? (3 marks)

Model Answer (a)

n = p × q
7 × 11 = 77
φ(n) = (p-1)(q-1)
(7-1)(11-1) = 6 × 10 = 60
Public key (e, n)
(13, 77)
gcd(13, 60) = 1 ✓
Find d: 13d ≡ 1 (mod 60)
d = 37
13×37 = 481 = 8×60+1 ✓
Private key (d, n)
(37, 77)

Model Answer (b)

Encrypt M=5
C = 5^13 mod 77
5^2 = 25
25 mod 77 = 25
5^4 = 625
625 mod 77 = 9 (625 = 8×77+9)
5^8 = 81
81 mod 77 = 4
5^13 = 5^8 × 5^4 × 5^1
4 × 9 × 5 = 180 mod 77 = 26
Ciphertext C = 26
Decrypt: 26^37 mod 77
= 5 ✓

Model Answer (c)

  • The alphabet is small (e.g., 26 letters or 128 ASCII characters)
  • An attacker can encrypt ALL possible characters with the public key and build a lookup table mapping ciphertext → plaintext
  • This lookup table attack recovers the entire message in O(alphabet_size) time
  • Fix: Use OAEP (Optimal Asymmetric Encryption Padding) to add randomness, or encrypt entire blocks/messages together

Q6. MITM Attack — prevention using PKI
8 marks

(a) Describe a Man-in-the-Middle attack in the context of symmetric key exchange (3 marks)

(b) Explain how Public Key Infrastructure (PKI) prevents MITM attacks (5 marks)

Model Answer (a)

Alice wants to share a symmetric key with Bob. She sends her public key. The attacker intercepts and replaces Alice's key with their own. Bob encrypts with the attacker's key. The attacker decrypts, reads the message, re-encrypts with Alice's real public key, and forwards it. Neither party is aware of the interception. Both Confidentiality and Integrity are violated.

Model Answer (b)

  • Certificate Authority (CA): A trusted third party signs certificates binding a public key to an identity
  • Certificate contents: {Subject, Public Key, Issuer CA, Expiry, Digital Signature by CA}
  • Verification: When Bob receives Alice's certificate, he verifies the CA's digital signature using the CA's pre-installed root certificate
  • MITM prevention: The attacker cannot forge a CA-signed certificate for Alice's domain (would need CA's private key)
  • Certificate Transparency: CAs log all issued certificates publicly, so rogue certificates can be detected
Q7. RBAC — Calculate number of permission assignments
8 marks

A hospital system has 500 staff members and 200 resources. There are 5 roles: Doctor, Nurse, Admin, Pharmacist, Manager.

(a) In a pure DAC system, what is the maximum number of permission assignments? (2 marks)

(b) In RBAC with 5 roles, what is the maximum? (2 marks)

(c) Explain why RBAC reduces administrative overhead. (4 marks)

Model Answers

(a) DAC: 500 × 200 = 100,000 possible assignments

(b) RBAC: 500 (user-role) + 5×200 (role-resource) = 500 + 1,000 = 1,500 max assignments

(c)

  • New employee: Assign to role (1 operation) → automatically inherits all role permissions. DAC requires setting permissions on each of 200 resources individually.
  • Role change: Change user's role assignment (1 operation). DAC requires modifying potentially hundreds of ACL entries.
  • Policy update: Change permissions for "Doctor" role → affects all 100 doctors immediately. DAC requires updating 100 × n individual entries.
  • Audit: Review Doctor role permissions once. DAC requires reviewing per-user per-resource.
Q8. Password Security — Salting and Hashing
8 marks

(a) Why is storing passwords as unsalted MD5 hashes insecure? (3 marks)

(b) Explain how salting prevents rainbow table attacks. (3 marks)

(c) Why is bcrypt preferred over SHA-256 for password hashing? (2 marks)

Model Answers

(a)

  • Rainbow tables: pre-computed tables of hash→password can reverse MD5 hashes instantly
  • Identical passwords: if two users have same password, they have same hash — reveals this relationship
  • Fast: MD5 can be computed billions of times per second on modern GPUs — brute force becomes feasible

(b)

  • Salt is a random value appended/prepended to the password before hashing: H(password + salt)
  • Rainbow tables are built for H(word) — adding a salt means an attacker would need a separate rainbow table for every possible salt value (computationally infeasible)
  • Two users with same password get different hashes (different salts) → no correlation visible

(c)

  • bcrypt has a configurable cost factor — makes hashing intentionally slow (100ms per hash)
  • SHA-256 is designed to be fast (billions/sec) — unsuitable for passwords. bcrypt's slowness makes brute force 10^7× harder.
Q9. DDoS — Types, impact, and mitigation
9 marks

(a) Describe the difference between a DoS and a DDoS attack. (2 marks)

(b) Explain how a SYN flood attack works and how SYN cookies mitigate it. (4 marks)

(c) List THREE other DDoS mitigation techniques. (3 marks)

Model Answers

(a) DoS (Denial of Service): single source sends flood of traffic to overwhelm target. DDoS (Distributed DoS): thousands of compromised machines (botnet) coordinate to overwhelm target from many IPs — making IP-based blocking ineffective.

(b)

  • SYN Flood: Attacker sends thousands of SYN packets with spoofed source IPs. Server allocates memory (TCB entry) for each half-open connection, waiting for ACK that never comes. Server's connection table fills up; legitimate connections are rejected.
  • SYN Cookies: Instead of allocating state, server encodes connection info in the ISN (sequence number) as a cryptographic cookie. State only allocated when valid ACK is received. Prevents table exhaustion while maintaining protocol compatibility.

(c)

  • Rate limiting per IP / per subnet
  • Traffic scrubbing via CDN (Cloudflare, Akamai) — filters traffic before it reaches server
  • Anycast IP distribution — attack traffic distributed across global PoPs
  • Blackholing — drop all traffic to target IP (last resort)
  • BGP flowspec — upstream router filters malicious traffic patterns

Q11. CBC Mode Error Propagation Analysis
6 marks

A message is encrypted using CBC mode and transmitted. A single bit in ciphertext block C_3 is corrupted during transmission.

(a) Which plaintext blocks will be affected upon decryption? Justify. (3 marks)

(b) Would the same corruption affect ECB mode differently? (3 marks)

Model Answer (a) — CBC Error Propagation

  • P_3 is completely garbled: D_K(C_3) produces wrong output since C_3 is corrupted — all bits of P_3 are affected
  • P_4 has exactly one bit flipped: P_4 = D_K(C_4) ⊕ C_3. Since D_K(C_4) is correct, and C_3 has exactly one bit flipped, P_4 has exactly one bit flipped in the same position
  • P_5, P_6, ... are correct: P_5 = D_K(C_5) ⊕ C_4. Since C_4 is uncorrupted, decryption of subsequent blocks is unaffected. CBC self-synchronises after 2 blocks.

Model Answer (b) — ECB Error Propagation

  • In ECB: P_i = D_K(C_i). Each block is independent.
  • Only P_3 is affected — D_K(C_3) produces garbled P_3
  • P_1, P_2, P_4, P_5, ... are completely unaffected
  • Difference: ECB error affects only one block; CBC error affects two blocks (the corrupted block and the next block partially)
Q12. Web Application Security — Secure Session Management
10 marks

Describe FIVE security mechanisms a web application should implement to ensure secure session management. For each, state which security aspect it addresses.

Model Answer

  1. HttpOnly; Secure Cookies (Confidentiality): Session tokens stored in cookies with HttpOnly flag (prevents JS access → mitigates XSS theft) and Secure flag (only sent over HTTPS)
  2. Session Regeneration after Login (Authenticity): Generate new session ID after successful authentication to prevent session fixation attacks
  3. Session Timeout (Confidentiality/Availability): Short inactivity timeouts (15-30 min) limit window for stolen session exploitation
  4. HTTPS Everywhere / HSTS (Confidentiality): All session tokens transmitted over encrypted channel; HSTS prevents SSL stripping attacks
  5. CSRF Tokens (Integrity): Per-request tokens prevent cross-site request forgery — attacker's site cannot forge authenticated requests on behalf of victim
  6. SameSite Cookie Attribute (Integrity): SameSite=Strict prevents CSRF by not sending cookies on cross-origin requests

1. Which security property is violated when an attacker intercepts and reads a private message without modifying it?
Integrity
Confidentiality
Availability
Non-repudiation
Confidentiality ensures data is only accessible to authorised parties. Reading without modification doesn't affect Integrity. The data is still available.
2. In CBC mode, if ciphertext block C₅ is corrupted, which plaintext blocks are affected?
Only P₅
P₅ through P_n (all subsequent)
P₅ (garbled) and P₆ (1-bit error only)
P₄, P₅, P₆
CBC: P_i = D_K(C_i) ⊕ C_{i-1}. Corruption in C_5 garbles P_5 completely and introduces a 1-bit error in P_6 (since P_6 = D_K(C_6) ⊕ C_5). From P_7 onwards, no effect since C_6 is uncorrupted.
3. In RSA with p=5, q=7, what is φ(n)?
35
24
30
12
φ(n) = (p-1)(q-1) = (5-1)(7-1) = 4 × 6 = 24. Note: n = 5×7 = 35, but φ(n) ≠ n.
4. What is the MAIN security weakness of ECB mode encryption?
It requires an IV (initialisation vector)
It cannot be parallelised
Identical plaintext blocks produce identical ciphertext, revealing patterns
It is too slow for practical use
ECB encrypts each block independently: C_i = E_K(P_i). If P_i = P_j, then C_i = C_j. This leaks information about the structure and content of the plaintext. The famous ECB penguin image demonstrates this vulnerability visually.
5. Which security aspect does NON-REPUDIATION primarily address?
Ensuring data is not modified in transit
Keeping data confidential from eavesdroppers
Preventing an entity from denying they performed an action
Ensuring a service remains accessible to users
Non-repudiation means an entity cannot deny performing an action. Implemented via digital signatures — the private key holder cannot deny signing something since only they have the key. Relevant in e-commerce, legal documents, etc.
6. Why is encrypting each character of a message independently with RSA insecure?
RSA cannot handle individual characters
An attacker can build a lookup table by encrypting all possible characters with the public key
It makes the ciphertext longer than necessary
RSA is non-deterministic so results vary
Since the alphabet is finite (26 letters, or 128 ASCII), an attacker uses the public key to encrypt every possible character. This creates a dictionary: ciphertext → plaintext. They then look up each character individually. Fix: use OAEP padding or hybrid encryption.
7. A company has 1000 employees and 500 resources. Switching from DAC to RBAC with 10 roles reduces max permission assignments from __ to __.
500,000 → 10,000
500,000 → 6,000
500,000 → 5,010
100,000 → 1,000
DAC: 1000 × 500 = 500,000 max assignments. RBAC: 1000 (user-role) + 10×500 (role-resource) = 1000 + 5000 = 6,000. This is a ~83× reduction in assignment complexity.
8. The THREE-WAY TCP handshake consists of:
SYN → ACK → FIN
SYN → SYN-ACK → ACK
SYN → ACK → SYN
SYN → DATA → ACK
1. Client sends SYN (synchronise) with sequence number x. 2. Server responds SYN-ACK (synchronise-acknowledge) with its own seq y and ack=x+1. 3. Client sends ACK (acknowledge) with ack=y+1. Connection established. SYN flood attacks exploit step 2 by never sending step 3.
9. Optical fibre is more secure than copper cable primarily because:
It transmits data faster
It is cheaper to install
It emits no electromagnetic radiation, making passive eavesdropping infeasible
It supports stronger encryption
Copper cables emit electromagnetic radiation that can be passively detected with inductive probes (no physical tapping needed). Optical fibre transmits light — no EM radiation escapes, so passive eavesdropping is extremely difficult. Physical tapping of fibre causes measurable signal loss, making it detectable.
10. Which hash property ensures that even a 1-bit change in input produces a completely different hash output?
Pre-image resistance
Collision resistance
Avalanche effect
Second pre-image resistance
The Avalanche Effect: a small change in input (even 1 bit) causes ~50% of output bits to change. This makes it impossible to make incremental modifications to a hash. Pre-image resistance = can't find input from output. Collision resistance = can't find two inputs with same output.
