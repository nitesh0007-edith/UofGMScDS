🗺️ Access Control: The Big Picture

An access control mechanism mediates between a user/entity and system resources (applications, OSes, firewalls, routers, files, databases). The full AAA pipeline is:

🔍 Authentication

Verification that the credentials of a user or other system entity are valid. Answers: "Are you who you claim to be?"

✅ Authorization

Granting of rights/permissions to a system entity to access a resource. Answers: "Are you allowed to do this?"

📋 Audit

Independent review and examination of system records and activities. Answers: "What did you do?"

🪪 Identity, Identification & Authentication

Identity

  • Representation of an entity inside a computer system
  • Often implies a unique name
  • Can change or be falsified (e.g. last name)

Identification

  • Claim by a user/application
  • User ID, process ID, smart card, etc.
  • The card/ID could be stolen

Authentication

  • Verification/proof of the identity claim
  • Must verify the ID through different methods
  • More factors = stronger assurance

Purposes of Identity

For Access Control

Determines what resources an entity can access and what operations it can perform.

For Accountability (Logging & Auditing)

Tracks who did what and when — essential for forensics and compliance.

Authentication Components (Formal)

Creating & Storing Authentication Info

  • Authentication information A — what the entity uses to prove identity (e.g., password)
  • Complementary information C — what the system stores alongside the identity
  • Complementary functions F — transforms A into C (one-way)
\[ C = f(A), \quad f \in F, \quad f: A \rightarrow C \]

For passwords: f is a one-way hash function (e.g., bcrypt, SHA-256 with salt). You can compute \(C\) from \(A\) but not the reverse.

Performing Authentication

  • Authentication functions L — verify an identity claim
\[ l(A, C) \rightarrow \{ \text{true}, \text{false} \}, \quad l \in L \]

The system checks whether the submitted \(A\) (e.g., typed password) matches the stored \(C\) (e.g., stored hash).

Example: Password Hashing Flow

# Registration (store phase)
password = "MySecret123!"        # A: authentication info
salt      = generate_random_salt()
stored_hash = bcrypt(password, salt)   # C = f(A): complementary info
db.store(username, salt, stored_hash)

# Login (verify phase)
attempt = get_user_input()
retrieved = db.lookup(username)        # retrieve C
result = bcrypt(attempt, retrieved.salt) == retrieved.hash  # l(A,C) → bool

🔒 Passwords

Purpose: Use information an entity knows to verify that a stated identity really belongs to that entity.

Key rule: Passwords must never be transmitted without proper protection (always use TLS/HTTPS). The hash function \(f \in F\) must be one-way.

Strong Password Properties

Theorem: When a password is selected uniformly at random from the full set of possible passwords, the expected time needed to guess it is maximised.

✅ At least one digit

0–9 adds 10 symbols to the alphabet

✅ Upper + lower case

a–z + A–Z adds 52 symbols

✅ Special character

Punctuation / control chars (e.g. !@#$%)

⚔️ Password Attacks

🗂️ Dictionary Attack

  • Most real passwords are not random — they're real words, names, acronyms
  • Attacker uses a list of likely passwords (e.g., "rockyou.txt" with 14M entries)
  • Examples targeted: Betty23, ChocolateFrog
  • Vastly reduces the search space
Also includes rainbow table attacks — precomputed hash→password mappings. Defeated by salting.

💥 Brute-Force Attack (Exhaustive)

  • Try every possible combination of characters
  • Time depends on password length and alphabet size
  • Making password standards public can be a security risk

Brute-Force Calculation

If a password is 1–8 characters from: uppercase (26) + lowercase (26) + digits (10) + special (32) = 94 characters:

\[ \text{Total possible passwords} = \sum_{i=1}^{8} 94^i = 6.1 \times 10^{15} \]
\[ \text{For exactly 8 chars: } 94^8 = 6.1 \times 10^{15} \text{ (the dominant term)} \]

Anderson's Formula

Variables

SymbolMeaning
PProbability of correctly guessing in the time period (e.g. 0.5)
GNumber of guesses per time unit
TNumber of time units
NTotal number of possible passwords
\[ P \geq \frac{TG}{N} \quad \Longleftrightarrow \quad N \geq \frac{TG}{P} \]

This gives the minimum N required so that an attacker has at most probability P of success.

Worked Example — Minimum Password Length

Given: Alphabet \(A = 96\) chars, \(G = 10^4\) guesses/sec, \(P = 0.5\), \(T = 365\) days

Convert T to seconds: \(T = 365 \times 24 \times 60 \times 60 = 31.536 \times 10^6 \text{ sec}\)
Apply Anderson's formula: \(N \geq \frac{TG}{P} = \frac{31.536 \times 10^6 \times 10^4}{0.5} = 6.31 \times 10^{11}\)
Find minimum password length \(S\):
\[ N = \sum_{i=1}^{S} 96^i \geq 6.31 \times 10^{11} \]
Check: \(96^5 = 8.15 \times 10^9 \approx 8 \times 10^9\) — too small. \(96^6 = 7.83 \times 10^{11}\) — sufficient!
Answer: S ≥ 6 characters minimum (with 96-char alphabet, 10,000 guesses/sec over 1 year)

Other Threat Models for Password Authentication

  • Phishing — fake login page captures credentials
  • Keylogging/Malware — records keystrokes in real-time
  • Shoulder surfing — physical observation
  • Credential stuffing — reusing leaked username/password pairs
  • Pass-the-hash — reusing a stolen hash without cracking
  • Social engineering — tricking user into revealing password

🛡️ Countermeasures to Password Guessing

Goal: Maximise the time consumed before the password is correctly guessed.

⏳ Exponential Back-off

After the n-th failed attempt, wait \(t^{n-1}\) seconds before allowing the next login:

\[ \text{Wait} = t^{n-1} \text{ seconds} \]

where t is a system-defined parameter (e.g., t = 2 means waits of 1, 2, 4, 8, 16 seconds...)

🔌 Disconnection

Disconnect the session entirely after a specified number of failed attempts.

🚫 Disabling

Permanently disable the account after N failed attempts. Requires admin intervention to re-enable.

🍯 Jailing (Honeypot)

Instead of disconnecting, fool the attacker — allow them into a fake/sandboxed environment and record all their activities. Useful for threat intelligence.

Example: After 3 failed attempts, let the attacker "succeed" into a fake shell with fake data while logging every command they run.

Web Application Defences

Secure Authentication

  • Hash + salt passwords on server
  • Two-factor authentication (2FA)
  • Login attempt rate-limiting
  • CAPTCHA after N failures

Session Management

  • Generate secure session ID after login
  • Use HttpOnly + Secure cookie flags
  • Session timeout on inactivity
  • Regenerate session ID after privilege change

Data Encryption

  • HTTPS (TLS) for all communication
  • Encrypts credentials, tokens, cookies in transit
  • Certificate pinning for mobile apps

Session Hijacking & Defences

URL Session ID Vulnerability

When session IDs appear in URLs (e.g. example.com/page?sessionid=abc123), they can be:

  • Logged in browser history, proxy logs, referrer headers
  • Stolen via XSS that extracts the URL
  • Shared accidentally (copy-paste of URL)

Defences: Store session IDs in HttpOnly cookies only, use CSRF tokens, set session expiry, regenerate IDs after login, validate IP/User-Agent binding.

👁️ Biometrics

Purpose: Automated measurement of biological or behavioural features to characterize and hence identify an entity. Requires special sensors.

Physiological Biometrics

  • 👆 Fingerprints — unique ridge patterns, widely deployed
  • 👀 Retina/Iris scan — highly accurate, but invasive
  • 😊 Face recognition — convenient, affected by lighting/ageing
  • 🧬 DNA — most accurate, not real-time

Behavioural Biometrics

  • 🎤 Voice recognition — affected by illness, background noise
  • ⌨️ Keystroke dynamics — pressure, interval, duration, position
  • 🚶 Gait and motion biometrics — how you walk
  • 🖊️ Signature dynamics — speed and pressure of signing
Exam tip — Continuous Authentication: Behavioural biometrics (keystroke dynamics, gait) are uniquely suited for continuous authentication because they can be monitored throughout a session without interrupting the user.

Biometric Error Rates

FAR — False Acceptance Rate

Probability that the system incorrectly accepts an impostor. Security risk — lower is better.

\[ FAR = \frac{\text{False Accepts}}{\text{Total Impostor Attempts}} \]

FRR — False Rejection Rate

Probability that the system incorrectly rejects a legitimate user. Usability issue — lower is better.

\[ FRR = \frac{\text{False Rejects}}{\text{Total Genuine Attempts}} \]

EER — Equal Error Rate (Crossover Point)

The threshold where FAR = FRR. Used to compare systems — lower EER = better biometric system.

Critical Limitation: Unlike passwords, biometric templates cannot be revoked. If a fingerprint database is compromised, you cannot change your fingerprint. This is a fundamental security risk compared to knowledge-based auth.

Single Sign-On (SSO)

✅ Advantages of SSO

  • User logs in once, accesses all services transparently
  • Reduces password fatigue — fewer passwords to remember
  • Centralised security policy enforcement
  • Faster user experience (fewer login screens)
  • Easier audit trail — single point of logging
  • Easier revocation — disable one account to cut all access

❌ Disadvantages of SSO

  • Single point of failure — if the SSO provider is down, all services are inaccessible
  • Single point of compromise — one leaked credential gives access to everything
  • Implementation complexity
  • Requires a trusted identity provider (IdP)
  • Privacy: IdP sees all user activity across services

SSO and the CIA+ Properties

PropertyImpact of SSO
ConfidentialityCentralised auth reduces password exposure; BUT a compromise exposes all services
IntegritySingle token/session can be tampered with or replayed if not protected properly
AuthenticityStronger — one robust authentication replaces many weak logins
AccountabilityImproved — all activity traced to one identity via a central log
Non-repudiationStrong if SSO uses digital signatures; weak if session tokens can be stolen

🎟️ Kerberos Authentication Protocol

Foundation: Needham-Schroeder protocol + Denning-Sacco modification. Uses symmetric key cryptography and time-limited tickets to provide SSO across many services.

Kerberos Entities

AS — Authentication Server

Authenticates a user to the Kerberos system. Shares a secret key with every user and with the TGS. Issues a Ticket-Granting Ticket (TGT).

TGS — Ticket-Granting Server

Issues service tickets to authenticated users. Shares a secret key with each application server. Users present TGT to get service tickets.

Application Servers S₁…Sₙ

The actual resources (file servers, email, etc.). Accept service tickets from TGS. No direct contact with AS needed for each access.

Client (Alice)

The user. Authenticates once to AS, then uses TGT to get service tickets for any application server — transparent to the user.

Kerberos Ticket Structure

Ticket \(T_{Alice, Server}\)

Encrypted with the server's secret key \(K_{server}\):

\[ T_{Alice,Server} = \{ Alice \| Alice's\text{ }address \| valid\text{ }time \| K_{Alice,Server} \}_{K_{server}} \]
  • \(K_{Alice,Server}\) — session key for this particular access (generated by the ticket issuer)
  • \(K_{server}\) — server's long-term secret key (only server can decrypt the ticket)
  • Alice presents this ticket to Server; Server decrypts it to get the session key

Authenticator \(A_{Alice, Server}\)

Encrypted with the session key \(K_{Alice,Server}\):

\[ A_{Alice,Server} = \{ Alice \| t \}_{K_{Alice,Server}} \]
  • \(t\) — current timestamp (proves freshness, prevents replay)
  • Alice creates this herself to prove she has the session key
  • Without \(K_{Alice,Server}\), an attacker cannot create a valid authenticator

Significance of Kerberos

Single Sign-On

  • User logs in once to AS → gets TGT
  • TGS issues service tickets → login to each app is transparent
  • No password re-entry for each service

Security Properties

  • Protected against eavesdropping (all messages encrypted)
  • Protected against replay attacks (timestamps in authenticators)
  • Widely used in financial systems and large-scale e-commerce
  • Forms basis of Windows Active Directory authentication

How Does AS Share Keys With All Users?

During user registration, the AS and user establish a shared secret (derived from the user's password via a one-way function). This shared key never travels over the network. When the user authenticates, the AS uses this key to encrypt the TGT response.

📝 Exam Questions & Full Solutions

Q1 — May 2025 (Actual Exam) • Session ID Vulnerability

5 marks

A company called Travelsafe operates through a website that uses URL session IDs (unique numbers assigned by a website server to a user for a specific session after the user's account has been authenticated). Mention and describe a potential vulnerability and the corresponding defence mechanisms that can be employed to mitigate it.

Full Answer (5 marks)

Vulnerability: Session Hijacking via Exposed Session ID

URL session IDs are vulnerable because they are included in the URL string (e.g., https://travelsafe.com/search?sessid=abc123). This exposes the session ID in:

  • Browser history (accessible to anyone with physical or remote access)
  • Server access logs and proxy logs (readable by third parties)
  • HTTP Referer header (sent when clicking a link to another site)
  • Screen / shoulder surfing

An attacker who obtains a valid session ID can impersonate the authenticated user without needing their password — this is a session hijacking attack.

Additional vulnerability: Cross-Site Scripting (XSS) — if the search box does not sanitise input, an attacker can inject JavaScript that reads and exfiltrates the URL (including session ID) to a remote server.

Defence Mechanisms:

  • Store session IDs in HttpOnly cookies (not in URLs) — inaccessible to JavaScript
  • Use Secure flag on cookies — only transmitted over HTTPS
  • Implement session timeout — expire sessions after inactivity
  • Use SameSite cookie attribute — prevents CSRF attacks
  • Regenerate session ID after login to prevent session fixation
  • Sanitise all user input to prevent XSS (escaping, Content Security Policy)
  • Bind sessions to IP address or User-Agent string

Q2 — May 2025 (Actual Exam) • Single Sign-On

7 marks

Some companies, including Google and Amazon, use Single Sign-On (SSO) for user authentication. 1) Argue for the advantages and disadvantages of using SSO over traditional approaches. 2) Explain what SSO means for Confidentiality, Integrity, Authenticity, Accountability, and Non-repudiation.

Full Answer (7 marks)

Part 1 — Advantages vs Disadvantages [~4 marks]

Advantages:

  • Convenience: One login gives access to all integrated services (Google: Gmail, Drive, Maps, YouTube)
  • Reduced password fatigue: Users remember one strong password instead of many weak ones
  • Centralised security control: Security updates (MFA requirements, lockouts) apply universally
  • Easier revocation: Disabling one account immediately cuts all access
  • Better audit: All activity correlates to one identity

Disadvantages:

  • Single point of failure: SSO provider outage locks out all services
  • Single point of compromise: One stolen credential (or cookie) grants access to everything
  • Privacy: The IdP can track all user activity across all connected services
  • Implementation complexity: Requires careful setup (OAuth, SAML, OpenID Connect)

Q3 — Practice: Authentication Components

6 marks

a) Define the authentication components A, C, and the function \(f: A \rightarrow C\). Why must \(f\) be a one-way function? [3 marks]

b) An authentication system stores SHA-256(password) without a salt. An attacker gains read access to the database. Explain TWO specific attacks possible and how to defend against them. [3 marks]

Full Answer (6 marks)

Part a)

  • A (Authentication information): What the entity provides to prove identity — e.g., the plaintext password
  • C (Complementary information): What the system stores — e.g., the hashed password
  • f: A → C maps the authentication info to stored form. \(f\) must be one-way so that even if the database is stolen, an attacker cannot reverse \(C\) back to \(A\) (the original password). Computing \(f(A)\) is easy; inverting is computationally infeasible.

Part b)

  • Attack 1 — Rainbow Table Attack: Attacker uses a precomputed table of SHA-256(common passwords) → password. Without a salt, identical passwords produce identical hashes, so a single lookup reveals many passwords at once.
    Defence: Add a unique random salt to each password before hashing — salt is stored with the hash; same password produces different hashes.
  • Attack 2 — Dictionary/Brute-Force Offline: Attacker can hash millions of candidate passwords per second offline (GPUs can do billions of SHA-256s/sec) and compare against the stolen hashes.
    Defence: Use a slow, memory-hard hash function like bcrypt, scrypt, or Argon2 — these are deliberately slow, making offline cracking impractical.

Q4 — Practice: Kerberos Protocol

8 marks

a) Describe the five-step Kerberos protocol. State the message sent in each step and what cryptographic key is used to protect it. [5 marks]

b) Explain how Kerberos protects against replay attacks. What would happen if system clocks were not synchronised? [3 marks]

Full Answer (8 marks)

Part a) Five Steps

Alice → AS: \(Alice \| TGS\) — plaintext request saying "I am Alice, I want access to TGS"
AS → Alice: \(\{K_{Alice,TGS}\}_{K_{Alice}} \| T_{Alice,TGS}\) — session key encrypted with Alice's key (only Alice can read it) + TGT encrypted with TGS's key (Alice cannot read this part)
Alice → TGS: \(T_{Alice,TGS} \| A_{Alice,TGS} \| S\) — TGT (from step 2) + Authenticator (Alice proves she has \(K_{Alice,TGS}\)) + name of service S wanted
TGS → Alice: \(\{K_{Alice,S}\}_{K_{Alice,TGS}} \| T_{Alice,S}\) — new session key for S (encrypted with \(K_{Alice,TGS}\)) + service ticket (encrypted with S's key)
Alice → S: \(T_{Alice,S} \| A_{Alice,S}\) — service ticket + fresh authenticator proving Alice has \(K_{Alice,S}\). S grants access.

Part b) Replay Attack Protection

  • Authenticators include a timestamp \(t\). Servers reject any authenticator whose timestamp is outside an acceptable window (typically ±5 minutes)
  • Servers can also cache recently used authenticators — same authenticator submitted twice is rejected
  • If clocks are not synchronised (clock skew > 5 min), legitimate users will be rejected (their fresh authenticators will be deemed expired). This is why Kerberos requires NTP (Network Time Protocol) or similar clock synchronisation across all machines.

Q5 — Practice: Anderson's Formula & Password Length

6 marks

A security policy requires that the probability of an attacker guessing a password within 30 days is no more than 0.01. The system allows \(10^6\) guesses per second. The password alphabet contains 94 characters. What is the minimum password length required? Show all working. [6 marks]

Full Answer (6 marks)

Convert T to seconds: \(T = 30 \times 24 \times 60 \times 60 = 2.592 \times 10^6 \text{ seconds}\)
Apply Anderson's formula with \(G = 10^6\), \(P = 0.01\): \[ N \geq \frac{TG}{P} = \frac{2.592 \times 10^6 \times 10^6}{0.01} = 2.592 \times 10^{14} \]
Find minimum S such that \(\sum_{i=1}^{S} 94^i \geq 2.592 \times 10^{14}\):
  • \(94^7 = 6.46 \times 10^{13}\) — insufficient
  • \(94^8 = 6.07 \times 10^{15}\) — sufficient! (dominant term)
Answer: Minimum password length = 8 characters

Q6 — May 2025 (Actual Exam) • Brute Force Key Attack

5 marks

Assuming you can do \(2^8\) encryptions per second and the key size is 16 bits, how long would a brute force attack take? Give a scenario where this would be practical and another where it wouldn't. What happens if you double the key size? [5 marks]

Full Answer (5 marks)

Total key space for 16-bit key: \(2^{16} = 65{,}536\) possible keys
Time (worst case) = \(\frac{2^{16}}{2^8} = 2^{16-8} = 2^8 = 256\) seconds ≈ 4.3 minutes
Expected time (on average, find key halfway): \(\approx\) 128 seconds ≈ 2 minutes
Practical scenario: Recovering a lost personal encryption key in a controlled environment (e.g., recovering your own encrypted file). 4 minutes is trivial.
Impractical scenario: This would NOT be acceptable for protecting financial data or medical records — any serious system would use at least 128-bit keys, making brute force infeasible.
Doubling key size (16 → 32 bits): \(2^{32} / 2^8 = 2^{24} \approx 1.67 \times 10^7\) seconds ≈ 194 days. Every extra bit doubles the brute-force time. Doubling key size squares the keyspace.

Q7 — Practice: Biometrics vs Password

5 marks

a) Explain the difference between FAR and FRR in biometric systems. What is the EER and why is it useful? [3 marks]
b) Why can biometric data not be "revoked" in the same way as passwords? What security implication does this have? [2 marks]

Full Answer (5 marks)

Part a)

  • FAR (False Acceptance Rate): Rate at which the system incorrectly accepts an impostor. High FAR = low security.
  • FRR (False Rejection Rate): Rate at which the system incorrectly rejects a legitimate user. High FRR = poor usability.
  • EER (Equal Error Rate): The operating threshold where FAR = FRR. Lower EER indicates a more accurate biometric system. It provides a single comparable metric.
  • There is a trade-off: making the system stricter reduces FAR but increases FRR, and vice versa.

Part b)

  • A password can be changed if compromised. Biometric traits (fingerprints, iris patterns) are permanent — you cannot change them.
  • If a biometric database is breached, affected users are permanently at risk. This is why biometric templates must be stored securely (ideally as irreversible transforms, not raw biometric data). This is a fundamental limitation not shared by knowledge-based authentication.

Q8 — Practice: Password Attack Countermeasures

6 marks

a) Explain the exponential back-off mechanism. Give the formula and a concrete example with t=2 for the first 5 failed attempts. [3 marks]
b) Compare and contrast Disconnection, Disabling, and Jailing as countermeasures to brute-force attacks, stating one advantage of each. [3 marks]

Full Answer (6 marks)

Part a)

After the \(n\)-th failed attempt, the system waits \(t^{n-1}\) seconds before allowing another attempt:

\[ \text{Wait after attempt } n = t^{n-1} \text{ seconds} \]

With t = 2:

Attempt (n)Wait time \(2^{n-1}\)
11 second
22 seconds
34 seconds
48 seconds
516 seconds

This rapidly increases the time per guess, making brute force impractical.

Part b)

  • Disconnection: Terminates the session after N failures. Advantage: user can try again by reconnecting — less disruptive than permanent disable. Good for interactive sessions.
  • Disabling: Permanently locks the account. Advantage: completely stops automated brute-force tools. Best for protecting high-value accounts (requires admin to re-enable).
  • Jailing (Honeypot): Fools attacker into a fake environment. Advantage: allows intelligence gathering — logs attacker's tools, techniques, targets. Provides threat intelligence without alerting the attacker.

4. Why is storing passwords as plain SHA-256 hashes (without salt) insecure?

ASHA-256 is not a one-way function
BAllows rainbow table attacks and reveals when two users share the same password

Without a salt, identical passwords produce identical hashes. This enables precomputed rainbow table lookups and reveals common passwords across users.

2³²/2⁸ = 2²⁴ = 16,777,216 seconds ≈ 194 days. Doubling key size squares the keyspace — exponential increase in brute-force time.
