Overview — The Big Picture

Before any resource can be accessed, a system must answer two questions:

1. Who are you?
→ Identification — the claim made by an entity
2. Can you prove it?
→ Authentication — verification of that claim

Authentication gates access to resources via Access Control

Identity, Identification & Authentication

Identity

Definition: A representation of an entity inside a computer system; often a unique name.
⚠ A person's identity can change or be falsified (e.g. last name)

Purposes

  • Access control
  • Accountability (logging & auditing)

Examples

ObjectIdentity used by…
File nameHuman

FAT entryOS Kernel (MS-DOS)
UsernameAuthentication system

Identification

Definition: The claim made by a user or application that it is a particular entity.
⚠ The ID or smart card could be stolen

Methods of Identification

  • User ID / username
  • Process ID (PID)
  • Smart card
  • Any unique identifier

Authentication

Definition: The verification / proof of the claimed identity.

A system must verify the claimed identity through one or more methods before granting access.

✓ Authentication ≠ Identification. You can identify without authenticating.

Groups and Roles

Groups

An identity can represent a group — a collection of entities. Convenient for applying access control to many entities at once.

  • Static group: An alias for a fixed set of entities
  • Dynamic group: Construct that groups entities based on conditions

Roles

An identity can represent a role — a set of rights or security functions. Entities are assigned to roles.

Example: admin, read-only, editor

ℹ Roles are the foundation of RBAC (Role-Based Access Control)

Identity & Certificates

A Certificate Authority (CA) acts as a trusted third party that issues certificates to verify identity.

Class 1 — Ownership
Authenticates an email address or web application domain.
Class 2 — Personal
Verifies real name and physical address via online database (e.g. online purchasing).
Class 3 — Rigorous
Background check by investigative service; highest level of assurance.
⚠ Trust of a certificate depends on the trustworthiness of the CA. Even certificates have trust issues.

Authentication Components — Formal Model

The authentication system has three formal components:

A — Authentication Information
The credential used by an entity to prove identity (e.g. password typed by user).
A = password entered
C — Complementary Information
Stored version of the credential used by the system to verify A.
C = hash(password) stored
F — Complementary Functions
Transforms A into C. For f ∈ F:
f: A → C
Usually a one-way hash function.
L — Authentication Functions
Verifies the identity. For l ∈ L:
l(A, C) → {true, false}
Flow: User types A → System computes f(A) → Compares with stored C → l(A, C) = true/false
# f is a one-way hash: easy to compute forward, infeasible to reverse

Password Hashing — How It Works

// Registration: store hash, not plaintext
function registerUser(username, plaintext_password):
    salt = generateRandomSalt()        // random bytes to prevent rainbow tables
    hash = bcrypt(plaintext_password + salt)  // one-way hash: A → C
    db.store(username, hash, salt)

// Login: verify without storing plaintext
function loginUser(username, plaintext_password):
    (hash, salt) = db.lookup(username)
    candidate_hash = bcrypt(plaintext_password + salt)
    return candidate_hash == hash   // l(A, C) → {true, false}

Passwords

Password Selection & Strength

Theorem: When passwords are chosen uniformly at random from the full password space, the expected time to crack is maximised.

Strong Password Requirements

  • At least one digit (0–9)
  • At least one uppercase and one lowercase letter
  • At least one special character (punctuation, symbols)
  • Sufficient length (minimum 8–12 characters)

Methods Against Password Guessing

1. Exponential Back-off

After the n-th failed attempt, wait tn−1 seconds before allowing the next attempt. Grows rapidly:

t=2: delays → 1s, 2s, 4s, 8s, 16s, 32s, …
# Effective against online brute-force

2. Disconnection

Disconnect the session after k consecutive failed login attempts. Forces attacker to reconnect, adding latency.

3. Account Disabling

Permanently (or temporarily) disable the account after k failures. Requires manual re-activation or admin intervention.

4. Jailing (Honeypot)

Appear to accept the wrong password and let the attacker "in" to a fake environment. Record all their activities for forensic analysis.

ℹ Also called a Honeypot — deceives attacker while logging evidence.

Brute-Force Attack

Tries every possible combination of characters until the correct password is found (exhaustive attack).

Password space N = Σᵢ₌₁ˢ Aⁱ
# A = alphabet size, S = max length
A=94 chars (all printable), S=8:
N = 94¹+94²+…+94⁸ ≈ 6.1 × 10¹⁵
⚠ Knowing password standards (min length, required characters) reduces the search space and is a security risk!

Dictionary Attack

Passwords are rarely truly random. People use common words, names, and acronyms like "Betty23" or "ChocolateFrog".

Uses a pre-compiled list of likely passwords to systematically try each one.

⚠ Vastly reduces the search space compared to brute force. Even 8-char passwords fall quickly if dictionary words.

Defence: Salted Hashing

# Salt makes pre-computed rainbow tables useless
H = hash(password + salt)
# Attacker must re-hash entire dictionary per user

Phishing

Attacker creates a fake login page resembling a legitimate site. User unknowingly submits credentials to attacker's server.

Red Flags to Check

  • Sender email address doesn't match domain
  • Grammar mistakes / urgency in message
  • Unexpected requests (login, download, pay)
  • Suspicious attachment names
  • URL doesn't use HTTPS or looks spoofed

Keylogging / Malware

Malware installed on the victim's device records keystrokes including passwords as they are typed.

Defences

  • Antivirus / endpoint detection
  • Don't store passwords in plaintext on clients
  • Use password managers (encrypted vault)
  • Audit system for malware presence
  • Virtual keyboard for critical logins

Anderson's Formula — Password Length Calculation

Anderson's formula lets you calculate the minimum required password length to resist guessing attacks with acceptable probability.

Variables:
P = probability of correctly guessing in time T (e.g. P = 0.5)
G = guesses per time unit (e.g. 10⁴/second)
T = time period in time units (e.g. 365 days in seconds)
N = total number of possible passwords
Anderson's Formula:
P ≥ TG/N ⟺ N ≥ TG/P
# Rearrange to find minimum N, then find minimum password length S
Worked Example — Lecture Example

Find minimum password length S for a system with A=96 characters:

1
Given: A = 96 characters, G = 10⁴ guesses/sec, P = 0.5, T = 365 days
2
Convert T to seconds: T = 365 × 24 × 60 × 60 = 31,536,000 seconds = 3.1536 × 10⁷
3
Apply Anderson's Formula: N ≥ TG/P = (3.1536 × 10⁷ × 10⁴) / 0.5 = 6.31 × 10¹¹
4
Find minimum S: N = Σᵢ₌₁ˢ 96ⁱ ≥ 6.31 × 10¹¹
5
Try S = 5: Σ = 96¹+96²+96³+96⁴+96⁵ ≈ 8.15 × 10⁹  ❌ too small
Try S = 6: Σ ≈ 7.83 × 10¹¹  ✓ ≥ 6.31 × 10¹¹
Answer: Minimum password length S = 6
Exam-Style Example — Key Size Brute Force

Q: Speed = 2⁸ encryptions/sec, key size = 16 bits. How long does a brute-force attack take?

1
Total keys = 2¹⁶ = 65,536
2
Speed = 2⁸ = 256 encryptions/second
3
Worst case time = 2¹⁶ / 2⁸ = 2⁸ = 256 seconds ≈ 4.3 minutes
4
Practical scenario: Protecting a low-value resource (a casual game save). 4 min is too long to wait, so attacker gives up. ✓
Not practical: Bank encryption — 4 min is completely feasible. A determined attacker will try all keys. ✗
5
Double key size to 32 bits: 2³² / 2⁸ = 2²⁴ ≈ 16.7 million seconds ≈ 194 days. Now impractical.
Key doubling rule: each extra bit DOUBLES the search space
16 bits → 32 bits: multiply time by 2¹⁶ = 65,536×

Kerberos Authentication Protocol

Foundation: Based on the Needham-Schroeder protocol + Denning-Sacco modification. Uses secret-key cryptography.

Use: Widely used in corporate networks (Windows Active Directory), financial systems, and large-scale e-commerce.

Key Components

AS — Authentication Server
Shares a secret key with every user and with TGS. Authenticates users to the Kerberos system.
TGS — Ticket-Granting Server
Shares a secret key with each application server S₁…Sₙ. Issues service tickets.
S₁…Sₙ — Application Servers
The actual resources. Accessed by presenting tickets issued by TGS.

Kerberos Data Structures

Ticket

T_Alice,Server = {Alice ‖ Alice's address ‖ valid time ‖ K_Alice,Server} K_server
# K_Alice,Server = session key (shared between Alice & Server)
# K_server = Server's secret key (encrypts the whole ticket)
# Only Server can decrypt — Alice cannot read the ticket contents

Authenticator

A_Alice,Server = {Alice ‖ t ‖ K_t} K_Alice,Server
# t = timestamp (prevents replay attacks)
# K_t = alternative session key
# Proves Alice possesses the session key — freshly created

The Protocol — Step by Step (Click to Animate)

1
Alice → AS
Alice ‖ TGS
Alice sends her identity and the name of the TGS she wants access to.
2
AS → Alice
{K_Alice,TGS} K_Alice ‖ T_Alice,TGS
AS sends: (1) session key encrypted with Alice's key, (2) a Ticket-Granting Ticket (TGT) for TGS.
3
Alice → TGS
S ‖ A_Alice,TGS ‖ T_Alice,TGS
Alice presents her TGT + an Authenticator (proving she has the session key) + name of desired server S.
4
TGS → Alice
Alice ‖ {K_Alice,S} K_Alice,TGS ‖ T_Alice,S
TGS issues a service ticket T_Alice,S for server S, plus a new session key K_Alice,S encrypted for Alice.
5
Alice → Server S
A_Alice,S ‖ T_Alice,S
Alice presents service ticket + fresh Authenticator. Server decrypts ticket to get K_Alice,S and verifies authenticator. Access granted!

Security Properties of Kerberos

Mutual Authentication
Both client and server verify each other's identity.
Replay Attack Prevention
Timestamps and limited ticket validity periods prevent replayed messages.
Eavesdropping Protection
All sensitive data is encrypted; session keys are never transmitted in plaintext.
Single Sign-On
Log in once to AS; TGT allows transparent access to all application servers.

Single Sign-On (SSO)

SSO allows a user to authenticate once and gain access to multiple services without re-authenticating. Used by Google, Amazon, Microsoft, university systems, etc.

Advantages of SSO

Convenience: Single password reduces cognitive load and password fatigue.
Fewer weak passwords: Users don't need to create many passwords, so the one password is (usually) stronger.
Centralised control: IT can revoke all access for a user in one action.
Reduced attack surface: Fewer passwords means fewer credential stores to breach.
Audit trail: Central point means comprehensive logging of authentication events.

Disadvantages of SSO

Single point of failure: If the SSO provider goes down, ALL services become inaccessible.
Single point of compromise: One stolen credential → access to ALL linked services.
Third-party dependency: Reliance on Google/Microsoft raises vendor lock-in and data concerns.
Session hijacking risk: A stolen SSO token grants access to everything (vs. one service).
Privacy: The SSO provider can track user logins across all connected services.

SSO and the Security Properties (CIA + AN)

✗ Single token compromise exposes all services

URL Session IDs — Vulnerability

Some websites pass the session ID in the URL (e.g. example.com/search?session=ABC123).

Vulnerability: Session Hijacking
  • URL stored in browser history → leaked
  • URL appears in server access logs → leaked
  • URL shared via Referer headers to other sites
  • Predictable session IDs can be guessed
  • Session fixation attacks possible
  • XSS can extract session IDs from URLs
Defence Mechanisms
  • Store session ID in HttpOnly cookies (not URL)
  • Use cryptographically random session IDs
  • Implement session timeouts
  • Regenerate session ID after login
  • Use HTTPS to encrypt transmission
  • Validate IP / user-agent binding
  • CSRF tokens for state-changing requests
⚠ The Travelsafe scenario (exam Q1): the search box could also be vulnerable to SQL injection or XSS, which in turn could steal session IDs.

Exam Questions & Model Answers

Q1 — May 2025 (5 marks) | Travelsafe URL Session IDs 5 marks
A company called Travelsafe operates through a website that uses URL session IDs. The website has a search box for visitors to identify holiday locations. Mention and describe a potential vulnerability and the corresponding defence mechanisms that can be employed to mitigate it.
Model Answer

Vulnerability 1: Session Hijacking via Exposed URL Session ID

URL session IDs are visible in browser history, server logs, and Referer headers. An attacker who obtains the session ID can impersonate the authenticated user without knowing their password. This is called session hijacking.

Defence mechanisms:

  • Store session IDs in HttpOnly, Secure cookies instead of the URL — prevents JavaScript access and ensures HTTPS-only transmission.
  • Generate cryptographically random, sufficiently long session IDs — prevents guessing.
  • Implement session timeouts — limits the window of exploitation if a session ID is leaked.
  • Regenerate session ID after login to prevent session fixation.
  • Bind sessions to IP address / user-agent for additional verification.

Vulnerability 2: Cross-Site Scripting (XSS) via Search Box

The search box may be vulnerable to reflected or stored XSS. An attacker could inject malicious JavaScript that steals the session cookie or redirects users to a phishing page.

Defence mechanisms:

  • Input validation and output encoding — encode all user input before rendering in HTML.
  • Implement a Content Security Policy (CSP) to restrict script execution.
  • Use HttpOnly cookies so JavaScript cannot access session tokens.
Q2 — May 2025 (7 marks) | Single Sign-On (SSO) 7 marks
Some companies, including Google and Amazon, use Single Sign-On (SSO) for user authentication. 1) Argue for the advantages and disadvantages of using SSO over traditional approaches. 2) Explain what SSO means for Confidentiality, Integrity, Authenticity, Accountability, and Non-repudiation.
Model Answer

Part 1: SSO Advantages vs. Traditional (3 marks)

Advantages:

  • Usability: One login for many services — reduces password fatigue and forgotten passwords.
  • Stronger security posture: Users maintain one strong password rather than many weak ones.
  • Centralised administration: Admins can revoke all access in one action (e.g. on employee departure).
  • Reduced attack surface: Fewer credential stores to protect against breaches.

Disadvantages:

  • Single point of failure: SSO system downtime → all services inaccessible.
  • Single point of compromise: One stolen SSO credential → all linked services compromised.
  • Vendor dependency: Reliance on Google/Microsoft creates lock-in and privacy concerns.

Part 2: CIA + AN Implications (4 marks)

PropertyImpact of SSO
ConfidentialityReduced risk of credential interception (fewer passwords). But a compromised SSO token exposes all services simultaneously.
IntegrityCentralised auth allows early anomaly detection. But if SSO is breached, the attacker can manipulate data across all connected services.
AuthenticityA strong, MFA-protected SSO system raises the authentication bar. However, if the SSO IdP is compromised, authenticity fails everywhere.
AccountabilitySSO provides a single, comprehensive audit log of all authentication events — strengthens accountability and forensic capability.
Non-repudiationThe central log serves as authoritative, time-stamped evidence. However, if the SSO is compromised, an attacker's actions may be misattributed to a legitimate user.

Tutorial Q2 — Web App Security: Login & Email Access Tutorial
Explain how web application security mechanisms protect the user during the login and email access process. Include: Secure authentication, Session management, Data encryption.
Model Answer

Secure Authentication

  • User submits username + password over HTTPS login page.
  • Password stored server-side as a salted hash (bcrypt/Argon2), never plaintext.
  • Many systems enforce 2FA/MFA: code sent to mobile or authenticator app.
  • Login attempt limits prevent automated brute-force attacks.

Session Management

  • After authentication, server generates a cryptographically random session ID.
  • Stored in a Secure, HttpOnly cookie — not visible to JavaScript, only sent over HTTPS.
  • Session ID is used for subsequent requests (stateless auth).
  • Session timeout after inactivity invalidates the session.
  • Session ID regeneration after login prevents session fixation.

Data Encryption

  • All communication over HTTPS / TLS — encrypts credentials, email content, attachments.
  • Prevents interception and MITM attacks, especially on public Wi-Fi.
  • TLS certificates validated by the browser to prevent spoofing.
Tutorial Q3 — Aleta's Phishing Email (XCORP Scenario) Tutorial
Aleta received an email urging her to log in and download "salary summary 2019.pdf". She suspects it could be malicious. a) What should Aleta check and do? b) What 3 things can the risk management team identify?
Model Answer

Part a) Advice for Aleta

Red flags to check in the email:

  • Is the sender address from the official company domain?
  • Are there grammar mistakes or unusual language?
  • Does it create urgency or ask for unusual action (login, download, payment)?
  • Is the link URL actually the company's official site?
  • Is the PDF filename unusual? ("2019" when it's a different year is suspicious.)
  • Is the attachment password-protected to evade antivirus?

  1. Sender IP address: Examine email headers (via email client investigation / pseudocode) to extract originating IP.
  2. Sender domain: Check the domain via open-source intelligence tools (e.g. ipvoid, VirusTotal) to determine if it is blacklisted or recently registered.
  3. Attachment malware analysis: Extract and sandbox the PDF in an isolated environment; check for embedded scripts, macros, or exploit code using tools like PDF analysers or VirusTotal.

Additional Q2 — Kerberos Protocol Understanding Exam-Style
Explain the role of each component in Kerberos (AS, TGS, Application Server). Why does the Kerberos protocol use timestamps in the Authenticator? How does Kerberos provide Single Sign-On?
Model Answer

Component Roles

  • AS (Authentication Server): Verifies the user's identity using their long-term secret key (derived from password). Issues a Ticket-Granting Ticket (TGT) encrypted for TGS.
  • TGS (Ticket-Granting Server): Issues service-specific tickets for application servers. Validates the TGT presented by the user — does not re-check the password.
  • Application Server: The resource being accessed. Validates the service ticket (containing session key and validity window) and the Authenticator.

Why Timestamps in Authenticator?

The Authenticator contains a timestamp t freshly created by Alice. The server checks that t is within a small window (e.g. 5 minutes) of its own clock. This prevents replay attacks: an intercepted Authenticator becomes invalid after the time window expires, so an attacker cannot reuse it.

How Kerberos Provides SSO

The user authenticates to AS once (entering their password). AS returns a TGT valid for the session duration (e.g. 8 hours). The user then presents the TGT to TGS to obtain tickets for any application server — without re-entering their password. From the user's perspective, access to all services is seamless and transparent.

Additional Q3 — Biometrics: FAR, FRR, Continuous Auth Exam-Style
Define FAR and FRR in biometric systems. Explain the trade-off between them. Which biometric methods can be used for continuous authentication and why?
Model Answer

FAR (False Acceptance Rate): The probability that the system incorrectly grants access to an imposter. High FAR = security risk.

FRR (False Rejection Rate): The probability that the system incorrectly denies access to a genuine user. High FRR = usability problem.

Trade-off: Tightening the acceptance threshold decreases FAR but increases FRR (and vice versa). The Equal Error Rate (EER) is the operating point where FAR = FRR — a useful single metric for comparing systems.

Continuous Authentication

Continuous authentication verifies identity throughout a session, not just at login. It requires methods that operate passively and non-intrusively:

  • Keystroke dynamics — analyses typing rhythm, pressure, key hold durations in the background.
  • Gait / motion biometrics — uses accelerometer/gyroscope data on mobile devices to recognise walking pattern.
  • Mouse dynamics — analyses cursor movement patterns.

These do not require the user to stop and re-authenticate, making them suitable for continuous verification. In contrast, fingerprint and face recognition require active user interaction.

Additional Q4 — Authentication Components: Formal Model Exam-Style
Define the three formal components A, C, and F in the authentication model. Why must the function f: A → C be a one-way function? Give a concrete example.
Model Answer

A (Authentication Information): The credential the user provides to prove their identity (e.g. the password they type).

C (Complementary Information): The stored representation of the credential on the server (e.g. the stored password hash).

F (Complementary Functions): The set of functions f: A → C used to transform A into C. The authentication function l: l(A, C) → {true, false} verifies the claim.

Why One-Way?

If f were reversible, an attacker who reads the database (e.g. SQL injection) could recover the original password A from C. A one-way hash makes this computationally infeasible: given C = f(A), it is practically impossible to find A.

Concrete Example

A = "MyPassword123!"          # what user types
salt = "a8f3bc9d"             # random, stored alongside hash
C = SHA-256(A + salt)         # stored in database
# f: A → C is SHA-256 (one-way)
# l(A, C): recompute SHA-256(input + salt), compare with C
Additional Q5 — MFA Scenario Analysis Exam-Style
A bank requires customers to enter (1) a password and (2) a one-time code sent to their registered phone. Which authentication factors does this represent? Analyse the security benefits and remaining vulnerabilities.
Model Answer

Factors Used

  • Password = "What you know" (knowledge-based)
  • OTP to phone = "What you have" (token-based)

This is Two-Factor Authentication (2FA) combining factors 1 (knowledge) and 2 (possession).

Security Benefits

  • Password alone is insufficient — a stolen password doesn't grant access.
  • Physical possession of the phone is required — remote attackers are blocked.
  • OTPs expire quickly — replay attacks are ineffective.

Remaining Vulnerabilities

  • SIM swapping: Attacker socially engineers the phone company to transfer the victim's number.
  • Phishing in real-time: Attacker proxies the login page and forwards the OTP immediately.
  • Phone malware: If the phone is infected, both password and OTP can be intercepted.
  • Social engineering: Attacker calls victim pretending to be the bank and asks for the OTP.
✓ Adding a third factor (biometric: fingerprint) would make this a 3FA system, significantly raising the attack complexity.
