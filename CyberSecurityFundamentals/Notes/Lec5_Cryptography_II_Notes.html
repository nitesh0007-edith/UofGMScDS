Quick Recap — Symmetric vs Asymmetric: In symmetric encryption (AES, DES, RC4) both parties share the same secret key. In asymmetric encryption each party has a key pair: a public key (freely shared) and a private key (never shared).

🔐 Asymmetric Encryption

The Core Idea

Each user generates a mathematically related key pair. The public key is published; the private key is kept secret. The keys are complementary — what one encrypts, only the other can decrypt.

Mode 1: Confidentiality (Secret Message)

Alice wants to send Bob a secret message. Only Bob can read it.

Plaintext M
→
Encrypt
Bob's PUBLIC key
→
Ciphertext C
──── network ────→
Decrypt
Bob's PRIVATE key
→
Plaintext M

Eve intercepts the ciphertext but cannot decrypt it — she does not have Bob's private key.

Mode 2: Authentication (Digital Signature)

Bob signs a message with his private key. Anyone with Bob's public key can verify it came from Bob.

Plaintext M
→
Encrypt
Bob's PRIVATE key
→
Ciphertext C
──── network ────→
Decrypt
Bob's PUBLIC key
→
Plaintext M ✓

If decryption with Bob's public key succeeds → message must have come from Bob (authentication).

Requirements for a Public-Key System

Computationally EASY (Must Be)

  • Generate a key pair (public key e,n — private key d)
  • Encrypt: given public key + plaintext → ciphertext
  • Decrypt: given private key + ciphertext → plaintext

Computationally INFEASIBLE (Must Be)

  • Determine private key from public key alone
  • Recover plaintext knowing only public key + ciphertext
  • Find factorisation of a large semi-prime n = pq

The Trap-Door Property

Public-key schemes rely on one-way trap-door functions: easy to compute in one direction, computationally hard to reverse unless you know the trap-door (private key).

  • RSA — trap-door = factoring large integers
  • Diffie-Hellman — trap-door = discrete logarithm problem
  • Elliptic Curve (ECC) — trap-door = elliptic curve discrete log

Either Key Can Play Either Role

$$\text{Decrypt}(K_{priv},\; \text{Encrypt}(K_{pub},\; M)) = M$$ $$\text{Decrypt}(K_{pub},\; \text{Encrypt}(K_{priv},\; M)) = M$$

The same mathematical relationship works in both directions. Which key is "public" and which is "private" is a policy decision, not a mathematical one.

🔢 RSA Algorithm

Invented

1977 — Ron Rivest, Adi Shamir, Len Adleman (MIT)

Type

Asymmetric block cipher

Used In

SSL/TLS, HTTPS, SSH, PGP, digital certificates

Mathematical Foundations

Euler's Totient Function φ(n)

φ(n) counts the number of integers from 1 to n−1 that are coprime to n (share no common factor with n).

$$\varphi(n) = n \prod_{p \mid n} \left(1 - \frac{1}{p}\right)$$
For RSA where n = pq (p,q prime):
$$\varphi(pq) = (p-1)(q-1)$$

Example

p = 7, q = 17 → n = 119 → φ(119) = (7−1)(17−1) = 6 × 16 = 96

p = 41, q = 43 → n = 1763 → φ(1763) = 40 × 42 = 1680

Multiplicative Inverse mod φ(n)

In RSA, the private key d must satisfy:

$$d \cdot e \equiv 1 \pmod{\varphi(n)}$$

This means d is the modular multiplicative inverse of e modulo φ(n). It exists if and only if gcd(e, φ(n)) = 1.

RSA Key Relationship: e · d = 1 + k·φ(n) for some integer k ≥ 0.

Extended Euclidean Algorithm (to find d)

Given e and φ(n), find d such that e·d ≡ 1 (mod φ(n)).

ALGORITHM: Extended Euclidean
Input: a = e, b = φ(n)
Step 1: Run Euclidean algorithm — divide repeatedly
b = q₀·a + r₀
a = q₁·r₀ + r₁ ← keep until remainder = 1
Step 2: Back-substitute to express 1 = x·e + y·φ(n)
Result: d = x mod φ(n) (make positive if negative)

Worked Example: e=17, φ(n)=1680

-- Euclidean Algorithm (forward pass) --
1680 = 98 × 17 + 14      → r₀ = 14
  17 = 1  × 14 + 3       → r₁ = 3
  14 = 4  × 3  + 2       → r₂ = 2
   3 = 1  × 2  + 1       → r₃ = 1 ← GCD = 1 ✓

-- Back-substitution (reverse pass) --
1 = 3 - 1×2
1 = 3 - 1×(14 - 4×3)    = 5×3 - 1×14
1 = 5×(17 - 1×14) - 14  = 5×17 - 6×14
1 = 5×17 - 6×(1680 - 98×17)
1 = 5×17 - 6×1680 + 588×17
1 = 593×17 - 6×1680

→ d = 593  (since 593 × 17 mod 1680 = 1 ✓)

Fast Modular Exponentiation

Direct computation of M^e mod n is impractical for large exponents. Use the property:

$$(A \times B) \bmod C = ((A \bmod C) \times (B \bmod C)) \bmod C$$

Example: Compute 2³ mod 5 step-by-step

Compute 7^3 mod 11:
7^1 mod 11 = 7
7^2 mod 11 = (7 × 7) mod 11 = 49 mod 11 = 5
7^3 mod 11 = (5 × 7) mod 11 = 35 mod 11 = 2

Compute 88^3 mod 187 (RSA decryption):
88^2 mod 187 = 7744 mod 187 = 77
88^3 mod 187 = (77 × 88) mod 187 = 6776 mod 187 = 88  ← back to plaintext!

RSA Security — Why Is It Hard to Break?

The Factoring Problem

An attacker knows n and e (both public). To find d, they need φ(n) = (p−1)(q−1), which requires factoring n into p and q.

For 2048-bit RSA: n ≈ 10⁶¹⁷ . No known polynomial-time algorithm for factoring.

Brute Force Key Size

$$\text{Time} = \frac{2^k}{\text{encryptions/second}}$$

For 16-bit key at 2⁸ enc/sec: 2¹⁶/2⁸ = 256 seconds (~4 min)

For 32-bit key: 2³²/2⁸ = 2²⁴ sec ≈ 194 days

Doubling key size squares the search space!

🤝 Diffie-Hellman Key Exchange

Purpose

Two parties establish a shared secret key over an insecure channel without ever transmitting the key itself. That shared key is then used for symmetric encryption (e.g., AES).

Primitive Roots

A number α is a primitive root mod q if its powers generate all values from 1 to q−1 (mod q). This is required for DH security.

Example: Is 2 a primitive root mod 7?

2^1 mod 7 = 2
2^2 mod 7 = 4
2^3 mod 7 = 8 mod 7 = 1   ← cycles after 3 steps, NOT all of {1..6}

→ 2 is NOT a primitive root mod 7

Is 3 a primitive root mod 7?
3^1=3,  3^2=2,  3^3=6,  3^4=4,  3^5=5,  3^6=1
→ Generates {1,2,3,4,5,6} — 3 IS a primitive root mod 7 ✓

DH Key Exchange Protocol

1
Public parameters (known to everyone including Eve)
q = \text{large prime},\quad \alpha = \text{primitive root of } q
2
Alice chooses secret X_A, computes and sends Y_A
Y_A = \alpha^{X_A} \bmod q
3
Bob chooses secret X_B, computes and sends Y_B
Y_B = \alpha^{X_B} \bmod q
4
Both compute the shared key K independently
\text{Alice:}\quad K = Y_B^{X_A} \bmod q = \alpha^{X_B \cdot X_A} \bmod q \\[6pt] \text{Bob:}\quad K = Y_A^{X_B} \bmod q = \alpha^{X_A \cdot X_B} \bmod q
Both computations give the same K because α^(X_A·X_B) = α^(X_B·X_A)!

Numerical Example

Public parameters:
  q = 23   (prime)
  α = 5    (primitive root mod 23)

Alice's secret: X_A = 6
  Y_A = 5^6 mod 23
       = 15625 mod 23
       = 8

Bob's secret:   X_B = 15
  Y_B = 5^15 mod 23
       = 5^8 × 5^4 × 5^2 × 5^1 mod 23
       = 16 × 4 × 2 × 5 mod 23  (computing powers)
       = 19

Shared Key — Alice computes:
  K = Y_B^X_A mod 23 = 19^6 mod 23
    = (19^2)^3 mod 23 = 16^3 mod 23 = 4096 mod 23 = 2

Shared Key — Bob computes:
  K = Y_A^X_B mod 23 = 8^15 mod 23
    = 8^8 × 8^4 × 8^2 × 8^1 mod 23
    = 4 × 2 × 18 × 8 mod 23 = 2  ✓

Both arrive at K = 2 without ever transmitting it!

Why Eve Cannot Find K (Discrete Log Problem)

Eve knows: q=23, α=5, Y_A=8, Y_B=19

To find K she needs X_A (or X_B). This requires solving the discrete logarithm:

5^{X_A} \equiv 8 \pmod{23} \implies X_A = ?

For small numbers she can brute force. For q with hundreds of digits → computationally infeasible.

Man-in-the-Middle Attack on DH

DH does NOT provide authentication! An adversary can impersonate both parties.
AliceAdversary (Eve)Bob
Sends Y_A → Intercepts Y_A
Generates own X_E, Y_E
Forwards Y_E to Bob		 ← Receives Y_E (thinks it's Y_A)
← Receives Y_E (thinks it's Y_B) Computes K_AE using Y_A + X_E
Computes K_BE using Y_B + X_E		 Sends Y_B →
K_Alice = Y_E^X_A = K_AE Decrypts/re-encrypts all messages K_Bob = Y_E^X_B = K_BE

Alice and Bob each think they share a key — but Eve sits in the middle, reading and modifying everything.

Mitigation: Authenticated DH

  • Digital certificates — bind public keys to identities (PKI)
  • Pre-shared secret — out-of-band verification
  • TLS uses DH with server certificates to prevent MITM

Is One Outdated?

"Is asymmetric more secure than symmetric?"

No. Security depends on key length and the computational work to break the cipher — not on the type. A 128-bit AES key provides comparable security to a 3072-bit RSA key.

"Is symmetric encryption outdated?"

No. Symmetric encryption has much lower computational overhead. AES is used everywhere for bulk data: SSD encryption, Google Cloud, TLS data transfer.

Hybrid Approach (used in practice): TLS uses asymmetric encryption (RSA/DH) only to securely exchange a symmetric session key. Then uses AES for the actual data — best of both worlds.

Is Public Key Distribution Trivial?

No! You still need to trust that the public key genuinely belongs to who you think it does. Without infrastructure (PKI / Certificate Authorities), MITM attacks are possible. This is why HTTPS uses certificates signed by trusted CAs.

Q1 (May 2025) — Travelsafe website vulnerability (URL session IDs)
5 marks

A company called Travelsafe operates through a website that uses URL session IDs. The website serves as an exploratory environment where visitors are provided with a search box to identify possible holiday locations. Mention and describe a potential vulnerability and the corresponding defence mechanisms that can be employed to mitigate it.

Model Answer (5 marks)

Vulnerability 1: SQL Injection (via search box)

The search box allows user input to be sent to a database query. If input is not sanitised, an attacker can inject SQL commands like ' OR '1'='1 to access or delete unauthorised data.

Vulnerability 2: Session Hijacking / Predictable Session IDs

URL session IDs are visible in browser history, server logs, and Referer headers. If IDs are predictable or short, an attacker can guess or brute-force them. If URLs are shared (e.g. by copy-paste), session tokens are exposed.

Defence Mechanisms:

  • Use parameterised queries / prepared statements to prevent SQL injection
  • Use cryptographically random session IDs (long, unpredictable)
  • Store session IDs in HTTP-only cookies instead of URLs
  • Implement session expiration and re-authentication
  • Use HTTPS to prevent network sniffing of session tokens
  • Apply input validation and output encoding
Q2 (May 2025) — Single Sign-On (SSO) — CIA + advantages
7 marks

1) Argue for the advantages and disadvantages of using SSO over traditional approaches. 2) Explain what SSO means for Confidentiality, Integrity, Authenticity, Accountability, and Non-repudiation.

Model Answer

Advantages of SSO:

  • Single authentication event → less password fatigue → users choose stronger passwords
  • Centralised access control — easy to revoke access (leave company → one account disabled)
  • Reduced attack surface — fewer credentials to steal
  • Better user experience, lower IT support costs

Disadvantages of SSO:

  • Single point of failure — if SSO provider is compromised, ALL systems are compromised
  • Complexity — harder to implement correctly
  • Vendor lock-in

SSO and Security Properties:

  • Confidentiality: Improved — fewer credentials to protect; but breach of SSO = breach of everything
  • Integrity: Improved — centralised policy enforcement ensures consistent data access rules
  • Authenticity: Improved — one strong auth (e.g., MFA) covers all services
  • Accountability: Improved — unified audit log across all services
  • Non-repudiation: Improved — single identity for all actions makes it harder to deny actions; tokens are signed
Q3 (May 2025) — Block cipher ECB/CBC decryption + ECB weakness
13 marks

Block cipher on 4-bit blocks. Permutation table:

m0123456789abcdef
E_K(m)1b5c7e2a49fd0368

(a) ECB: decrypt 187b06 [2 marks]

(b) CBC: decrypt 301be with IV=1 [4 marks]

(c) Explain why ECB is not secure; suggest a better mode [7 marks]

Part (a) — ECB Decryption of 187b06 [2 marks]

ECB decrypts each block independently. We need the inverse permutation: find m given E_K(m).

Inverse table (swap rows of original):

Cipher0123456789abcdef
Plainc06d82e4f9713b5a
Ciphertext: 1  8  7  b  0  6
Decrypt:    0  f  4  1  c  e

Plaintext: 0f41ce

Part (b) — CBC Decryption of 301be, IV=1 [4 marks]

In CBC decryption: P_i = D_K(C_i) XOR C_{i-1}, with C_0 = IV.

Ciphertext blocks: C₁=3, C₂=0, C₃=1, C₄=b, C₅=e
IV = C₀ = 1

P₁ = D_K(3) XOR C₀ = d XOR 1 = c    (d=1101, 1=0001 → 1100=c)
P₂ = D_K(0) XOR C₁ = c XOR 3 = f    (c=1100, 3=0011 → 1111=f)
P₃ = D_K(1) XOR C₂ = 0 XOR 0 = 0
P₄ = D_K(b) XOR C₃ = 1 XOR 1 = 0
P₅ = D_K(e) XOR C₄ = 5 XOR b = e    (5=0101, b=1011 → 1110=e)

Plaintext: cf00e

Part (c) — ECB Weakness + Better Mode [7 marks]

Why ECB is insecure:

  • ECB encrypts identical plaintext blocks to identical ciphertext blocks using the same key
  • An attacker can detect patterns — if two ciphertext blocks are identical, the corresponding plaintext blocks are identical
  • This leaks structural information about the message (e.g., repeated words, headers)
  • A famous example: the "ECB penguin" — an encrypted image in ECB mode still reveals the outline
  • Vulnerable to replay attacks — attacker can cut and paste ciphertext blocks

Better Mode: Cipher Block Chaining (CBC)

$$C_i = E_K(P_i \oplus C_{i-1}) \quad \text{(encrypt)}$$ $$P_i = D_K(C_i) \oplus C_{i-1} \quad \text{(decrypt)}$$

How CBC fixes ECB's weakness:

  • Each plaintext block is XORed with the previous ciphertext block before encryption
  • Identical plaintext blocks produce different ciphertext (because they XOR with different previous ciphertext)
  • An Initialisation Vector (IV) randomises the first block, so even identical messages produce different ciphertexts
  • Errors propagate: a single bit error in C_i corrupts exactly P_i and P_{i+1} (limited propagation)
Q4a (May 2025) — RSA private key derivation: e=17, n=1763
10 marks

RSA encryption key is public: e=17, n=1763. Find the private key d. Show all steps.

Full Step-by-Step Solution [10 marks]

Step 1: Factor n = 1763
1763 ÷ 41 = 43  →  p = 41, q = 43

Step 2: Compute φ(n)
φ(1763) = (41-1)(43-1) = 40 × 42 = 1680

Step 3: Verify gcd(e, φ(n)) = gcd(17, 1680) = 1
1680 = 98×17 + 14
  17 =  1×14 + 3
  14 =  4×3  + 2
   3 =  1×2  + 1   ← GCD = 1 ✓

Step 4: Find d = 17⁻¹ mod 1680 (Extended Euclidean back-substitution)
1 = 3 - 1×2
  = 3 - 1×(14 - 4×3)   = 5×3 - 14
  = 5×(17 - 14) - 14   = 5×17 - 6×14
  = 5×17 - 6×(1680 - 98×17)
  = 593×17 - 6×1680

Step 5: d = 593 (coefficient of 17, taken mod 1680)

Step 6: Verify: 17 × 593 = 10081 = 6×1680 + 1 ✓

Q4b (May 2025) — Brute force attack: 16-bit key, 2⁸ enc/sec
5 marks

Assuming 2⁸ encryptions/second, key size = 16 bits. How long for brute force? Give a practical and impractical scenario. What happens when key size doubles?

Model Answer

$$\text{Search space} = 2^{16} = 65536 \text{ keys}$$ $$\text{Time (worst case)} = \frac{2^{16}}{2^8} = 2^8 = 256 \text{ seconds} \approx 4 \text{ minutes}$$ $$\text{Time (average)} = \frac{2^{16}}{2 \times 2^8} = 128 \text{ seconds} \approx 2 \text{ minutes}$$

Practical scenario: Legacy system with 16-bit key (e.g., early Bluetooth pairing, old IoT device). An attacker could break this in minutes on a laptop.

Impractical scenario: In a military command system or bank transaction processor with high authentication overhead, you might get only 1-2 attempts per second, but that still only extends it to days — not secure.

Doubling key size (16 → 32 bits):

$$\text{Search space} = 2^{32} = 4 \text{ billion keys}$$ $$\text{Time} = \frac{2^{32}}{2^8} = 2^{24} \approx 16.7 \text{ million seconds} \approx \mathbf{194 \text{ days}}$$

Doubling key size squares the search space. Each additional bit doubles the time. Going from 16 to 32 bits makes it 2^16 = 65,536 times harder.

AQ3 — CBC error propagation: what happens if C₃ is corrupted?
Practice

CBC Error Propagation Analysis

$$P_i = D_K(C_i) \oplus C_{i-1}$$

If C₃ is corrupted (contains an error):

  • P₃ = D_K(C₃_corrupted) ⊕ C₂ → completely wrong (D_K of wrong input)
  • P₄ = D_K(C₄) ⊕ C₃_corrupted → wrong in the bits that were corrupted (XOR with corrupted C₃)
  • P₅ and beyond → correct (C₄ onwards are uncorrupted)
Key exam answer: A single corrupted ciphertext block C_i damages exactly P_i (completely) and P_{i+1} (partially, in the bits where C_i was corrupted). All subsequent blocks are correctly decrypted.

Compare with ECB: a corrupted C_i in ECB damages only P_i. No propagation.

AQ4 — Why is public key distribution NOT trivial? How is it solved?
Practice

Even though public keys are meant to be shared, there is no guarantee that a public key you receive actually belongs to who you think it does.

Problem: Without authentication, an adversary can intercept your request for Alice's public key and substitute their own key. You'd then encrypt messages for Alice using the adversary's key.

Solution — Public Key Infrastructure (PKI):

  • Certificate Authority (CA): A trusted third party that digitally signs certificates binding a public key to an identity
  • X.509 certificate: Contains subject name, public key, and CA signature
  • Browser has a list of trusted CAs pre-installed
  • Chain of trust: Root CA → Intermediate CA → End-entity certificate
  • Certificate Revocation Lists (CRL) and OCSP for revoked certificates

This is why HTTPS certificates are issued by CAs — they prevent MITM attacks on public key exchange.

AQ5 — Describe the RSA confidentiality and authentication scenarios with diagrams.
Practice

Confidentiality (Secret Message):

  • Alice wants to send a message ONLY Bob can read
  • Alice encrypts with Bob's public key: C = E(K_Bob_pub, M)
  • Only Bob can decrypt: M = D(K_Bob_priv, C)
  • Anyone can encrypt for Bob, but only Bob can decrypt

Authentication (Digital Signature):

  • Bob wants to prove a message came from him
  • Bob encrypts (signs) with his private key: C = E(K_Bob_priv, M)
  • Anyone can verify using Bob's public key: M = D(K_Bob_pub, C)
  • If decryption succeeds → message must have come from Bob (only he has the private key)

Combined (Confidentiality + Authentication):

  1. Bob signs with his private key: S = E(K_Bob_priv, M)
  2. Alice encrypts for Charlie: C = E(K_Charlie_pub, S)
  3. Charlie decrypts: S = D(K_Charlie_priv, C)
  4. Charlie verifies Bob's signature: M = D(K_Bob_pub, S)
AQ6 — Explain the Man-in-the-Middle attack on Diffie-Hellman. How does TLS prevent it?
Practice

The Attack:

  1. Alice sends Y_A to Bob over network
  2. Eve intercepts Y_A and generates her own X_E, Y_E
  3. Eve forwards Y_E to Bob (pretending to be Alice)
  4. Bob sends Y_B to Alice; Eve intercepts and forwards her Y_E
  5. Alice computes shared key with Eve K_AE; Bob computes K_BE
  6. Eve can decrypt from Alice using K_AE, re-encrypt for Bob using K_BE

Alice and Bob believe they have a secure channel, but Eve reads and modifies all traffic.

Why DH is vulnerable: DH provides no authentication. Y_A is just a number — there is nothing that proves it came from Alice.

TLS Prevention:

  • Server sends its X.509 certificate signed by a trusted CA
  • Certificate contains server's public key
  • Client verifies certificate chain up to root CA
  • This authenticates the server's DH public key
  • Result: Authenticated DH (used in TLS 1.3 as ECDHE)
