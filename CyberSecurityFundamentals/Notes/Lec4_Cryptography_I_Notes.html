Brute Force Attack

Try all possible keys until intelligible plaintext is found. Impractical when key space is large.

Computationally Secure means either: (1) the cost of breaking exceeds the value of the information, OR (2) the time to break exceeds the useful lifetime of the information.

Key size = \(n\) bits → Key space = \(2^n\) possible keys

Average brute force attempts = \(2^{n-1}\)

With \(r\) attempts/sec, time = \(\dfrac{2^{n-1}}{r}\) seconds

Feistel Cipher Structure

Key Properties

  • Plaintext split into two halves: L₀, R₀
  • Passes through n rounds of processing
  • Each round uses subkey Kᵢ derived from main key
  • Substitution on left half using round function F on right half
  • Halves are swapped after each round

Design Decisions

  • Block size: 128 bits (common)
  • Key size: 128 bits (common)
  • Rounds: 16 typical
  • Subkey gen: complex = more secure
  • Round function: more complex = harder to cryptanalyse

Encryption: \(C = E(K_3,\, D(K_2,\, E(K_1, P)))\)

AES – Advanced Encryption Standard

Block Size

128 bits (fixed)

Key Size

128, 192, or 256 bits

Rounds

10 (128-bit key), 12 (192), 14 (256)

Structure

NOT Feistel – operates on entire block in parallel

Key AES vs DES difference: DES uses Feistel structure (half block modified at a time). AES processes the ENTIRE block in each round using 4 operations.

AES Four Operations (Each Round)

1. Substitute Bytes

16×16 S-box lookup. Left 4 bits = row, right 4 bits = column. Provides non-linearity (confusion).

2. Shift Rows

Row 1: no shift. Row 2: 1-byte left shift. Row 3: 2-byte shift. Row 4: 3-byte shift. Provides diffusion.

3. Mix Columns

Matrix multiplication in GF(2⁸). Each column treated as polynomial. Mixes data across columns.

4. Add Round Key

XOR state with round key. Self-inverse: XOR same key again returns original. The only key-dependent step.

Mix Column Math

Mix Columns matrix multiplication:

\[\begin{bmatrix} S'_{0j} \\ S'_{1j} \\ S'_{2j} \\ S'_{3j} \end{bmatrix} = \begin{bmatrix} 02 & 03 & 01 & 01 \\ 01 & 02 & 03 & 01 \\ 01 & 01 & 02 & 03 \\ 03 & 01 & 01 & 02 \end{bmatrix} \begin{bmatrix} S_{0j} \\ S_{1j} \\ S_{2j} \\ S_{3j} \end{bmatrix}\]

All arithmetic in GF(2⁸): 01·x = x, 02·x = shift left (XOR 0x1B if overflow), 03·x = 02·x ⊕ x

Mix Column Worked Example

For byte 87 (= 1000 0111):

02 × 87: leftmost bit=1, so shift left → 0000 1110, then XOR 0001 1011 = 0001 0101

For byte 6E (= 0110 1110):

02 × 6E: leftmost bit=0, so shift left → 1101 1100 = DC

03 × 6E = 02×6E ⊕ 6E = DC ⊕ 6E = B2

Add Round Key – Why It's Its Own Inverse

\(A \oplus K = C\) (encrypt)

\(C \oplus K = A \oplus K \oplus K = A \oplus 0 = A\) (decrypt)

This is why no separate decryption key needed for this step.

RC4 – Rivest Cipher 4 (Stream Cipher)

Type

Stream cipher (variable key length)

Key Length

1–256 bytes

State

256-byte permutation vector S[0..255]

Output

Pseudorandom keystream XORed with plaintext

Phase 1: State Initialization (KSA)

// Key Scheduling Algorithm (KSA)
for i = 0 to 255:
    S[i] = i
    T[i] = K[i mod keylen]   // T is temporary vector

j = 0
for i = 0 to 255:
    j = (j + S[i] + T[i]) mod 256
    swap(S[i], S[j])          // Initial permutation of S

Phase 2: Keystream Generation (PRGA)

// Pseudo-Random Generation Algorithm (PRGA)
i = 0; j = 0
while (plaintext bytes remain):
    i = (i + 1) mod 256
    j = (j + S[i]) mod 256
    swap(S[i], S[j])
    t = (S[i] + S[j]) mod 256
    k = S[t]              // One keystream byte
    output k XOR plaintext_byte

Stream Cipher Security Vulnerability

NEVER reuse a stream cipher key!
If C₁ = P₁ ⊕ K and C₂ = P₂ ⊕ K, then C₁ ⊕ C₂ = P₁ ⊕ P₂.
If one plaintext is known/guessable, the other is immediately recoverable.

Stream Cipher Advantages

  • As secure as block cipher (comparable key)
  • Faster, uses less code
  • Good for real-time applications

Stream Cipher Disadvantages

  • Key reuse is catastrophic
  • Block ciphers can safely reuse keys
  • No authentication by itself

The problem: DES/3DES blocks = 64 bits. AES blocks = 128 bits. What about longer messages? We need block modes.

Same key, no chaining. P₁=P₂ → C₁=C₂ !

Encryption: \(C_j = E(K, P_j)\)

Decryption: \(P_j = D(K, C_j)\)

Properties

  • Each block encrypted independently
  • Same plaintext → same ciphertext
  • Blocks can be encrypted in parallel
  • Random access possible

Weaknesses

  • Patterns in plaintext visible in ciphertext
  • Block replay attacks possible
  • Not semantically secure
  • Classic example: ECB penguin image

Encryption: \(C_j = E(K,\, C_{j-1} \oplus P_j)\)   where \(C_0 = IV\)

Decryption: \(P_j = D(K, C_j) \oplus C_{j-1}\)

Proof: \(D(K, C_j) = D(K, E(K, C_{j-1} \oplus P_j)) = C_{j-1} \oplus P_j\)

So: \(C_{j-1} \oplus D(K, C_j) = C_{j-1} \oplus C_{j-1} \oplus P_j = P_j\) ✓

CBC Error Propagation: If ciphertext block Cᵢ is corrupted, it affects decryption of BOTH Pᵢ (fully corrupted) and Pᵢ₊₁ (partially corrupted — only the corresponding bit positions affected). Pᵢ₊₂ onwards are fine.

IV Requirements

  • Must be known to both sender and receiver
  • Must be protected like a key
  • Adversary changing IV bits → corrupts P₁
  • Should be random and unpredictable

CBC Properties

  • Identical plaintext blocks → different ciphertext
  • Chaining hides patterns
  • Cannot parallelize encryption
  • Can parallelize decryption

3. Cipher Feedback (CFB)

Converts a block cipher into a stream cipher. Uses the encryption function for BOTH encryption AND decryption.

Encryption: \(C_1 = P_1 \oplus S_s[E(K, IV)]\)

Decryption: \(P_1 = C_1 \oplus S_s[E(K, IV)]\)

Note: decryption also uses E (not D)!

RSA – Asymmetric Encryption

Type

Public-key (asymmetric) cryptosystem

Security Basis

Difficulty of factoring large integers

Keys

Public key (e, n) for encryption; Private key (d, n) for decryption

RSA Key Generation Steps

Choose two large primes p and q
Compute \(n = p \times q\) (the modulus)
Compute \(\phi(n) = (p-1)(q-1)\) (Euler's totient)
Choose integer e such that \(\gcd(e, \phi(n)) = 1\) and \(1 < e < \phi(n)\)
Find d such that \(d \cdot e \equiv 1 \pmod{\phi(n)}\) (use Extended Euclidean Algorithm)
Public key: (e, n)   Private key: (d, n)

Encryption: \(C = M^e \bmod n\)

Decryption: \(M = C^d \bmod n\)

Modular arithmetic shortcut: \((A \times B) \bmod C = [(A \bmod C) \times (B \bmod C)] \bmod C\)

Power mod: \(A^B \bmod C = (A \bmod C)^B \bmod C\)

RSA Worked Example 1: p=3, q=11, e=7, M=5

\(n = 3 \times 11 = 33\)
\(\phi(n) = (3-1)(11-1) = 2 \times 10 = 20\)
Find d: \(d \times 7 \equiv 1 \pmod{20}\) → d = 3 (since 3×7=21≡1 mod 20)
Encrypt: \(C = 5^7 \bmod 33\)
5² = 25, 5⁴ = 625 mod 33 = 625 - 18×33 = 625 - 594 = 31
5⁷ = 5⁴ × 5² × 5¹ = 31 × 25 × 5 mod 33
31×25 = 775 mod 33 = 775 - 23×33 = 775 - 759 = 16
16×5 = 80 mod 33 = 80 - 2×33 = C = 14
Decrypt: \(M = 14^3 \bmod 33\)
14² = 196 mod 33 = 196 - 5×33 = 196 - 165 = 31
14³ = 31 × 14 mod 33 = 434 mod 33 = 434 - 13×33 = 434 - 429 = M = 5 ✓

RSA Worked Example 2: p=5, q=11, e=3, M=9

\(n = 55\), \(\phi(n) = 40\)
Find d: \(3d \equiv 1 \pmod{40}\) → d = 27 (3×27=81=2×40+1)
Encrypt: \(C = 9^3 \bmod 55 = 729 \bmod 55\)
729 = 13×55 + 14 → C = 14
Decrypt: \(M = 14^{27} \bmod 55\)
14¹⁰ mod 55 = 1 (can verify), 14⁷ mod 55 = 9
14²⁷ = 14¹⁰ × 14¹⁰ × 14⁷ mod 55 = 1 × 1 × 9 = M = 9 ✓

RSA Extended Euclidean Algorithm: e=31, n=3599

Factor n=3599: note 3599 = 60² - 1 = (60-1)(60+1) → p=59, q=61
\(\phi(n) = 58 \times 60 = 3480\)
Find d: \(31d \equiv 1 \pmod{3480}\) — use Extended Euclidean:
// Extended Euclidean Algorithm
3480 = 112(31) + 8
  31 = 3(8)  + 7
   8 = 1(7)  + 1   ← GCD = 1

// Back-substitute
1 = 8 - 7
  = 8 - (31 - 3×8) = 4×8 - 31
  = 4×(3480 - 112×31) - 31
  = 4×3480 - 449×31

// So 31×(-449) ≡ 1 (mod 3480)
d = -449 mod 3480 = 3480 - 449 = 3031
Private key d = 3031

RSA Security: Dictionary Attack on Small Alphabet

Vulnerability: If plaintext is encoded character-by-character (A=0..Z=25), attacker can precompute all 26 encryptions {0^e mod n, 1^e mod n, …, 25^e mod n} using the PUBLIC key, then look up each ciphertext block to get plaintext. This is NOT secure!

Question 1 — Session ID Vulnerability

A company called Travelsafe operates a website using URL session IDs (unique numbers assigned after authentication). Visitors can search for holiday locations. Mention and describe a potential vulnerability and corresponding defence mechanisms.

Vulnerability: Session Hijacking via URL Session ID

When session IDs are embedded in URLs (e.g., travelsafe.com/search?sessionid=12345), they are exposed in:

  • Browser history
  • Server access logs
  • HTTP Referer headers when navigating to external sites
  • Shared links (users may accidentally share their session)

An attacker (Eve) who obtains the URL can assume the victim's identity without knowing their password.

Defence Mechanisms:

  • Use cookies (HttpOnly, Secure flag) instead of URL parameters to store session IDs
  • Use HTTPS to encrypt the session ID in transit
  • Implement session expiry (short-lived sessions)
  • Use cryptographically random session IDs with sufficient entropy (≥128 bits)
  • Bind sessions to IP address or user-agent (though this can break with NAT/proxies)
  • Implement CSRF tokens for state-changing requests
[7 marks]

Question 2 — Single Sign-On (SSO)

Companies like Google and Amazon use SSO for authentication. 1) Argue for the advantages and disadvantages of SSO over traditional approaches. 2) Explain what SSO means for Confidentiality, Integrity, Authenticity, Accountability, and Non-repudiation.

1) Advantages of SSO:

  • Users authenticate once — reduces password fatigue and encourages strong unique passwords
  • Reduced attack surface (fewer credential stores to protect)
  • Centralised authentication makes auditing and revocation easier
  • Better user experience → fewer support tickets for password resets

Disadvantages of SSO:

  • Single point of failure: if SSO provider is compromised or unavailable, ALL services fail
  • Credential compromise gives attacker access to ALL linked services
  • Vendor lock-in to identity provider
  • Privacy: identity provider can track all service usage

2) Security Properties with SSO:

  • Confidentiality: SSO tokens transmitted over HTTPS protect credentials; risk if token intercepted
  • Integrity: Signed tokens (JWT with digital signature) ensure tokens haven't been tampered with
  • Authenticity: Identity provider verifies the user's identity; relying parties trust the provider's assertion
  • Accountability: Centralised logs at identity provider enable tracking of all authentication events
  • Non-repudiation: Signed authentication assertions mean users cannot deny having authenticated

Question 3(c) — ECB Security Analysis (Eve's Attack)

Alice encrypts messages using ECB mode. Eve notices ciphertext blocks repeat. Explain why ECB is insecure and suggest a better mode.

Why ECB is insecure (4 marks):

  • ECB encrypts each block independently using the same key: \(C_j = E(K, P_j)\)
  • If \(P_i = P_j\) then \(C_i = C_j\) — identical plaintext always produces identical ciphertext
  • This means the pattern structure of the plaintext is preserved in the ciphertext
  • Eve can identify when the same plaintext block appears in different messages without knowing the key
  • Classic demonstration: encrypting a bitmap image — the outline of the image remains visible in ECB ciphertext
  • Also vulnerable to block replay attacks: attacker can reorder/replay ciphertext blocks to alter plaintext

Better alternative — CBC mode (3 marks):

  • CBC chains blocks: \(C_j = E(K, C_{j-1} \oplus P_j)\)
  • Even if \(P_i = P_j\), their ciphertext will differ because different preceding ciphertext blocks XOR in
  • A random, unpredictable IV ensures ciphertext varies even for the same message encrypted twice
  • Patterns in plaintext are completely hidden in ciphertext
  • Mitigates Eve's pattern analysis attack

Question 4(b) — Brute Force Analysis

You can do 2⁸ encryptions/second, key size = 16 bits. How long does a brute force attack take? What if key size doubles to 32 bits?

Key space: \(2^{16} = 65,536\) keys

Average attempts: \(2^{16}/2 = 2^{15} = 32,768\) attempts

Rate: \(2^8 = 256\) encryptions/second

Time: \(\dfrac{2^{15}}{2^8} = 2^7 = 128 \text{ seconds} \approx 2 \text{ minutes}\)

Where this IS practical: Short-lived data (e.g., a time-sensitive OTP that expires in 2 minutes), low-value data where attacker invests minimal resources, or during a race condition attack window.

Where this is NOT practical: Banking credentials (by the time cracked, session may be expired but damage requires immediate action), long-term secrets, or when keys change frequently.

Doubling key size to 32 bits:

Key space: \(2^{32} = 4,294,967,296\)

Average attempts: \(2^{31}\)

Time: \(\dfrac{2^{31}}{2^8} = 2^{23} \approx 8.4 \text{ million seconds} \approx 97 \text{ days}\)

Doubling key size from 16→32 bits squares the key space, making brute force completely impractical with the same hardware.

Extra Q1 — Linear Block Cipher Attack

A linear block cipher EL satisfies: EL(k, m₁ ⊕ m₂) = EL(k, m₁) ⊕ EL(k, m₂). Using 128 chosen ciphertexts, explain how an adversary can decrypt ANY ciphertext without knowing key k.

Attack strategy:

For i = 1 to 128, choose cᵢ = the 128-bit string with a 1 in position i and zeros elsewhere (standard basis vectors).
Obtain decryptions: let mᵢ = D(k, cᵢ) for each i (these are our 128 chosen ciphertext decryptions).
Given any target ciphertext c, express it as XOR of a subset I(c) of the basis vectors: c = ⊕ᵢ∈I(c) cᵢ (every non-zero binary string has a unique such representation).
By linearity: c = ⊕cᵢ = ⊕E(mᵢ) = E(⊕mᵢ), so plaintext = ⊕ᵢ∈I(c) mᵢ
Special case c=0: E(0⊕0) = E(0)⊕E(0) = 0, so plaintext of 0 is 0.
This breaks the cipher with only 128 chosen ciphertext queries — demonstrating why non-linearity (S-boxes) is essential.

Extra Q2 — CBC Error Propagation

An error occurs in ciphertext block Cᵢ during transmission in CBC mode. What effect does this have on the recovered plaintext blocks?

CBC decryption: \(P_j = D(K, C_j) \oplus C_{j-1}\)

Effect of a corrupted block Cᵢ:

  • Block Pᵢ: Completely garbled — D(K, Cᵢ) is wrong, so Pᵢ = D(K, Cᵢ) ⊕ Cᵢ₋₁ is entirely incorrect
  • Block Pᵢ₊₁: Partially corrupted — Pᵢ₊₁ = D(K, Cᵢ₊₁) ⊕ Cᵢ. Since Cᵢ is wrong, exactly the corrupted BIT POSITIONS in Cᵢ will flip the corresponding bits in Pᵢ₊₁. If a single bit in Cᵢ is flipped, exactly one bit of Pᵢ₊₁ is wrong.
  • Blocks Pᵢ₊₂ onwards: Perfectly recovered — they depend on Cᵢ₊₁, Cᵢ₊₂, etc., which are intact.

Conclusion: A single corrupted ciphertext block in CBC causes damage to 2 plaintext blocks (one fully, one partially), then self-corrects.

Extra Q4 — Stream Cipher Key Reuse Attack

Alice encrypts two messages P1 and P2 using RC4 with the same key K. Eve intercepts C1 and C2. Eve knows that P1 starts with "LOGIN". Describe how Eve can recover P2.

Attack:

C1 = P1 ⊕ K    C2 = P2 ⊕ K

C1 ⊕ C2 = P1 ⊕ K ⊕ P2 ⊕ K = P1 ⊕ P2

If P1 is known: P2 = (P1 ⊕ P2) ⊕ P1 = (C1 ⊕ C2) ⊕ P1

Eve computes C1 ⊕ C2, then XORs with known P1 bytes ("LOGIN" = 0x4C4F47494E) to recover corresponding bytes of P2. With crib-dragging techniques, even partial knowledge of P1 can reveal much of P2.

Lesson: NEVER reuse a stream cipher key. This is why protocols like TLS use a new key (derived from key exchange) for each session.

Extra Q5 — AES vs DES Comparison

Compare AES and DES across: block size, key size, rounds, and structure. Why is AES considered more secure?

FeatureDESAES
Block size64 bits128 bits
Key size56 bits (effective)128, 192, or 256 bits
Rounds1610, 12, or 14

Why AES is more secure:

  • 56-bit DES key: 2⁵⁶ ≈ 7×10¹⁶ possible keys — broken in hours with modern hardware
  • 128-bit AES key: 2¹²⁸ ≈ 3.4×10³⁸ — computationally infeasible to brute force
  • AES operates on full 128-bit block simultaneously — greater diffusion per round
  • No known practical attacks on full AES-128 (as of 2025)

Interactive Quiz

Test your understanding. Click an option to check your answer.

Q1. In CBC mode, what is XORed with each plaintext block before encryption?

Correct! CBC: Cⱼ = E(K, Cⱼ₋₁ ⊕ Pⱼ). The chaining is what prevents identical plaintext producing identical ciphertext.

Q2. Which block mode allows both encryption AND decryption to be parallelised?

Correct! CTR has no dependencies between blocks — encryption and decryption are fully parallelisable. CBC can only parallelise decryption, not encryption.

Q3. In RSA, if φ(n) = 40 and e = 7, what is d?

Correct! d×7 ≡ 1 (mod 40). Check: 23×7 = 161 = 4×40 + 1 ✓. (Note: for p=3,q=11, the answer is d=3; this uses φ(n)=40 with e=7 generally).

Q4. What is the fundamental weakness of ECB mode?

Correct! ECB's deterministic block-by-block encryption means patterns in plaintext are visible in ciphertext — a critical security flaw.

Q5. In the AES Mix Columns operation, what does 02 × 0x87 equal? (0x87 = 10000111)

Correct! 0x87 has leftmost bit=1, so: shift left → 0x0E, then XOR 0x1B → 0x0E ⊕ 0x1B = 0x15.

Q6. In 3DES EDE encryption with K1=K2=K3=K, what is the result?

Correct! E(K,D(K,E(K,P))) = E(K,P). The decrypt undoes one encrypt, leaving a single encrypt. This is backward compatibility with DES.

Q7. Which AES operation uses the same function for both encryption and decryption?

Correct! XOR is self-inverse: A ⊕ K ⊕ K = A. So Add Round Key is identical in encryption and decryption.

Q8. Alice encrypts 26 letters (A=0…Z=25) character by character with RSA. Why is this insecure?

Correct! Since the plaintext space is only 26 values and the public key is known, Eve computes M^e mod n for M=0..25 and creates a lookup table. No key needed.

Q9. A ciphertext block Cᵢ is corrupted in CBC. Which plaintext blocks are affected?

Correct! Pᵢ = D(K,Cᵢ) ⊕ Cᵢ₋₁ — fully wrong. Pᵢ₊₁ = D(K,Cᵢ₊₁) ⊕ Cᵢ — only the corrupted bits of Cᵢ affect Pᵢ₊₁. All later blocks use intact ciphertext blocks.

Q10. Key size is 16 bits, rate = 2⁸ encrypt/sec. How long for brute force (worst case)?

Correct! Worst case = try ALL 2¹⁶ keys. Time = 2¹⁶ / 2⁸ = 2⁸ = 256 seconds. Average case = 2¹⁵/2⁸ = 2⁷ = 128 seconds.
